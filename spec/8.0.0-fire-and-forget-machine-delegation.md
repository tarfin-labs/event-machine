# Fire-and-Forget Machine Delegation

**Status:** Draft v3
**Author:** Claude
**Date:** 2025-03-17
**Scope:** Enable fire-and-forget machine delegation via absence of `@done`

---

## Problem

The `machine` key always establishes a managed parent-child relationship. In async mode (`queue`), the parent creates a `MachineChild` tracking record and waits for `ChildMachineCompletionJob` to route `@done`/`@fail`. There is no way to spawn a child machine in the background without tracking its lifecycle.

### Real-World Use Case

In `CarSalesMachine`, when an application is "prevented", a TÜRMOB identity verification should run in the background. The parent doesn't need the result. Today this requires a manual `dispatch()` in an entry action or a custom job wrapper — both break the declarative nature of machine delegation.

---

## Design: `@done` Absence = Fire-and-Forget

The fire-and-forget signal is the **absence of `@done`**, not a new keyword. This is already how sync delegation works (verified in code: `routeChildDone()` returns early at line 1388 when `onDoneTransition` is null). We extend this natural behavior to async mode.

### Execution Order (verified at `MachineDefinition` line 2370-2402)

```
1. Entry actions run                    (line 2374)
2. handleMachineInvoke() runs           (line 2378)
3. @always transitions checked          (line 2387)
```

This order means: fire-and-forget dispatches the child in step 2, then `@always` can transition the parent in step 3. The parent doesn't get "stuck" — it either stays in the state (with its own `on` events) or transitions via `@always`.

### The Five Modes

| Config | Behavior | Status |
|--------|----------|--------|
| `machine + @done` | **Managed sync.** Child runs inline, `@done` routes parent. | Existing |
| `machine + queue + @done` | **Managed async.** Child dispatched, parent waits for `ChildMachineCompletionJob`. | Existing |
| `machine` (no `@done`, no `queue`) | **Sync fire-and-forget.** Child runs inline, result ignored, parent stays. | Existing (undocumented) |
| `machine + queue` (no `@done`) | **Async fire-and-forget.** Child dispatched, parent stays in state. | **NEW** |
| `machine + queue + target` (no `@done`) | **Async fire-and-forget + transition.** Child dispatched, parent transitions to `target`. | **NEW (optional)** |

### Primary API

```php
// Parent stays in 'prevented', child runs independently
'prevented' => [
    'machine' => TurmobMachine::class,
    'with'    => ['tckn'],
    'queue'   => true,
    // No @done → fire-and-forget
    'on' => ['RETRY' => 'retrying'],
],
```

### With `@always` (alternative to `target`)

```php
// Parent spawns child, then immediately transitions via @always
'dispatching_turmob' => [
    'machine' => TurmobMachine::class,
    'with'    => ['tckn'],
    'queue'   => true,
    'on'      => ['@always' => 'prevented'],  // runs AFTER handleMachineInvoke
],
```

### With `target` (optional, consistent with `job` key)

```php
// Explicit fire-and-forget + immediate transition
'dispatching_turmob' => [
    'machine' => TurmobMachine::class,
    'with'    => ['tckn'],
    'queue'   => true,
    'target'  => 'prevented',
],
```

`target` is supported for consistency with the `job` key fire-and-forget pattern. But the primary/recommended API is "no `@done` + `on` events" or "no `@done` + `@always`".

### Detection Logic

```php
// In handleAsyncMachineInvoke():
$isFireAndForget = $stateDefinition->onDoneTransition === null;
```

Not on `MachineInvokeDefinition` — the signal comes from the state definition, not the invoke config.

When `$isFireAndForget`:
- No `MachineChild` tracking record
- No `$state->addActiveChild()`
- `ChildMachineJob` receives `fireAndForget: true`
- No `ChildMachineCompletionJob` dispatched by child
- If `$invokeDefinition->target !== null` → transition to target
- If `$invokeDefinition->target === null` → stay in state (let `@always` or `on` handle flow)

### No MachineChild Tracking

Fire-and-forget does not create a `MachineChild` record:
- `MachineChild` exists for lifecycle management (parent queries it, completion job uses it). Fire-and-forget has none of this.
- The child still persists its own `MachineEvent` records — observability preserved.
- Mirrors the `job` fire-and-forget which also creates no tracking record.

### Child Gets Parent Identity

The child still receives `parentRootEventId` and `parentMachineClass` via `setMachineIdentity()` — useful for tracing and `sendToParent()`.

---

## Validation Rules

Fire-and-forget is detected as: `queue` present AND `@done` absent.

```php
$isFireAndForget = isset($stateConfig['queue']) && !isset($stateConfig['@done']);
```

| Config | Valid? | Reason |
|--------|--------|--------|
| `machine + @done` | ✅ | Managed sync (existing) |
| `machine + queue + @done` | ✅ | Managed async (existing) |
| `machine` (no `@done`, no `queue`) | ✅ | Sync fire-and-forget (existing, undocumented) |
| `machine + queue` (no `@done`) | ✅ | **Async fire-and-forget, stay in state (NEW)** |
| `machine + queue + target` (no `@done`) | ✅ | **Async fire-and-forget + transition (NEW)** |
| `machine + queue + @done + target` | ❌ | Ambiguous — pick managed (`@done`) or fire-and-forget (`target`) |
| `machine + target` (no `queue`) | ❌ | Sync fire-and-forget with transition is contradictory |
| Fire-and-forget + `@fail` | ❌ | Fire-and-forget doesn't track failures |
| Fire-and-forget + `@timeout` | ❌ | Fire-and-forget doesn't track timing |
| Fire-and-forget + `output` | ❌ | Fire-and-forget doesn't receive results |
| Fire-and-forget + `forward` | ❌ | Fire-and-forget doesn't communicate with running child |

### Step 1: Update `StateConfigValidator::validateMachineConfig()`

**File:** `src/StateConfigValidator.php` (lines 489-494)

Replace the blanket `machine + target` ban with nuanced rules:

```php
private static function validateMachineConfig(array $stateConfig, string $path): void
{
    // ... existing validations (FQCN, parallel) ...

    // forward requires queue (existing, unchanged)
    if (!empty($stateConfig['forward']) && !isset($stateConfig['queue'])) {
        throw new InvalidArgumentException(
            message: "State '{$path}' has 'forward' without 'queue'. Event forwarding is only valid in async mode."
        );
    }

    // Detect fire-and-forget: async + no @done
    $isFireAndForget = isset($stateConfig['queue']) && !isset($stateConfig['@done']);

    if ($isFireAndForget) {
        // @fail is invalid — fire-and-forget doesn't track child failures
        if (isset($stateConfig['@fail'])) {
            throw new InvalidArgumentException(
                message: "State '{$path}' has '@fail' without '@done'. "
                       . "Fire-and-forget machine delegation does not support failure callbacks. "
                       . "Add '@done' for managed delegation, or remove '@fail'."
            );
        }

        // @timeout is invalid — fire-and-forget doesn't track timing
        if (isset($stateConfig['@timeout'])) {
            throw new InvalidArgumentException(
                message: "State '{$path}' has '@timeout' without '@done'. "
                       . "Fire-and-forget machine delegation does not support timeout callbacks. "
                       . "Add '@done' for managed delegation, or remove '@timeout'."
            );
        }

        // output is invalid — fire-and-forget doesn't receive results
        if (isset($stateConfig['output'])) {
            throw new InvalidArgumentException(
                message: "State '{$path}' has 'output' without '@done'. "
                       . "Fire-and-forget machine delegation does not produce output for the parent."
            );
        }

        // forward is invalid — no running child to forward to
        if (!empty($stateConfig['forward'])) {
            throw new InvalidArgumentException(
                message: "State '{$path}' has 'forward' without '@done'. "
                       . "Fire-and-forget machine delegation does not support event forwarding."
            );
        }
    }

    // target validation
    if (isset($stateConfig['target'])) {
        // target requires queue
        if (!isset($stateConfig['queue'])) {
            throw new InvalidArgumentException(
                message: "State '{$path}' has 'machine' with 'target' but no 'queue'. "
                       . "Fire-and-forget with immediate transition requires async execution."
            );
        }

        // target + @done is ambiguous
        if (isset($stateConfig['@done'])) {
            throw new InvalidArgumentException(
                message: "State '{$path}' cannot have both '@done' and 'target' with 'machine'. "
                       . "Use '@done' for managed delegation or 'target' for fire-and-forget."
            );
        }
    }
}
```

### Step 2: Update `handleAsyncMachineInvoke()`

**File:** `src/Definition/MachineDefinition.php` (lines 1194-1256)

Add fire-and-forget branch inside the existing method. Same flag pattern as `handleJobInvoke()`.

```php
protected function handleAsyncMachineInvoke(
    State $state,
    StateDefinition $stateDefinition,
    MachineInvokeDefinition $invokeDefinition,
): void {
    $childMachineClass = $invokeDefinition->machineClass;
    $childContext      = $invokeDefinition->resolveChildContext($state->context);
    $isFireAndForget   = $stateDefinition->onDoneTransition === null;

    // Record child machine start event
    $state->setInternalEventBehavior(
        type: InternalEvent::CHILD_MACHINE_START,
        placeholder: $childMachineClass,
    );

    // Create tracking record (managed only)
    $machineChildId = '';
    if (!$isFireAndForget) {
        $childRecord = MachineChild::create([
            'parent_root_event_id' => $state->history->first()->root_event_id,
            'parent_state_id'      => $stateDefinition->id,
            'parent_machine_class' => $this->machineClass,
            'child_machine_class'  => $childMachineClass,
            'status'               => MachineChild::STATUS_PENDING,
            'created_at'           => now(),
        ]);

        $state->addActiveChild($childRecord->id);
        $machineChildId = $childRecord->id;
    }

    // Dispatch child machine job
    $job = new ChildMachineJob(
        parentRootEventId: $state->history->first()->root_event_id,
        parentMachineClass: $this->machineClass,
        parentStateId: $stateDefinition->id,
        childMachineClass: $childMachineClass,
        machineChildId: $machineChildId,
        childContext: $childContext,
        retry: $invokeDefinition->retry ?? 1,
        fireAndForget: $isFireAndForget,
    );

    if ($invokeDefinition->queue !== null) {
        $job->onQueue($invokeDefinition->queue);
    }

    if ($invokeDefinition->connection !== null) {
        $job->onConnection($invokeDefinition->connection);
    }

    dispatch($job)->afterCommit();

    // Fire-and-forget: optionally transition to target, then return
    if ($isFireAndForget) {
        if ($invokeDefinition->target !== null) {
            $targetState = $this->idMap[$this->id.'.'.$invokeDefinition->target]
                ?? $this->idMap[$invokeDefinition->target]
                ?? null;

            if ($targetState instanceof StateDefinition) {
                $state->setCurrentStateDefinition($targetState);
            }
        }
        // If no target: method returns, @always check runs at line 2387
        return;
    }

    // Managed: dispatch timeout job if configured
    if ($invokeDefinition->timeout !== null && $stateDefinition->onTimeoutTransition instanceof TransitionDefinition) {
        $timeoutJob = new ChildMachineTimeoutJob(
            parentRootEventId: $state->history->first()->root_event_id,
            parentMachineClass: $this->machineClass,
            parentStateId: $stateDefinition->id,
            machineChildId: $childRecord->id,
            childMachineClass: $childMachineClass,
            timeoutSeconds: $invokeDefinition->timeout,
        );

        if ($invokeDefinition->queue !== null) {
            $timeoutJob->onQueue($invokeDefinition->queue);
        }

        dispatch($timeoutJob)->afterCommit()->delay($invokeDefinition->timeout);
    }
}
```

**No changes to `handleMachineInvoke()` router.** Fire-and-forget naturally flows through the async path because `queue` sets `async = true`.

**New section after `@timeout`:**

````markdown
## Fire-and-Forget

When you need to spawn a child machine without tracking its result, omit `@done`. The child runs independently — its completion or failure does not affect the parent.

### Stay in State (primary pattern)

The state spawns the child on entry and continues functioning normally with its own `on` events:

<!-- doctest-attr: ignore -->
```php
'prevented' => [
    'machine' => VerificationMachine::class,
    'with'    => ['tckn'],
    'queue'   => 'verifications',
    // No @done → fire-and-forget
    'on' => ['RETRY' => 'retrying'],
],
```

**Reads as:** "When entering `prevented`, spawn `VerificationMachine` in the background. The state handles its own events normally."

### Spawn and Move On (with @always)

Use `@always` to immediately transition after spawning the child:

<!-- doctest-attr: ignore -->
```php
'dispatching_verification' => [
    'machine' => VerificationMachine::class,
    'with'    => ['tckn'],
    'queue'   => 'verifications',
    'on'      => ['@always' => 'prevented'],  // runs after child is dispatched
],
```

### Spawn and Move On (with target)

Alternatively, use `target` for an explicit fire-and-forget transition (consistent with the `job` key pattern):

<!-- doctest-attr: ignore -->
```php
'dispatching_verification' => [
    'machine' => VerificationMachine::class,
    'with'    => ['tckn'],
    'queue'   => 'verifications',
    'target'  => 'prevented',
],
```

### Rules

- Fire-and-forget requires `queue` — the child must run asynchronously.
- `@done` absence is the signal — no new keyword needed.
- `@fail`, `@timeout`, `output`, and `forward` are not valid without `@done`.
- The child still persists its own `MachineEvent` records (observability).
- The child receives parent identity (`sendToParent()` still works).
````

## Fire-and-Forget Pattern

Fire-and-forget means spawning work without tracking the result. Use it for side effects where the parent doesn't care about the outcome.

### Machine Delegation (stay in state)

Omit `@done` to make a machine delegation fire-and-forget. The parent stays in the state and handles its own events:

<!-- doctest-attr: ignore -->

'dispatching_verification' => [

### Choosing the Right Mechanism

| Mechanism | Tracks Result | Parent Waits | Use Case |
|-----------|--------------|-------------|----------|
| `machine + queue` (no `@done`) | No | No | Stateful child (multiple states, webhooks) |
| `job` + `target` | No | No | Single-step async (logging, notification) |
| `dispatchTo()` | No | No | Event to existing machine |

## Edge Cases

### `sendToParent()` from Fire-and-Forget Child
Valid — `parentRootEventId` is injected. Parent may or may not handle the event.

### Child Failure
Child's failure is its own concern. `ChildMachineJob::failed()` returns early. Recorded in Laravel's `failed_jobs` table.

### `@always` on Managed Async State
Technically possible but would break delegation (parent leaves before child completes). This is a pre-existing edge case, not introduced by our change. Could add a validator warning in a future release.

### Named Queue / Connection / Retry
All work as expected — they configure `ChildMachineJob` dispatch, not lifecycle.

## Migration Path

Purely additive:
- `machine + queue` without `@done` was previously valid but useless (parent waits forever). Now it's fire-and-forget. No sane config uses the old behavior.
- `machine + target` was previously invalid. Now it's optionally valid with `queue`.
- No database migrations.
- No breaking API changes.
- All existing tests pass unchanged.
