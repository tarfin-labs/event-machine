# MachineRouter Endpoint Filtering (`only` / `except`)

**Status:** Draft v4 ‚Äî Final
**Author:** Claude
**Date:** 2026-03-21
**Scope:** Add `only` and `except` options to `MachineRouter::register()` to control which event endpoints are registered per route group

---

## Problem

When the same machine serves multiple consumer groups with different auth requirements, there is no way to split endpoints across route registrations. `MachineRouter::register()` always registers **all** endpoints from the machine definition.

### Real-World Scenario

Two consumers for the same `CarSalesMachine`:

1. **Customer (public, no auth):** SMS link ‚Äî consent, personal info, findeks pin
2. **Retailer (auth:retailer):** Dealer panel ‚Äî vehicle pricing, payment options, allocation

```php
// Both calls register ALL endpoints ‚Äî only the binding pattern differs
MachineRouter::register(CarSalesMachine::class, [
    'prefix'       => 'car-sales',            // intended: public
    'machineIdFor' => [StatusRequestedEvent::class],
]);

MachineRouter::register(CarSalesMachine::class, [
    'prefix'       => 'machines/car-sales',   // intended: retailer-only
    'middleware'    => ['auth:retailer'],
    'machineIdFor' => [VehicleSubmittedEvent::class],
]);
```

**Result:** Every endpoint appears under both prefixes. Customer endpoints leak behind auth, retailer endpoints are exposed publicly.

`machineIdFor` / `modelFor` only control **binding mode** (how the machine instance is resolved from the URL parameter), not which endpoints are registered.

---

## Design Principles

1. **Minimal surface area** ‚Äî Two new options (`only`, `except`), mutually exclusive, same type signature as `machineIdFor`
2. **Consistent API** ‚Äî Accept both event class FQCNs and `SCREAMING_SNAKE_CASE` strings, exactly like `machineIdFor` / `modelFor`
3. **Filter everything** ‚Äî Apply to both regular endpoints and forwarded endpoints uniformly
4. **Validate early, fail fast** ‚Äî All misconfigurations throw at route registration time, never silently ignored. This includes `machineIdFor`/`modelFor` referencing nonexistent or forwarded endpoints, even without `only`/`except`.
5. **Backwards compatible** ‚Äî When neither `only` nor `except` is provided, endpoint registration behavior is identical to current code.
6. **Stricter validation** ‚Äî `machineIdFor`/`modelFor` referencing nonexistent or forwarded endpoints now throws instead of being silently ignored (D6). This surfaces pre-existing misconfigurations.

## API Design

### `only` ‚Äî Whitelist

Register **only** the listed event endpoints. All others are excluded.

```php
MachineRouter::register(Machine::class, [
    'prefix'       => 'public',
    'only'         => [EventA::class, EventB::class],
    'machineIdFor' => [EventA::class, EventB::class],
    'name'         => 'machine.public',
]);
```

### `except` ‚Äî Blacklist

Register all endpoints **except** the listed ones.

```php
MachineRouter::register(Machine::class, [
    'prefix'       => 'admin',
    'middleware'    => ['auth:admin'],
    'except'       => [EventA::class, EventB::class],
    'create'       => true,
    'machineIdFor' => [EventC::class, EventD::class],
    'name'         => 'machine.admin',
]);
```

### Mutual Exclusivity

```php
// Throws InvalidArgumentException at route registration time
MachineRouter::register(Machine::class, [
    'prefix' => 'api',
    'only'   => [EventA::class],
    'except' => [EventB::class],
]);
```

### Type Signature

```php
@param array{
    prefix: string,
    model?: string,
    attribute?: string,
    create?: bool,
    machineIdFor?: string[],
    modelFor?: string[],
    middleware?: string[],
    name?: string,
    only?: string[],      // ‚Üê NEW
    except?: string[],    // ‚Üê NEW
} $options
```

## Design Decisions

### D1: `create` is orthogonal to `only`/`except`

`create` is a synthetic endpoint ‚Äî it initializes a new machine instance, not an event transition. It has no event type, so it cannot appear in `only`/`except` lists. The `create: bool` flag already provides independent control over its registration.

```php
// Create-only registration (valid):
MachineRouter::register(Machine::class, [
    'prefix' => 'admin',
    'create' => true,
    'only'   => [],          // no event endpoints, only create
]);

// Create + subset (valid):
MachineRouter::register(Machine::class, [
    'prefix' => 'admin',
    'create' => true,
    'only'   => [EventA::class],  // create + one event endpoint
]);
```

### D2: `machineIdFor` / `modelFor` must reference registered endpoints (strict)

Every event type in `machineIdFor` and `modelFor` must exist in the **filtered** endpoint set. Referencing a filtered-out event is a configuration error ‚Äî throw immediately.

**Why strict:** Event-machine validates everything at registration time (modelFor without model ‚Üí throw, machineIdFor ‚à© modelFor overlap ‚Üí throw). Silent ignore masks configuration drift and makes debugging harder.

```php
// Throws ‚Äî CANCEL is excluded by 'only' but referenced in 'machineIdFor'
MachineRouter::register(Machine::class, [
    'prefix'       => 'api',
    'only'         => ['START'],
    'machineIdFor' => ['START', 'CANCEL'],
]);
// ‚Üí "MachineRouter: 'machineIdFor' references event types not in the registered
//    endpoint set (filtered by 'only'): CANCEL. Remove from 'machineIdFor'
//    or add to 'only'."
```

**Practical implication:** Each `register()` call is a self-contained, explicit declaration. No shared arrays across registrations ‚Äî each call declares exactly what it registers and how it binds.

### D3: Empty arrays are valid

- **`only: []`** ‚Äî register zero event endpoints. Valid for create-only registrations.
- **`except: []`** ‚Äî exclude nothing, equivalent to omitting `except`. Allowed because filter arrays may be built dynamically and an empty result should not crash.

### D4: Forwarded endpoints filter by parent event type

Forwarded endpoints are filtered by their **parent-facing event type** ‚Äî the key in the `$forwardedEndpoints` array, and the same identifier used in route names and URIs. When a forward config renames a child event (`'PARENT_CONFIRM' => ['child' => 'CHILD_CONFIRM']`), the user writes `PARENT_CONFIRM` in `only`/`except`, not `CHILD_CONFIRM`.

**Why:** The user thinks in terms of their API surface (URLs, route names), not child implementation details.

### D5: Forwarded endpoints inherit binding mode independently

Forwarded endpoints do not appear in `machineIdFor`/`modelFor`. Their binding mode is inherited from the parent's global model config (`$model` + `$attribute` options). If a forwarded event type appears in `machineIdFor`/`modelFor`, it is a configuration error ‚Äî forwarded endpoints cannot be individually assigned a binding mode. Filtering does not change this ‚Äî it only controls which forwarded endpoints are registered, not how they bind.

### D6: `machineIdFor` / `modelFor` validation always runs

The orphan validation (event types in `machineIdFor`/`modelFor` must exist in the registered regular endpoint set) runs **unconditionally** ‚Äî not just when `only`/`except` is active. This catches pre-existing misconfigurations that were previously silently ignored:

```php
// Today: silently ignored (TYPO_EVENT never matches any endpoint)
// After this change: throws immediately
MachineRouter::register(Machine::class, [
    'prefix'       => 'api',
    'machineIdFor' => ['TYPO_EVENT'],
]);
// ‚Üí "MachineRouter: 'machineIdFor' references event types not in the registered
//    endpoint set: TYPO_EVENT. Available: START, COMPLETE, CANCEL"
```

Similarly, forwarded event types in `machineIdFor`/`modelFor` are caught with a specific error message explaining that forwarded endpoints inherit binding mode from parent config.

**Validation order rationale (Steps 10 ‚Üí 11 ‚Üí 12 ‚Üí 13):**
- Step 10 first: forwarded events in `machineIdFor`/`modelFor` get a specific, actionable error ("forwarded endpoints inherit binding from parent config"). Most specific error, catches a conceptual misunderstanding.
- Step 11 second: orphaned events (not in endpoint set) get caught next. If an event is both orphaned AND overlapping, the user gets the orphan error (root cause) instead of the misleading overlap error. If an event is orphaned AND `modelFor` lacks model/attribute, the user gets the orphan error first ‚Äî fixing it (removing from `modelFor`) may empty `modelFor` entirely, making the model/attribute check moot.
- Step 12 third: `modelFor` requires model + attribute. Only runs after orphan check confirms all `modelFor` events are valid and registered.
- Step 13 last: overlap check runs on remaining valid, registered events only.

### Step 5‚Äì6: Parse and Resolve `only` / `except`

Insert after existing option parsing, before the existing validations:

```php
// ‚îÄ‚îÄ Endpoint filtering ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
$only   = $options['only'] ?? null;
$except = $options['except'] ?? null;

if ($only !== null && $except !== null) {
    throw new \InvalidArgumentException(
        "MachineRouter: 'only' and 'except' cannot be used together."
    );
}

$onlyTypes   = $only !== null
    ? array_map(EndpointDefinition::resolveEventType(...), $only)
    : null;
$exceptTypes = $except !== null
    ? array_map(EndpointDefinition::resolveEventType(...), $except)
    : null;
```

Note: `$only !== null` instead of `$only ?` ‚Äî an empty array `[]` is a valid value (D3) and must not be treated as "not set."

### Step 7: Validate Against Known Endpoints

```php
if ($onlyTypes !== null || $exceptTypes !== null) {
    $allKnownTypes = array_merge(
        array_keys($endpoints),
        array_keys($forwardedEndpoints),
    );

    $filterTypes = $onlyTypes ?? $exceptTypes;
    $filterKey   = $onlyTypes !== null ? 'only' : 'except';
    $unknown     = array_diff($filterTypes, $allKnownTypes);

    if ($unknown !== []) {
        throw new \InvalidArgumentException(
            sprintf(
                "MachineRouter: unknown event types in '%s': %s. Available: %s",
                $filterKey,
                implode(', ', $unknown),
                implode(', ', $allKnownTypes),
            )
        );
    }
}
```

Error message includes available event types ‚Äî helps the user fix the mistake without looking up the machine definition.

### Step 8: Filter Endpoint Arrays

```php
if ($onlyTypes !== null) {
    $endpoints = array_filter(
        $endpoints,
        fn (string $eventType): bool => in_array($eventType, $onlyTypes, true),
        ARRAY_FILTER_USE_KEY,
    );
    $forwardedEndpoints = array_filter(
        $forwardedEndpoints,
        fn (string $eventType): bool => in_array($eventType, $onlyTypes, true),
        ARRAY_FILTER_USE_KEY,
    );
} elseif ($exceptTypes !== null) {
    $endpoints = array_filter(
        $endpoints,
        fn (string $eventType): bool => !in_array($eventType, $exceptTypes, true),
        ARRAY_FILTER_USE_KEY,
    );
    $forwardedEndpoints = array_filter(
        $forwardedEndpoints,
        fn (string $eventType): bool => !in_array($eventType, $exceptTypes, true),
        ARRAY_FILTER_USE_KEY,
    );
}
```

Filtering happens **outside** the route group closure, before the closure captures `$endpoints` and `$forwardedEndpoints` via `use`. The closure receives the already-filtered arrays.

### Step 9: Post-Filter Early Return

```php
if ($endpoints === [] && $forwardedEndpoints === [] && !$create) {
    return;
}
```

The `!$create` check ensures a create-only registration (`only: []` + `create: true`) still registers the route group.

### Step 10: NEW ‚Äî Forwarded Events Cannot Be in `machineIdFor` / `modelFor`

Forwarded endpoints inherit their binding mode from the parent's global model config (D5). Putting them in `machineIdFor`/`modelFor` has no effect and indicates a misunderstanding. This validation runs unconditionally (D6).

```php
$forwardedEventTypes = array_keys($forwardedEndpoints);

if ($forwardedEventTypes !== []) {
    $forwardedInMachineId = array_intersect($machineIdFor, $forwardedEventTypes);
    if ($forwardedInMachineId !== []) {
        throw new \InvalidArgumentException(
            sprintf(
                "MachineRouter: 'machineIdFor' cannot reference forwarded endpoints "
                ."(they inherit binding mode from parent model config): %s",
                implode(', ', $forwardedInMachineId),
            )
        );
    }

    $forwardedInModel = array_intersect($modelFor, $forwardedEventTypes);
    if ($forwardedInModel !== []) {
        throw new \InvalidArgumentException(
            sprintf(
                "MachineRouter: 'modelFor' cannot reference forwarded endpoints "
                ."(they inherit binding mode from parent model config): %s",
                implode(', ', $forwardedInModel),
            )
        );
    }
}
```

### Step 11: NEW ‚Äî `machineIdFor` / `modelFor` Must Reference Registered Endpoints

Every event type in `machineIdFor` and `modelFor` must exist in the regular endpoint set (after filtering, if active). This validation runs unconditionally (D6).

```php
$registeredEventTypes = array_keys($endpoints);

$machineIdOrphans = array_diff($machineIdFor, $registeredEventTypes);
if ($machineIdOrphans !== []) {
    if ($onlyTypes !== null) {
        $context = " (filtered by 'only'). Remove from 'machineIdFor' or add to 'only'";
    } elseif ($exceptTypes !== null) {
        $context = " (filtered by 'except'). Remove from 'machineIdFor' or adjust 'except'";
    } else {
        $context = sprintf(". Available: %s", implode(', ', $registeredEventTypes));
    }

    throw new \InvalidArgumentException(
        sprintf(
            "MachineRouter: 'machineIdFor' references event types not in the registered endpoint set: %s%s",
            implode(', ', $machineIdOrphans),
            $context,
        )
    );
}

$modelOrphans = array_diff($modelFor, $registeredEventTypes);
if ($modelOrphans !== []) {
    if ($onlyTypes !== null) {
        $context = " (filtered by 'only'). Remove from 'modelFor' or add to 'only'";
    } elseif ($exceptTypes !== null) {
        $context = " (filtered by 'except'). Remove from 'modelFor' or adjust 'except'";
    } else {
        $context = sprintf(". Available: %s", implode(', ', $registeredEventTypes));
    }

    throw new \InvalidArgumentException(
        sprintf(
            "MachineRouter: 'modelFor' references event types not in the registered endpoint set: %s%s",
            implode(', ', $modelOrphans),
            $context,
        )
    );
}
```

Error messages adapt based on context:
- With `only`/`except`: tells the user which filter caused the issue and how to fix it.
- Without filtering: lists available endpoint types so the user can spot the typo.

### Step 12: Existing ‚Äî `modelFor` Requires Model + Attribute

```php
if ($modelFor !== [] && ($model === null || $attribute === null)) {
    throw new \InvalidArgumentException(
        "MachineRouter: 'model' and 'attribute' are required when 'modelFor' is set."
```

Unchanged from current code. Now runs after orphan check (Step 11) ‚Äî if a `modelFor` event was orphaned, the user fixes that first; removing it may empty `modelFor`, making this check moot.

### Step 13: Existing ‚Äî `machineIdFor` ‚à© `modelFor` Overlap

```php

if ($overlap !== []) {
```

Unchanged from current code. Now runs last so orphaned/forwarded events are caught first with more specific messages.

## Interaction with Other Options

| Option | Interaction with `only`/`except` |
|--------|----------------------------------|
| `create` | **Orthogonal.** Controlled by `create: bool`, not affected by `only`/`except` (D1). |
| `machineIdFor` | Must reference endpoints in the **filtered** regular endpoint set. Referencing a filtered-out, forwarded, or nonexistent event throws (D2, D5, D6). |
| `modelFor` | Must reference endpoints in the **filtered** regular endpoint set. Referencing a filtered-out, forwarded, or nonexistent event throws (D2, D5, D6). |
| `middleware` | Unaffected. Applied to all routes in the group, including `create`. |
| `name` | Unaffected. Filtered endpoints simply don't generate routes. |
| Forwarded endpoints | Filtered by parent-facing event type (D4). Binding mode inheritance unaffected (D5). |

## Duplicate Route Scenario

When the same event appears in two `register()` calls with different `prefix` and `name`:

- `GET /car-sales/{machineId}/status-requested` ‚Üí `car-sales.public.status_requested`
- `GET /machines/car-sales/{machineId}/status-requested` ‚Üí `machines.car-sales.status_requested`

**No conflict.** Different prefix + different name = different URI + different route name. Same machine, different access points with different auth is a valid pattern.

## Alternatives Considered

### A: Manual Route Definitions

**Rejected:** Loses MachineRouter conventions (URI generation, handler selection, result behavior resolution, forwarded endpoint handling, route defaults). Error-prone and high maintenance.

### B: Middleware Override on Subset

**Rejected:** Laravel route override is "last wins" for same URI. Cannot reliably add auth to a subset of already-registered routes. Fragile.

### C: Separate Machine Definitions

**Rejected:** Same state machine with shared transitions, context, and event history. Splitting creates artificial boundaries and duplicates logic.

### D: Include `create` in `only`/`except`

**Rejected:** `create` is not an event ‚Äî it has no event type string. Including it would require a magic identifier (`'@create'`, `MachineRouter::CREATE`) that breaks the homogeneous event type list. The `create: bool` flag already provides clean, independent control. See D1.
