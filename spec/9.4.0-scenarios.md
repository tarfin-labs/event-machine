## 1. Problem

Complex machines like `CarSalesMachine` have deep state hierarchies with parallel child delegations, API integrations, and multi-step guard chains. To test a new feature that affects behavior at state `checking_protocol`, a human must:

1. Trigger `START_APPLICATION` with valid retailer/farmer models
2. Grant consent → pass eligibility check
3. Wait for `FindeksMachine` to complete (6+ steps, external API calls)
4. Wait for `TurmobMachine` to complete (external API call)
5. Both parallel regions must finish → guard check → exit to `checking_protocol`

This requires 10+ manual steps, 2 child machines, multiple API calls, and deep domain knowledge. Product teams and QA cannot do this in staging without developer assistance.

**Scenarios solve this:** define behavior overrides and delegation outcomes once, activate from existing frontend endpoints via a `scenario` field, arrive at the desired state with a fully functional machine.

## 2. Design Principles

| Principle | Rationale |
|-----------|-----------|
| **Behavior is real** | Machine transitions, event history, context mutations, entry/exit actions — all real. The resulting machine is indistinguishable from one that arrived at that state organically. |
| **Override, don't fake** | External dependencies return predetermined responses via `plan()`. The behavior still runs in the engine — it just gets scenario-defined data instead of calling real services. |
| **State-route-keyed plan** | Every `plan()` key is a full state route — you see WHERE each override or delegation outcome applies. No mixed key types. |
| **Child delegation as outcomes or scenarios** | Parent declares child outcomes (`@done.report_saved`) for fast-forward, or references a child scenario for interactive states (e.g., OTP entry). Child internals are never directly overridden from the parent. |
| **Identity = Machine × Source × Event × Target × Slug** | A scenario defines a journey: which machine, from which source state, via which event, to which target state. The slug disambiguates when multiple scenarios share the same trigger point. `@continue` directives auto-advance through intermediate interactive states to reach the target. |
| **Endpoint-native** | Scenarios are activated through existing machine endpoints via a `scenario` field in the request body. |
| **Async-safe** | Scenario info is persisted to the database, so async queue jobs (parallel regions, listeners, child completion) restore overrides automatically. |
| **Environment-gated** | Disabled by default (`MACHINE_SCENARIOS_ENABLED=true`). Zero overhead in production. |

## 3. Scenario Identity — Machine × Source × Event × Target × Slug

A scenario defines a **journey** through a machine — from a source state, via a triggering event, to a target state. Five coordinates:

| Coordinate | Property | Example | Purpose |
|-----------|----------|---------|---------|
| **Machine** | `$machine` | `CarSalesMachine::class` | Which machine this scenario targets |
| **Source** | `$source` | `'awaiting_customer_start'` | Full state route — where the machine is BEFORE the event (start) |
| **Event** | `$event` | `CustomerStartedEvent::class` | Which event triggers this scenario (class or string) |
| **Target** | `$target` | `'checking_protocol'` | Full state route — where the machine should end up (end) |
| **Slug** | (from class name) | `'at-checking-protocol-scenario'` | Unique identifier — multiple scenarios can share the same source+event |

The first three determine **where** the scenario can be activated. The target declares **where it aims**. The slug determines **which** scenario is selected when multiple share the same start point.

Multiple scenarios can share the same `(source, event)` with different targets — e.g., `AtVerificationScenario` (target: `verification`) and `AtCheckingProtocolScenario` (target: `checking_protocol`) both activate from `awaiting_customer_start` + `CustomerStartedEvent`, but their `plan()` and `@continue` chains take the machine to different endpoints.

This quintuple determines:
- **Discovery** — which scenarios are available at a given source state for a given event
- **Validation** — the machine must be at `$source` when the scenario activates, `$event` must match the event being sent, and the machine must reach `$target` after execution
- **`availableScenarios`** — grouped by event in endpoint responses, each entry includes `target`

A machine can be running normally without any scenario and then at **any state**, the user can activate a scenario — as long as a scenario exists for that source + event combination.

### Full State Route Requirement

`$source`, `$target`, and `plan()` keys must use the **full route** (dot-notation path), not just the leaf key:

```php
// ❌ Ambiguous — could exist in multiple parallel regions
protected string $source = 'awaiting_vehicle_and_pricing';

// ✅ Unambiguous — full path from machine root
protected string $source = 'data_collection.retailer.awaiting_vehicle_and_pricing';
protected string $target = 'data_collection.retailer.pricing_complete';
```

This matches how `$state->value` returns state paths in endpoint responses. For root-level states (e.g., `routing`, `eligibility_check`), the leaf key IS the full route.

## 4. MachineScenario Class

```php
abstract class MachineScenario
{
    /** Which machine this scenario targets (class-string). */
    protected string $machine;

    /** Full state route — the state the machine must be in BEFORE the event (start). */
    protected string $source;

    /** The event that triggers this scenario (class-string or event type string). */
    protected string $event;

    /** Full state route — where the machine should end up after execution (end). */
    protected string $target;

    /** Human-readable description shown in endpoint responses. */
    protected string $description;

    /**
     * The scenario's plan — every key is a full state route.
     * Value type determines meaning: array → behavior overrides (may include @continue),
     * string starting with @ → delegation outcome,
     * MachineScenario class → child scenario reference.
     */
    protected function plan(): array { return []; }

    /**
     * Scenario parameter definitions.
     * Each key is a parameter name. Value is either:
     *   - plain array: Laravel validation rules only (e.g., ['required', 'string'])
     *   - assoc array: rich definition with optional type/values/label + rules
     * Validated before plan() is called.
     */
    protected function params(): array { return []; }

    /**
     * Access a validated scenario parameter.
     * Values come from frontend's scenarioParams, validated against params() rules.
     */
    protected function param(string $key, mixed $default = null): mixed
    {
        return $this->resolvedParams[$key] ?? $default;
    }
}
```

**5 properties** (`$machine`, `$source`, `$event`, `$target`, `$description`) — set as simple property assignments.
**2 optional methods** (`plan()`, `params()`) — `plan()` defines behavior overrides, delegation outcomes, and `@continue` directives; `params()` defines parameter definitions with optional type metadata.

Base class validates all properties are set at instantiation. After execution, ScenarioPlayer validates the machine reached `$target`.

### Properties vs Methods

Properties are the default for identity fields. Methods can be used when computed values are needed:

```php
// Default — property assignment (recommended)
protected string $machine = CarSalesMachine::class;
protected string $source  = 'awaiting_customer_start';
protected string $target  = 'checking_protocol';

// Alternative — method override (when computation is needed)
protected function machine(): string
{
    return config('machines.car_sales.class');
}
```

The base class checks: method first, property fallback.

### Example

```php
class AtVerificationScenario extends MachineScenario
{
    protected string $machine     = CarSalesMachine::class;
    protected string $source      = 'awaiting_customer_start';
    protected string $event       = CustomerStartedEvent::class;
    protected string $target      = 'verification';
    protected string $description = 'At verification — farmer eligible';

    protected function plan(): array
    {
        return [
            'eligibility_check' => [
                IsFarmerNotEligibleGuard::class => false,
            ],
        ];
    }
}
```

### Example with Parameters

```php
class AtRejectedScenario extends MachineScenario
{
    protected string $machine     = CarSalesMachine::class;
    protected string $source      = 'checking_protocol';
    protected string $event       = ProtocolRejectedEvent::class;
    protected string $target      = 'rejected';
    protected string $description = 'Application rejected with specific reason';

    protected function params(): array
    {
        return [
            // Rich definition — frontend renders a dropdown
            'reason' => [
                'type'   => 'enum',
                'values' => ['GENERAL', 'INSUFFICIENT_INCOME', 'CREDIT_SCORE_LOW'],
                'label'  => 'Rejection Reason',
                'rules'  => ['required'],
            ],
            // Plain rules — frontend renders a generic number input
            'findeksScore' => ['integer', 'min:0', 'max:1900'],
        ];
    }

    protected function plan(): array
    {
        return [
            'allocation' => [
                RejectAction::class => [
                    'rejectionReason' => $this->param('reason'),
                    'findeksScore'    => $this->param('findeksScore', 750),
                ],
            ],
        ];
    }
}
```

### Parameter Definition Format

Each `params()` entry value is either a **plain array** (validation rules only) or an **assoc array** (rich definition with optional type metadata):

| Format | Detection | Example |
|--------|-----------|---------|
| Plain rules | Sequential array | `['required', 'string', 'max:500']` |
| Rich definition | Assoc array with `rules` key | `['type' => 'enum', 'values' => [...], 'rules' => ['required']]` |

**Rich definition keys:**

| Key | Required | Description |
|-----|----------|-------------|
| `rules` | Yes | Laravel validation rules array |
| `type` | No | Hint for frontend widget: `enum`, `number`, `string`, `boolean` |
| `values` | No | Allowed values (for `enum` type) |
| `label` | No | Human-readable label for the frontend |
| `min` / `max` | No | Range constraints (for `number` type) |

When `type` is not provided, the frontend falls back to the raw `rules` array or renders a generic text input. The `required` flag in the response is auto-derived from the `rules` array.

### Naming Convention

| Type | Base class | Concrete example | Suffix |
|------|-----------|-----------------|--------|
| Guard scenario | `GuardScenarioBehavior` | `HasConsentGuardScenario` | `{OriginalName}Scenario` |
| Action scenario | `ActionScenarioBehavior` | `StoreApplicationActionScenario` | `{OriginalName}Scenario` |
| Calculator scenario | `CalculatorScenarioBehavior` | `CustomerContextCalculatorScenario` | `{OriginalName}Scenario` |
| Output scenario | `OutputScenarioBehavior` | `OrderSummaryOutputScenario` | `{OriginalName}Scenario` |
| Machine scenario | `MachineScenario` | `AtVerificationScenario` | Descriptive + `Scenario` suffix |

**Behavior scenario names** mirror the original behavior name with `Scenario` suffix. **Search advantage:** `HasConsent` finds both `HasConsentGuard` and `HasConsentGuardScenario` side by side.

**Machine scenario names** are descriptive — typically the target state. When multiple scenarios target the same state via different paths, disambiguate:
- `AtVerificationScenario` — unique target, sufficient
- `AtVerificationViaConsentScenario` — disambiguated: same target, different state/event
- ❌ `AtVerificationCarSalesMachineScenario` — machine name is redundant

## 5. plan() — Unified State-Route-Keyed Plan

Every `plan()` key is a **full state route**. The **value type** determines what happens at that state:

| Value type | Detection | Meaning |
|-----------|-----------|---------|
| `array` without `'outcome'` key | `is_array($value) && !isset($value['outcome'])` | Behavior overrides — guard/action/calculator/output overrides |
| `string` starting with `@` | `str_starts_with($value, '@')` | Delegation outcome — simulate child completion/failure/timeout |
| `array` with `'outcome'` key | `isset($value['outcome'])` | Delegation outcome with optional output AND/OR behavior overrides for `@done`/`@fail` transition guards/actions |
| `class-string<MachineScenario>` | `is_subclass_of($value, MachineScenario::class)` | Child scenario reference — run child with its own scenario |

ScenarioPlayer validates each key against the machine definition:
- State exists? If not → `ScenarioConfigurationException`
- State has no delegation? Value must be behavior override array (no `'outcome'` key)
- State has delegation? Value must be outcome string, child scenario, or `'outcome'`-keyed array

### Behavior Overrides

For states without delegation — override guards, actions, calculators, and outputs:

```php
protected function plan(): array
{
    return [
        'routing' => [
            // Calculator override — runs BEFORE guards, prepares data
            CustomerContextCalculator::class => ['farmer' => $mockFarmer],

            // Guard override
            HasConsentGuard::class => true,
        ],
        'eligibility_check' => [
            IsFarmerNotEligibleGuard::class => false,
        ],
    ];
}
```

All behavior types (`GuardBehavior`, `ActionBehavior`, `CalculatorBehavior`, `OutputBehavior`) that extend `InvokableBehavior` can be overridden. Both class-based behaviors (`ClassName::class => value`) and inline behaviors (`'camelCaseKey' => value`) are supported — the override mechanism is identical for both.

**Override value forms by behavior type:**

| Behavior | Bool | Array (context write) | Closure | Class |
|----------|------|----------------------|---------|-------|
| **Guard** | ✅ return value | ❌ | ✅ must return bool | ✅ `GuardScenarioBehavior` |
| **Action** | ❌ | ✅ key-value to context | ✅ void | ✅ `ActionScenarioBehavior` |
| **Calculator** | ❌ | ✅ key-value to context | ✅ void | ✅ `CalculatorScenarioBehavior` |
| **Output** | ❌ | ✅ return as output | ✅ must return mixed | ✅ `OutputScenarioBehavior` |

**Guards:**

```php
'eligibility_check' => [
    // 1. Bool shorthand
    IsFarmerNotEligibleGuard::class => false,

    // 2. Closure — inline with DI
    HasConsentGuard::class => function (ContextManager $ctx): bool {
        return $ctx->get('farmer')->hasValidConsent();
    },

    // 3. Class — reusable scenario behavior
    IsCustomerInfoCompleteGuard::class => IsCustomerInfoCompleteGuardScenario::class,
],
```

**Actions:**

```php
'data_collection.retailer.calculating_prices' => [
    // 1. Array shorthand — key-value pairs written to context
    CheckProtocolAction::class => ['protocolEligible' => true],

    // 2. Closure — inline with DI
    StoreApplicationAction::class => function (ContextManager $ctx) {
        $ctx->set('applicationId', 'APP-'.Str::random());
    },

    // 3. Class — reusable scenario behavior
    CalculatePricesAction::class => CalculatePricesActionScenario::class,
],
```

**Calculators:**

```php
'routing' => [
    // 1. Array shorthand — pre-set context values that calculator would compute
    CustomerContextCalculator::class => [
        'farmer'   => $mockFarmer,
        'retailer' => $mockRetailer,
    ],

    // 2. Closure — inline with DI
    RetailerContextCalculator::class => function (ContextManager $ctx) {
        $ctx->set('retailer', Retailer::find(7));
    },

    // 3. Class — reusable scenario behavior (alternative to array shorthand above)
    // CustomerContextCalculator::class => CustomerContextCalculatorScenario::class,
],
```

**Outputs:**

```php
'approved' => [
    // 1. Array shorthand — return as output data
    OrderSummaryOutput::class => ['orderId' => 'ORD-001', 'status' => 'approved'],

    // 2. Closure — inline with DI (alternative to above)
    // OrderSummaryOutput::class => function (ContextManager $ctx): array {
    //     return ['orderId' => $ctx->get('orderId'), 'status' => 'approved'];
    // },

    // 3. Class — reusable scenario behavior (alternative to above)
    // OrderSummaryOutput::class => OrderSummaryOutputScenario::class,
],
```

### Scenario Behavior Base Classes

All behavior types have a corresponding scenario base class:

```php
abstract class GuardScenarioBehavior extends GuardBehavior {}
abstract class ActionScenarioBehavior extends ActionBehavior {}
abstract class CalculatorScenarioBehavior extends CalculatorBehavior {}
abstract class OutputScenarioBehavior extends OutputBehavior {}
```

DI is free (inherited from `InvokableBehavior`), type-compatible (can replace the original in the container).

```php
class CustomerContextCalculatorScenario extends CalculatorScenarioBehavior
{
    public function __invoke(ContextManager $ctx): void
    {
        $ctx->set('farmer', Farmer::find(42));
        $ctx->set('retailer', Retailer::find(7));
    }
}
```

### Same Behavior, Different Values Per State

When the same behavior appears under multiple states with **different values**, ScenarioPlayer creates a **state-aware override** that checks `$state->currentStateDefinition->route` at invocation time:

```php
'routing' => [
    HasConsentGuard::class => true,
],
'data_collection.customer_info.info_checking' => [
    HasConsentGuard::class => false,
],
```

| Situation | What ScenarioPlayer does |
|-----------|-------------------------|
| Behavior in one state only | Simple bind (state key is documentary) |
| Same value across states | Simple bind (state key is documentary) |
| Different values across states | State-aware bind (checks `->route`) |
| Behavior runs in un-overridden state | Original behavior executes |

### Delegation Outcomes

For states with `machine` or `job` delegation — declare what the child/job produces:

```php
protected function plan(): array
{
    return [
        // Child machine outcome — simple string
        'verification.findeks.running' => '@done.report_saved',
        'verification.turmob.verifying' => '@done',

        // Job actor outcome
        'querying_phones' => '@done',
    ];
}
```

Outcome with output data:

```php
'verification.findeks.running' => [
    'outcome' => '@done.report_saved',
    'output'  => ['reportId' => 'RPT-001', 'score' => 1400],
],
```

Outcome with output AND behavior overrides for `@done` guards:

```php
'polling' => [
    'outcome' => '@done',
    IsPinRequiredGuard::class => true,  // override @done transition guard
],
```

**Outcome formats:**

| Format | Example | When to use |
|--------|---------|-------------|
| Simple string | `'@done.report_saved'` | Parent only cares about routing, no guard overrides needed |
| With output | `['outcome' => '@done.report_saved', 'output' => [...]]` | Parent's `@done` action reads output |
| With guard override | `['outcome' => '@done', GuardClass::class => true]` | `@done`/`@fail` transition has guards that need overriding |
| Failure | `'@fail'` | Test failure path |
| Failure with context | `['outcome' => '@fail', 'output' => ['errorCode' => 'INVALID']]` | Parent's `@fail` action reads error context |
| Timeout | `'@timeout'` | Test timeout path |

**How it works:**
1. ScenarioPlayer intercepts delegation dispatch (`ChildMachineJob` / `ChildJobJob`)
2. Does NOT run the delegated machine/job
3. Simulates the completion by sending the declared outcome to the parent
4. Parent's `@done`/`@fail`/`@timeout` transition fires with the declared output

### Child Machine Scenario Reference

Instead of an outcome, reference a **child machine's own scenario** — the child runs and may pause at an interactive state:

```php
protected function plan(): array
{
    return [
        'eligibility_check' => [
            IsFarmerNotEligibleGuard::class => false,
        ],
        'verification.turmob.verifying' => '@done',
        'verification.findeks.running'  => AtAwaitingPinScenario::class,
    ];
}
```

The child's own scenario:

```php
class AtAwaitingPinScenario extends MachineScenario
{
    protected string $machine     = FindeksMachine::class;
    protected string $source      = 'idle';
    protected string $event       = 'MACHINE_START';
    protected string $target      = 'awaiting_pin';
    protected string $description = 'FindeksMachine at awaiting_pin';

    protected function plan(): array
    {
        return [
            'checking_existing_report' => [
                HasExistingReportGuard::class => false,
            ],
            'querying_phones' => '@done',
            'requesting'      => '@done',
            'polling' => [
                'outcome' => '@done',
                IsPinRequiredGuard::class => true,  // force awaiting_pin branch
            ],
        ];
    }
}
```

**What happens:**
1. ScenarioPlayer intercepts FindeksMachine dispatch
2. Creates FindeksMachine, applies child scenario's `plan()` overrides
3. Engine processes automatic transitions with overrides
4. FindeksMachine reaches `awaiting_pin` — interactive state, waits for `PinConfirmedEvent`
5. Child **pauses** — parent stays at `verification.findeks.running`
6. Forward endpoints become active — QA can send `PinConfirmedEvent` via `/findeks/pin-confirmed`

**Child completion:**
- Child reaches a **final state** during replay → ScenarioPlayer simulates `@done` to parent
- Child reaches an **interactive state** → child stays running, parent stays in delegating state, forward endpoints active

**`$source` and `$event` in parent-referenced context:**
When referenced from a parent's `plan()`, the child always starts from its **initial state**. The child scenario's `$source` and `$event` are for **independent usage** — when QA interacts with the child directly.

### Parallel States

For parallel states, specify each region's delegation separately. Only regions you mention are controlled — unmentioned delegations run normally:

```php
protected function plan(): array
{
    return [
        'eligibility_check' => [
            IsFarmerNotEligibleGuard::class => false,
        ],
        // Findeks: complete with report
        'verification.findeks.running' => '@done.report_saved',
        // Turmob: complete
        'verification.turmob.verifying' => '@done',
        // Parallel @done guard
        'verification' => [
            'isFindeksRegionCompletedGuard' => true,
        ],
    ];
}
```

**Partial control — mix outcomes and scenarios:**

```php
// Findeks pauses at OTP, Turmob completes
'verification.findeks.running'  => AtAwaitingPinScenario::class,
'verification.turmob.verifying' => '@done',

// Findeks completes, Turmob runs normally (no plan entry)
'verification.findeks.running' => '@done.report_saved',
// verification.turmob.verifying → NOT in plan → real behavior

// Findeks completes, Turmob fails → test @fail path
'verification.findeks.running'  => '@done.report_saved',
'verification.turmob.verifying' => '@fail',
```

### Fire-and-Forget Delegation

Fire-and-forget delegation (machine with `queue` + no `@done`, or job with `target`) does NOT need entries in `plan()`. The parent transitions immediately — the child/job is a background side effect. In scenario mode, fire-and-forget dispatches are **suppressed**.

### Forward Events and Scenario Interplay

| plan() value | Child state | Forward endpoints |
|-------------|-------------|-------------------|
| `'@done.report_saved'` | Completed (simulated) | **Not active** — child didn't run |
| `AtAwaitingPinScenario::class` | Running, paused at interactive state | **Active** — child is real, waiting for input |

### `@continue` — Auto-Advancing Through Interactive States

When a scenario needs to traverse **multiple interactive states** in a single activation, `@continue` auto-sends events at intermediate stops:

```php
class AtCompletedScenario extends MachineScenario
{
    protected string $machine     = CarSalesMachine::class;
    protected string $source      = 'awaiting_customer_start';
    protected string $event       = CustomerStartedEvent::class;
    protected string $target      = 'allocation';
    protected string $description = 'Full journey — all checks passed, protocol approved';

    protected function plan(): array
    {
        return [
            'eligibility_check' => [
                IsFarmerNotEligibleGuard::class => false,
            ],
            'verification.findeks.running'  => '@done.report_saved',
            'verification.turmob.verifying' => '@done',
            'verification' => [
                'isFindeksRegionCompletedGuard' => true,
            ],
            // Machine arrives at checking_protocol — interactive state.
            // Auto-send ProtocolPassedEvent and continue.
            'checking_protocol' => [
                '@continue' => ProtocolPassedEvent::class,
                CheckProtocolAction::class => ['protocolEligible' => true],
            ],
            // Machine arrives at allocation — no @continue, stops here.
        ];
    }
}
```

**Flow:**

```
1. QA sends CustomerStartedEvent with scenario
2. Machine: awaiting_customer_start → eligibility_check → verification (parallel)
3. Delegations simulated → checking_protocol
4. ScenarioPlayer: @continue at checking_protocol → auto-send ProtocolPassedEvent
5. Machine: checking_protocol → allocation
6. ScenarioPlayer: no @continue at allocation → stop
7. Target validation: machine at allocation === $target ✅
8. Response returns allocation state
```

**`@continue` formats:**

```php
// Event class only — no payload
'@continue' => ProtocolPassedEvent::class,

// Event class + payload
'@continue' => [ProtocolPassedEvent::class, 'payload' => ['source' => 'auto']],

// With scenario params
'@continue' => [OtpSubmittedEvent::class, 'payload' => [
    'otp' => $this->param('otp', '123456'),
]],
```

**`@continue` + behavior overrides in the same state:**

```php
'checking_protocol' => [
    '@continue'                    => ProtocolPassedEvent::class,
    CheckProtocolAction::class     => ['protocolEligible' => true],
    HasValidDocumentsGuard::class  => true,
],
```

Behavior overrides apply to transitions triggered FROM this state (including the `@continue` event's transition). Overrides are registered first, then `@continue` fires.

**Rules:**
- `@continue` is only valid on **non-delegation states** — delegation states use outcomes/child scenarios
- ScenarioPlayer loops: after each `$machine->send()`, check current state for `@continue` → send → check again → until no match
- Infinite loop protection: respects `max_transition_depth`
- If a `@continue` event fails (guard rejection, invalid transition), `ScenarioFailedException` is thrown with the failing state and event

## 6. Parameterized Behaviors in Scenarios

EventMachine supports parameterized behaviors via named parameters: `'guards' => [[HasRoleGuard::class, 'role' => 'admin']]`. Closures and class-based scenario behaviors receive these parameters via DI:

```php
// Machine: 'guards' => [[HasRoleGuard::class, 'role' => 'admin'], [HasRoleGuard::class, 'role' => 'editor']]
'state_a' => [
    HasRoleGuard::class => function (ContextManager $ctx, string $role): bool {
        return match ($role) {
            'admin'  => true,
            'editor' => false,
        };
    },
],
```

| Override form | Parameterized behavior support |
|---------------|-------------------------------|
| `bool` | Returns same value for ALL calls regardless of parameters |
| `array` | Writes same context for ALL calls regardless of parameters |
| `Closure` | Receives parameters via DI — can differentiate |
| `class-string` | Receives parameters via `__invoke` — can differentiate |

## 7. Engine Feature Interactions

### Timers (after/every)

Timers are lazy in EventMachine — `MachineTimerFire` records are created by `ProcessTimersCommand` during scheduled sweeps, not during state entry. Since scenario replay is synchronous (milliseconds), the sweep command doesn't run during replay. Intermediate state timers never fire. No special handling needed.

### Scheduled Events

Scheduled events are processed by `machine:process-scheduled` on cron. Since scenario replay is synchronous, scheduled events don't fire during replay. No special handling needed.

### Queued Listeners

Queued listeners (`ListenerJob`) dispatch to the queue during scenario replay. They restore the machine via `Machine::create()` — scenario overrides propagate via `machine_current_states.scenario_class` + `scenario_params` (§9). Listener behavior respects scenario overrides automatically.

### ValidationGuardBehavior

`ValidationGuardBehavior` throws `MachineValidationException` on failure (returns 422 via endpoints). Scenario overrides work identically to regular guards — the override return value determines whether the guard passes. If the scenario overrides a validation guard to return `true`, no exception is thrown. If overridden to return `false`, the exception fires normally.

### Machine Locking

Scenario activation doesn't change lock semantics. Locks are acquired/released per existing rules during `Machine::send()`. Async jobs (parallel regions, listeners) acquire their own locks when restored. No special handling needed.

### Machine::fake() Compatibility

`Machine::fake()` stubs machine execution entirely — no real transitions, no event history. Scenarios require **real execution** with overridden behaviors. These are incompatible: scenarios should NOT be activated on faked machines. `ScenarioPlayer` throws `ScenarioConfigurationException` if the target machine is faked.

### Event History

Scenario replay creates **real event history**. All transitions, entry/exit actions, @always chains, listeners — recorded as normal `MachineEvent` records. The resulting machine is indistinguishable from one that arrived at the state organically. Full restoration from any point is supported.

### Event Archival

Event archival (`ArchiveService`) is transparent to scenarios. Archived machines are auto-restored when new events arrive. No special handling needed.

### Machine Persistence (should_persist)

Scenario respects the machine's `should_persist` config. If `should_persist=false`, scenario-generated events are not persisted (in-memory only). The `machine_current_states.scenario_class` and `scenario_params` columns are only written when `should_persist=true`.

### Entry/Exit Actions

Entry and exit actions execute normally during scenario replay — they are part of the real state transitions. Scenario behavior overrides apply to entry/exit actions the same way they apply to transition actions:

```php
'checking_protocol' => [
    // This overrides CheckProtocolAction whether it's a transition action or entry action
    CheckProtocolAction::class => ['protocolEligible' => true],
],
```

If the scenario does NOT override an entry/exit action, the original action runs with real behavior. This is by design — scenarios only override what they explicitly declare.

### Event Bubbling in Hierarchical States

When an event is sent to a machine in a nested state, the engine bubbles the event up the parent chain until a handler is found. Scenario overrides are matched to the state where the transition is **defined**, not the state where the event was sent:

```php
// Machine config:
'data_collection' => [
    'on' => ['CANCEL' => 'cancelled'],  // handler defined on parent
    'states' => [
        'retailer' => [...],            // event sent while here
    ],
],

// Scenario plan():
'data_collection' => [
    CancelGuard::class => true,  // override applies when CANCEL bubbles to data_collection
],
```

If the guard is on the parent state's transition, the override key must be the parent's route (`data_collection`), not the child's route (`data_collection.retailer`).

### raise() / sendTo() / dispatchTo()

Actions executed during scenario replay can call `raise()`, `sendTo()`, and `dispatchTo()`. These work normally:

- **`raise()`** — raised events are processed within the same macrostep. Scenario overrides apply to transitions triggered by raised events (same container scope).
- **`sendTo()`** — sends an event to another machine synchronously. The target machine does NOT have scenario overrides — scenarios are scoped to the machine they're activated on.
- **`dispatchTo()`** — dispatches an event to another machine asynchronously via `SendToMachineJob`. Same scoping as `sendTo()` — no scenario propagation to other machines.
- **`sendToParent()` / `dispatchToParent()`** — sends to the parent machine. Parent's scenario overrides (if any) are separate from the child's.

**Key principle:** Scenarios are scoped to a single machine instance. Cross-machine communication is unaffected.

### Computed Context (`computedContext()`)

Custom `ContextManager` subclasses with `computedContext()` methods execute normally during scenario replay. Computed fields are **not overridable** via scenarios — they're methods on the context class, not `InvokableBehavior` subclasses.

If a computed field depends on external data (e.g., API call, database query), it will run with real data during scenario replay. To control computed output, override the actions that populate the context fields the computed method depends on.

### Lock Contention During Scenario Activation

Scenario activation respects existing machine locks. If the machine is already processing an event (lock held), the scenario request receives the same lock contention handling as any other request:

- **POST endpoint** → HTTP 423 Locked + `isProcessing: true`
- The scenario is NOT activated — QA must retry after the lock is released

This prevents race conditions during scenario activation.

### Machine::query() and Scenario Filtering

The `machine_current_states` table has `scenario_class` and `scenario_params` columns (§9). `Machine::query()` can filter by active scenario:

```php
// Find machines currently running under a specific scenario
CarSalesMachine::query()
    ->whereNotNull('scenario_class')
    ->get();

// Find machines with a specific scenario
CarSalesMachine::query()
    ->where('scenario_class', AtCheckingProtocolScenario::class)
    ->get();
```

This is useful for QA dashboards — showing which machines are in active scenario testing.

### Transient State as Target

If `$target` is a transient state (has `@always` transitions), the machine will enter the state and immediately fire `@always` — it won't stop there. `ScenarioTargetMismatchException` will be thrown because the machine moved past the target.

**Rule:** `$target` must be a state where the machine can **settle** — interactive states, delegation states (waiting for child), parallel states (waiting for regions), or final states. Transient states are not valid targets. `machine:scenario-validate` (§16) catches this at validation time.

### Path Coverage

Scenario replay creates real state transitions, so paths exercised during scenario activation are recorded by `PathCoverageTracker` (if enabled via `TracksPathCoverage` trait). Scenario-driven paths are **indistinguishable** from normally-reached paths in coverage reports — this is correct behavior since scenarios exercise real engine code paths.

## 8. Endpoint Integration

### QA Workflow

1. QA opens frontend, sees scenario selector (staging only)
2. Endpoint response includes `availableEvents` and `availableScenarios` (grouped by event)
3. QA selects a scenario from the dropdown for a specific event
4. QA triggers the event — frontend sends `{ type: "...", scenario: "...", scenarioParams: {...} }`

**What happens:**
1. Controller detects `scenario`, validates `$source` = current state, `$event` = event being sent
2. Persists scenario to `machine_current_states.scenario_class` + `scenario_params`
3. Registers behavior overrides, prepares delegation handling
4. `$machine->send(CustomerStartedEvent)` — transitions to `eligibility_check`
5. `IsFarmerNotEligibleGuard` → scenario returns `false` → `verification` (parallel)
6. `FindeksMachine` intercepted → simulated as `@done.report_saved`
7. `TurmobMachine` intercepted → simulated as `@done`
8. `isFindeksRegionCompletedGuard` → scenario returns `true` → `checking_protocol`
9. `@continue` loop — no `@continue` at `checking_protocol` → stop
10. Target validation: `checking_protocol` === `$target` ✅
11. Cleanup, response with final state + `availableScenarios`

### Endpoint Flow

```
MachineController::executeEndpoint()
│
├─ 1. maybeRegisterScenarioOverrides(Request, machineClass)
│     ├─ Read 'scenario' slug and 'scenarioParams' from request
│     ├─ If null → clear scenario_class + scenario_params from machine_current_states, skip
│     ├─ If disabled → skip
│     ├─ Resolve scenario class (via ScenarioDiscovery)
│     ├─ Validate $source matches current state
│     ├─ Validate $event matches event being sent
│     ├─ Validate scenarioParams against params() rules (extracted from plain or rich definitions)
│     ├─ Hydrate scenario with validated params
│     ├─ Persist scenario_class + scenario_params to machine_current_states
│     ├─ Register state-keyed overrides in container
│     ├─ Prepare delegation handling (outcomes + child scenario references)
│     └─ Return scenario instance
│
├─ 2. $state = $machine->send(event)
│     ├─ @always guards use scenario behaviors ✅
│     ├─ Delegation intercepted → outcomes simulated or child scenarios run ✅
│     └─ Async jobs carry scenario via DB (see §9)
│
├─ 3. @continue loop (see §5)
│     └─ While current state has @continue → $machine->send(@continue event)
│
├─ 4. Validate $target — machine must be at scenario's target state
│
├─ 5. finally: cleanup — unbind scenario behaviors from container
│
└─ 6. buildResponse() — final state, availableEvents, availableScenarios
```

### `availableScenarios` in Endpoint Response

When `scenarios.enabled=true`, `buildResponse()` adds `availableScenarios` to every endpoint response. This field is state-aware — only scenarios whose `$source` matches the current state are included, grouped by event type. See [Scenario Endpoints](/advanced/scenario-endpoints) for response structure details.

## 9. Async Propagation

Scenario overrides live in the Laravel container — process-scoped. Queue jobs run in separate processes with their own containers. The scenario must propagate to async jobs.

### The Problem

During `$machine->send()`, the engine may dispatch async jobs:
- `ParallelRegionJob` — parallel regions processed by queue workers
- `ListenerJob` — queued listeners
- `ChildMachineCompletionJob` — child completion routing on parent

These jobs run in separate processes without scenario overrides.

### The Solution: `machine_current_states.scenario_class`

The `machine_current_states` table already has one row per machine instance. A nullable `scenario_class` column stores the active scenario's class name.

**Migration:**

```php
Schema::table('machine_current_states', function (Blueprint $table) {
    $table->string('scenario_class')->nullable()->after('machine_value');
    $table->json('scenario_params')->nullable()->after('scenario_class');
});
```

**Activation** — when QA sends an event with `scenario` field:

```php
MachineCurrentState::where('root_event_id', $rootEventId)
    ->update([
        'scenario_class'  => AtVerificationScenario::class,
        'scenario_params' => $validatedParams ?: null,
    ]);
```

**Restoration** — when any job restores a machine via `Machine::create()`:

```php
if (config('machine.scenarios.enabled')) {
    $record = MachineCurrentState::where('root_event_id', $rootEventId)
        ->first(['scenario_class', 'scenario_params']);
    if ($record?->scenario_class) {
        $scenario = new $record->scenario_class();
        $scenario->hydrateParams($record->scenario_params ?? []);
        ScenarioPlayer::registerOverrides($scenario);
    }
}
```

**Deactivation** — when QA sends the next event WITHOUT a `scenario` field:

```php
MachineCurrentState::where('root_event_id', $rootEventId)
    ->update(['scenario_class' => null, 'scenario_params' => null]);
```

### Write Always, Read on Restore

`scenario_class` is **always written** during scenario activation — regardless of whether the machine runs sync or async. Writing is nearly free (one extra column in an existing UPDATE).

**Reading** happens on every `Machine::create()` restoration when `scenarios.enabled=true`. The machine doesn't distinguish sync from async — it always checks. In sync context, overrides may already be registered (idempotent re-registration is harmless). In async context, this is how overrides propagate to queue workers. In production, `scenarios.enabled=false` prevents the query entirely.

**`@continue` runs in ScenarioPlayer only.** The `@continue` loop executes in two contexts: (1) the controller flow when QA sends an event with a scenario, and (2) child scenario replay when ScenarioPlayer intercepts a delegation and runs the child with its own scenario. In both cases, ScenarioPlayer controls the loop. Async jobs (ParallelRegionJob, ListenerJob, ChildMachineCompletionJob) restore scenario behavior overrides but do NOT run the `@continue` loop — they simply apply overrides for whatever event they're processing.

### Scenario Lifecycle

```
1. Machine running normally (no scenario)
   → scenario_class = null, scenario_params = null

2. QA sends event with scenario field
   → scenario_class = 'AtRejectedScenario'
   → scenario_params = {"reason": "INSUFFICIENT_INCOME"}
   → overrides active for this event + any async jobs

3. Async jobs pick up work
   → restore machine → find scenario_class + scenario_params → hydrate → register overrides

4. QA sends next event WITHOUT scenario field
   → scenario_class = null, scenario_params = null
   → machine continues with real behavior

5. QA sends next event with DIFFERENT scenario
   → scenario_class = 'AtApprovedScenario'
   → scenario_params = null (no params needed)
   → new overrides replace old
```

## 10. ScenarioPlayer — Runtime Engine

```
execute(scenario, eventPayload, machine)
│
├─ 1. Validate environment (scenarios enabled?)
├─ 2. Validate and hydrate params
│     ├─ Extract validation rules from params() (plain or rich definitions)
│     ├─ Validate scenarioParams against extracted rules
│     └─ Store validated values → $this->param('key') ready
├─ 3. Parse plan()
│     ├─ Validate $source and $target exist in machine definition
│     ├─ Validate all plan() state routes against machine definition
│     ├─ Separate behavior overrides (array values on non-delegation states)
│     ├─ Extract @continue directives from behavior override arrays
│     ├─ Separate delegation outcomes (@ strings on delegation states)
│     └─ Separate child scenario references (MachineScenario classes on delegation states)
├─ 4. Persist scenario_class + scenario_params to machine_current_states
├─ 5. Register behavior overrides in container (state-aware bindings)
├─ 6. Prepare delegation handling
│     ├─ Outcomes → intercept dispatch, simulate @done/@fail/@timeout
│     └─ Child scenarios → intercept dispatch, create child, apply scenario, run
│
├─ 7. Send trigger event
│     └─ $event with payload from endpoint request
│
├─ 8. @continue loop
│     ├─ Check: does current state have @continue in plan()?
│     ├─ Yes → $machine->send(@continue event with payload)
│     ├─ Repeat from check (new current state)
│     ├─ No → exit loop
│     └─ Safety: max_transition_depth prevents infinite loops
│
├─ 9. Validate target
│     └─ Machine's current state matches $target? If not → ScenarioTargetMismatchException
│
├─ 10. Build ScenarioOutput
│
└─ 11. Cleanup (finally)
       └─ Unbind overrides from container (DB record stays for async jobs)
```

## 13. Error Handling & requiredContext Hints

### Exceptions

| Exception | When |
|-----------|------|
| `ScenariosDisabledException` | `MACHINE_SCENARIOS_ENABLED=false` |
| `ScenarioFailedException` | Guard rejection during replay, `@continue` event rejected, `$source` mismatch, `$event` mismatch. Includes `eventType`, `currentState`, `rejectionReason`. |
| `ScenarioTargetMismatchException` | Machine did not reach `$target` after execution. Includes `expectedTarget`, `actualState`. Indicates a misconfigured `plan()` or missing `@continue`. |
| `ScenarioConfigurationException` | Invalid state route in `plan()`. Delegation outcome on non-delegation state. Behavior override on delegation state. `@continue` on delegation state. Child scenario `$machine` doesn't match delegation target. Missing required properties (`$machine`, `$source`, `$event`, `$target`). Invalid scenario params (validation failure). |
| `NoScenarioPathFoundException` | `machine:scenario` command: no path connects source to target in the machine definition. Includes `source`, `target`, `machineClass`. |
| `AmbiguousScenarioPathException` | `machine:scenario` command: multiple paths connect source to target — use `--path=N` or interactive selection. Includes `paths` list with signatures. |

### requiredContext Hints

When a `MissingMachineContextException` is thrown during scenario replay, the error is enriched with `requiredContext` hints:

```
`farmer` is missing in context.

Required context for behaviors at this state:
  → IsFarmerNotEligibleGuard: farmer (App\Models\Farmer), retailer (App\Models\Retailer)
```

Most structural errors can be caught **before deployment** with `machine:scenario-validate` (§16). Runtime exceptions serve as a safety net for errors that slip through.

## 15. `machine:scenario` — Scaffold Command

Generate a `MachineScenario` class by analyzing the machine definition and resolving the path from source to target.

### Command Signature

```
php artisan machine:scenario
    {name : Scenario class name (Scenario suffix auto-added if missing)}
    {machine : Machine class name, FQCN, or file path}
    {source : Source state route (full or unambiguous partial)}
    {event : Triggering event (class FQCN or event type string)}
    {target : Target state route (full or unambiguous partial)}
    {--dry-run : Print generated file without writing}
    {--force : Overwrite existing scenario file}
    {--path=0 : Select path by index when multiple paths exist}
```

**Example:**

```bash
php artisan machine:scenario AtAllocation CarSalesMachine \
    awaiting_customer_start CustomerStartedEvent allocation
```

Writes `app/Machines/CarSales/Scenarios/AtAllocationScenario.php`.

### Architecture

Three components with single responsibilities, built on a shared graph utility:

```
machine:scenario command
│
├─ MachineGraph                    — shared graph utility (reusable)
│   ├─ resolveState()              — partial/full route → StateDefinition
│   ├─ transitionsFrom()           — all transitions from a state (parent chain)
│   ├─ classifyState()             — transient / delegation / parallel / interactive / final
│   ├─ availableEventsFrom()       — event types available from a state
│   ├─ delegationOutcomes()        — @done.{state} options for delegation states
│   └─ alwaysBranchGuards()        — guard classes on @always branches
│
├─ ScenarioPathResolver            — source→target path finder
│   ├─ resolve(source, target)     — single best path
│   └─ resolveAll(source, target)  — all paths (for selection)
│
└─ ScenarioScaffolder              — PHP file generator
    └─ scaffold(name, machine, source, event, target, steps)
```

### MachineGraph — Shared Graph Utility

Extracted from `PathEnumerator`'s internal methods into a reusable component. Both `PathEnumerator` (coverage analysis) and `ScenarioPathResolver` (scenario scaffolding) use `MachineGraph` as their foundation.

```php
class MachineGraph
{
    public function __construct(private readonly MachineDefinition $definition) {}

    /**
     * Find a state by full or partial route.
     * Partial routes are resolved by suffix matching — 'allocation' matches
     * 'car_sales.allocation' if unambiguous. Throws if ambiguous or not found.
     */
    public function resolveState(string $route): StateDefinition;

    /**
     * All transitions available from a state, including inherited parent chain.
     * Equivalent to PathEnumerator::collectAllTransitions().
     */
    public function transitionsFrom(StateDefinition $state): array;

    /** Classify a state by its structural role. */
    public function classifyState(StateDefinition $state): StateClassification;

    /** Event types that can be sent to this state (from own + parent transitions). */
    public function availableEventsFrom(StateDefinition $state): array;

    /** @done.{finalState} options for a delegation state. */
    public function delegationOutcomes(StateDefinition $state): array;

    /** Guard class names on each @always branch. */
    public function alwaysBranchGuards(StateDefinition $state): array;
}
```

**`StateClassification` enum:**

```php
enum StateClassification: string
{
    /** Has @always transitions — machine passes through automatically. */
    case TRANSIENT = 'transient';

    /** Has machine or job delegation (machine/job key). */
    case DELEGATION = 'delegation';

    /** type === 'parallel' — concurrent regions. */
    case PARALLEL = 'parallel';

    /** No @always, no delegation, not parallel, not final — waits for external event. */
    case INTERACTIVE = 'interactive';

    /** type === 'final' — terminal state. */
    case FINAL = 'final';
}
```

**Partial route resolution:**

The command accepts partial routes for convenience. `MachineGraph::resolveState()` resolves by suffix matching:

| Input | Machine has | Resolves to | Notes |
|-------|-----------|-------------|-------|
| `allocation` | `car_sales.allocation` | `car_sales.allocation` | Unambiguous leaf match |
| `retailer.pricing` | `car_sales.data_collection.retailer.pricing` | Full route | Unambiguous suffix match |
| `running` | `verification.findeks.running`, `verification.turmob.running` | **Error** | Ambiguous — multiple matches |
| `verification.findeks.running` | Same | Full route | Exact match, no ambiguity |
| `nonexistent` | — | **Error** | Not found |

### ScenarioPathResolver — Source → Target Path Finding

Finds the path a machine would take from source to target, classifying each intermediate state:

```php
class ScenarioPathResolver
{
    public function __construct(private readonly MachineGraph $graph) {}

    /**
     * Find the shortest path from source to target via the trigger event.
     * Returns classified steps for scaffold generation.
     *
     * @throws NoScenarioPathFoundException  No path connects source to target.
     * @throws AmbiguousScenarioPathException  Multiple paths exist (use resolveAll).
     */
    public function resolve(string $source, string $event, string $target): ScenarioPath;

    /**
     * Find ALL paths from source to target via the trigger event (for user selection).
     * @return list<ScenarioPath>
     */
    public function resolveAll(string $source, string $event, string $target): array;
}
```

**`ScenarioPath` — resolved path with classified steps:**

```php
readonly class ScenarioPath
{
    /**
     * @param  list<ScenarioPathStep>  $steps  Ordered steps from source to target.
     */
    public function __construct(
        public array $steps,
    ) {}

    /** Human-readable signature: "source→[event]→state→[event]→target" */
    public function signature(): string;
}
```

**`ScenarioPathStep` — one state in the resolved path:**

```php
readonly class ScenarioPathStep
{
    public function __construct(
        /** Full state route (e.g., 'car_sales.verification.findeks.running'). */
        public string $stateRoute,

        /** Short state key for display (e.g., 'running'). */
        public string $stateKey,

        /** How this state should be handled in plan(). */
        public StateClassification $classification,

        /** Event that enters this state (null for source state). */
        public ?string $event,

        /** Guard class names on the transition into this state. */
        public array $guards,

        /** Action class names on the transition into this state. */
        public array $actions,

        /** Child machine or job FQCN (DELEGATION states only). */
        public ?string $invokeClass,

        /** Events available FROM this state (INTERACTIVE states — for @continue selection). */
        public array $availableEvents,

        /** @done.{state} options (DELEGATION states — for outcome selection). */
        public array $availableDoneStates,
    ) {}
}
```

**Algorithm — BFS with classification:**

```
resolve(source, event, target):
1. sourceState = graph.resolveState(source)
2. targetState = graph.resolveState(target)

   // Start from the state(s) reachable via the trigger event from source.
   // Source itself doesn't need a plan() entry — the user sends $event to trigger it.
3. firstTransition = sourceState.transitions[event]
4. queue = [(branch.target, []) for branch in firstTransition.branches]
5. visited = {}

6. while queue not empty:
   a. (state, path) = dequeue
   b. if state.route == target → return ScenarioPath(path)
   c. if state in visited → skip (cycle)
   d. visited.add(state)
   e. classification = graph.classifyState(state)

   f. switch classification:
      TRANSIENT:
        — for each @always branch:
            step = ScenarioPathStep(stateRoute, TRANSIENT, '@always', guards, ...)
            enqueue(branch.target, path + step)

      DELEGATION:
        — for each @done/@fail/@timeout transition:
            step = ScenarioPathStep(stateRoute, DELEGATION, invokeType, invokeClass, ...)
            enqueue(transition.target, path + step)

      PARALLEL:
        — resolve region delegations
        — follow @done transition
        step = ScenarioPathStep(stateRoute, PARALLEL, '@done', guards, ...)
        enqueue(@done.target, path + step)

      INTERACTIVE:
        — for each event transition:
            step = ScenarioPathStep(stateRoute, INTERACTIVE, event, ...)
            enqueue(transition.target, path + step)

      FINAL:
        — dead end (not target) → skip

7. throw NoScenarioPathFoundException
```

**Multiple paths:** When BFS finds multiple paths to the target, the command presents them:

```
Found 2 paths from awaiting_customer_start to allocation:

  [0] awaiting_customer_start → eligibility_check → verification → checking_protocol → allocation
      3 overrides, 2 delegation outcomes, 1 @continue

  [1] awaiting_customer_start → eligibility_check → manual_review → allocation
      2 overrides, 0 delegation outcomes, 0 @continue

Select path [0]:
```

Use `--path=N` to skip the prompt.

### ScenarioScaffolder — PHP File Generator

Transforms a `ScenarioPath` into a complete `MachineScenario` PHP class:

```php
class ScenarioScaffolder
{
    /**
     * Generate PHP file content for a MachineScenario class.
     *
     * @param  list<ScenarioPathStep>  $steps  Classified path steps.
     * @return string  PHP file content (ready to write).
     */
    public function scaffold(
        string $scenarioName,
        string $machineClass,

```

**Classification → plan() entry mapping:**

| Classification | Generated plan() entry | Comment |
|---------------|----------------------|---------|
| `TRANSIENT` | `'route' => [Guard::class => false]` | Lists all guards with default `false`. `// TODO: adjust` |
| `DELEGATION` | `'route' => '@done'` | `// Available: @done.report_saved, @done.no_report, @fail, @timeout` |
| `PARALLEL` | Region outcomes + `@done` guard override | Each region as separate entry, parent guard as override |
| `INTERACTIVE` | `'route' => ['@continue' => Event::class]` | `// Also: OtherEvent, AnotherEvent` |
| `FINAL` | Not included | Target state — machine stops here |

**Guard override defaults:**
- Single guard: `Guard::class => false` (skip)
- Multiple guards on same @always branch: all listed with `=> false`
- Multiple @always branches (guarded): first branch's guards overridden, others as comments

**Delegation outcome defaults:**
- `'@done'` as default (most common — child completes successfully)
- Available outcomes listed as comment: `@done.{state}` variants, `@fail`, `@timeout`

**`@continue` event selection:**
- Single event from the state toward target: auto-selected
- Multiple events: first event toward target selected, others as comments

**`@continue` payload extraction:**

When the `@continue` event is an `EventBehavior` subclass, the scaffolder reads its `rules()` method to extract payload field names and types. Generated plan() includes TODO-annotated payload:

```php
// EventBehavior with rules(): ['phone' => 'required|string', 'queryId' => 'required|string']
'awaiting_report_request' => [
    '@continue' => [ReportRequestedEvent::class, 'payload' => [
        'phone'   => '', // TODO: required (string)
        'queryId' => '', // TODO: required (string)
    ]],
],
```

If the event class has no `rules()` or is a string type (not a class), the payload is omitted:

```php
'some_state' => [
    '@continue' => SomeEvent::class,
],
```

### Deep Target — Cross-Delegation Targeting

When the target path crosses a delegation boundary, the command resolves it into a parent scenario + child scenario reference:

```bash
php artisan machine:scenario AtFindeksBirthDateCorrection CarSalesMachine \
    idle CarSalesApplicationStartedEvent findeks.awaiting_birth_date_correction
```

**Resolution steps:**

1. Try `findeks.awaiting_birth_date_correction` in CarSalesMachine → not found
2. Prefix match: `findeks` → `verification.findeks` (parallel region)
3. `verification.findeks.running` is a delegation state → delegates to `FindeksMachine`
4. `awaiting_birth_date_correction` exists in `FindeksMachine` → cross-delegation target confirmed
5. Parent target resolved: `verification` (parallel parent stays here while child is active)
6. Child target resolved: `awaiting_birth_date_correction` in `FindeksMachine`

**Child scenario discovery:**

The command scans `FindeksMachine`'s `Scenarios/` directory for scenarios where `$target === 'awaiting_birth_date_correction'`:

| Found | Behavior |
|-------|----------|
| Exactly one match | Auto-referenced in parent's plan(): `'verification.findeks.running' => AtAwaitingBirthDateCorrectionScenario::class` |
| Multiple matches | User selects (same UX as path selection) |
| No match | Error with scaffold suggestion: `php artisan machine:scenario AtAwaitingBirthDateCorrection FindeksMachine idle MACHINE_START awaiting_birth_date_correction` |

**Intended workflow:**

```bash
# 1. Create child scenario first
php artisan machine:scenario AtAwaitingBirthDateCorrection FindeksMachine \
    idle MACHINE_START awaiting_birth_date_correction

# 2. Create parent scenario — auto-discovers child scenario
php artisan machine:scenario AtFindeksBirthDateCorrection CarSalesMachine \
    idle CarSalesApplicationStartedEvent findeks.awaiting_birth_date_correction
```

The parent's generated plan() references the discovered child scenario:

```php
'verification.findeks.running' => AtAwaitingBirthDateCorrectionScenario::class,
```

**Deep target format:**

`{region_key}.{child_state}` — the region key identifies the delegation state, the child state is the target inside the child machine:

| Deep target | Region | Delegation state | Child machine | Child target |
|-------------|--------|-----------------|---------------|-------------|
| `findeks.awaiting_birth_date_correction` | `verification.findeks` | `verification.findeks.running` | `FindeksMachine` | `awaiting_birth_date_correction` |
| `findeks.awaiting_pin` | `verification.findeks` | `verification.findeks.running` | `FindeksMachine` | `awaiting_pin` |
| `turmob.verified` | `verification.turmob` | `verification.turmob.verifying` | `TurmobMachine` | `verified` |

Ambiguity rules apply: if `awaiting_birth_date_correction` is unique across ALL child machines, the region prefix can be omitted. If ambiguous, the command requires the prefix.

### Edge Cases

| Situation | Behavior |
|-----------|----------|
| Source = initial state | Normal — scenario starts from machine's initial state |
| Source ≠ initial state | Normal — scenario starts from any state in the machine |
| Target = final state | Normal — no @continue needed at target |
| Target = intermediate state | Normal — machine stops at target, @continue reaches it |
| No path exists source → target | `NoScenarioPathFoundException` with explanation |
| Ambiguous partial route | Error with disambiguation suggestions: "Did you mean: `verification.findeks.running` or `verification.turmob.running`?" |
| Scenario file already exists | Error. Use `--force` to overwrite |
| Machine class not found | Error with helpful message |
| Event not valid from source state | Warning: "Event `X` is not available from `source`. Available events: Y, Z" |
| Parallel state in path | Regions scaffolded as separate delegation outcome entries |
| @always with unguarded fallback | Single entry — fallback always taken, no guard override needed |
| @always with ALL branches guarded | All branches listed, first branch overridden by default |
| Fire-and-forget delegation in path | Skipped — parent transitions immediately, no plan() entry needed |
| Path passes through compound state | Compound entry/exit tracked, child states handled normally |
| Delegation with guarded @done branches | `'outcome'`-keyed array with guard overrides: `['outcome' => '@done', Guard::class => true]` |
| Deep target (cross-delegation) | Resolves delegation boundary, discovers child scenario, references in parent plan() |
| Deep target — no child scenario | Error with `machine:scenario` command suggestion for child |
| Deep target — multiple child scenarios | Presents selection (same UX as path selection) |
| `@continue` event has `EventBehavior` with `rules()` | Payload fields extracted and scaffolded with TODO comments |

## 16. `machine:scenario-validate` — Scenario Validation

Validate all scenarios against their machine definitions — catches structural errors and broken paths without running machines or requiring a database.

### Command Signature

```
php artisan machine:scenario-validate
    {machine? : Validate scenarios for a specific machine (optional — all machines if omitted)}
    {--scenario= : Validate a single scenario by class name or slug}
```

### What It Validates

Two validation levels, both run by default:

**Level 1 — Static validation (structural correctness):**

| Check | Error |
|-------|-------|
| `$machine` class exists | `Machine class not found: App\Machines\OrderMachine` |
| `$source` state route exists in machine definition | `Source state 'awaiting_start' not found in CarSalesMachine` |
| `$target` state route exists in machine definition (or child machine for deep targets) | `Target state 'allocation' not found in CarSalesMachine` |
| `$target` is not a transient state (has @always, machine won't settle) | `Target 'eligibility_check' is transient (@always) — machine cannot stop here` |
| `$event` is valid from `$source` state | `Event 'CustomerStartedEvent' not available from 'idle'. Available: CarSalesApplicationStartedEvent` |
| All `plan()` state routes exist in machine definition | `State route 'eligibilty_check' not found (typo?)` |
| Behavior classes in overrides exist | `Guard class 'IsFarmerNotEligibleGard' not found` |
| Delegation outcomes only on delegation states | `'eligibility_check' has outcome '@done' but is not a delegation state` |
| Behavior overrides only on non-delegation states | `'verification.findeks.running' has guard override but is a delegation state` |
| `@continue` only on non-delegation states | `'verification.findeks.running' has @continue but is a delegation state` |
| `@continue` event classes exist | `Event class 'ProtocolPasedEvent' not found` |
| `@continue` events valid from their state | `'ProtocolPassedEvent' not available from 'checking_protocol'` |
| Child scenario `$machine` matches delegation target | `AtAwaitingPinScenario targets FindeksMachine but 'verification.turmob.verifying' delegates to TurmobMachine` |
| `params()` rules are valid | `Invalid validation rule 'requred' in param 'reason'` |
| Required properties set | `Missing required property: $target` |

**Level 2 — Path validation (reachability):**

| Check | Error |
|-------|-------|
| Path exists from `$source` to `$target` via `$event` | `No path from 'idle' to 'allocation' via 'StartEvent' in CarSalesMachine` |
| `@continue` events lead toward `$target` | `@continue at 'checking_protocol' sends 'ProtocolRejectedEvent' which leads to 'rejected', not toward target 'allocation'` |
| Deep target child scenario exists | `No scenario found for FindeksMachine targeting 'awaiting_pin'. Run: php artisan machine:scenario ...` |

Test-runner style output:

```

  ✓ AtCheckingProtocolScenario            idle → checking_protocol
  ✗ AtAllocationScenario                  idle → allocation

  ✓ AtFindeksBirthDateCorrectionScenario  idle → verification (child: FindeksMachine → awaiting_birth_date_correction)
```

### CI/CD Integration

Add to `composer.json` quality gate:

```json
{
    "scripts": {
        "test:scenarios": "php artisan machine:scenario-validate",
        "quality": [
            "@pint",
            "@rector",
            "@test",
            "@test:scenarios"
        ]
    }
}
```

Or run standalone in CI pipeline:

```bash
php artisan machine:scenario-validate --ansi
```

Exit code 0 = all valid, exit code 1 = failures found.

### When to Run

| Trigger | Why |
|---------|-----|
| After machine definition changes | State renames, guard additions, transition restructuring can break scenarios |
| After scenario creation/modification | Catch typos and structural errors immediately |
| CI/CD pipeline (pre-deploy) | Prevent broken scenarios from reaching staging |
| `composer quality` | Part of the standard quality gate |

### Relationship to Runtime Validation

`machine:scenario-validate` catches errors **before deployment** via static analysis. Runtime validation (§13 exceptions) catches errors **during activation** in staging:

| Validation | When | Catches |
|-----------|------|---------|
| `machine:scenario-validate` | CI/CD, pre-deploy | Structural errors, broken paths, missing classes, typos |
| `ScenarioConfigurationException` | plan() parse time | Same structural errors, at runtime |
| `ScenarioFailedException` | Execution time | Guard rejections, invalid transitions during replay |
| `ScenarioTargetMismatchException` | Post-execution | plan() logic doesn't reach $target |

Static validation is a **superset** of `ScenarioConfigurationException` checks — it runs the same validations plus path reachability. If `machine:scenario-validate` passes, `ScenarioConfigurationException` will never be thrown for those scenarios.
