# Callable Delegation Outcome in Scenarios

**Status:** Draft
**Date:** 2026-04-01

## Problem

Scenario `plan()` and `continuation()` support static delegation outcomes: `'@done'`, `'@fail'`, `'@timeout'`. But real QA flows need **runtime-conditional outcomes** where the result depends on context data.

### The Findeks PIN Example

FindeksMachine has a `confirming_pin` job actor state:

```
confirming_pin (job: ConfirmFindeksPinJob)
  @done → polling → saving_report → report_saved
  @fail → [IsPinRetryableGuard=true] → awaiting_pin (retry)
        → [IsPinRetryableGuard=false] → failed
```

QA needs to test both paths. The PIN is entered by the user and stored in `context->pin` via `StorePinAction`. In a scenario, we want:

- PIN = today's date (DDMMYY format, e.g., `010426` for April 1, 2026) → `@done` (correct PIN)
- Any other PIN → `@fail` (wrong PIN, retryable)

### Current Limitation

```php
// Can only do ONE of these — not both conditionally:
'confirming_pin' => '@done',     // Always succeeds
'confirming_pin' => '@fail',     // Always fails
```

QA must use two separate scenarios (`AtReportSavedWithPinScenario` for success, a manual flow for failure) and know which to activate at the right time.

### Additional Limitation: Guard Overrides in Outcome Arrays

Even with static outcomes, guard overrides in outcome arrays don't work:

```php
'confirming_pin' => [
    'outcome' => '@fail',
    IsPinRetryableGuard::class => true,  // ← IGNORED today
],
```

`classifyPlanValues()` puts the entire array into `$outcomes` but never extracts or registers the guard override. When the `@fail` routing runs, `IsPinRetryableGuard` evaluates with real behavior instead of the override.

## Proposed Solution

Two changes:

1. **Callable outcome** — `'outcome'` accepts a `Closure` resolved via `InvokableBehavior` parameter injection. Type-hint what you need (`ContextManager`, `State`, `EventBehavior`), return an outcome string.

2. **Guard override extraction** (bug fix) — Class-keyed entries in outcome arrays (`IsPinRetryableGuard::class => true`) are extracted and registered as behavior overrides. Currently silently ignored.

## Implementation

### 1. `MachineDefinition::handleScenarioOutcome()` — Resolve Callable

```php
// Current (line 1863):
$outcomeString = is_array($outcome) ? ($outcome['outcome'] ?? '@done') : $outcome;

// New:
$rawOutcome    = is_array($outcome) ? ($outcome['outcome'] ?? '@done') : $outcome;
$outcomeString = $rawOutcome instanceof \Closure
    ? $this->resolveCallableOutcome($rawOutcome, $state)
    : $rawOutcome;
```

The Closure is resolved via `InvokableBehavior::injectInvokableBehaviorParameters()` — the same parameter injection mechanism used by actions, guards, and calculators. This means the Closure can type-hint any injectable:

```php
// Minimal — just context:
'outcome' => function (ContextManager $context): string {
    return $context->pin === now()->format('dmy') ? '@done' : '@fail';
},

// Full access — state, event, context:
'outcome' => function (ContextManager $context, State $state, EventBehavior $event): string {
    // ...
},
```

Helper method on MachineDefinition:

```php
private function resolveCallableOutcome(\Closure $closure, State $state): string
{
    $parameters = InvokableBehavior::injectInvokableBehaviorParameters(
        actionBehavior: $closure,
        state: $state,
        eventBehavior: $state->triggeringEvent,
    );

    return $closure(...$parameters);
}
```

### 2. `ScenarioPlayer::classifyPlanValues()` — Extract Behavior Overrides

When an outcome array contains class keys (guards, actions), extract them into `$overrides`:

```php
} elseif (is_array($value) && isset($value['outcome'])) {
    // Delegation outcome with optional behavior overrides
    $outcomes[$stateRoute] = $value;

    // Extract behavior overrides from outcome array (guard/action class keys)
    $behaviorKeys = array_filter(
        $value,
        fn (mixed $v, int|string $k): bool => is_string($k) && $k !== 'outcome' && $k !== 'output' && class_exists($k),
        ARRAY_FILTER_USE_BOTH,
    );

    if ($behaviorKeys !== []) {
        $overrides[$stateRoute] = array_merge($overrides[$stateRoute] ?? [], $behaviorKeys);
    }
}
```

These extracted overrides are then registered via `registerOverrides()` alongside the regular behavior overrides, so guard/action fakes are bound in the container when `@fail`/`@done` routing evaluates them.

### 3. `ScenarioValidator` — Accept Callable Outcome

In `checkContinuation()` and `checkPlanStructure()`, when validating outcome values:

```php
// Current: only accepts string outcomes
if (is_string($outcomeValue) && str_starts_with($outcomeValue, '@')) { ... }

// New: also accept Closure
if ($outcomeValue instanceof \Closure) {
    // Valid — callable outcome, skip string validation
} elseif (is_string($outcomeValue) && str_starts_with($outcomeValue, '@')) { ... }
```

### 4. Return Value Validation

The Closure must declare `: string` return type — PHP enforces this at runtime (`TypeError` on violation). The returned string must be a valid outcome (`@done`, `@done.{state}`, `@fail`, `@timeout`). No additional validation needed — the existing routing logic handles invalid outcomes the same way as invalid static outcomes.

## Closure Signature

Uses the standard `InvokableBehavior` parameter injection. Type-hint what you need:

```php
// Minimal — ContextManager only:
function (ContextManager $context): string

// With State:
function (ContextManager $context, State $state): string

// With Event:
function (ContextManager $context, EventBehavior $event): string
```

Must return a valid outcome string: `'@done'`, `'@done.{state}'`, `'@fail'`, or `'@timeout'`.

**Why InvokableBehavior injection?**
- Consistent with actions, guards, calculators — same parameter injection pattern
- Type-hint what you need, ignore the rest
- Access to `ContextManager`, `State`, `EventBehavior`, `EventCollection` — same as any behavior
- No new mechanism to learn

## Deactivation / Cleanup

No changes. Callable outcomes are resolved at delegation time and produce a standard outcome string. All downstream behavior (routing, event recording, scenario deactivation) works identically to static outcomes.

## What Doesn't Change

- Static string outcomes (`'@done'`, `'@fail'`, `'@timeout'`) — unchanged
- Static outcome arrays (`['outcome' => '@done', 'output' => [...]]`) — unchanged
- Child scenario references (`ChildScenario::class`) — unchanged
- `ScenarioPlayer::getOutcome()` return type — still returns `string|array|null` (Closure is inside the array)
- `handleScenarioOutcome()` downstream routing — unchanged after outcome string resolution
- `InvokableBehavior::injectInvokableBehaviorParameters()` — reused as-is, no changes

## What Gets Added

| Component | Change |
|-----------|--------|
| `MachineDefinition::handleScenarioOutcome()` | Resolve Closure outcome via `InvokableBehavior::injectInvokableBehaviorParameters()` |
| `MachineDefinition::resolveCallableOutcome()` | New private helper — resolves Closure with parameter injection, returns outcome string |
| `ScenarioPlayer::classifyPlanValues()` | Extract class-keyed behavior overrides from outcome arrays into `$overrides` |
| `ScenarioValidator` | Accept `Closure` as valid outcome value |

## Breaking Change?

**No.** All changes are additive. Existing static outcomes work identically. The behavior override extraction from outcome arrays is a **bug fix** — previously these overrides were silently ignored.

## Documentation Patterns (no code changes)

### Selective Pause Pattern

When designing a scenario, intentionally omitting `@continue` from an interactive state creates a **selective pause** — the scenario loop stops, the machine stays in that state, and QA interacts with the real endpoint. After QA acts, the continuation overrides resume from the next state.

**Example:** Findeks consent flow — QA must give real consent, but the rest of the flow (PIN confirmation, report saving) is automated by continuation overrides.

Document in `scenario-plan.md` under the `@continue` section.

### Continuation Branching Pattern

At a selective pause point, QA sees both `activeScenario` (current continuation) and `availableScenarios` (other scenarios activatable from this state). QA can:
- Continue the active scenario (send event without slug)
- Branch to a different scenario (send event with different slug)
- Deactivate and go manual (send `scenario: null`)

This creates **branch points** in QA flows where the same pause state can lead to different test paths.

Document in `scenario-endpoints.md` under the "Scenario Switching" section.
