# Release Notes — 9.14.1 startingAt Parallel Regions

Bug-fix release for testing DX. No consumer changes required — existing tests keep passing; upgrade with `composer update tarfin-labs/event-machine`.

## Fixed

- **`Machine::startingAt('<parallel state>')` now enters the regions.** Previously the machine was placed on the bare parallel parent with all regions inactive, so region events could not route and parallel gating could not be tested without hand-copied `TestMachine::define` region configs — the exact anti-pattern the 9.14.0 guidance warns against. Now every region activates at its initial leaf, matching real parallel-entry semantics: region events route, gating guards see live regions, and `@done` fires when all regions reach their final states.

  ```php
  OrderMachine::startingAt('data_collection')            // regions live at their initials
      ->send(ProductSelectedEvent::forTesting())
      ->send(CustomerInfoSubmittedEvent::forTesting())
      ->send(ApplicationSubmittedEvent::forTesting())     // gating guard sees both regions final
      ->assertState('submitted');
  ```

- **Starting at a leaf inside one region auto-initializes sibling regions.** `startingAt('data_collection.product.product_selected')` anchors the machine on the parallel parent, keeps the target leaf active, and brings every sibling region up at its initial leaf — so cross-region flows are drivable from any starting point.

- **`startingAt()` accepts machine-relative dotted paths.** `'parent.region.leaf'` now resolves without the machine-id prefix (full ids and bare leaf names keep working).

Docs: https://eventmachine.dev/testing/test-machine#parallel-states
