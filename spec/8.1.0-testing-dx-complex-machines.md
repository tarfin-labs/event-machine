# Testing DX Improvements for Complex Machines

**Status:** Draft (reviewed)
**Author:** Claude
**Date:** 2026-03-22
**Scope:** Reduce testing boilerplate for machines with many behaviors and deep state paths

---

## Problem

Testing complex machines (e.g., FindeksMachine with 7 actions, 5 guards, 8 states) requires excessive boilerplate:

1. **Action faking lists** — 7-8 actions listed in every test
2. **Guard setup** — 3-5 static `shouldReturn()` calls before machine creation
3. **Path replay** — 5+ steps to reach a deep state for testing
4. **Fragile maintenance** — adding a new action to the machine breaks all tests

---

## Proposals Evaluated

| Proposal | Decision | Why |
|----------|----------|-----|
| `fakingAllActions(except: [])` | ✅ | Natural extension of `faking()`. Introspects `$definition->behavior['actions']`, auto-fakes all class-based actions. |
| `fakingAllGuards(except: [])` | ✅ | Same pattern for guards. |
| `fakingAllBehaviors(except: [])` | ✅ | Combo wrapper — fakes actions + guards + calculators. |
| `withGuards([Class => value])` | ✅ | Solves the @always timing problem. Add `guards:` parameter to `withContext()`. |
| `startingAt(machine, state, context)` | ✅ | New factory that skips init lifecycle. |
| `replayTo(state, via: [...])` | ❌ | `startingAt()` eliminates the need. `via:` array is a fragile DSL. |

---

## Implementation Details

### 1. `fakingAllActions(except: [])` / `fakingAllGuards(except: [])` / `fakingAllBehaviors(except: [])`

**How it works:** Introspect `$this->machine->definition->behavior['actions']`, filter to class-based entries (FQCN strings, not Closures), call `$class::spy()` for each.

```php
public function fakingAllActions(array $except = []): self
{
    $actions = $this->machine->definition->behavior['actions'] ?? [];

    foreach ($actions as $key => $value) {
        // Skip if in except list — match by FQCN or by behavior key
        if (in_array($value, $except, true) || in_array($key, $except, true)) {
            continue;
        }

        if (is_string($value) && is_subclass_of($value, InvokableBehavior::class)) {
            $value::spy();
            $this->fakedBehaviors[] = $value;
        }
    }

    return $this;
}
```

**`except:` parameter:**
- Accepts both FQCN and behavior key name: `except: [StorePinAction::class]` or `except: ['storePinAction']`
- Optional — calling without `except` fakes everything
- Skipped items run their real logic

**`fakingAllGuards`** — same pattern with `behavior['guards']`. Creates spies via `$class::spy()`. Note: Mockery spy default return is `null` → in guard evaluation `null` is falsy → **guard fails by default**. To make guards pass, use `guards:` parameter with explicit return values, or `except:` to run real logic on specific guards.

**`fakingAllBehaviors`** — calls all three: actions + guards + calculators.

**Inline behaviors:** `fakingAll*` only affects class-based behaviors. Inline closures are skipped (can't be introspected). Use `faking(['inlineKey'])` for those.

**Cleanup:** Uses existing `$this->fakedBehaviors` tracking → `resetFakes()` works unchanged.

### 2. `guards:` Parameter — @always Timing Fix

**The timing problem:** `withContext()` calls `getInitialState()` which fires `@always` transitions. Guards in `@always` must be faked BEFORE this.

**Solution:** Add optional `guards:` parameter to `withContext()` and `create()`:

```php
public static function withContext(string $machineClass, array $context, array $guards = []): self
{
    // Set guard fakes BEFORE machine creation
    $guardClasses = [];
    foreach ($guards as $guardClass => $returnValue) {
        $guardClass::shouldReturn($returnValue);
        $guardClasses[] = $guardClass;
    }

    // ... existing withContext logic ...

    $instance->fakedBehaviors = array_merge($instance->fakedBehaviors, $guardClasses);
    return $instance;
}
```

### 3. `startingAt()` — Skip Init Lifecycle

New static factory that creates the machine at a specific state WITHOUT running any lifecycle:

```php
public static function startingAt(
    string $machineClass,
    string $stateId,
    array $context = [],
    array $guards = [],
): self
```

**Does NOT:** run entry actions, fire @always, dispatch jobs, create history.
**DOES:** use real definition, resolve compound states, support all TestMachine methods after creation.

**`Queue::fake()` still needed:** `startingAt` only skips init. Subsequent transitions that enter job states will call `dispatch()`. Use `Queue::fake()` if the flow passes through any `job:` state.

## What's NOT Included (and Why)

| Proposal | Why Not |
|----------|---------|
| `replayTo(state, via: [...])` | `startingAt()` is better. `via:` array is a fragile DSL within a DSL. |
| `fakingAll*` for inline closures | Can't be introspected. Use `faking(['inlineKey'])` explicitly. |
| `withGuards()` as separate builder | Requires new builder class. `guards:` parameter is simpler. |
| Mandatory `except:` | Over-restrictive. Doc warnings are sufficient. |

## API Summary

```php
// Fake all class-based actions (except specified ones)
->fakingAllActions()
->fakingAllActions(except: [StorePinAction::class])
->fakingAllActions(except: ['storePinAction'])         // by behavior key

// Fake all class-based guards
->fakingAllGuards()
->fakingAllGuards(except: [IsEligibleGuard::class])

// Fake everything (actions + guards + calculators)
->fakingAllBehaviors()
->fakingAllBehaviors(except: [SendEmailAction::class])

// Guards as parameter (solves @always timing)
TestMachine::withContext(FindeksMachine::class, context: [...], guards: [
    HasExistingReportGuard::class => false,
])

// Start at specific state (skip path replay)
TestMachine::startingAt(FindeksMachine::class,
    stateId: 'awaiting_pin',
    context: ['requestId' => 'REQ-1'],
    guards: [IsPinRetryableGuard::class => true],
)
```

## Documentation Plan

### Warning: The "Testing Nothing" Anti-Pattern

These convenience methods are powerful but can lead to tests that verify nothing meaningful. The documentation must explicitly address this risk.

**Core principle to document:** `fakingAll*()` without `except:` means "I'm not testing any behavior logic — only transition wiring." This is valid for some tests but dangerous as a default habit.

### Anti-Pattern Table (for docs)

| Pattern | What It Actually Tests | Risk |
|---------|----------------------|------|
| `fakingAllActions()` + `fakingAllGuards()` + `startingAt()` | Only transition config (`@done → state_x`) | Very high — almost nothing tested |
| `fakingAllActions(except: [X])` + `startingAt()` | Action X runs with real logic + transition routing | Low — clear intent |
| `fakingAllActions()` + real guards | Guard logic + transition routing | Medium — guards validated, actions skipped |
| `startingAt()` + no faking | All behaviors run from that state forward | Good — focused state testing |
| Full path (no `startingAt`, no `fakingAll`) | Everything: path + guards + actions + transitions | Best coverage, most boilerplate |

### When to Use Which (for docs)

| Goal | Recommended Approach |
|------|---------------------|
| Test a single action's effect on state transition | `fakingAllActions(except: [MyAction::class])` |
| Test a guard chain on a specific transition | `fakingAllActions()` + guards with controlled return values via `guards:` parameter |
| Test job @done/@fail routing from deep state | `startingAt()` + `simulateChildDone()` |
| Test the full machine flow end-to-end | `withContext()` + explicit `faking()` — NO shortcuts |
| Smoke test that machine definition is valid | `fakingAllBehaviors()` + traverse all paths |

### Rule of Thumb (for docs)

> **If your test has `fakingAllActions()` without `except:`, ask yourself: "What behavior am I actually testing?"** If the answer is "just the transition config," consider whether that's worth a test — or whether your effort is better spent on a test that exercises real logic.

### Doc Content: "Bulk Faking" Section Draft

```markdown
## Bulk Faking

Fake all class-based behaviors in one call instead of listing each one:

\```php
->fakingAllActions()                                   // all actions → spy
->fakingAllActions(except: [StorePinAction::class])    // all except this one
->fakingAllActions(except: ['storePinAction'])          // by behavior key

->fakingAllGuards()                                    // all guards → spy
->fakingAllBehaviors()                                 // actions + guards + calculators
\```

The `except:` parameter accepts both class FQCNs and behavior key strings. Excluded behaviors run their real logic.

::: warning Testing Nothing Trap
`fakingAllActions()` without `except:` means NO action logic runs.
Combined with `fakingAllGuards()`, your test only verifies transition wiring
(`@done → state_x`) — which is already visible in the machine config.

**Ask yourself: "What behavior am I actually testing?"**

✅ Good — tests StorePinAction logic:
\```php
->fakingAllActions(except: [StorePinAction::class])
\```

⚠️ Questionable — tests nothing but config:
\```php
->fakingAllActions()
->fakingAllGuards()
\```
:::
```

### Doc Content: "Starting at a Specific State" Section Draft

```markdown
## Starting at a Specific State

Skip path replay and start the machine at any state:

\```php
TestMachine::startingAt(FindeksMachine::class,
    stateId: 'awaiting_pin',
    context: ['requestId' => 'REQ-1', 'phone' => '05321234567'],
    guards: [IsPinRetryableGuard::class => true],
)
->fakingAllActions(except: [StorePinAction::class])
->send(PinConfirmedEvent::make(['pin' => '123456']))
->assertState('confirming_pin');
\```

`startingAt()` creates the machine at the given state without running
any lifecycle (no entry actions, no @always, no job dispatch). The machine
uses the real definition — all transitions, guards, and actions are available.

::: tip When to use startingAt() vs withContext()
- **`withContext()`** — tests the full path from initial state. Entry actions and @always run.
  Use for integration tests that validate the complete flow.
- **`startingAt()`** — tests behavior FROM a specific state. No path validation.
  Use for focused unit tests of a single transition or state behavior.
:::
```
