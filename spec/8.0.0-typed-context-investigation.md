# Typed ContextManager vs Array Context — Investigation

## Summary

`setupContextManager()` is the pivot point: it detects whether `config['context']`
is a ContextManager subclass and, if so, **moves it to `behavior['context']` and
resets `config['context']` to an empty array**. Every downstream path (initialization,
restore, persist) checks `behavior['context']` first.

---

## 1. Array Context — How It Works

### Configuration shape

```php
MachineDefinition::define(
    config: [
        'context' => ['count' => 0, 'total' => 0],
        ...
    ]
)
```

### `setupContextManager()` (line 526–533)

```
config['context'] = ['count' => 0, ...]   // raw array
behavior['context'] is not set / empty
→ is_subclass_of(array, ContextManager) === false
→ method body is a no-op: nothing changes
```

### `initializeContextFromState()` (line 499–518)

`behavior['context']` is empty, so the method falls through to:

```php
$contextConfig = $this->config['context'] ?? [];           // ['count' => 0, ...]
return ContextManager::validateAndCreate(['data' => $contextConfig]);
```

The array is stored inside the base `ContextManager::$data` property.
Access via `$context->get('count')` hits the `Arr::get($this->data, $key)` branch.

---

## 2. Typed ContextManager — How It Works

### Configuration shape

```php
MachineDefinition::define(
    config: [
        'context' => TrafficLightsContext::class,   // FQCN string, not an array
        ...
    ]
)
```

`TrafficLightsContext` is a `ContextManager` subclass with typed constructor
properties (`public int|Optional $count`).

### `setupContextManager()` (line 526–533)

```php
if (isset($this->config['context']) &&
    is_subclass_of($this->config['context'], ContextManager::class)) {

    $this->behavior['context'] = $this->config['context'];  // move FQCN to behavior
    $this->config['context']   = [];                        // CLEAR config['context']
}
```

After this call:

| key | before | after |
|-----|--------|-------|
| `config['context']` | `'TrafficLightsContext'` | `[]` |
| `behavior['context']` | (empty) | `'TrafficLightsContext'` |

### `initializeContextFromState()` (lines 506–512)

`behavior['context']` is now populated:

```php
$contextClass = $this->behavior['context'];                // 'TrafficLightsContext'
return $contextClass::validateAndCreate($this->config['context'] ?? []);
//                                        ^^^^^^^^^^^^^^^^^^^^^^^^^
//                                        always [] after setupContextManager()
```

`TrafficLightsContext::from([])` is called via Spatie Data's `validateAndCreate`.
The constructor has `Optional` defaults, so missing keys become `Optional` instances
which the constructor coerces to their defaults (e.g., `$count = 0`).

Access via `$context->get('count')` hits the `$this->$key` branch (typed property).

---

## 3. `getInitialState()` — Entry Point

```php
public function getInitialState(...): ?State
{
    $context = $this->initializeContextFromState();   // no $state passed
    ...
}
```

`$state` is null here, so `initializeContextFromState()` always builds the
context from scratch using whichever path above applies.

---

## 4. Where `withContext()` Must Inject

`withContext()` needs to provide pre-start context values — values that should be
present **before** `getInitialState()` is called, so entry actions on the initial
state already see them.

### Path A — Array context

The default context lives in `config['context']`. Injection is straightforward:

```php
$this->config['context'] = array_merge(
    $this->config['context'] ?? [],
    $overrides
);
```

This works because `initializeContextFromState()` reads `config['context']` directly
when `behavior['context']` is empty.

### Path B — Typed ContextManager

`config['context']` is `[]` after `setupContextManager()` runs (constructor).
`validateAndCreate([])` initialises all typed properties to their `Optional`/default
values. To inject overrides, they must arrive inside the array passed to
`validateAndCreate`.

The injection point is therefore:

```php
// behavior['context'] holds the FQCN; config['context'] holds the seed array
$this->config['context'] = array_merge(
    $this->config['context'] ?? [],   // always [] at this point
    $overrides
);
```

Calling `$contextClass::validateAndCreate(['count' => 5])` will then set
`$count = 5` through Spatie Data's constructor injection.

**Both paths use the exact same one-liner.** The difference is only how the data
is used downstream.

---

## 5. Edge Cases & Recommendations

### 5.1 `setupContextManager()` runs in the constructor

By the time any public API can call `withContext()`, `setupContextManager()` has
already run. So `config['context']` is guaranteed to be an array (possibly `[]`),
never a class string. Safe to `array_merge` unconditionally.

### 5.2 Typed ContextManager constructor validation

`validateAndCreate()` runs Spatie Data validation rules (e.g. `#[Min(0)]`).
If `withContext()` injects a value that violates a validation rule, the call will
throw `MachineContextValidationException` — this is correct behavior, not a bug.

### 5.3 Unexpected keys in typed context

For typed subclasses, keys that don't correspond to constructor parameters are
silently ignored by Spatie Data's `from()`. No runtime error, but the value is
lost. Document that `withContext(['unknownKey' => 1])` is a silent no-op for
typed contexts.

### 5.4 The `Optional` pattern for defaults

`TrafficLightsContext` uses `int|Optional $count` with a default assignment in
the constructor body. Passing `['count' => 5]` to `validateAndCreate` causes
Spatie Data to supply `5` to the constructor, bypassing the `Optional` guard.
This is the correct behavior.

### 5.5 Do NOT call `setupContextManager()` again

Any `withContext()`-like method must not re-invoke `setupContextManager()`, as
doing so a second time would find `config['context'] = []` and correctly do
nothing (it's idempotent), but is unnecessary overhead.

### 5.6 Timing: before `getInitialState()`, not after

The existing `TestMachine::create()` applies context **after** `Machine::create()`
(which calls `getInitialState()` internally), meaning entry actions on the initial
state run with default context. A proper `withContext()` on `MachineDefinition`
must mutate `config['context']` **before** `getInitialState()` is called, i.e.,
before `Machine::start()` / `Machine::create()`.

---

## 6. Recommended `withContext()` Implementation Sketch

```php
// On MachineDefinition — called before getInitialState()
public function withContext(array $overrides): static
{
    // config['context'] is always an array at this point
    // (setupContextManager() in constructor already normalized it)
    $this->config['context'] = array_merge(
        $this->config['context'] ?? [],
        $overrides
    );

    return $this;
}
```

Caller pattern:

```php
$definition = MyMachine::definition()->withContext(['count' => 5]);
$machine    = Machine::withDefinition($definition);
$machine->start();
// entry actions now see count=5
```

This single implementation handles both array context and typed ContextManager
subclasses because in both cases `initializeContextFromState()` ultimately calls
`validateAndCreate($this->config['context'])`.
