# Named Parameters for Behaviors

**Status:** Implemented
**Date:** 2026-03-28
**Target:** v9.0.0

---

## Problem

Behaviors (guards, actions, calculators) that need configuration parameters use a string-based `:arg1,arg2` syntax:

```php
'guards' => 'isAmountInRangeGuard:100,10000'
```

The guard receives these as an untyped positional array:

```php
class IsAmountInRangeGuard extends GuardBehavior
{
    public function __invoke(ContextManager $ctx, array $args): bool
    {
        return $ctx->get('amount') >= (int) $args[0]   // string → manual cast
            && $ctx->get('amount') <= (int) $args[1];   // positional, unnamed
    }
}
```

**Problems:**
- All arguments are strings — manual casting required
- Positional only — `$args[0]` says nothing about what it is
- No IDE navigation to see what params a behavior accepts
- Comma in values is impossible (JSON, arrays)
- No default values
- No validation — missing or extra args silently ignored

---

## Solution

Array-tuple syntax with named parameters, resolved via the existing `__invoke` parameter injection system:

```php
// Config — parameterized behavior is always an inner array (tuple)
'guards' => [[IsAmountInRangeGuard::class, 'min' => 100, 'max' => 10000]],
```

```php
// Guard
class IsAmountInRangeGuard extends GuardBehavior
{
    public function __invoke(ContextManager $ctx, int $min, int $max): bool
    {
        return $ctx->get('amount') >= $min
            && $ctx->get('amount') <= $max;
    }
}
```

Framework resolves `__invoke` parameters by type first, then by name:
1. `ContextManager $ctx` → known type → inject from framework
2. `int $min` → unknown type → match `'min'` from config params → inject `100`
3. `int $max` → unknown type → match `'max'` from config params → inject `10000`

Definition remains **pure data** (strings, arrays, primitives) — serializable, cacheable, XState-exportable.

## Syntax

### Parameterless behavior (unchanged)

```php
'guards' => HasBalanceGuard::class,
'guards' => 'hasBalanceGuard',
```

### Inline closure (unchanged)

```php
'guards' => fn (ContextManager $ctx): bool => $ctx->get('amount') > 100,
```

### Parameterized behavior — always an inner array

A parameterized behavior is a **tuple**: `[ClassOrKey, 'param' => value, ...]`. This tuple is always an element inside the behavior list — even when it's the only behavior:

```php
// Class reference with named params
'guards' => [
    [IsAmountInRangeGuard::class, 'min' => 100, 'max' => 10000],
],

// Inline key with named params
'guards' => [
    ['isAmountInRangeGuard', 'min' => 100, 'max' => 10000],
],
```

### Multiple behaviors

```php
// All parameterless
'guards' => [HasBalanceGuard::class, IsActiveGuard::class],

// Mixed
'guards' => [
    [IsAmountInRangeGuard::class, 'min' => 100, 'max' => 10000],
    HasBalanceGuard::class,
],

// Multiple parameterized
'guards' => [
    [IsAmountInRangeGuard::class, 'min' => 100, 'max' => 10000],
    [HasMinimumBalanceGuard::class, 'threshold' => 50],
],
```

### Old syntax (deprecated)

```php
'guards' => 'isAmountInRangeGuard:100,10000'
```

Still works. Behaviors using `array $args` type-hint continue receiving positional string array.

## Parsing Rule

**An outer array is always a list of behaviors. A parameterized behavior is always an inner array (tuple).**

No disambiguation needed — the framework never inspects string keys at the top level:

| Shape | Interpretation |
|-------|----------------|
| `Class::class` | Single parameterless behavior |
| `'inlineKey'` | Single parameterless inline behavior |
| `fn () => ...` | Single inline closure |
| `[A::class, B::class]` | List of parameterless behaviors |
| `[[A::class, 'x' => 1]]` | List of one parameterized behavior |
| `[[A::class, 'x' => 1], B::class]` | List — first parameterized, second plain |

Each element in the list is resolved as:
- `string` → class reference or inline key (parameterless)
- `Closure` → inline closure
- `array` → parameterized tuple: `[0]` = class/key, string keys = params

Invalid formats (throw `InvalidBehaviorDefinitionException`):
- `[['min' => 100]]` — tuple with no class reference at `[0]`.
- `[[]]` — empty tuple.
- `[[fn () => true, 'x' => 1]]` — closure in tuple. Closures cannot receive named params; use a class instead.

## Applies To All Behavior Keys

```php
// Transition guards
'guards' => [[IsAmountInRangeGuard::class, 'min' => 100, 'max' => 10000]],

// Transition actions
'actions' => [[SendNotificationAction::class, 'channel' => 'sms']],

// Transition calculators
'calculators' => [[ApplyDiscountCalculator::class, 'rate' => 0.15]],

// Entry / exit actions
'entry' => [[LogTransitionAction::class, 'level' => 'info']],

// Outputs — behavior.outputs map (final states): inner array for parameterized
'outputs' => [
    'completed' => [[FormatOutput::class, 'format' => 'json']],
],

// Outputs — state-level config (any state): inner array for parameterized
'output' => [[FormatOutput::class, 'format' => 'json']],

// Full transition example
'SUBMIT' => [
    'target'  => 'processing',
    'guards'  => [[IsAmountInRangeGuard::class, 'min' => 100, 'max' => 10000]],
    'actions' => SendEmailAction::class,
],
```

**Output uses the same inner-array rule as guards/actions.** Parameterized output = inner array (tuple). Context key filter = plain array of strings. No special disambiguation needed:

| `output` value | Interpretation |
|----------------|----------------|
| `FormatOutput::class` | Class behavior (parameterless) |
| `fn ($ctx) => [...]` | Inline closure |
| `['orderId', 'total']` | Context key filter (array of strings) |
| `[[FormatOutput::class, 'format' => 'json']]` | Parameterized tuple (inner array) |

Detection: `is_array($value) && isset($value[0]) && is_array($value[0])` → inner array → parameterized tuple. Otherwise → context key filter.

## Listener Config Format (Breaking Change)

The current listener format uses class names as **array keys** — this makes parameterized tuples impossible (PHP arrays only allow `int|string` keys). This release replaces the listener format entirely. **No backward compatibility.**

### Old format (removed)

```php
'listen' => [
    'entry' => [
        SyncMarkerAction::class,                          // sync (numeric key)
        QueuedMarkerAction::class => ['queue' => true],   // queued (class = key)
    ],
    'exit' => [
        QueuedExitAction::class => ['queue' => true],
    ],
    'transition' => [
        QueuedTransitionAction::class => ['queue' => true],
    ],
]
```

### New format

Listener list follows the same **list-of-items** pattern as other behavior keys. Queue option moves into the tuple via `@queue` (reserved prefix, consistent with `@always`, `@done`, `@fail`):

```php
'listen' => [
    'entry' => [
        SyncMarkerAction::class,                                              // sync, no params
        [QueuedMarkerAction::class, '@queue' => true],                        // default queue
        [AuditAction::class, 'verbose' => true, '@queue' => 'audit-queue'],  // specific queue + named params
    ],
    'exit' => [
        [QueuedExitAction::class, '@queue' => true],
    ],
    'transition' => [
        [QueuedTransitionAction::class, '@queue' => 'transition-queue'],
    ],
]
```

**`@queue` type:** `bool|string`. `true` = default queue, `'queue-name'` = specific queue, `false`/omitted = sync. Consistent with `'queue'` key in machine delegation config.

**Shorthand (unchanged):**

```php
// Single sync listener — string, not array
'listen' => [
    'entry' => 'childCountAction',
]
```

### Parsing rule

Each element in a listener list is resolved as:
- `string` → class reference or inline key, sync (`@queue` defaults to `false`)
- `Closure` → inline closure, sync
- `array` → tuple: `[0]` = class/key, string keys = behavior params, `@queue` = queue option (`bool|string`)

`@queue` is extracted before behavior param injection — it never reaches `__invoke`.

### Implementation — `parseListenConfig` rewrite

```php
foreach ($raw as $element) {
    if (is_string($element)) {
        // Shorthand: plain class or inline key — sync
        $this->listen[$key][] = ['action' => $element, 'queue' => false, 'configParams' => []];
    } elseif (is_array($element)) {
        // Tuple: [Class::class, 'param' => val, '@queue' => true|'queue-name']
        $class        = $element[0];
        $queue        = $element['@queue'] ?? false;
        $configParams = array_filter(
            $element,
            fn ($k) => is_string($k) && !str_starts_with($k, '@'),
            ARRAY_FILTER_USE_KEY,
        );
        $this->listen[$key][] = ['action' => $class, 'queue' => $queue, 'configParams' => $configParams];
    }
}
```

### Affected code

- `MachineDefinition::parseListenConfig()` — rewrite parser
- `MachineDefinition::runEntryListeners()` — pass `configParams` to `runAction()`
- `MachineDefinition::runExitListeners()` — same
- `MachineDefinition::runTransitionListeners()` — same
- `MachineDefinition::dispatchListenerJob()` — serialize `configParams`, pass to `ListenerJob`
- `ListenerJob` — accept and forward `configParams` to `runAction()`
- `StateConfigValidator::validateListenConfig()` — validate new format

## Error Handling

**Missing required param:** If `__invoke` declares `int $min` but config has no `'min'` key and there is no default value, the framework throws `MissingBehaviorParameterException`:

```
IsAmountInRangeGuard requires parameter 'min' (int) but it was not provided in the definition.
```

**Extra config params:** Params in config that don't match any `__invoke` parameter are silently ignored.

**Type mismatch:** PHP's native type coercion applies. `'min' => '100'` injected into `int $min` works (PHP coerces). `'min' => 'abc'` into `int $min` throws `TypeError`.

**Definition-time validation:** `machine:validate-config` should catch missing params, invalid formats, and type mismatches before runtime. Invocation-time errors are the fallback for configs that bypass static validation.

Both syntaxes coexist. Colon syntax will be removed in v10.0.0 (see Implementation §6).

## Implementation

### 1. Parsing — extract class and config params from tuple

Each element in the behavior list is resolved. When an element is an array, it's a parameterized tuple:

```php
// Element: [IsAmountInRangeGuard::class, 'min' => 100, 'max' => 10000]
// Detection: is_array($element)

$class        = $element[0];  // class reference or inline key
$configParams = array_filter($element, fn ($k) => is_string($k) && !str_starts_with($k, '@'), ARRAY_FILTER_USE_KEY);
```

### 2. Injection — extend `injectInvokableBehaviorParameters()`

Add `?array $configParams = null` parameter. After known-type matching, match remaining params by name:

```php
$value = match (true) {
    // Existing: framework-provided (matched by type-hint, not by name)
    is_a($typeName, ForwardContext::class, true)  => $forwardContext,
    is_a($typeName, ContextManager::class, true)  => $state->context,
    is_a($typeName, EventBehavior::class, true)   => $eventBehavior,
    $state instanceof $typeName                    => $state,
    is_a($state->history, $typeName)               => $state->history,

    // Existing: positional args (deprecated `:arg` path)
    $typeName === 'array' && $actionArguments !== null => $actionArguments,

    // New: named config params — match by parameter name
    $configParams !== null
        && array_key_exists($parameter->getName(), $configParams)
        => $configParams[$parameter->getName()],

    // Default value from __invoke signature
    $parameter->isDefaultValueAvailable() => $parameter->getDefaultValue(),

    default => null,
};
```

**Key rules:**
- Colon syntax and named params are **mutually exclusive per definition** — a definition is either a string (colon path) or a tuple (named params path). They cannot coexist in the same definition.
- Closure tuples are **not supported** — if `[0]` is a Closure, throw `InvalidBehaviorDefinitionException`. Closures are for one-off inline logic; named params are for reusable class-based behaviors.
- `@`-prefixed keys are **framework-reserved** — stripped before injection. PHP parameter names cannot start with `@`, so collision is impossible. Currently only `@queue` is used (listeners).

### 3. Affected resolution points

- `TransitionDefinition::resolveGuards()` — guard resolution
- `TransitionDefinition::resolveCalculators()` — calculator resolution
- `TransitionBranch::runActions()` — transition action resolution
- `MachineDefinition::runAction()` — entry/exit/listener action resolution
- `Machine::resolveOutputBehavior()` — output behavior resolution (colon parsing + injection). Must also handle inner-array tuples: `initializeResults()` stores raw config in the registry, so `resolveOutputBehavior()` can receive `[[FormatOutput::class, 'format' => 'json']]` from registry path. Detect inner array → extract class + configParams before resolving.
- `Machine::resolveOutputDefinition()` — state-level output resolution: if `is_array` and `[0]` is array → inner array → extract tuple, delegate to `resolveOutputBehavior`. Otherwise → context key filter.
- `MachineController::handleEndpoint()` — endpoint output resolution: currently `is_string` → behavior, `is_array` → filter. Must add inner-array check for tuple support.
- `MachineController::resolveAndRunOutput()` — endpoint output behavior invocation (colon parsing + injection)

### 4. Testing/Faking

`fakingAllOfType()` iterates the **behavior registry** (`definition->behavior[$type]`), not transition configs. For guards/actions/calculators, registry values are class names or closures — these don't change with named params (params live in config, not registry).

**Exception: outputs.** The output registry (`behavior['outputs']`) maps state IDs to output definitions. These CAN be inner-array tuples: `'completed' => [[FormatOutput::class, 'format' => 'json']]`. `fakingAllOfType('outputs', ...)` must detect inner arrays and extract `[0][0]` as the class to spy.

For explicit faking (`faking([Class::class])`), no change needed — the class is already known.

### 5. XState Export

Named params map to XState v5's parameterized action format:

```json
{
  "type": "IsAmountInRangeGuard",
  "params": { "min": 100, "max": 10000 }
}
```

### 6. Deprecation tracking for `:arg` syntax

Two-part deprecation:

1. **Code:** Add `@deprecated` annotation and `trigger_deprecation()` to the colon-parsing paths (6 locations listed in §3). Runtime warning surfaces usage in logs/tests.
2. **Spec:** Create `spec/v10-removals.md` — single file tracking all planned v10 removals. Add the `:arg1,arg2` colon syntax as the first entry with affected code locations.

Removal target: v10.0.0.

### B. Injection — parameter resolution in `__invoke`

11. **Named params injected by name:**
    `__invoke(ContextManager $ctx, int $min, int $max)` with `'min' => 100, 'max' => 10000`
    → `$min = 100`, `$max = 10000`.

12. **Framework types + named params coexist:**
    `__invoke(ContextManager $ctx, EventBehavior $event, int $min)` with `'min' => 5`
    → `$ctx` from framework, `$event` from framework, `$min = 5` from config.

13. **Default values used when param not in config:**
    `__invoke(ContextManager $ctx, int $min, int $max = 999)` with `'min' => 100`
    → `$min = 100`, `$max = 999`.

14. **All default values — tuple with class only, no params:**
    `__invoke(ContextManager $ctx, int $min = 0, int $max = 100)` with `[]` config params
    → `$min = 0`, `$max = 100`.

15. **Missing required param → `MissingBehaviorParameterException`:**
    `__invoke(ContextManager $ctx, int $min, int $max)` with only `'min' => 100`
    → exception naming `'max'`.

16. **Extra config params silently ignored:**
    `__invoke(ContextManager $ctx, int $min)` with `'min' => 100, 'max' => 200, 'extra' => 'ignored'`
    → `$min = 100`, no error.

17. **Type coercion (PHP native):**
    `'min' => '100'` into `int $min` → coerced to `100`.
    `'min' => 'abc'` into `int $min` → `TypeError`.

18. **Complex value types:**
    - `'channels' => ['sms', 'email']` into `array $channels` — array param
    - `'status' => OrderStatus::Pending` into `OrderStatus $status` — enum param

19. **Parameter order independence:**
    `__invoke(int $max, ContextManager $ctx, int $min)` with `'min' => 1, 'max' => 10`
    → param order in `__invoke` doesn't matter — matched by name.

75. **`ListenerJob` serializes and deserializes `configParams`:**

    New tuple format passes. Old `Class => ['queue' => true]` format fails validation.

78. **Queued listener with params under Horizon (LocalQA):**
    Full round-trip: machine persisted → `ListenerJob` dispatched → Horizon picks up → action runs with named params → context updated. Verify via `waitFor`.

### J. Edge cases

79. **Config param with `null` value:**
    `'guards' => [[FlexibleGuard::class, 'limit' => null]]`
    → `$limit` receives `null`. `array_key_exists` matches (key exists), unlike `isset`.

80. **Config param with `false` value:**
    `'actions' => [[ToggleAction::class, 'enabled' => false]]`
    → `$enabled` receives `false`, not skipped.

81. **Same inline key, different params on different transitions:**
    ```php
    'states' => [
        'a' => ['on' => ['X' => ['guards' => [['rangeGuard', 'min' => 0, 'max' => 100]]]]],
        'b' => ['on' => ['Y' => ['guards' => [['rangeGuard', 'min' => 500, 'max' => 9999]]]]],
    ]
    ```
    → same inline key reused with different params. Each transition's configParams is independent.

82. **Parameterless tuple — class only, no string keys:**
    `'guards' => [[IsActiveGuard::class]]`
    → valid tuple, empty configParams. Equivalent to `IsActiveGuard::class`. All `__invoke` params use defaults or framework injection.

83. **Output inner-array with parameterless class:**
    `'output' => [[FormatOutput::class]]`
    → inner array detected, no configParams. Equivalent to `'output' => FormatOutput::class`.

84. **Multiple `@`-prefixed keys in listener tuple:**
    `[AuditAction::class, 'verbose' => true, '@queue' => true, '@connection' => 'redis']`
    → all `@` keys stripped. Only `'verbose'` reaches `__invoke`. `@connection` stored for future use (currently ignored, no error).

85. **Config param value is an enum:**
    `'actions' => [[SetStatusAction::class, 'status' => OrderStatus::Pending]]`
    → `OrderStatus $status` receives enum value directly. PHP type system handles it.

86. **Config param value is a nested associative array:**
    `'actions' => [[ConfigAction::class, 'settings' => ['retries' => 3, 'timeout' => 30]]]`
    → `array $settings` receives the full nested array.

87. **Deep delegation chain — named params at every level:**
    Grandparent uses named params on `@done` action. Parent uses named params on entry. Child uses named params on guard.
    → each level resolves its own configParams independently. No cross-level leakage.

88. **Named params on `@always` chain — multiple hops:**
    State A → `@always` with parameterized guard → State B → `@always` with different parameterized guard → State C.
    → each `@always` hop resolves its own params. `triggeringEvent` preserved across chain (not overwritten).

89. **Behavior with only framework types + named params (no ContextManager):**
    `__invoke(EventBehavior $event, int $threshold): bool`
    → framework injects `$event`, named param injects `$threshold`. Works without ContextManager.

### New: Named Parameters for Behaviors

Behaviors now accept named parameters via array-tuple syntax. The old `:arg1,arg2` colon syntax is deprecated (removed in v10).

**Before (still works, deprecated):**

```php ignore
'guards' => 'isAmountInRangeGuard:100,10000',

// Behavior receives untyped positional array
public function __invoke(ContextManager $ctx, ?array $arguments = null): bool {
    return $ctx->get('amount') >= (int) $arguments[0]
        && $ctx->get('amount') <= (int) $arguments[1];
}
```

**After:**

```php ignore
'guards' => [[IsAmountInRangeGuard::class, 'min' => 100, 'max' => 10000]],

// Behavior receives typed named parameters
public function __invoke(ContextManager $ctx, int $min, int $max): bool {
    return $ctx->get('amount') >= $min
        && $ctx->get('amount') <= $max;
}
```

Works with all behavior keys — guards, actions, calculators, entry/exit, outputs, listeners.

**Output with named params** (inner-array rule, same as guards/actions):

```php ignore
// Parameterized output — inner array
'output' => [[FormatOutput::class, 'format' => 'json']],

// Context key filter — plain array (unchanged)
'output' => ['orderId', 'totalAmount'],
```

**Migration pitfall:** When migrating, update BOTH config AND behavior signature. If only config is changed, old `?array $arguments` gets `null` — silent failure.

### New: Listener Config Format (breaking)

The listener config format has changed. Class-as-key syntax is replaced with tuple syntax. `@`-prefixed keys are framework-reserved (never reach `__invoke`).

**Before (no longer works):**

```php ignore
'listen' => [
    'entry' => [
        SyncAction::class,
        QueuedAction::class => ['queue' => true],
    ],
]
```

**After:**

```php ignore
'listen' => [
    'entry' => [
        SyncAction::class,
        [QueuedAction::class, '@queue' => true],
    ],
]
```

**With named params:**

```php ignore
'listen' => [
    'entry' => [
        [AuditAction::class, 'verbose' => true, '@queue' => true],
    ],
]
```

**Migration steps:**
1. Find all `'listen'` config blocks in your machine definitions.
2. Replace `ClassName::class => ['queue' => true]` with `[ClassName::class, '@queue' => true]`.
3. Sync listeners (numeric key, no options) remain unchanged: `ClassName::class`.
```

## Benefits

| Aspect | Old (`:args`) | New (named array) |
|--------|--------------|-------------------|
| Type safety | All strings | Native PHP types |
| Named parameters | No (`$args[0]`) | Yes (`'min' => 100`) |
| Default values | Manual in behavior | PHP parameter defaults |
| Complex values | Impossible | Arrays, enums, any PHP value |
| Definition is data | Yes | Yes (preserved) |
| Serializable | Yes | Yes |
| XState export | Positional only | Named params in JSON |
| Error messages | Silent null | `MissingBehaviorParameterException` |
