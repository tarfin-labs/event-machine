# 9.12.0 — `reads` (read-only HTTP projections) + persist write-path hardening

## Motivation

EventMachine machines expose their current state over HTTP through the `endpoints:`
config. Every endpoint maps an event type to a route, and every request flows through
`MachineController` → `Machine::send()` → `transition()` → `persist()`. That is the
**write path**: each call appends rows to `machine_events`.

Many clients do not want to change the machine — frontends polling for current status,
dashboards, and "resume where I left off" screens only want to **read** its current state
(reconnect fallbacks for missed websocket broadcasts, form-prefill on resume).
Today the only way to answer those polls over HTTP is to model them as **targetless
events** (e.g. `StatusRequestedEvent => ['uri' => '/status', 'method' => 'GET']`). Even
though such an event causes no state change, it still runs the full `send()` → `persist()`
pipeline and writes ~4 rows per call (the external event + `transition.<…>.start/finish`
+ `state.<…>.enter`).

This is wrong on two axes that every state-machine and event-sourcing formalism agrees on:

1. **State-machine theory (XState/SCXML):** reading the current configuration is not an
   event. An event triggers a microstep/macrostep; inspecting `state.value`,
   `state.nextEvents` (our `availableEvents()`), or `state.can(EVENT)` is a pure read.
2. **CQRS / event sourcing:** queries must not produce events. The event store is the
   append-only log of facts that happened. A poll is not a fact about the domain — nothing
   happened to the entity, the UI merely looked.

The production consequence is concrete: a long-open screen polling `/status` every ~10s
and `/resume` every ~2 min bloated one machine's history to ~5458 rows. `persist()`
re-upserts the entire history on every event, and MySQL caps a prepared statement at
65,535 placeholders; at 5458 rows × 12 columns the next event threw `QueryException` and
wedged the machine (state desynced from the `status` column because the action committed
but the events could not persist).

This spec introduces **`reads`**: a first-class, declarative, zero-write read concept that
is the blessed alternative to read-as-event. `endpoints:` are commands (write); `reads`
are queries (read). A read never constructs an event, never calls `send()`/`transition()`/
`persist()`, never acquires a lock, and never writes a row — by construction, not by
configuration.

This spec covers the `reads` concept plus persist write-path hardening (incremental persist,
atomicity, prune-safe sequencing), together with detailed user documentation and the agent
skill update required for the new concept.

## Concept

A **read** is a named, read-only HTTP projection of a single machine instance's current
state. It is identified by its key (which becomes the URI suffix), not by an event type.

A read request:

1. Restores the machine from its `root_event_id` (`Machine::create(state: $machineId)` —
   pure restore, no writes).
2. Reads `state->value`, `availableEvents()`, and computes `output()` (optionally shaped
   by the read's `output`).
3. Returns the standard response envelope via the existing `buildResponse()` path.

A read MUST NOT: construct an `EventBehavior`, call `send()`/`transition()`/`persist()`,
acquire a machine lock, run any lifecycle behavior (entry/exit/transition actions, guards,
calculators), or write to any table. These guarantees are structural — there is no config
flag to disable them.

`reads` are defined entirely in `MachineRouter::register()` options (not in the machine's
`define()`), and always bind by machine id: `GET {prefix}/{machineId}/{read}`.

## API Surface

`reads` is a new key in the `MachineRouter::register()` `$options` array. It accepts:

| Form | Meaning |
|------|---------|
| (absent) | No read routes registered (default — no silent exposure). |
| `'reads' => true` | Register a single default read named `status` with the standard envelope (state + availableEvents + output). URI: `GET {prefix}/{machineId}/status`. Exactly equivalent to `['status' => null]`; the boolean and array forms are mutually exclusive inputs (not merged). |
| `'reads' => ['status' => null, 'resume' => [...]]` | Register one read route per entry. Key = URI suffix; value = `null` (defaults), a `string` URI-shorthand, or an options array (see Read Definition Options). |

Example:

```php
MachineRouter::register(CarSalesMachine::class, [
    'prefix' => 'machines/car-sales-application',
    'reads'  => [
        'status' => null,                                       // GET …/{machineId}/status
        'resume' => ['output' => ResumeEndpointOutput::class],  // GET …/{machineId}/resume
    ],
]);
```

`reads` are independent of the `only` / `except` event filters (those operate on event
types; reads have none). `reads` participate in the registration's group `prefix` and group
`middleware`, plus any per-read `middleware`.

**Reads always bind by machine id**, even when the registration's endpoints are model-bound
(`model` + `attribute` + `modelFor`). A read targets a machine instance by its `root_event_id`,
not a domain model, so its URI is `{prefix}/{machineId}/{uri}` regardless of how commands bind.
Model-bound reads are not supported — the machineId binding is the stated rule.

## Read Definition Options

A read entry's value is `null` (all defaults), a `string` (URI-shorthand, equivalent to
`['uri' => <string>]`), or an options array. The options array surface deliberately mirrors
`EndpointDefinition` minus the keys that imply a side effect or an event.

| Option | Type | Default | Meaning |
|--------|------|---------|---------|
| `uri` | `string` | the read key | Override the URI suffix. Normalized as below. |
| `output` | `null\|string\|array` | `null` | Shape the response `output` field (the 3rd `output` semantic — shape HTTP response). See Output Integration for the string-vs-array routing. |
| `middleware` | `array<int,string>` | `[]` | Per-read middleware, applied in addition to the group middleware. |
| `status` | `int` | `200` | HTTP status code for the response. Passed through unvalidated (no range check). |
| `available_events` | `bool\|null` | `null` | Include `availableEvents` in the envelope. `null` (default) and `true` both include it; `false` omits the key (requires the `buildResponse()` fix in Response Envelope). |

URI normalization: the effective URI is `'/'.trim($uri ?? $key, '/')` — strip surrounding
slashes, then prepend exactly one `/`. A value that normalizes to empty or just `/` (e.g. `uri`
of `''` or `'/'`), or that contains a route placeholder (`{...}`), is rejected with
`InvalidRouterConfigException`. Multi-segment URIs are allowed (`status/full` → `/status/full`).
The duplicate-URI check (Router Config Validation) runs on the normalized URI.

Excluded / rejected keys (throw `InvalidRouterConfigException` at registration):

| Key | Reason |
|-----|--------|
| `action` | An action is a side effect; reads are pure. |
| `method` | Reads are `GET` only. A non-GET read is a smell. |
| any unrecognized key | Strict validation — a typo (`midleware`, `outputs`) fails fast at registration, not silently at request time. |

## ReadDefinition Value Object

Add `src/Routing/ReadDefinition.php`, a value object mirroring `EndpointDefinition` for the
read surface.

1. Constructor with readonly props: `string $name`, `string $uri`, `null|string|array $output`,
   `array $middleware`, `int $statusCode`, `?bool $availableEvents`.
2. `public static function fromConfig(string $key, null|string|array $config = null): self` —
   normalizes the three value forms (`null`, `string` URI-shorthand, options array) into a
   `ReadDefinition`. Defaults applied: `uri` = normalized per Read Definition Options
   (`'/'.trim($config['uri'] ?? $key, '/')`), `output` = `null`, `middleware` = `[]`,
   `statusCode` = `200`, `availableEvents` = `null`.
3. `fromConfig` throws `InvalidRouterConfigException` when the options array contains `action`,
   `method`, any unrecognized key, or an invalid URI (empty / contains `{...}`).

## Route Registration

Extend `MachineRouter::register()` to materialize read routes inside the existing
`Route::prefix($prefix)->middleware($middleware)->group(...)` block.

1. Read `$options['reads'] ?? null`; normalize `true` to `['status' => null]`; treat absent as
   no reads.
2. `register()` has three early returns: the `package:discover` guard
   (`app()->runningConsoleCommand('package:discover')`), the top content guard
   (`$endpoints === [] && $forwardedEndpoints === []`), and the post-filter guard
   (`$endpoints === [] && $forwardedEndpoints === [] && !$create`). The `package:discover` guard
   still short-circuits (reads are not registered during discovery, like everything else). The
   two content guards MUST be relaxed when `reads` is non-empty so a reads-only registration
   proceeds, and `$prefix`, `$middleware`, and `$namePrefix` (`$options['name'] ?? $definition->id`)
   must be resolved before the route group is built.
3. For each entry, build a `ReadDefinition` via `ReadDefinition::fromConfig()`.
4. Register `Route::match(['GET'], "/{machineId}{$read->uri}", [MachineController::class, 'handleSnapshot'])`
   with the read's `middleware`.
5. Name the route `"{$namePrefix}.read.{$readName}"` (distinct from event route names, which
   are `"{$namePrefix}.<eventtype>"`).
6. Set route defaults: `_machine_class`, `_read_output`, `_read_status_code`,
   `_read_available_events`.
7. `model` / `attribute` / `modelFor` / `machineIdFor` and their orphan validations concern
   **event endpoints only** — reads never appear in those lists (reads are always machineId-bound
   and have no event type). A reads-only registration carries no model binding for its reads; if
   such a registration also passes `modelFor`/`machineIdFor`, the existing orphan checks apply to
   those event-type lists as today (independent of reads).

## Controller — handleSnapshot

Add `MachineController::handleSnapshot(Request $request): JsonResponse`, the zero-write read
path.

1. Resolve `$machineId` via `$request->route('machineId')` (matching the `{machineId}`
   placeholder in the read route URI) and `_machine_class` from route defaults.
2. Restore the machine read-only: `$machine = $machineClass::create(state: $machineId)`. No
   event is constructed; `send()` is never called.
3. Split `_read_output` onto `buildResponse()`'s two output parameters using the **same**
   discriminator as `handleEndpoint()` (`MachineController` L109–113): a `string` or an
   inner-array behavior tuple (`[[Class::class, ...]]`) → `outputKey`; a plain list of context
   keys (`array` whose first element is not an array) → `outputKeys`; `null` → neither (falls
   back to `$machine->output()`).
4. Call `buildResponse()` with the split `outputKey`/`outputKeys`, `_read_status_code` as
   `statusCode`, `_read_available_events` (the `ReadDefinition` default is `null`, which
   includes `availableEvents`) as `includeAvailableEvents`, and `isProcessing: false`.
5. Map restore failures to `404` by catching `RestoringStateException`. This covers both the
   "not found" case and the wrong-machine-class case: the restore path is hardened so an
   `idMap` miss (a `machineId` whose `machine_value` is not a state of this machine's
   definition) throws `RestoringStateException` rather than a raw undefined-key error (see
   Error Handling).
6. The handler MUST NOT acquire a lock, call `send()`, or call `persist()`.

## Response Envelope

A read returns the same base envelope as event endpoints (via `buildResponse()`):

```json
{ "data": { "id", "machineId", "state", "availableEvents", "output", "isProcessing" } }
```

1. `isProcessing` is always `false` for reads (no async processing is triggered).
2. **`buildResponse()` fix (required):** today `buildResponse()` accepts `$includeAvailableEvents`
   but ignores it — it sets `'availableEvents' => $state->availableEvents()` unconditionally.
   Change `buildResponse()` to honor the flag: `null` (default) and `true` include
   `availableEvents`; `false` **omits the `availableEvents` key entirely** (key absent — not
   present-but-`[]`). This also fixes endpoints' `available_events`, currently a silent no-op.
   Reads pass `_read_available_events` (the `ReadDefinition` default `null` → include).
3. `output` is resolved via `buildResponse()` exactly as for endpoints, using the split from
   Controller step 3 (see Output Integration).
4. Scenario keys: when `config('machine.scenarios.enabled')` is true, `buildResponse()` also
   appends `availableScenarios` (and `activeScenario` when one is active). Reads reuse
   `buildResponse()` verbatim, so a read response carries these keys too in scenario-enabled
   apps. The base envelope above is the scenarios-disabled shape.

## Output Integration

Reads reuse the existing `output` resolution (the 3rd `output` semantic — shape HTTP response)
and the **same** parsing endpoints use. `handleSnapshot` routes the configured `output` onto
`buildResponse()`'s two parameters identically to `handleEndpoint()`:

| `output` form | Routed as | Behavior |
|---------------|-----------|----------|
| `null` | neither | Falls back to `$machine->output()` (the machine's own output, or `null`). |
| plain list of context keys (`array`, first element not an array) | `outputKeys` | Context-key filter via `toResponseArray()` merge + `array_intersect_key`. |
| `string` (OutputBehavior class) | `outputKey` | Computed output with DI via `resolveAndRunOutput()`; may return array or `MachineOutput`. |
| inner-array behavior tuple (`[[Class::class, 'param' => val]]`) | `outputKey` | Parameterized OutputBehavior via `resolveAndRunOutput()`. |

The discriminator is exactly `handleEndpoint()`'s (`MachineController` L109–113):
`is_string($output) || isInnerArrayTuple($output)` → `outputKey`; plain array → `outputKeys`;
`null` → neither. Reads introduce no new discriminator. Closures are not supported via route
defaults (route caching cannot serialize them), matching existing endpoint behavior.

Note (non-normative): a read's `output` is expected to be a pure read shaper; reads do not
enforce `OutputBehavior` purity, so an impure shaper is a caller bug, not a framework-rejected
error.

## Error Handling

| Condition | Response |
|-----------|----------|
| `machineId` not found (active table + archive both empty) | `404` — restore throws `RestoringStateException`, caught by `handleSnapshot`. |
| `machineId` belongs to a different machine class than the route's `_machine_class` | `404`. **Restore-path hardening (required):** make `restoreCurrentStateDefinition()` throw `RestoringStateException` on an `idMap` miss (the restored `machine_value` is not a state of this machine's definition, as happens for a foreign machine) instead of a raw undefined-key error — so the wrong-class case is caught and mapped to `404` uniformly. |
| Archived machine | Restored transparently by `restoreStateFromRootEventId()` (existing behavior); read returns 200. The auto-restore may move rows out of the archive — a pre-existing property of restore, not a write produced by the read. |
| Read definition contains `action`, `method`, or an unknown key | `InvalidRouterConfigException` at registration time (fail fast). |

The "zero-write" guarantee is precise: a read produces **no new `machine_events` rows and no
lock row** of its own. It does not suppress the pre-existing transparent auto-restore of an
archived machine (a property of restore shared with every code path that restores).

`MachineAlreadyRunningException` is not reachable from a read (no lock is acquired).

## Backward Compatibility & Migration

1. The `reads` concept is purely additive — absent `reads` means no new routes and no
   behavior change. Persist Write-Path Hardening changes the engine but stays
   backward-compatible: incremental persist and `max()`-based sequencing are transparent for
   normal (contiguous) histories, and atomicity only tightens an existing failure mode.
2. Existing read-as-event endpoints continue to work unchanged (they still write); `reads`
   is the recommended replacement, not a forced migration.
3. Migration is optional and owned by the consuming app, not this package. To convert a
   read-event endpoint to a read: (a) remove the event entry from the machine's `endpoints:`
   config, (b) delete the now-unused `*RequestedEvent` class, (c) add the read to
   `MachineRouter::register(['reads' => ...])`, (d) move any endpoint `output` shaper to the
   read's `output`. An app that hits no errors need not migrate.

## Router Config Validation

1. `ReadDefinition::fromConfig()` validates each read definition at registration time: throws
   `InvalidRouterConfigException` on a forbidden key (`action`, `method`), any unrecognized key
   (strict), or an invalid URI (empty / contains `{...}`). Recognized keys: `uri`, `output`,
   `middleware`, `status`, `available_events`.
2. Duplicate read URIs within one registration (compared on the **normalized** URI) throw
   `InvalidRouterConfigException` — so keys `status` and `/status` collide.
3. A read route that collides with another route built **within this `register()` call's route
   group** (the endpoints, forwarded endpoints, and `create` route assembled in the same group)
   throws `InvalidRouterConfigException`. Collision key = (HTTP method, fully-resolved route
   URI). Because reads are always `GET` and machineId-bound (`{prefix}/{machineId}/{uri}`), a
   read collides only with a `GET` machineId-bound endpoint of the same resolved path. Example
   collision: read `status` + a `GET` machineId-bound endpoint with uri `/status`. Example
   non-collision: read `status` + a `POST` endpoint `/status` (different method), or a
   model-bound endpoint (different URI shape). The check considers only this registration's
   group, not the global Laravel route table. The contract is **exact equality** of (method,
   resolved path); placeholder/route-ordering shadows (e.g. a multi-segment read vs an endpoint
   with a `{param}` segment) are out of scope and the caller's responsibility.

## Persist Write-Path Hardening

These fix the root scalability and correctness problems behind the incident, independent of
`reads`. `reads` stops poll-driven bloat; these protect legitimately long-lived machines
(thousands of real events) and fix a latent desync bug that affects every machine. Reading
the full history on restore is **not** changed — it is acceptable and always correct. Only
the write path changes.

### Incremental persist

Today `persist()` maps the **entire** `state->history` into one `MachineEvent::upsert(...)`,
re-writing every row on every event. This is O(n) writes per event and hits MySQL's 65,535
placeholder ceiling on long histories. Only two kinds of rows actually carry new data each
persist; the rest are byte-identical re-writes.

1. Keep the full in-memory walk that rebuilds the incremental-context baseline
   (`recursiveDiff`/`recursiveMerge`) — it is cheap (in-memory array merges) and required to
   compute the diff for the rows that change.
2. Upsert **only the dirty rows**: every event appended during this macrostep, plus the one
   row that was the previous tail (it held a full snapshot and must be rewritten as a diff now
   that it is no longer last) — **except index 0 (the root event), which always keeps its full
   context and is never diffed** (matching the existing `$index > 0` guard in `persist()`), even
   when it falls inside the dirty slice. All rows before the previous tail are byte-identical to
   the DB and MUST NOT be re-written.
3. Persisted watermark: capture, at restore time, the count of rows loaded from the DB (the
   number of already-persisted events) on the `State`/history. A freshly-created (non-restored)
   machine has watermark `0`. The dirty slice is `history[max(0, watermark - 1) .. end]` — the
   previous tail (index `watermark - 1`, included in the upsert with its now-diffed context)
   plus all events appended after it. When watermark is `0`, the whole history is written (all
   new). **After a successful `persist()`, advance the watermark to the current history count**,
   so a subsequent `persist()` on the same (non-restored) in-memory instance writes only rows
   appended since — without this, a long-lived in-memory instance would re-write the full
   history on every persist and lose the incremental benefit.
4. Concurrency: the dirty-slice computation assumes no other writer appended rows since
   restore. `Machine::send()` already serializes concurrent mutation via its lock (documented
   lock strategy), so within a locked `send()` the in-memory history is authoritative.
5. Preserve the storage invariant: the last row stores the **full** context snapshot (restore
   reads only the last row); all earlier rows store diffs. Deletion-safe — no merge on restore,
   so `recursiveDiff`'s inability to capture key removals is never exercised.
6. Result: upsert placeholder count is O(new events + 1) × columns, not O(total). A machine
   with thousands of events persists a new event without `QueryException`.

Worked example — restore a machine with 1000 persisted rows, then `send()` one external event
that produces 3 internal rows: the upsert writes 4 rows (row 1000 downgraded full→diff, plus
rows 1001–1003), not 1003. Rows 1–999 are untouched.

### Persist atomicity

Today `persist()` runs **outside** the `DB::transaction` that wraps `transition()` (`send()`
wraps `transition()` in a transaction only when `$event->isTransactional` is true — the default
for `EventBehavior`, false for raw string/array events). So an action that commits a DB write
(e.g. updating an external `status` column) can succeed while `persist()` fails, leaving the
row committed but `machine_events` unwritten — desyncing state from the log and wedging the
machine.

1. For a persisted `send()` of a transactional event (`isTransactional === true`), `persist()`
   MUST run inside the **same** `DB::transaction` that wraps `transition()`, so a `persist()`
   failure rolls back the transition's committed DB side-effects. Concretely: move the
   `if (shouldPersist) persist();` call inside the `DB::transaction(...)` closure rather than
   after it.
2. For a non-transactional event (`isTransactional === false`, an explicit opt-out), behavior
   is unchanged: `transition()` and `persist()` run unwrapped. The atomicity guarantee applies
   **only to transactional events** (the default for `EventBehavior`) — this is the firm
   requirement, chosen to preserve the explicit opt-out while fixing the default/incident case.
   Note: moving `persist()` inside the transaction also brings `syncCurrentStates()` (the
   `machine_current_states` write) under the same rollback boundary — intended, since
   current-state sync must not commit when the events it reflects do not.
3. Atomicity covers DB writes only — external non-DB I/O in actions (HTTP, etc.) cannot be
   transactional and is out of this guarantee (another reason to keep external I/O in
   delegations).
4. Test: see Testing item 9.

### Sequence number assignment

New events derive `sequence_number` from the loaded history count/position
(`count($this->history) + 1` in `State::setCurrentEventBehavior`) rather than
`max(sequence_number)`. After pruning/compaction (e.g. an operator deletes poll rows), new
events collide with retained higher sequence numbers, producing duplicates and incorrect
hydration.

1. Assign a new event's `sequence_number` from `max(existing sequence_number) + 1`, derived
   from the in-memory `state->history` already loaded for this macrostep (not a separate
   `SELECT max()` query), so it is consistent with the rows being persisted and serialized by
   the same `Machine::send()` lock that guards concurrent mutation. No extra query, no new race
   beyond what `send()` already serializes.
2. Normal (contiguous) histories are unaffected — `count == max` — so this only changes
   behavior after pruning/compaction.
3. Test: see Testing item 10.
## Deliberately Not Included

1. **Read-model / snapshot cache** — reads restore the full history on each request. This is
   acceptable: the full-history *read* is not the bottleneck (the *write*-side full-history
   upsert was, fixed by Incremental persist). Serving reads from a denormalized cache (e.g.
   `MachineCurrentState`) would reintroduce the documented parallel-dispatch staleness, so a
   read could return stale state. Restoring from history is always correct — no cache.
2. **No-op persist guardrail** (skipping persistence of targetless/no-change transitions) —
   superseded by `reads`, which never enters the write path at all.
