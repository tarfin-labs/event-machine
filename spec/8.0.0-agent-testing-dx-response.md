# Event-Machine Testing DX — Agent Feedback Response

We evaluated your feedback and made a decision for each item. Here's the result:

---

## 1. ChildMachineDoneEvent::forTesting() / ChildMachineFailEvent::forTesting() ✅ DONE

**Added.** Now works with sensible defaults:

```php
// Before — 6 lines of boilerplate
$event = ChildMachineDoneEvent::forChild([
    'result'        => ['statusCode' => 3],
    'output'        => [],
    'machine_id'    => 'test',
    'machine_class' => 'Test',
]);

// Now — only the data you care about
$event = ChildMachineDoneEvent::forTesting(['result' => ['statusCode' => 3]]);

// Fail event
$event = ChildMachineFailEvent::forTesting(['error_message' => 'Gateway timeout']);

// With output
$event = ChildMachineFailEvent::forTesting([
    'error_message' => 'API error',
    'output'        => ['errorCode' => 'E311'],
]);

// With final state (done routing)
$event = ChildMachineDoneEvent::forTesting([
    'result'      => ['status' => 'ok'],
    'final_state' => 'approved',
]);

// Zero config — all defaults
$event = ChildMachineDoneEvent::forTesting();
$event = ChildMachineFailEvent::forTesting();
```

Defaults: `machine_id: 'test'`, `machine_class: 'TestMachine'`, `result: []`, `output: []`.

## 2. result() and output() will NOT be added to EventBehavior base class ⚠️ DESIGN DECISION

**Not adding.** Reasons:

**Semantic problem:** Not every event has `result()` and `output()` — what would `ORDER_SUBMITTED`'s result be? Adding default methods that return `null` on the base class hides incorrect API usage by silently returning `null` instead of erroring.

**Type safety loss:** Having guards type-hint `ChildMachineDoneEvent $event` instead of `EventBehavior $event` lets the IDE and PHPStan protect you.

**Correct approach:** Use the concrete event class as your type-hint in guards/actions:

```php
// ✅ Correct — IDE/PHPStan help, methods are defined
public function __invoke(ContextManager $context, ChildMachineDoneEvent $event): bool
{
    return $event->result('statusCode') === 3;
}

// ❌ Wrong — result() not on base class, IDE can't help
public function __invoke(ContextManager $context, EventBehavior $event): bool
{
    return $event->result('statusCode') === 3; // Method not found
}
```

EventMachine's injection system (`InvokableBehavior::injectInvokableBehaviorParameters()`) already resolves the `ChildMachineDoneEvent` type-hint correctly — it injects the right event subclass when guards/actions are invoked during `@done` transitions.

**Summary:** Use `ChildMachineDoneEvent $event`, not `EventBehavior $event`.

## 3. Job states with TestMachine::define() → simulateChild* already works ℹ️ ALREADY SUPPORTED

**TestMachine already provides `simulateChildDone/Fail/Timeout`:**

```php
// With Machine subclass (recommended)
$testMachine = FindeksMachine::test(['tcno' => '12345678901'])
    ->send('START')
    ->simulateChildDone(QueryPhonesChild::class, result: ['phones' => ['555...']])
    ->assertState('requesting')
    ->simulateChildFail(RequestChild::class, 'API timeout')
    ->assertState('failed');
```

It also works with `TestMachine::define()`, as long as the state definition has a `machine` key:

```php
$testMachine = TestMachine::define([
    'id'      => 'test',
    'initial' => 'querying',
    'states'  => [
        'querying' => [
            'machine' => SomeChildMachine::class,
            '@done'   => 'processing',
            '@fail'   => 'failed',
        ],
        'processing' => ['type' => 'final'],
        'failed'     => ['type' => 'final'],
    ],
]);

$testMachine
    ->simulateChildDone(SomeChildMachine::class, result: ['data' => 'ok'])
    ->assertState('processing');
```

**Important:** `simulateChildDone/Fail/Timeout` methods do NOT touch the database. They route events entirely through definition-level logic (`routeChildDoneEvent`/`routeChildFailEvent`), so they work with `define()` and `withContext()` (no persistence) just as well as with `create()`.

**Our recommendation:** For isolated guard/action testing, use `forTesting()` (#1). For testing transition flow, use `simulateChildDone/Fail`. These are different test levels:

| Level | Tool | What It Tests |
|-------|------|---------------|
| Unit (guard/action) | `forTesting()` + direct invoke | Guard logic, action side effects |
| Integration (flow) | `simulateChildDone/Fail` | Transition routing, guard → state chain |

## 4. advanceTimers() in-memory support 🔜 PLANNED

**Plan is ready, will be implemented.** Currently `advanceTimers()` with `define()` throws `RuntimeException`. Plan:

- In-memory timer state tracking added to `TestMachine`
- `advanceTimers()` auto-detects persistence mode — DB path for persisted, in-memory path otherwise
- `assertTimerFired/NotFired` also work in in-memory mode
- `assertHasTimer()` gets optional `Timer $expectedDuration` parameter

Timer tests will look like:

```php
TestMachine::define([
    'id'      => 'pin_flow',
    'initial' => 'waiting',
    'states'  => [
        'waiting' => [
            'on' => [
                'PIN_EXPIRED' => [
                    'target' => 'expired',
                    'after'  => Timer::seconds(120),
                ],
            ],
        ],
        'expired' => ['type' => 'final'],
    ],
])
->assertHasTimer('PIN_EXPIRED', Timer::seconds(120))
->advanceTimers(Timer::seconds(60))   // 60s < 120s
->assertState('waiting')              // not triggered yet
->advanceTimers(Timer::seconds(61))   // total 121s > 120s
->assertState('expired')              // triggered
->assertTimerFired('PIN_EXPIRED');
```

`@every` timers (including max/then) will also be supported.

## General Recommendations

1. **Use concrete event type-hints in guards** — `ChildMachineDoneEvent` not `EventBehavior`
2. **Separate unit vs integration tests** — `forTesting()` for unit, `simulateChildDone/Fail` for integration
3. **For timer tests in the meantime** — continue using `send('PIN_EXPIRED')` directly — in-memory support is coming soon
