# Forward-Aware Endpoints

**Status:** Draft v5 ŌĆö Final review complete, ready for implementation
**Author:** Claude
**Date:** 2025-03-17
**Scope:** Forward-aware endpoints with child response customization, HATEOAS `available_events`, `ForwardContext` type-safe injection, FQCN support, endpoint customization, and binding mode inheritance

---

## Problem

When a parent machine delegates to an async child via the `machine` key, the child may need user input (e.g., payment details, document approval). The `forward` mechanism in `MachineInvokeDefinition` handles internal event routing correctly (`tryForwardEventToChild`), but the **HTTP routing layer is completely unaware of it**.

### The Gap

```
POST /orders/{order}/provide-card  ŌåÆ  MachineController
                                          Ōöé
                                          Ōö£ŌöĆŌöĆ resolveEvent(): looks up PROVIDE_CARD in parent's behavior['events']
                                          Ōöé         Ōöé
                                          Ōöé    Not found ŌåÆ 422 "Event type not found"  ŌåÉ BREAKS HERE
                                          Ōöé
                                          ŌĢ│  Machine::send() never called
                                          ŌĢ│  tryForwardEventToChild() never reached
```

The controller resolves events exclusively from the parent machine's behavior definitions (`MachineController.php:112-121`). Forwarded events, which belong to the child machine, are invisible at this layer.

### Current Workaround (Manual Duplication)

The parent must re-declare child events in three places:

```php
// 1. Parent's behavior must include child's EventBehavior classes
behavior: [
    'events' => [
        'PROVIDE_CARD'    => ProvideCardEvent::class,    // ŌåÉ duplicate of child
        'CONFIRM_PAYMENT' => ConfirmPaymentEvent::class, // ŌåÉ duplicate of child
    ],
],
// 2. Parent's endpoints must list them
endpoints: ['PROVIDE_CARD', 'CONFIRM_PAYMENT'],
// 3. Parent's state must forward them
'processing' => ['forward' => ['PROVIDE_CARD', 'CONFIRM_PAYMENT']],
```

This creates tight coupling, DRY violation, and maintenance burden ŌĆö when the child changes, the parent must change too.

### Response Gap

When `tryForwardEventToChild()` succeeds, it returns `$state` (parent's unmodified state). The HTTP response shows:

```json
{ "data": { "machine_id": "parent-abc", "value": "payment_processing", "context": { ... } } }
```

The child transitioned (e.g., `idle` ŌåÆ `card_provided`) but the response gives **zero feedback** about it. The user sent card details and sees no confirmation that anything happened.

---

## Design Principles

1. **`forward` is the single source of truth** ŌĆö no duplicate declarations in behavior or endpoints
2. **Parent is the API contract** ŌĆö user always talks to `/orders/{order}/...`, child is an implementation detail
3. **Child validates its own events** ŌĆö forwarded events use the child's `EventBehavior`, not the parent's
4. **Response reflects what happened** ŌĆö forwarded endpoint responses include child state feedback
5. **Overlap is an error, not a fallback** ŌĆö forward events MUST NOT appear in parent's `behavior['events']` or `endpoints`. Validator rejects overlap. No dual code paths, no ambiguity.
6. **Breaking change accepted** ŌĆö this is a major feature, existing workaround patterns must migrate. Clean design over compatibility.

## API Design

### Minimal (Just `forward`)

```php
'payment_processing' => [
    'machine' => PaymentMachine::class,
    'queue'   => 'default',
    'forward' => [
        ProvideCardEvent::class,                 // FQCN ŌĆö auto-resolved to 'PROVIDE_CARD'
        'CONFIRM_PAYMENT',                       // string ŌĆö also works
        CancelOrderEvent::class => 'ABORT',      // FQCN key + string child event
    ],
    '@done' => 'completed',
],
// No need to add PROVIDE_CARD to behavior['events'] or endpoints[]
```

The `forward` key alone is sufficient. No manual duplication needed. Both FQCN and string event types work ŌĆö FQCN entries are resolved to SCREAMING_SNAKE via `getType()` at parse time, consistent with `endpoints`, `machineIdFor`, `modelFor`, and `schedules`.

### Extended (Forward with Full Config)

```php
'forward' => [
    ProvideCardEvent::class => [
        // ŌöĆŌöĆ Child event routing ŌöĆŌöĆ
        'child_event'  => ConfirmPaymentEvent::class,  // FQCN ŌĆö resolved to 'CONFIRM_PAYMENT'

        // ŌöĆŌöĆ Endpoint customization (parent overrides) ŌöĆŌöĆ
        'uri'          => '/enter-payment-details',    // override child's auto-generated URI
        'method'       => 'PATCH',                      // override HTTP method (default: POST)
        'middleware'    => ['throttle:10'],              // additive middleware for this endpoint
        'action'       => CardEntryAction::class,       // parent-level MachineEndpointAction

        // ŌöĆŌöĆ Response customization ŌöĆŌöĆ
        'result'       => PaymentStepResult::class,    // parent's ResultBehavior (receives both contexts)
        'contextKeys'  => ['card_last4', 'status'],    // filter child context in default response
        'status'       => 200,                          // HTTP status code
    ],
    'CONFIRM_PAYMENT',                                 // simple forward, default response
    CancelOrderEvent::class => AbortEvent::class,      // FQCN rename both sides
],
```

#### Format 3 Array Configuration Options

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `child_event` | `string` | Same as parent event | Child event type (string or FQCN) |
| `uri` | `string` | Auto-generated from parent event type | Override endpoint URI |
| `method` | `string` | `'POST'` | Override HTTP method |
| `middleware` | `array` | `[]` | Additive per-endpoint middleware |
| `action` | `string` | `null` | `MachineEndpointAction` subclass FQCN |
| `result` | `string` | `null` | ResultBehavior (parent or child) FQCN or inline key |
| `contextKeys` | `array` | `null` (all keys) | Filter child context keys in default response |
| `status` | `int` | `200` | HTTP status code |
| `available_events` | `bool` | `true` | Include `available_events` in default response (set `false` to opt out) |

Three forward syntax levels (all support both string and FQCN):
1. **Plain**: `'PROVIDE_CARD'` or `ProvideCardEvent::class` ŌĆö forward as-is, default response, auto-generated URI
2. **Rename**: `'CANCEL_ORDER' => 'ABORT'` or `CancelOrderEvent::class => AbortEvent::class` ŌĆö rename, default response
3. **Full array**: `ProvideCardEvent::class => [...]` ŌĆö full endpoint and response customization

### Response Customization

#### Parent-Level ResultBehavior (Recommended)

The `result` in forward config is a **parent-level** concept. The ResultBehavior runs with access to **both** parent and child state via parameter injection:

```php
class PaymentStepResult extends ResultBehavior
{
    public function __invoke(
        ContextManager $context,           // parent context (always available)
        ForwardContext $forwardContext,     // child context + state (forward mode only)
    ): array {
        return [
            'order_id'   => $context->get('order_id'),                      // from parent
            'card_last4' => $forwardContext->childContext->get('card_last4'), // from child
            'child_step' => $forwardContext->childState->value[0],           // child's current state
            'next_step'  => 'confirm_payment',                               // parent-level business logic
        ];
    }
}
```

`ForwardContext` is **only injected** in forward endpoint context. In regular endpoint context, it is not available ŌĆö type-hinting it will result in `null` injection (TypeError if not nullable). This is intentional ŌĆö a ResultBehavior that uses `ForwardContext` should only be referenced from forward endpoint configs.

This follows the existing `InvokableBehavior::injectInvokableBehaviorParameters()` pattern ŌĆö `ForwardContext` is added as a new injectable type alongside `ContextManager`, `EventBehavior`, `State`, etc.

#### Default Response (no `result`)

When no `result` is specified, the response reflects **both** parent and child state plus `available_events`:

```json
{
    "data": {
        "machine_id": "parent-root-event-id",
        "value": "payment_processing",
        "child": {
            "value": "card_provided",
            "context": { "card_last4": "4242", "card_brand": "visa" }
        },
        "available_events": [
            { "type": "CONFIRM_PAYMENT", "source": "forward" },
            { "type": "CANCEL", "source": "parent" }
        ]
    }
}
```

#### With `contextKeys`

Only specified keys from child context are included (same structure, filtered `child.context`):

```json
{
    "data": {
        "machine_id": "parent-root-event-id",
        "value": "payment_processing",
        "child": {
            "value": "card_provided",
            "context": { "card_last4": "4242" }
        },
        "available_events": [
            { "type": "CONFIRM_PAYMENT", "source": "forward" },
            { "type": "CANCEL", "source": "parent" }
        ]
    }
}
```

#### With `result` (Parent ResultBehavior)

The parent's ResultBehavior transforms both parent and child data into a custom response:

```json
{
    "data": {
        "order_id": "ord_12345",
        "card_last4": "4242",
        "child_step": "card_provided",
        "next_step": "confirm_payment"
    }
}
```

### Forward vs Reverse: Existing Mechanisms

The reverse of `forward` already exists via established mechanisms:

| Direction | Mechanism | Trigger | Use Case |
|-----------|-----------|---------|----------|
| Parent ŌåÆ Child | `forward` | User HTTP request | User input to child |
| Child ŌåÆ Parent (mid-lifecycle) | `sendToParent()` | Child's action | Child notifies parent during execution |
| Child ŌåÆ Parent (completion) | `@done` / `@fail` | Child reaches final state | Lifecycle completion |
| Child ŌåÆ Parent (data) | `output` key on final state | Child reaches final state | Filter what parent sees at completion |

No new reverse mechanism needed ŌĆö the current tools are complementary and cover all use cases.

## Detailed Design

### Parse-Time: Forward Endpoint Discovery

**File:** `MachineDefinition::parseEndpoints()` (extend, not replace)

After parsing explicit endpoints, iterate all states with `forward` config:

```
For each state with machineInvokeDefinition.forward:
  1. Load child machine definition: $childDef = $invokeDefinition->machineClass::definition()
  2. For each forward entry:
     a. Resolve parent event type and child event type
     b. Look up child's EventBehavior class from $childDef->behavior['events'][$childEventType]
     c. Auto-generate EndpointDefinition (URI from parent event type, method POST)
     d. Store in $this->forwardedEndpoints[$parentEventType] = ForwardedEndpointDefinition
  3. Validate: no overlap between parsedEndpoints and forwardedEndpoints
```

New value object: `ForwardedEndpointDefinition`

```php
class ForwardedEndpointDefinition
{
    public function __construct(
        public readonly string $parentEventType,    // PROVIDE_CARD (resolved from FQCN if needed)
        public readonly string $childEventType,     // PROVIDE_CARD (or renamed)
        public readonly string $childMachineClass,  // PaymentMachine::class
        public readonly string $childEventClass,    // ProvideCardEvent::class
        public readonly string $uri,                // /provide-card (or overridden)
        public readonly string $method,             // POST (or overridden)
        public readonly ?string $actionClass,       // parent-level MachineEndpointAction (optional)
        public readonly ?string $resultBehavior,    // parent's ResultBehavior (optional)
        public readonly ?array $contextKeys,        // child context keys to include (optional)
        public readonly ?int $statusCode,           // HTTP status code (optional)
        public readonly array $middleware,           // additional middleware
        public readonly ?bool $availableEvents,     // opt-out for available_events (null = enabled)
    ) {}
}
```

### Route Registration: Auto-Register Forwarded Endpoints

**File:** `MachineRouter::register()`

After registering explicit endpoints, also register forwarded endpoints.

**Binding mode rule:** If `model` + `attribute` are configured on the router ŌåÆ forwarded endpoints are **model-bound**. Otherwise ŌåÆ **machineId-bound**. This is the natural inheritance ŌĆö the parent's router config determines binding for all endpoints.

```php
// Determine binding mode for forwarded endpoints
$hasModelBinding = $model !== null && $attribute !== null;

foreach ($definition->forwardedEndpoints as $eventType => $fwdEndpoint) {
    if ($hasModelBinding && $modelParam !== null) {
        // Model-bound: /{model}/{uri} ŌĆö same as parent's modelFor endpoints
        $routeUri = "/{$modelParam}{$fwdEndpoint->uri}";
        $handler  = 'handleForwardedModelBound';
    } else {
        // MachineId-bound: /{machineId}/{uri} ŌĆö same as parent's machineIdFor endpoints
        $routeUri = "/{machineId}{$fwdEndpoint->uri}";
        $handler  = 'handleForwardedMachineIdBound';
    }

    Route::match(
        [$fwdEndpoint->method],
        $routeUri,
        [MachineController::class, $handler]
    )
        ->name("{$namePrefix}.".strtolower($eventType))
        ->middleware($fwdEndpoint->middleware)
        ->setDefaults([
            '_machine_class'         => $machineClass,
            '_event_type'            => $eventType,
            '_child_event_type'      => $fwdEndpoint->childEventType,
            '_child_machine_class'   => $fwdEndpoint->childMachineClass,
            '_child_event_class'     => $fwdEndpoint->childEventClass,
            '_action_class'          => $fwdEndpoint->actionClass,
            '_result_behavior'       => $fwdEndpoint->resultBehavior,
            '_context_keys'          => $fwdEndpoint->contextKeys,
            '_status_code'           => $fwdEndpoint->statusCode ?? 200,
            '_available_events'      => $fwdEndpoint->availableEvents,
            '_model_attribute'       => $attribute,  // only used for model-bound
        ]);

    // Register model binding if model-bound (same as explicit endpoints)
    if ($hasModelBinding && $modelParam !== null) {
        Route::model($modelParam, $model);
    }
}
```

**Two forwarded handlers** (mirrors the existing `handleModelBound`/`handleMachineIdBound` split):

```php
// MachineController
public function handleForwardedModelBound(Request $request): JsonResponse
{
    // Resolve parent machine via route model binding (same as handleModelBound)
    $route          = $request->route();
    $modelParam     = $route->parameterNames()[0];
    $model          = $route->parameter($modelParam);
    $modelAttribute = $route->defaults['_model_attribute'];

{
    // Resolve parent machine via machineId (same as handleMachineIdBound)

    $machineClass = $route->defaults['_machine_class'];
    $machine      = $machineClass::create(state: $route->parameter('machineId'));

    return $this->executeForwardedEndpoint($machine, $request);
}

// Shared forwarded endpoint logic
protected function executeForwardedEndpoint(Machine $machine, Request $request): JsonResponse
{
    // ... event resolution, action lifecycle, send, response building ...
}
```

### Request-Time: Shared Forwarded Endpoint Logic

**File:** `MachineController` ŌĆö new shared method `executeForwardedEndpoint()`

Parent machine resolution is handled by `handleForwardedModelBound()` / `handleForwardedMachineIdBound()` (see router section above). The shared logic:

```php
protected function executeForwardedEndpoint(Machine $machine, Request $request): JsonResponse
{
    $defaults = $request->route()->defaults;

    // 1. Resolve event using CHILD's EventBehavior class
    $childEventClass = $defaults['_child_event_class'];
    $event = $childEventClass::validateAndCreate($request->all());

    // 2. Run parent-level action.before() if configured
    $action = ($defaults['_action_class'] ?? null) !== null
        ? resolve($defaults['_action_class'])->withMachineContext($machine, $machine->state)
        : null;

    $action?->before();

    // 3. Send to parent (triggers tryForwardEventToChild internally)
    try {
        $state = $machine->send(event: $event);
    } catch (\Throwable $e) {
        $response = $action?->onException($e);
        if ($response !== null) {
            return $response;
        }
        throw $e;
    }

    // 4. Run parent-level action.after()
    if ($action !== null) {
        $action->withMachineContext($machine, $state);
    }
    $action?->after();

    // 5. NOTE: dispatchChildCompletionIfFinal is NOT called here.
    // tryForwardEventToChild() already dispatches ChildMachineCompletionJob
    // if the child reaches a final state (MachineDefinition.php:1396-1412).
    // Calling it again would cause double-dispatch.

    // 6. Build response with child state info
    return $this->buildForwardedResponse($machine, $state, $defaults);
}
```

### Response Building: Child State in Response

**File:** `MachineController` ŌĆö new method `buildForwardedResponse()`

```php
protected function buildForwardedResponse(
    Machine $machine,
    State $parentState,
    array $defaults,
): JsonResponse {
    $resultKey   = $defaults['_result_behavior'] ?? null;
    $contextKeys = $defaults['_context_keys'] ?? null;
    $statusCode  = $defaults['_status_code'] ?? 200;

    // Get child state from forwarded state (stashed by tryForwardEventToChild)
    $childState = $parentState->getForwardedChildState();

    // Custom result behavior ŌĆö runs on PARENT with both contexts injected
    if ($resultKey !== null && $childState !== null) {
        $result = $this->resolveAndRunForwardedResult(
            resultKey: $resultKey,
            parentState: $parentState,
            parentMachine: $machine,
            childState: $childState,
        );

        return response()->json(['data' => $result], $statusCode);
    }

    // Default response: parent state + child state summary + available_events
    $rootEventId = $parentState->history->first()?->root_event_id;

    $response = [
        'machine_id' => $rootEventId,
        'value'      => $parentState->value,
    ];

    if ($childState !== null) {
        $childContext = $childState->context->toArray();

        if ($contextKeys !== null) {
            $childContext = array_intersect_key($childContext, array_flip($contextKeys));
        }

        $response['child'] = [
            'value'   => $childState->value,
            'context' => $childContext,
        ];
    }

    // Add available_events unless opted out
    $includeAvailableEvents = $defaults['_available_events'] ?? true;
    if ($includeAvailableEvents !== false) {
        $response['available_events'] = $parentState->availableEvents();
    }

    return response()->json(['data' => $response], $statusCode);
}

/**
 * Resolve and run a ResultBehavior with ForwardContext injection.
 *
 * Unlike resolveAndRunResult(), this creates a ForwardContext wrapper
 * and passes it to the injection system. The ResultBehavior can type-hint
 * ContextManager (parent) and ForwardContext (child context + state).
 */
protected function resolveAndRunForwardedResult(
    string $resultKey,
    State $parentState,
    Machine $parentMachine,
    State $childState,
): mixed {
    $resultClass = class_exists($resultKey)
        ? $resultKey
        : ($parentMachine->definition->behavior['results'][$resultKey] ?? null);

    if ($resultClass === null) {
        throw new \RuntimeException("Result behavior '{$resultKey}' not found.");
    }

    $resultBehavior = resolve($resultClass);

    // Create ForwardContext wrapper for type-safe injection
    $forwardContext = new ForwardContext(
        childContext: $childState->context,
        childState: $childState,
    );

    // Inject with ForwardContext available
    $params = InvokableBehavior::injectInvokableBehaviorParameters(
        actionBehavior: $resultBehavior,
        state: $parentState,
        eventBehavior: $parentState->currentEventBehavior,
        forwardContext: $forwardContext,
    );

    return $resultBehavior(...$params);
}
```

**Note:** `injectInvokableBehaviorParameters()` receives `?ForwardContext $forwardContext = null`. The injection logic matches `ForwardContext` type-hint to this parameter. `ContextManager` always resolves to parent context (no ambiguity). See `ForwardContext` wrapper class section below for full injection design.

### `tryForwardEventToChild` Enhancement

**File:** `MachineDefinition.php`

Current: returns `bool` ŌĆö we only know if forwarding happened.
Enhanced: returns the child's `State` after forwarding (or `null`).

```php
// Before:
protected function tryForwardEventToChild(...): bool

// After:
protected function tryForwardEventToChild(...): ?State
```

This allows the controller to access the child's post-transition state for response building without a second DB query.

The `Machine::send()` catch block also changes:

```php
// Before:
if ($this->tryForwardEventToChild($state, $currentStateDefinition, $eventBehavior)) {
    return $state;
}

// After:
$childState = $this->tryForwardEventToChild($state, $currentStateDefinition, $eventBehavior);
if ($childState !== null) {
    $state->setForwardedChildState($childState);  // stash on parent state
    return $state;
}
```

### Forward Config Parsing Enhancement

**File:** `MachineInvokeDefinition`

#### FQCN Support (Bug Fix / Missing Feature)

**Current problem:** Forward entries are stored as raw strings and compared literally in `resolveForwardEvent()`. FQCN class references **do not work**:

```php
// ŌØī Currently broken ŌĆö stored as FQCN, never resolved to SCREAMING_SNAKE
'forward' => [ProvideCardEvent::class]

// Ō£ģ Currently works ŌĆö literal string comparison
'forward' => ['PROVIDE_CARD']
```

**Fix:** Normalize forward entries at parse time in `StateDefinition::initializeMachineInvoke()` (line 438), using the same `EndpointDefinition::resolveEventType()` that endpoints, schedules, and `machineIdFor`/`modelFor` already use:

```php
// In initializeMachineInvoke(), before passing to MachineInvokeDefinition:
$rawForward = $this->config['forward'] ?? [];
$forward    = [];

foreach ($rawForward as $key => $value) {
    if (is_int($key)) {
        // List entry: 'PROVIDE_CARD' or ProvideCardEvent::class
        $forward[] = EndpointDefinition::resolveEventType($value);
    } elseif (is_string($value)) {
        // Rename: key is parent event, value is child event
        $resolvedKey   = EndpointDefinition::resolveEventType($key);
        $resolvedValue = EndpointDefinition::resolveEventType($value);
        $forward[$resolvedKey] = $resolvedValue;
    } elseif (is_array($value)) {
        // Format 3: full config ŌĆö resolve key and child_event inside
        $resolvedKey = EndpointDefinition::resolveEventType($key);
        if (isset($value['child_event'])) {
            $value['child_event'] = EndpointDefinition::resolveEventType($value['child_event']);
        }
        $forward[$resolvedKey] = $value;
    }
}
```

This enables all these equivalent declarations:

```php
'forward' => [
    // Format 1: plain ŌĆö string or FQCN
    'PROVIDE_CARD',
    ProvideCardEvent::class,                            // ŌåÆ resolved to 'PROVIDE_CARD'

    // Format 2: rename ŌĆö string or FQCN for both sides
    'CANCEL_ORDER' => 'ABORT',
    CancelOrderEvent::class => AbortEvent::class,       // ŌåÆ resolved to 'CANCEL_ORDER' => 'ABORT'
    CancelOrderEvent::class => 'ABORT',                 // ŌåÆ mixed also works

    // Format 3: full config ŌĆö FQCN as key and/or child_event
    ProvideCardEvent::class => [
        'child_event'  => ConfirmPaymentEvent::class,   // ŌåÆ resolved to 'CONFIRM_PAYMENT'
        'result'       => PaymentSummaryResult::class,
        'contextKeys'  => ['card_last4'],
    ],
],
```

The `forward` array now supports three format levels (with FQCN support at every level):

```php
'forward' => [
    'APPROVE',                              // Format 1: plain string (existing)
    ProvideCardEvent::class,                // Format 1: plain FQCN (NEW)
    'PARENT_UPDATE' => 'CHILD_UPDATE',      // Format 2: rename string (existing)
    CancelOrderEvent::class => 'ABORT',     // Format 2: rename FQCN key (NEW)
    'PROVIDE_CARD' => [                     // Format 3: full config (NEW)
        'child_event'  => 'PROVIDE_CARD',
        'result'       => ProvideCardResult::class,
        'contextKeys'  => ['card_last4'],
        'status'       => 200,
    ],
],
```

Detection logic (after FQCN resolution):

```php

    if (is_int($key) && is_string($value)) {
```

## Validation Rules

| Config | Valid? | Reason |
|--------|--------|--------|
| `forward` with `queue` + `@done` | Yes | Managed async with forwarding (existing) |
| `forward` without `queue` | No | Forward requires async child (existing rule) |
| `forward` without `@done` | No | Fire-and-forget doesn't support forward (existing rule) |
| Forward event type exists in child definition | Yes | Required ŌĆö validated at parse time |
| Forward event type also in parent `endpoints` | **No** | Error: overlap rejected, `forward` is single source of truth |
| Forward event type also in parent `behavior['events']` | **No** | Error: overlap rejected, child event discovered from child definition |
| Forward event type NOT in child definition | No | Error: "Forward event 'X' not found in child machine 'Y'" |
| Forward result class not a ResultBehavior | No | Error: same as existing endpoint validation |
| Forward event name collides with parent event | No | Error: "Forward event 'X' conflicts with parent event" |

## Edge Cases

### 1. Multiple Delegating States with Different Children

```php
'states' => [
    'payment' => [
        'machine' => PaymentMachine::class,
        'forward' => ['PROVIDE_CARD'],
        ...
    ],
    'shipping' => [
        'machine' => ShippingMachine::class,
        'forward' => ['UPDATE_ADDRESS'],
        ...
    ],
]
```

All forwarded endpoints are registered at boot time. At runtime, only the active state's forwards work. If the parent is in `payment` and receives `UPDATE_ADDRESS`, `tryForwardEventToChild` finds no running child for ShippingMachine ŌåÆ 409.

### 2. Forward Event Name Collision Across States

```php
'payment'  => ['forward' => ['CANCEL']],   // PaymentMachine::CANCEL
'shipping' => ['forward' => ['CANCEL']],   // ShippingMachine::CANCEL
```

Same parent event type from different children ŌåÆ **invalid**. Validator must catch this at parse time. Solution: use rename syntax:

```php
'payment'  => ['forward' => ['CANCEL_PAYMENT'  => 'CANCEL']],
'shipping' => ['forward' => ['CANCEL_SHIPPING' => 'CANCEL']],
```

### 3. Forward Event Also Defined as Parent Endpoint ŌĆö REJECTED

If `PROVIDE_CARD` is in both `endpoints` and `forward`, the validator **rejects** with a clear error:

```
State 'processing' forwards 'PROVIDE_CARD' which is also declared in parent's
endpoints. Remove 'PROVIDE_CARD' from endpoints and behavior.events ŌĆö forward
is the single source of truth for child events.
```

**Rationale:** Two code paths for the same event creates ambiguity ŌĆö which EventBehavior validates? Which response format? Which controller handler? Forcing a single path eliminates an entire class of bugs. Endpoint customization (URI, method, middleware, action, result) is available via Format 3 on the `forward` config itself ŌĆö no need to duplicate in `endpoints`.

### 4. Child Machine Used Standalone AND as Child

```php
// Standalone registration ŌĆö child has its own endpoints
MachineRouter::register(PaymentMachine::class, [
    'prefix'       => 'payments',
    'machineIdFor' => ['DO', 'DONE'],
]);
// Routes: POST /payments/{machineId}/do, POST /payments/{machineId}/done

// Parent registration ŌĆö model-bound, child events forwarded
MachineRouter::register(OrderMachine::class, [
    'prefix'    => 'orders',
    'model'     => Order::class,
    'attribute' => 'order_mre',
    'modelFor'  => ['START', 'CONTINUE'],
]);
// Explicit: POST /orders/{order}/start, POST /orders/{order}/continue
// Forwarded: POST /orders/{order}/do, POST /orders/{order}/done (model-bound, inherited from parent)
```

Both route groups exist independently:

| Route | Binding | Handler | Resolves to |
|-------|---------|---------|-------------|
| `POST /payments/{machineId}/do` | machineId | `handleMachineIdBound` | PaymentMachine directly |
| `POST /payments/{machineId}/done` | machineId | `handleMachineIdBound` | PaymentMachine directly |
| `POST /orders/{order}/do` | model (Order) | `handleForwardedModelBound` | OrderMachine ŌåÆ forward ŌåÆ PaymentMachine |
| `POST /orders/{order}/done` | model (Order) | `handleForwardedModelBound` | OrderMachine ŌåÆ forward ŌåÆ PaymentMachine |

No conflict ŌĆö different prefix, different binding, different handler. Same child machine, two access paths.

**Key point:** Forwarded endpoints inherit the **parent's** binding mode. `OrderMachine` is model-bound ŌåÆ forwarded endpoints are model-bound too. The child's standalone registration (`machineIdFor`) does not influence the parent's forwarded endpoints.

### 5. Sync Child with Forward

Already validated as invalid: `forward` requires `queue`. Sync children complete in a single `send()` call ŌĆö no mid-execution user input possible.

### 6. Forward During Non-Delegating State

If user sends `PROVIDE_CARD` when parent is in `idle` (not `payment_processing`), no child is running. `tryForwardEventToChild()` returns null ŌåÆ 409 response.

### 7. Parent in Parallel State with Forward (BLOCKING GAP)

**Current code:** `transition()` (line 2295) checks `isInParallelState()` first:

```php
if ($state->isInParallelState()) {
    return $this->transitionParallelState(...);  // ŌåÉ forward NEVER reached
}

try {
    $transitionDefinition = $this->findTransitionDefinition(...);
} catch (NoTransitionDefinitionFoundException $e) {
    if ($this->tryForwardEventToChild(...)) {  // ŌåÉ only in non-parallel branch
        return $state;
    }
    throw $e;
}
```

`tryForwardEventToChild()` is **only called in the non-parallel branch**. If the parent machine is currently in a parallel state, forward events are never attempted ŌĆö they throw `NoTransitionDefinitionFoundException` directly.

**Scenario:**
```php
// Parent machine
'fulfillment' => [
    'type' => 'parallel',
    'states' => [
        'payment' => [
            'initial' => 'processing',
            'states' => [
                'processing' => [
                    'machine' => PaymentMachine::class,
                    'queue'   => 'default',
                    'forward' => ['PROVIDE_CARD'],
                    '@done'   => 'paid',
                ],
                'paid' => ['type' => 'final'],
            ],
        ],
        'documents' => [
            'initial' => 'uploading',
            'states' => [...],
        ],
    ],
],
```

Sending `PROVIDE_CARD` to the parent ŌåÆ `transitionParallelState()` ŌåÆ `selectTransitions()` ŌåÆ no transition found ŌåÆ exception. Forward never tried.

**Fix required:** `transitionParallelState()` must also attempt `tryForwardEventToChild()` when no transitions are found. The forward should check each active region's state definition for forward config.

**Complexity:** In a parallel state, multiple regions are active simultaneously. Multiple regions could theoretically have forward configs (e.g., payment region forwards `PROVIDE_CARD`, documents region forwards `UPLOAD_DOCUMENT`). The forward mechanism needs to:
1. Iterate active region states
2. Find which region (if any) has a forward matching the event
3. Forward to that region's child

This is a **non-trivial** code change ŌĆö `tryForwardEventToChild()` currently takes a single `StateDefinition`, but in parallel mode we need to check all active leaf states.

### 8. Child Machine Itself Has Parallel States

A child machine is a regular machine ŌĆö nothing prevents it from having parallel states internally:

```php
// Child machine definition
'states' => [
    'verifying' => [
        'type' => 'parallel',
        'states' => [
            'identity' => [...],
            'credit'   => [...],
        ],
        '@done' => 'verified',
    ],
    'verified' => ['type' => 'final'],
],
```

**Forward to parallel child:** When the parent forwards `PROVIDE_CARD` to this child, `tryForwardEventToChild()` restores the child and calls `$childMachine->send(...)`. Inside the child's `send()`, if the child is in a parallel state, `transitionParallelState()` handles it. This should work correctly ŌĆö the child processes the event within its own parallel state machine. **No issue here** as long as the child's parallel state has transitions for the forwarded event.

**Potential issue:** If the child has a parallel state where different regions handle different events, but the forwarded event only matches one region, the child's `transitionParallelState()` ŌåÆ `selectTransitions()` finds the matching region and processes it. This works because parallel event handling already routes to the correct region.

### 9. Nested Delegation (Grandchild)

Parent ŌåÆ Child ŌåÆ Grandchild:

```php
// Parent machine
'processing' => [
    'machine' => ChildMachine::class,
    'queue'   => 'default',
    'forward' => ['PROVIDE_CARD'],
    '@done'   => 'completed',
],

// Child machine
'payment' => [
    'machine' => GrandchildMachine::class,
    'queue'   => 'default',
    'forward' => ['PROVIDE_CARD'],
    '@done'   => 'paid',
],
```

When parent forwards `PROVIDE_CARD` to child, `tryForwardEventToChild()` calls `$childMachine->send(PROVIDE_CARD)`. Inside the child's `send()`, since the child itself has a `forward` config, it attempts `tryForwardEventToChild()` on the grandchild.

**This should chain naturally** ŌĆö each level's `send()` independently handles forwarding. However:
- Parent's HTTP endpoint resolves the event using child's EventBehavior (one level down)
- The child's `tryForwardEventToChild()` resolves the same event to the grandchild
- The response chain: grandchild state ŌåÆ child's `forwardedChildState` ŌåÆ parent's `forwardedChildState`

**Issue:** The parent's `forwardedChildState` would be the child's state (which hasn't changed because the child is still in its delegating state). The grandchild's state is stashed on the child's state, not the parent's. The HTTP response would show `child.value = "payment"` but not the grandchild's state.

**Decision:** Nested forward is an edge case. For v1, document that forward works one level deep. Multi-level forward response chaining can be addressed in a follow-up if demand exists.

### 10. Parent Cancels While Child Is Running (Orphan Risk)

When the parent has both `forward` and `on` events on the delegating state:

```php
'processing' => [
    'machine' => PaymentMachine::class,
    'queue'   => 'default',
    'forward' => ['PROVIDE_CARD', 'CONFIRM_PAYMENT'],
    'on'      => ['CANCEL' => 'cancelled'],  // ŌåÉ parent can leave!
    '@done'   => 'completed',
],
```

User can: (1) forward PROVIDE_CARD to child, (2) send CANCEL to parent. Parent moves to `cancelled`, child is orphaned in `awaiting_confirmation`.

`ChildMachineCompletionJob` has an idempotency check (line 77):
```php
if ($parentMachine->state->currentStateDefinition->id !== $this->parentStateId) {
    return;  // silently discards ŌĆö parent moved on
}
```

So the child's completion is silently ignored. This is **by design** ŌĆö the parent already moved on. But the developer must design for this scenario using one of these patterns:

| Pattern | Implementation | When to use |
|---------|---------------|-------------|
| Guard | `'guards' => 'noActiveChildGuard'` on CANCEL transition | Child must complete before cancel |
| Exit action | `'exit' => 'cancelChildAction'` on delegating state | Child is cancellable |
| Compensation | `'entry' => 'compensateAction'` on cancelled state | Side effects need rollback |
| Accept orphan | No special handling | Child is harmless (logging, audit) |

**Documentation required** in best practices: "When combining `forward` with `on` events on a delegating state, always design an orphan strategy."

## Migration Path

### Before (Manual Duplication)

```php
MachineDefinition::define(
    config: [
        'states' => [
            'processing' => [
                'machine' => PaymentMachine::class,
                'queue'   => 'default',
                'forward' => ['PROVIDE_CARD', 'CONFIRM_PAYMENT'],
                '@done'   => 'completed',
            ],
        ],
    ],
    behavior: [
        'events' => [
            'START'           => StartEvent::class,
            'PROVIDE_CARD'    => ProvideCardEvent::class,    // duplicate
            'CONFIRM_PAYMENT' => ConfirmPaymentEvent::class, // duplicate
        ],
    ],
    endpoints: ['START', 'PROVIDE_CARD', 'CONFIRM_PAYMENT'], // duplicate
);
```

### After (Forward Only)

```php
MachineDefinition::define(
    config: [
        'states' => [
            'processing' => [
                'machine' => PaymentMachine::class,
                'queue'   => 'default',
                'forward' => ['PROVIDE_CARD', 'CONFIRM_PAYMENT'],
                '@done'   => 'completed',
            ],
        ],
    ],
    behavior: [
        'events' => [
            'START' => StartEvent::class,
            // No child events needed here
        ],
    ],
    endpoints: ['START'], // Only parent's own endpoints
);
```

The `forward` key auto-discovers `ProvideCardEvent` and `ConfirmPaymentEvent` from `PaymentMachine::definition()->behavior['events']`.

### Breaking Change ŌĆö Migration Required

The "before" pattern (manual duplication) is **no longer valid**. The validator rejects forward events that also appear in `endpoints` or `behavior['events']`. Users must remove the duplicates:

1. Remove child event types from parent's `behavior['events']`
2. Remove child event types from parent's `endpoints`
3. Ensure `forward` config is the only place child events are referenced

If endpoint customization is needed (URI, method, middleware, action, result), use Format 3 on the `forward` config instead of the old `endpoints` array config.

**Upgrade guide entry required** in `docs/getting-started/upgrading.md`.

### Doc 5: Update `docs/getting-started/upgrading.md`

**What:** Add a "Forward-Aware Endpoints (Breaking Change)" section to the upgrade guide.

**Content:**
- **What changed:** Forward events are now auto-discovered from child machine definitions. The "manual duplication" pattern (declaring child events in parent's `behavior['events']` and `endpoints`) is no longer valid.
- **Validator now rejects overlap:** If a forward event appears in both `forward` config and parent's `endpoints` or `behavior['events']`, a parse-time error is thrown with a clear migration message.
- **Migration steps:**
  1. Find all parent machines that use `forward` config
  2. For each forwarded event type, remove it from parent's `behavior['events']`
  3. Remove it from parent's `endpoints` array
  4. If you had endpoint customization (action, result, middleware) on the old explicit endpoint, move it to Format 3 on the `forward` config:
     ```php
     // Before (old pattern)
     behavior: ['events' => ['PROVIDE_CARD' => ProvideCardEvent::class]],
     endpoints: ['PROVIDE_CARD' => ['action' => LockAction::class, 'result' => CardResult::class]],
     'processing' => ['forward' => ['PROVIDE_CARD'], ...],

     // After (new pattern)
     // behavior.events: REMOVED PROVIDE_CARD
     // endpoints: REMOVED PROVIDE_CARD
     'processing' => [
         'forward' => [
             'PROVIDE_CARD' => [
                 'action' => LockAction::class,
                 'result' => CardResult::class,
             ],
         ],
         ...
     ],
     ```
  5. Update tests: response format changed from parent-only to parent + child state
- **Response format change:**
  ```
  // Before: parent state only
  { "data": { "machine_id": "...", "value": "processing", "context": {...} } }

  // After: parent + child state
  { "data": { "machine_id": "...", "value": "processing", "child": { "value": "card_provided", "context": {...} } } }
  ```
- **Why:** Eliminates ambiguity (which EventBehavior validates? which response format?), enforces `forward` as single source of truth, removes duplicate code paths.

### Doc 7: New page `docs/laravel-integration/available-events.md`

**What:** Dedicated documentation page for `available_events` ŌĆö this is a cross-cutting feature that deserves its own page rather than being buried in the endpoints page.

**Content:**
- **Concept:** HATEOAS-style event discoverability ŌĆö machine tells the consumer what it can do next
- **Core API:**
  - `$machine->availableEvents()` ŌĆö convenience proxy
  - `$state->availableEvents()` ŌĆö core method
  - `$state->toArray()` ŌĆö automatically includes `available_events`
- **HTTP Response format:**
  - `available_events` array with `type`, `source`, and optional `region` fields
  - Response examples for: idle state, delegating state with forward, parallel state, final state
  - Opt-out: `'available_events' => false` in endpoint/forward config
  - Not included when `result` (ResultBehavior) is specified
- **Source annotations:**
  | Source | Meaning |
  |--------|---------|
  | `parent` | Direct `on` event on current state |
  | `forward` | Forwarded to running child machine |
  - Parallel state: `region` field added to disambiguate
- **Forward-aware behavior:**
  - Forward events only included when child's current state accepts them
  - PROVIDE_CARD disappears after child moves past `awaiting_card`
  - Performance: uses `MachineCurrentState` table for lightweight child state lookup
- **Testing:**
  - `assertAvailableEvent('PROVIDE_CARD')` ŌĆö event is sendable
  - `assertNotAvailableEvent('START')` ŌĆö event is not sendable
  - `assertAvailableEvents(['PROVIDE_CARD', 'CANCEL'])` ŌĆö exact set
  - `assertForwardAvailable('PROVIDE_CARD')` ŌĆö forward event specifically
  - `assertNoAvailableEvents()` ŌĆö final state
- **Eloquent integration:**
  ```php
  // In Blade template:
  @foreach($order->order_mre->availableEvents() as $event)
      <button>{{ $event['type'] }}</button>
  @endforeach
  ```
- **What's excluded:** `@always`, `@done`, `@fail`, `@timeout`, timer events (after/every) ŌĆö these are internal/time-triggered, not user-sendable
- **Guard handling:** Events with guards are included ŌĆö guards are runtime, not static. Frontend tries to send, machine rejects if guard fails.

## Resolved Questions

1. ~~**Should forwarded endpoints support `action` (MachineEndpointAction)?**~~ **Yes.** Format 3 includes `action` key. The parent's `MachineEndpointAction` runs `before()`/`after()`/`onException()` in the HTTP layer, receiving the parent machine and state. This enables rate limiting, audit logging, cache locks on forwarded endpoints.

2. ~~**Should parent be able to customize child endpoint properties?**~~ **Yes.** Format 3 includes `uri`, `method`, `middleware` overrides. The parent can completely customize how a child event is exposed without touching the child machine definition.

3. ~~**Should parent be able to transform child responses?**~~ **Yes.** The `result` key in forward config points to a **parent-level** ResultBehavior that receives both `ContextManager $context` (parent) and `ContextManager $childContext` + `State $childState` (child) via parameter injection. This lets the parent merge, transform, or enrich child responses.

4. ~~**Should forward support FQCN class references?**~~ **Yes.** All three forward formats support FQCN event class references, resolved via `EndpointDefinition::resolveEventType()` at parse time. This is consistent with `endpoints`, `machineIdFor`, `modelFor`, and `schedules`.

5. ~~**Is a reverse of `forward` needed?**~~ **No.** `sendToParent()` (mid-lifecycle) and `@done`/`@fail` (completion) + `output` key (data filtering) already cover childŌåÆparent communication. No new mechanism needed.

6. ~~**Should `available_events` be part of this plan?**~~ **Yes.** Core on `State::availableEvents()`, surfaced in HTTP responses, TestMachine assertions, `toArray()` serialization, and a dedicated docs page.

7. ~~**XState export**~~ **Deferred to follow-up.** Forward event list already exported via `invoke.meta.eventMachine.forward`. Endpoint-specific metadata (URI, method) is HTTP concern, not XState semantics.

8. ~~**`InvokableBehavior` injection: named params vs wrapper class?**~~ **`ForwardContext` wrapper class.** Type-safe, no convention-based naming fragility, single `ContextManager` ŌåÆ always parent (no ambiguity). See Step 7 details.

9. ~~**Should forwarded endpoints inherit parent's binding mode?**~~ **Yes.** If `model` + `attribute` are configured on the router ŌåÆ model-bound. Otherwise ŌåÆ machineId-bound. Two separate handlers: `handleForwardedModelBound` / `handleForwardedMachineIdBound`, each resolving the parent machine via the appropriate mechanism before delegating to shared `executeForwardedEndpoint()` logic.

10. ~~**Should there be a `forward_prefix` for grouping child endpoint URIs?**~~ **No (v1).** Format 3 `uri` override per-event already covers the need. A `forward_prefix` key would be syntactic sugar for applying a common prefix to all auto-generated URIs ŌĆö nice-to-have but not essential. Can be added in a follow-up if multiple forward events per state is common in practice.

    ```php
    // Instead of forward_prefix, use Format 3 per-event:
    'forward' => [
        'PROVIDE_CARD'    => ['uri' => '/payment/provide-card'],
        'CONFIRM_PAYMENT' => ['uri' => '/payment/confirm-payment'],
    ],
    // Equivalent to a hypothetical forward_prefix: '/payment'
    ```

## `available_events` ŌĆö Machine Event Introspection

### Problem

API consumers (frontend, mobile) must hardcode state ŌåÆ valid events mapping. When the machine definition changes, the frontend must be updated in sync. This tight coupling breaks independent deployability.

But `available_events` is NOT just an HTTP concern ŌĆö it's a **core machine introspection capability** that applies across multiple layers:

| Layer | Consumer | Use Case |
|-------|----------|----------|
| **HTTP Endpoints** | Frontend, mobile apps | HATEOAS ŌĆö show valid buttons/forms dynamically |
| **State object** | PHP code calling `$machine->send()` | Check if an event is valid before sending |
| **TestMachine** | Tests | Assert which events are available in a given state |
| **Eloquent/MachineCast** | Laravel models with `HasMachines` | `$order->order_mre->availableEvents()` in Blade |
| **Broadcasting** | WebSocket/Livewire | Push available events to frontend on state change |
| **XState export** | Stately Studio visualization | Show valid events per state in diagrams |
| **Artisan commands** | CLI debugging | `machine:status {id}` shows current state + available events |

### Design: Core on `State`, Surface on Multiple Layers

The resolution logic lives on `State` (core), not on `MachineController` (HTTP). This makes it available everywhere:

```
State::availableEvents()           ŌåÉ core method (returns array)
  Ōö£ŌöĆŌöĆ MachineController            ŌåÉ adds to HTTP response
  Ōö£ŌöĆŌöĆ TestMachine                  ŌåÉ assertAvailableEvent(), assertNotAvailableEvent()
  Ōö£ŌöĆŌöĆ Machine::availableEvents()   ŌåÉ convenience proxy
  Ōö£ŌöĆŌöĆ State::toArray()             ŌåÉ included in serialization
  ŌööŌöĆŌöĆ Artisan commands             ŌåÉ displayed in timer-status, etc.
```

### Solution: Multi-Layer `available_events`

#### Layer 1: Core ŌĆö `State::availableEvents()`

Every endpoint response includes the list of events that are **currently valid** for the machine's state:

```json
{
    "data": {
        "machine_id": "abc123",
        "value": "payment_processing",
        "child": {
            "value": "awaiting_card",
            "context": { ... }
        },
        "available_events": [
            { "type": "PROVIDE_CARD", "source": "forward" },
            { "type": "CANCEL", "source": "parent" }
        ]
    }
}
```

Frontend renders UI purely from `available_events` ŌĆö no state knowledge needed.

### Design

#### What's Included in `available_events`

For a given machine state, collect:

1. **Parent's own `on` transitions** ŌĆö events defined on the current state's `on` config
2. **Forward events** ŌĆö if current state has `forward` config and a running child, include forwarded events
3. **Internal events excluded** ŌĆö `@always`, `@done`, `@fail`, `@timeout` are internal, not user-sendable
4. **Timer events excluded** ŌĆö `after`/`every` are time-triggered, not user-sendable

```
available_events = parent_on_events(non-internal) + forward_events(if child running)
```

#### Source Annotation

Each event has a `source` field:

| Source | Meaning |
|--------|---------|
| `parent` | Direct event on parent's current state |
| `forward` | Forwarded to running child machine |

This helps the frontend understand the event's nature without knowing the machine definition.

#### Guard Handling ŌĆö Static vs Dynamic

Guards present a challenge: an event may have a guard that evaluates to `false` at runtime. Should the event be in `available_events`?

**Decision: Include all events with transitions, regardless of guards.** Rationale:
- Guards often depend on request data (not just state) ŌĆö can't evaluate statically
- Showing the event and getting a 422 guard rejection is better UX than hiding it
- The frontend can always try to send the event; the machine will reject if guard fails
- Alternative: `"guarded": true` annotation ŌĆö deferred (adds complexity, questionable value)

#### Parallel State Handling

When parent is in a parallel state, collect events from **all active regions**:

```
fulfillment (parallel)
  Ōö£ŌöĆŌöĆ payment.awaiting_card  ŌåÆ on: [PROVIDE_CARD, CANCEL_PAYMENT]
  ŌööŌöĆŌöĆ documents.uploading    ŌåÆ on: [UPLOAD_DOCUMENT]

available_events = [PROVIDE_CARD, CANCEL_PAYMENT, UPLOAD_DOCUMENT]
```

If any region has `forward` config, its forwarded events are included too.

#### Result Response Opt-Out

When `result` (ResultBehavior) is specified on an endpoint, the response is **fully custom** ŌĆö no `available_events` is added. The ResultBehavior controls the entire response. If the developer wants `available_events` in a custom result, they can compute it themselves via the injected State.

```
Endpoint with result ŌåÆ { "data": <ResultBehavior output> }     ŌåÉ NO available_events
Endpoint without result ŌåÆ { "data": { ..., "available_events": [...] } }  ŌåÉ included
```

#### Forwarded Endpoint Response

For forwarded endpoints, `available_events` is collected from the **parent's perspective** (not child's):

```
Parent in payment_processing, child in awaiting_card:

available_events = [
    { "type": "PROVIDE_CARD", "source": "forward" },     // from forward config
    { "type": "CONFIRM_PAYMENT", "source": "forward" },  // from forward config
    { "type": "CANCEL", "source": "parent" },             // from parent's on config
]
```

After forwarding PROVIDE_CARD, child moves to `awaiting_confirmation`:

```
available_events = [
    { "type": "CONFIRM_PAYMENT", "source": "forward" },  // still forwardable
    { "type": "CANCEL", "source": "parent" },             // still sendable
]
// PROVIDE_CARD is gone ŌĆö child no longer in awaiting_card state
```

Wait ŌĆö how do we know PROVIDE_CARD is no longer valid? The parent is still in `payment_processing` and `forward` config still lists PROVIDE_CARD. The parent doesn't know the child's current state.

**Solution:** For `source: "forward"` events, check if the child's current state actually accepts that event:

```php
foreach ($forwardEvents as $eventType) {
    $childEventType = $invokeDefinition->resolveForwardEvent($eventType);
    // Check if child's current state has a transition for $childEventType
    if ($childState !== null && $childState->currentStateDefinition->events !== null
        && in_array($childEventType, $childState->currentStateDefinition->events, true)) {
        $availableEvents[] = ['type' => $eventType, 'source' => 'forward'];
    }
}
```

This requires knowing the child's current state ŌĆö which we already have from `forwardedChildState` or a fresh restore.

**Performance concern:** Restoring the child machine just for `available_events` is expensive. Optimization: use `MachineCurrentState` table (denormalized) to get the child's current state ID without full restore, then look up events from the child's definition.

## `ForwardContext` Wrapper Class

### Design

**File:** `src/Routing/ForwardContext.php` (new)

```php
namespace Tarfinlabs\EventMachine\Routing;

use Tarfinlabs\EventMachine\Actor\State;
use Tarfinlabs\EventMachine\ContextManager;

/**
 * Wrapper that carries child machine state for forward endpoint ResultBehavior injection.
 *
 * Type-hint this in a ResultBehavior's __invoke() to access the child's
 * context and state when processing a forwarded endpoint response.
 * Only available in forward endpoint context ŌĆö not injected in regular endpoints.
 */
class ForwardContext
{
    public function __construct(
        public readonly ContextManager $childContext,
        public readonly State $childState,
    ) {}
}
```

### Injection Mechanism

**File:** `src/Behavior/InvokableBehavior.php` ŌĆö update `injectInvokableBehaviorParameters()`

Add a new match arm for `ForwardContext`:

```php
$value = match (true) {
    $typeName === null                                  => null,
    is_a($typeName, ContextManager::class, true) || ... => $state->context,    // ContextManager ŌåÆ parent
    is_a($typeName, EventBehavior::class, true) || ...  => $eventBehavior,     // EventBehavior
    is_a($typeName, ForwardContext::class, true)        => $forwardContext,     // ForwardContext ŌåÆ child (NEW)
    $state instanceof $typeName                         => $state,             // State ŌåÆ parent
    is_a($state->history, $typeName)                    => $state->history,    // EventCollection
    $typeName === 'array'                               => $actionArguments,   // array
    default                                             => null,
};
```

The `$forwardContext` parameter is passed into `injectInvokableBehaviorParameters()`:

```php
public static function injectInvokableBehaviorParameters(
    callable $actionBehavior,
    State $state,
    ?EventBehavior $eventBehavior = null,
    ?array $actionArguments = null,
    ?ForwardContext $forwardContext = null,      // NEW ŌĆö null in non-forward context
): array
```

### Usage in ResultBehavior


### Safety

- **Regular endpoint:** `$forwardContext` is `null` ŌåÆ if ResultBehavior type-hints `ForwardContext`, `null` is injected ŌåÆ PHP TypeError if not nullable. This is intentional ŌĆö a ResultBehavior that expects forward context should only be used on forward endpoints.
- **Forward endpoint:** `$forwardContext` is constructed from child state ŌåÆ injection works.
- **No ambiguity:** `ContextManager` always means parent. `ForwardContext` always means child. No naming conventions needed.
