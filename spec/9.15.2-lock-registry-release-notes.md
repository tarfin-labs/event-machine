# Release Notes — 9.15.2 Process-Local Lock Registry Centralization

Bug-fix release. No consumer changes required; upgrade with `composer update tarfin-labs/event-machine`.

## Fixed

- **Jobs that acquire a machine lock now register it in the process-local re-entrancy registry.** `Machine::send()` treats `Machine::$heldLockIds` as the record of which machine locks the current process holds, but `ChildMachineCompletionJob` and `ListenerJob` acquired locks without ever registering them. On the sync queue (`QUEUE_CONNECTION=sync` — the recommended setup for browser e2e since 9.15.0), any job dispatched inline while a completion job held the parent's lock — the next delegation's completion, a queued listener — blocked on a lock held by its own call stack, burned the full 10–30s lock timeout, and resolved as a spurious `@fail`: a held lock was reported as a failed operation and the machine ended in a wrong terminal state.

  Registration and deregistration now live inside `MachineLockManager::acquire()` and `MachineLockHandle::release()`, so every acquirer — `send()`, `ChildMachineCompletionJob`, `ListenerJob`, the timeout and parallel-region jobs — participates in re-entrancy detection by construction.

  <!-- doctest-attr: ignore -->
  ```php
  // Chained delegation on the sync queue: step_one's @done invokes step_two inline.
  // 9.15.1: step_two's completion deadlocks 30s on the parent's own lock → spurious @fail
  // 9.15.2: detected as re-entrant → the chain completes in milliseconds
  $machine = ChainedJobParentMachine::create();
  $machine->send(['type' => 'START']);   // → chained_job_parent.completed
  ```

- **`ListenerJob` releases its lock null-safely on the re-entrant path.** When the lock is already held higher in the current call stack, no handle is acquired; the `finally` block no longer calls `release()` on `null`.

- **`InteractsWithMachines` resets `Machine::$heldLockIds` between tests.** A test that acquires a lock without releasing it can no longer leak re-entrancy state into later tests in the same process.

Docs: https://eventmachine.dev/laravel-integration/persistence#re-entrant-lock-support
