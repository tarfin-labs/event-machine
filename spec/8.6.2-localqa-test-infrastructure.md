# LocalQA Test Infrastructure Overhaul

**Status:** Plan
**Date:** 2026-03-25
**Scope:** Make LocalQA tests deterministic, remove all sleep(), automate Horizon lifecycle

---

## Problems

### P1: Cross-test job pollution
A `ChildMachineTimeoutJob` dispatched with 30s delay in test A fires during test B, writing to freshly-truncated tables.

**Root cause:** `cleanTables()` drains pending/delayed queues but cannot stop in-flight (reserved) jobs. Worker finishes old job → writes to new test's tables → flaky assertion.

### P2: sleep() in 7 places
`sleep(1-3)` to "wait for Horizon". Non-deterministic — sometimes too short (flaky), sometimes too long (slow).

### P3: Manual Horizon lifecycle
`pkill -9 -f horizon`, `redis-cli FLUSHALL`, start Horizon, wait, run tests, kill. Error-prone.

### P4: No failure diagnostics
`waitFor()` timeout → "Expected true, got false". No queue state, no pending jobs, no clue what went wrong.

---

## Architecture Principles

1. **Each test sees a clean world** — empty tables, empty queues, idle workers.
2. **Never sleep, always wait for a condition** — `waitFor(condition)` is faster and reliable.
3. **Fail with context** — on timeout, dump queue sizes, last events, failed jobs, locks.

## Changes

### 1. `cleanTables()` — Guarantee clean slate

**Current:** Drain all queues (including reserved) → check reserved zset → truncate.
**Problem:** Deleting reserved stops tracking in-flight jobs. Worker still runs, writes to DB after truncate.

**New flow:** Drain pending + delayed only. Do NOT touch reserved — let workers finish naturally. Wait for DB write activity to settle. Then truncate.

```php
public static function cleanTables(): void
{
    $redis  = app('redis');
    $prefix = config('database.redis.options.prefix', 'laravel_database_');

    // 1. Drain pending + delayed (stop NEW jobs from being picked up).
    //    Do NOT delete reserved — workers will finish those naturally.
    foreach (['default', 'child-queue'] as $queue) {
        $redis->del("{$prefix}queues:{$queue}");
        $redis->del("{$prefix}queues:{$queue}:delayed");
        $redis->del("{$prefix}queues:{$queue}:notify");
    }

    // 2. Wait for chain settlement: no new DB writes for 500ms.
    //    Catches in-flight reserved jobs AND chained dispatches
    //    (e.g., ChildMachineJob → ChildMachineCompletionJob).
    static::waitForQuietPeriod(quietMs: 500, timeoutSeconds: 15);

    // 3. Truncate — safe because no jobs are writing
    DB::statement('SET FOREIGN_KEY_CHECKS=0;');
    DB::table('machine_events')->truncate();
    DB::table('machine_current_states')->truncate();
    DB::table('machine_timer_fires')->truncate();
    DB::table('machine_children')->truncate();
    DB::table('machine_locks')->truncate();
    DB::table('job_batches')->truncate();
    if (DB::getSchemaBuilder()->hasTable('jobs')) {
        DB::table('jobs')->truncate();
    }
    if (DB::getSchemaBuilder()->hasTable('failed_jobs')) {
        DB::table('failed_jobs')->truncate();
    }
    DB::statement('SET FOREIGN_KEY_CHECKS=1;');

    // 4. NOW drain reserved (workers finished, safe to clean up)
    foreach (['default', 'child-queue'] as $queue) {
        $redis->del("{$prefix}queues:{$queue}:reserved");
    }
}
```

**Key difference from current:** Reserved set deleted AFTER truncate (step 4), not before. Workers finish their jobs → write to DB → quiet period detects settlement → truncate → clean reserved.

### 2. `waitForQuietPeriod()` — DB-based settlement detection

```php
private static function waitForQuietPeriod(int $quietMs, int $timeoutSeconds): void
{
    $start      = time();
    $quietAccum = 0;
    $lastCount  = DB::table('machine_events')->count() + DB::table('failed_jobs')->count();

    while (time() - $start < $timeoutSeconds) {
        usleep(50_000); // 50ms poll

        $currentCount = DB::table('machine_events')->count() + DB::table('failed_jobs')->count();
        if ($currentCount === $lastCount) {
            $quietAccum += 50;
            if ($quietAccum >= $quietMs) {
                return;
            }
        } else {
            $quietAccum = 0;
            $lastCount  = $currentCount;
        }
    }

    // Timeout — workers may still be writing. Log warning.
    fwrite(STDERR, "[cleanTables] waitForQuietPeriod timed out after {$timeoutSeconds}s — truncating anyway\n");
}
```

**Why `machine_events + failed_jobs` count is sufficient:** Every EventMachine state change writes to `machine_events`. If a job fails and its `failed()` handler also fails, only `failed_jobs` gets written. Checking both tables catches all activity.

**Performance:** Best case 500ms, worst case 15s. With 68 tests + afterEach: 68 × 0.5s = 34s overhead. Acceptable for determinism guarantee.

### 3. `waitFor()` — Exponential backoff + always-on diagnostics

```php
public static function waitFor(
    callable $condition,
    int $timeoutSeconds = 45,
    string $description = '',
): bool {
    $start      = time();
    $intervalMs = 100;

    while (time() - $start < $timeoutSeconds) {
        if ($condition()) {
            return true;
        }
        usleep($intervalMs * 1000);
        $intervalMs = min((int) ($intervalMs * 1.5), 1000); // cap 1s
    }

    // Always dump diagnostics on timeout (no opt-in needed)
    static::dumpDiagnostics($description);

    return false;
}
```

**Changes from current:**
- Exponential backoff: 100ms → 150ms → 225ms → ... → 1s cap. Reduces DB pressure.
- Diagnostics dump on EVERY timeout (not optional).
- `$description` is a string with default empty (not nullable) — callers can add context.

### 4. `dumpDiagnostics()` — Actionable failure context

```php
private static function dumpDiagnostics(string $description): void
{
    $redis  = app('redis');
    $prefix = config('database.redis.options.prefix', 'laravel_database_');

    $diagnostics = [
        'description'    => $description ?: '(no description)',
        'machine_events' => DB::table('machine_events')->count(),
        'last_5_events'  => DB::table('machine_events')
            ->orderByDesc('sequence_number')
            ->limit(5)
            ->pluck('type')
            ->toArray(),
        'current_states' => DB::table('machine_current_states')
            ->pluck('state_id')
            ->toArray(),
        'children'       => DB::table('machine_children')
            ->get(['status', 'child_machine_class'])
            ->toArray(),
        'locks'          => DB::table('machine_locks')->count(),
        'failed_jobs'    => DB::table('failed_jobs')
            ->limit(3)
            ->pluck('exception')
            ->map(fn ($e) => mb_substr((string) $e, 0, 200))
            ->toArray(),
    ];

    foreach (['default', 'child-queue'] as $queue) {
        $diagnostics["queue:{$queue}:pending"]  = (int) $redis->llen("{$prefix}queues:{$queue}");
        $diagnostics["queue:{$queue}:delayed"]  = (int) $redis->zcard("{$prefix}queues:{$queue}:delayed");
        $diagnostics["queue:{$queue}:reserved"] = (int) $redis->zcard("{$prefix}queues:{$queue}:reserved");
    }

    fwrite(STDERR, "\n[waitFor TIMEOUT] ".json_encode($diagnostics, JSON_PRETTY_PRINT)."\n");
}
```

**Key additions vs original plan:**
- `last_5_events`: Last 5 event types (not just count) — shows what happened before timeout.
- `failed_jobs`: Actual exception message (first 200 chars) — shows WHY a job failed.

### 5. Remove all `sleep()` calls

| Test Name | Current | Replacement |
|-----------|---------|-------------|
| `TimerSweepTest: after timer does NOT fire before deadline` | `sleep(2)` | Keep `sleep(1)` — negative assertion, cannot `waitFor` absence (see special case below) |
| `TimerSweepTest: after timer dedup` | `sleep(2)` | `waitFor` fire record status == fired |
| `ParallelDispatchTest: parallel regions complete via events` | `sleep(1)` | `waitFor` machine_events count increased after dispatch |
| `ReviewFixesTest: completion job succeeds with retry config` | `sleep(3)` | `waitFor` child status == running |
| `ReviewFixesTest: every timer does not double-fire` | `sleep(2)` | `waitFor` fire record last_fired_at updated |
| `ReviewFixesTest: forward event routing delivers APPROVE` | `sleep(3)` | `waitFor` MachineChild record created |
| `FireAndForgetQATest: fire-and-forget child does not dispatch completion job` | `sleep(2)` | `waitFor` child machine_events exist for child root_event_id |

**Special case:** `after timer does NOT fire before deadline` uses `sleep(2)` to verify something does NOT happen. This cannot be replaced with `waitFor` (you can't wait for absence). Keep a short sleep (1s) + assertion that state hasn't changed. Document why.

### 6. `run-qa.sh` — Automated Horizon lifecycle

```bash
#!/bin/bash
set -e

SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
PACKAGE_DIR="$(cd "$SCRIPT_DIR/../.." && pwd)"
QA_PROJECT="${QA_PROJECT:-/tmp/qa-v7-review}"

# Preflight checks
mysql -u root -e "SELECT 1" > /dev/null 2>&1 || { echo "MySQL not running"; exit 1; }
redis-cli PING > /dev/null 2>&1 || { echo "Redis not running"; exit 1; }
[ -f "$QA_PROJECT/artisan" ] || { echo "QA project not found at $QA_PROJECT"; exit 1; }

# Kill old Horizon
pkill -9 -f horizon 2>/dev/null || true
sleep 1

# Flush Redis
redis-cli FLUSHALL > /dev/null

# Start Horizon
cd "$QA_PROJECT" && php artisan horizon &
HORIZON_PID=$!

# Wait for Horizon workers to register
sleep 5

# Run tests
cd "$PACKAGE_DIR"
vendor/bin/pest tests/LocalQA/ "$@"
EXIT_CODE=$?

# Cleanup
kill "$HORIZON_PID" 2>/dev/null
wait "$HORIZON_PID" 2>/dev/null || true

exit $EXIT_CODE
```

**Fixes from original plan:** No hardcoded paths. `PACKAGE_DIR` derived from script location. `QA_PROJECT` configurable via env var.

### Bug: ListenerJob action not wrapped in transaction

**Location:** `ListenerJob::handle()` (src/Jobs/ListenerJob.php)
**Problem:** Queued listener action modifies context → throws mid-execution → context changes persisted without `LISTEN_QUEUE_COMPLETED` event. On retry, action runs again with already-modified context.
**Impact:** Non-idempotent listener actions execute multiple times with corrupted context.

**Fix:** Wrap listener action execution + persist in `DB::transaction()`. On throw, transaction rolls back, retry sees original context.

**TDD:**
1. Write test: listener action that increments counter + throws on first call → expect counter=1 after retry, not counter=2
2. If test PASSES without fix → bug is not present (persist() not called before throw). Update plan, skip fix.
3. If test FAILS → add DB::transaction wrapper, verify test passes.

**Note:** The bug may not manifest because `persist()` is called AFTER the action. If action throws before persist, in-memory changes are lost and retry starts fresh. The TDD step will determine if the fix is needed.

**Dependency:** C3 (Listener QA tests) should run after this is resolved (either fixed or confirmed not-a-bug).
