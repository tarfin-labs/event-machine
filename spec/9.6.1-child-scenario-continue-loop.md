# Fix: executeChildScenario — @continue Loop + Child Persistence

**Status:** Draft
**Date:** 2026-04-02

## Problem

`ScenarioPlayer::executeChildScenario()` has two bugs that together make child scenario references unusable for real QA flows:

### Bug 1: Missing @continue Loop (FIXED in current branch)

`executeChildScenario()` only calls `$childMachine->start()`, which runs @always chains and outcome interception but stops at the first interactive state. No `@continue` loop — unlike `execute()` and `executeContinuation()`.

**Status:** Fixed. TDD test passing. 3 unit tests + 1 QA test written.

### Bug 2: Child Not Persisted When Pausing at Interactive State

When the child scenario pauses at an interactive state (returns `null`), the child machine was created with `shouldPersist=false`. Nothing is written to DB:
- No `machine_children` record (parent-child link)
- No `machine_current_states` record (child state)
- No `machine_events` (child history)

Forward endpoints depend on `machine_children.child_root_event_id` to restore the child. Without DB records, QA cannot interact with the paused child — PinConfirmedEvent via forward endpoint fails because there's no child to restore.

**TDD test written and FAILING:** `tests/Features/ScenarioPlayerChildScenarioTest.php` — "child scenario pausing at interactive state persists child to DB".

### The Findeks Flow

```
Parent (CarSalesMachine):
  plan: 'verification.findeks.running' => AtAwaitingPinFromStartScenario::class

Child (FindeksMachine) via executeChildScenario:
  idle → querying_phones(@done) → awaiting_report_request(@continue) → requesting(@done) → polling(@done) → awaiting_pin
  ↑ child pauses here — QA sends PinConfirmedEvent via forward endpoint

Current behavior: child reaches awaiting_pin in-memory but is volatile — no DB records.
Expected: child persisted at awaiting_pin, machine_children record created, forward endpoints work.
```

## Proposed Solution

When `executeChildScenario()` completes and the child is at an interactive state (not final):

1. Switch child to `shouldPersist = true`
2. Call `$childMachine->persist()`
3. Create `machine_children` record linking parent to child
4. Persist child's `scenario_class` + `scenario_params` if the child scenario has `continuation()`

### Key Design Decision: Persist Only on Pause

- Child runs entirely with `shouldPersist=false` during the @continue loop (performance: no DB writes for transient states)
- Persistence happens only at the end, when the child pauses at an interactive state
- If child reaches a FINAL state, no persistence needed — parent routes @done immediately

## Implementation

### `ScenarioPlayer::executeChildScenario()` — Add Persistence on Pause

After the @continue loop, before returning null:

```php
// Child paused at interactive state — persist for forward endpoint access
$childMachine->definition->shouldPersist = true;
$childMachine->persist();

$childRootEventId = $childMachine->state->history->first()->root_event_id;

// Create parent-child tracking record
MachineChild::create([
    'parent_root_event_id' => $parentRootEventId,
    'parent_machine_class' => $parentMachineClass,
    'parent_state_id'      => $parentStateId,
    'child_root_event_id'  => $childRootEventId,
    'child_machine_class'  => $childMachineClass,
]);

// Persist scenario for continuation (if child scenario has continuation())
if ($childScenario->hasContinuation()) {
    MachineCurrentState::where('root_event_id', $childRootEventId)
        ->update([
            'scenario_class'  => $childScenarioClass,
            'scenario_params' => null,
        ]);
}
```

### New Parameters for executeChildScenario

The method needs parent context for the `machine_children` record:

```php
public static function executeChildScenario(
    string $childScenarioClass,
    string $childMachineClass,
    array $input = [],
    ?string $parentRootEventId = null,    // NEW
    ?string $parentMachineClass = null,   // NEW
    ?string $parentStateId = null,        // NEW
): ?State
```

Optional parameters (null = no persistence, backward compatible). When all three are provided AND child pauses, persistence kicks in.

### Caller Update: `handleScenarioOutcome()`

`handleScenarioOutcome()` calls `executeChildScenario()` — it has access to parent context:

```php
$childState = ScenarioPlayer::executeChildScenario(
    childScenarioClass: $childScenario,
    childMachineClass: $invokeDefinition->machineClass,
    input: $childContext,
    parentRootEventId: $state->history->first()?->root_event_id,
    parentMachineClass: $this->machineClass,
    parentStateId: $stateDefinition->id,
);
```

## What Gets Added

| Component | Change |
|-----------|--------|
| `ScenarioPlayer::executeChildScenario()` | Persist child on interactive pause + create machine_children record |
| `MachineDefinition::handleScenarioOutcome()` | Pass parent context to executeChildScenario |

## What Doesn't Change

- Child scenarios reaching FINAL state — no persistence needed, parent routes @done immediately
- @continue loop (already fixed) — unchanged
- Outcome interception — unchanged
- Parent scenario flow — unchanged
- `shouldPersist=false` during child execution — unchanged (persist only at end)
