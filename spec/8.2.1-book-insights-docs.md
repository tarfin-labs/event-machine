## Changes by File

### 1. `docs/best-practices/transition-design.md` — Guard Priority in Multi-Branch Transitions

**Add new section:** "Guard Priority: Errors First"

**Content:**
- In multi-branch guarded transitions (same event or `@always` with multiple branches), EventMachine evaluates guards in array order — first passing guard wins
- Error/failure guards should come BEFORE happy-path fallback
- This does NOT apply to different event keys in the `on` array — each event key is independent and only evaluated when that specific event fires

**Important clarification:** Different event keys (`PAYMENT_CAPTURED`, `PAYMENT_FAILED`) in the same `on` array do NOT compete. Each event targets a specific key. Guard priority only matters for **multi-branch transitions** where the same trigger has multiple possible targets.

**Anti-pattern:**

```php ignore
// Anti-pattern: happy path first in multi-branch @always
'evaluating' => [
    'on' => [
        '@always' => [
            ['target' => 'processing'],                                  // happy path first — WRONG
            ['target' => 'retrying', 'guards' => 'canRetryGuard'],
            ['target' => 'failed', 'guards' => 'hasErrorGuard'],
        ],
    ],
],
```

The fallback (`processing`) matches first, so the guards are never evaluated.

**Fix:**

```php ignore
// Fix: error guards first, happy path fallback last
'evaluating' => [
    'on' => [
        '@always' => [
            ['target' => 'failed', 'guards' => 'hasErrorGuard'],       // error first
            ['target' => 'retrying', 'guards' => 'canRetryGuard'],     // retry second
            ['target' => 'processing'],                                  // happy path last (fallback)
        ],
    ],
],
```

Same principle for guarded transitions on a specific event:

```php ignore
// Multi-branch on same event — error guard first
'PAYMENT_RESULT' => [
    ['target' => 'failed', 'guards' => 'isPaymentDeclinedGuard'],
    ['target' => 'captured'],  // fallback — only if guard above fails
],
```

**Guideline to add:** "In multi-branch transitions, put error and failure guards before the happy-path fallback. The first passing guard wins — defensive ordering ensures failures are caught before the default path."

### 2. `docs/best-practices/time-based-patterns.md` — Timer Guard Rule

**Add new section:** "Timer as Reliability Guard"

**Content:**
- Every state that waits for an external response (webhook, API callback, user input) MUST have an `after` timeout
- This is not a convenience feature — it is a reliability requirement
- Without a timer, the machine can hang indefinitely in a state that no event will ever resolve
- Anti-pattern: waiting state with no timeout
- Fix: add `after` timer with explicit timeout transition

**Anti-pattern:**

```php ignore
'awaiting_webhook' => [
    'on' => [
        'WEBHOOK_RECEIVED' => 'processing',
    ],
    // No timeout — if webhook never arrives, machine hangs forever
],
```

**Fix:**

```php ignore
'awaiting_webhook' => [
    'on' => [
        'WEBHOOK_TIMEOUT'  => ['target' => 'timed_out', 'after' => Timer::hours(1)],
        'WEBHOOK_RECEIVED' => 'processing',
    ],
],
```

**Guideline to add:** "Every state that waits for an external response needs a timer guard. Ask yourself: 'If this response never arrives, what happens?' If the answer is 'the machine hangs forever,' you need an `after` transition."

**Estimated lines:** ~50

### 3. `docs/best-practices/state-design.md` — State Explosion Warning + Error State Pattern

**Add two new sections:**

#### 3a. "Avoid State Explosion"

- Don't encode input combinations in state names
- Use context to track current values, use states for control flow
- Anti-pattern: `payment_valid_stock_available`, `payment_valid_stock_unavailable`, `payment_invalid_stock_available`... (2^n states for n inputs)
- Fix: single `validating` state with guards checking context

**Anti-pattern:**

```php ignore
// 2^n state explosion — encoding input values in state names
'states' => [
    'payment_valid_stock_available'     => [...],
    'payment_valid_stock_unavailable'   => [...],
    'payment_invalid_stock_available'   => [...],
    'payment_invalid_stock_unavailable' => [...],
    // 4 states for 2 binary inputs — grows exponentially
],
```

**Fix:**

```php ignore
// Context stores values, guards check them
'context' => ['payment_valid' => false, 'stock_available' => false],
'states' => [
    'validating' => [
        'on' => [
            '@always' => [
                ['target' => 'ready', 'guards' => 'isPaymentValidAndStockAvailableGuard'],
                ['target' => 'waiting'],
            ],
        ],
    ],
],
```

**Estimated lines:** ~40

#### 3b. "Error States vs Alarm Actions"

- Don't proliferate error states for each failure cause
- Use a single error/failed state per concern area
- Differentiate failure reasons through entry actions and context — not state names

**Anti-pattern:**

```php ignore
// State proliferation — one state per error cause
'payment_gateway_timeout'       => ['type' => 'final'],
'payment_insufficient_funds'    => ['type' => 'final'],
'payment_card_declined'         => ['type' => 'final'],
'payment_fraud_detected'        => ['type' => 'final'],
```

**Fix:**

```php ignore
// Single error state, reason in context
'payment_failed' => [
    'type'  => 'final',
    'entry' => 'recordPaymentFailureAction',  // stores reason in context
],
```

**When to use separate final states instead:** When the machine is a child and the parent needs to route differently based on failure type. In that case, use `@done.{state}` routing — see [Machine System Design: Design Your Child States for the Parent](./machine-system-design#design-your-child-states-for-the-parent).

**The distinction:** Separate states are for **control** (the parent needs to branch). Context/actions are for **information** (logging, notification, debugging). Don't create states for information — create them for control.

**Estimated lines:** ~50

### 4. `docs/best-practices/machine-decomposition.md` — Completeness Rule

**Add one new section + brief cross-reference.**

#### 4a. "The Completeness Rule"

- If you model a domain with a machine, ALL control for that domain must flow through the machine
- Anti-pattern: half the transitions in the machine, half in Eloquent observers or middleware
- The machine definition should be the single source of truth for behavior

**Anti-pattern:**

```php ignore
// Machine handles some transitions...
'submitted' => ['on' => ['PAYMENT_RECEIVED' => 'paid']],
```

```php ignore
// ...but an Eloquent observer handles others — DANGEROUS
// Side effect OUTSIDE the machine — invisible to machine definition
class Order extends Model {
    protected static function booted() {
        static::updated(function ($order) {
            if ($order->status === 'expired') {
                $order->notifyCustomer();
            }
        });
    }
}
```

**Fix:** All control flows through the machine. Use a timer event for expiration:

```php ignore
// Expiration handled IN the machine — single source of truth
'awaiting_payment' => [
    'on' => [
        'ORDER_EXPIRED'    => ['target' => 'expired', 'after' => Timer::days(7)],
        'PAYMENT_RECEIVED' => 'paid',
    ],
],
'expired' => [
    'type'  => 'final',
    'entry' => 'notifyCustomerAction',  // notification is a machine action
],
```

#### 4b. Cross-reference only

Add a brief note at the end of the existing "Parent Orchestrates, Child Specialises" section:

> When designing a child machine for delegation, ensure its final states provide enough information for any future parent. A machine that works standalone may need additional states when integrated into a system. See [Machine System Design: Design Your Child States for the Parent](./machine-system-design#design-your-child-states-for-the-parent) for detailed guidance.

**Estimated lines:** ~50 (4a) + ~5 (4b cross-ref)

### 5. `docs/best-practices/machine-system-design.md` — Iterative Design Recipe

**Add new section:** "Design Recipe: Sunny Day First" (before Guidelines)

- Start with the happy path only
- Add error handling state by state: "What can go wrong here?"
- Add timer guards: "What if this response never arrives?"
- Test in isolation, then integrate into parent system

**Recipe (numbered steps):**
1. **Happy path** — model the ideal flow end-to-end
2. **Error states** — for each state, ask "what can go wrong?" and add error transitions
3. **Timer guards** — for each state waiting for external input, add `after` timeout
4. **Isolation test** — verify the machine works correctly standalone
5. **Integration check** — ensure final states provide enough information for the parent (see §3 above)

**Estimated lines:** ~40

### 6. NEW: `docs/reference/execution-model.md` — Execution Model Reference

**New file.** A precise reference for how EventMachine processes events.

**Note:** All claims in this section MUST be verified against the source code during implementation. The sequence below is based on book analysis + API exploration but has not been line-by-line verified.

**Content:**

#### Event Processing Sequence

```
1. Event arrives (via send() or HTTP endpoint)
2. Find matching transition:
   a. Look up event type in current state's transitions
   b. For multi-branch: evaluate guards in array order — first passing guard wins
   c. If no match in current state: try event bubbling to parent state
   d. If still no match: [verify: throw or silently ignore?]
3. Execute selected transition:
   a. Run exit actions of current state (bottom-up for nested states)
   b. Run transition actions
   c. Run entry actions of new state (top-down for nested states)
4. If new state has @always transitions:
   a. Repeat from step 2 with synthetic @always event
   b. Loop protection: max_transition_depth (default: 100)
5. If new state has machine delegation:
   a. Create/dispatch child machine (sync or async based on queue key)
   b. Parent stays in delegation state until @done/@fail/@timeout
6. If new state is final AND machine is a child:
   a. Notify parent via ChildMachineCompletionJob (async) or direct call (sync)
   b. Parent routes: @done.{state} (specific) → @done (catch-all) → no transition
```

#### Key Guarantees

- **Guard evaluation order:** Array order in config = evaluation priority (multi-branch only)
- **First match wins:** Only one transition per event
- **Event consumption:** Each event is processed once, then discarded
- **@always chaining:** Entry/exit actions fire for all intermediate states
- **Delegation handshake:** Parent enters delegation state (busy) → child runs → @done/@fail fires (done)
- **[Verify] Action atomicity:** Are all actions in a transition wrapped in a database transaction?
- **[Verify] Calculators:** Do calculators run alongside transition actions? If so, document their position in the sequence.

#### Delegation Lifecycle Signals

| Signal | Fires When | Use For |
|--------|-----------|---------|
| `@done` | Child reaches ANY final state | Business outcomes (catch-all) |
| `@done.{state}` | Child reaches SPECIFIC final state | Routing based on outcome |
| `@fail` | Exception thrown / job crashed | Infrastructure failures only |
| `@timeout` | Timeout seconds elapsed, child still running | Infrastructure protection |

For detailed usage, see [Machine System Design: Guidelines](./best-practices/machine-system-design#guidelines).

**Estimated lines:** ~100-120
