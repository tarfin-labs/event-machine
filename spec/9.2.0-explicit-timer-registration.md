# Explicit Timer Registration — Eliminate Boot-Time File Scanning

**Status:** Draft
**Date:** 2026-03-31

## Problem

`MachineServiceProvider::boot()` registers timer sweeps by scanning **every PHP file** in `app/` using PhpParser to find Machine subclasses with timer transitions. In a project with 4104 PHP files, this takes ~3.7 seconds per application boot.

PHPUnit boots the application per test. With 91 Findeks machine tests:

```
91 tests × 3.7s discovery = 5.6 minutes overhead (actual test execution: ~2s)
```

### Root Cause

The discovery exists because timer registration is "auto-magic" — define an `@after` or `@every` timer in your machine config and it "just works." But this magic requires knowing which Machine classes exist at boot time, which requires scanning the entire codebase.

Meanwhile, the two other registration systems are explicit:

| System | Registration | File Scanning |
|--------|-------------|--------------|
| **Endpoints** | `MachineRouter::register(OrderMachine::class, [...])` in `routes/api.php` | None |
| **Schedules** | `MachineScheduler::register(OrderMachine::class, 'CHECK_EXPIRY')` in `routes/console.php` | None |
| **Timers** | Auto-discovered via PhpParser file scan | 4104 files parsed |

Timer registration is the architectural outlier.

### Secondary Problem

The `machine:cache` / `machine:clear` commands exist solely to mitigate this discovery cost. They cache the list of timer machine classes to `bootstrap/cache/machines.php`. In production this works, but:

- Developers must remember to run `machine:cache` after adding/removing timers
- CI pipelines need the cache step
- Testing environment has no cache (intentionally)
- The cache contains class names, not the timer config — stale cache = wrong behavior

## Proposed Solution

Replace auto-discovery with explicit registration, matching the existing patterns for endpoints and schedules.

### New API: `MachineTimer::register()`

```php
// routes/console.php (or AppServiceProvider::boot)

use Tarfinlabs\EventMachine\Scheduling\MachineTimer;

// Register timer sweeps for specific machines
MachineTimer::register(OrderMachine::class);
MachineTimer::register(FindeksMachine::class);

// Optional: custom resolution (default: everyMinute)
MachineTimer::register(BillingMachine::class)
    ->everyFiveMinutes()
    ->environments(['production', 'staging']);
```

### Implementation

```php
// src/Scheduling/MachineTimer.php
class MachineTimer
{
    public static function register(string $machineClass): SchedulingEvent
    {
        /** @var Schedule $schedule */
        $schedule = resolve(Schedule::class);

        $resolution = TimerResolution::tryFrom(
            (string) config('machine.timers.resolution', 'everyMinute')
        ) ?? TimerResolution::EVERY_MINUTE;

        return $schedule->command("machine:process-timers --class={$machineClass}")
            ->{$resolution->value}()
            ->withoutOverlapping()
            ->runInBackground();
    }
}
```

### What Gets Removed

| Component | Action |
|-----------|--------|
| `MachineServiceProvider::registerTimerSweeps()` | Delete |
| `MachineServiceProvider::boot()` → `callAfterResolving(Schedule)` | Delete |
| `MachineDiscovery::findTimerMachines()` | Delete |
| `MachineDiscovery::findAllMachineClasses()` | Keep (used by `machine:validate --all` and `machine:cache`) |
| `MachineCacheCommand` (`machine:cache`) | Keep but repurpose — caches for `machine:validate --all` only |
| `MachineClearCommand` (`machine:clear`) | Keep (clears validate cache) |
| `MachineClassVisitor` | Keep (used by validate + cache) |
| PhpParser dependency | Keep (used by validate) |
| `config('machine.timers.resolution')` | Keep (default for `MachineTimer::register()`) |

### Migration

Before (v9.1):
```php
// Nothing needed — timers auto-discovered at boot
// (but costs 3.7s per boot without cache)
```

After (v9.2):
```php
// routes/console.php
MachineTimer::register(OrderMachine::class);
MachineTimer::register(FindeksMachine::class);
```

### Validation

`machine:process-timers --class=X` already validates that the class has timer transitions at runtime. If a user registers a machine without timers, the sweep command runs but finds nothing — harmless no-op.

For proactive validation, the `machine:validate` command can warn about:
- Timer machines not registered via `MachineTimer::register()`
- `MachineTimer::register()` calls for machines without timer transitions

### Comparison with Other Frameworks

| Framework | Timer Registration |
|-----------|-------------------|
| **Laravel** | Explicit in `routes/console.php` |
| **XState** | Timers are runtime-only (in-process delays) |
| **Temporal** | Workflows registered explicitly |
| **EventMachine (schedules)** | `MachineScheduler::register()` — explicit |
| **EventMachine (endpoints)** | `MachineRouter::register()` — explicit |
| **EventMachine (timers)** | Auto-discovered → **should be explicit** |

## Breaking Change?

Yes — machines with timers must add `MachineTimer::register()` to `routes/console.php`. Without it, timer sweeps won't run.

Mitigation: `machine:validate` can detect timer machines without registration and warn.
