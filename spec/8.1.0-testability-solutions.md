## Testing Pattern Evolution

### Complete API Surface

v5.1 added a dedicated testing layer. Here's the full inventory:

```
TestMachine (25 methods)              ← v5.1: Fluent machine-level testing
  ::create() / ::define() / ::for()
  ->send() / ->sendMany()
  ->assertState() / ->assertNotState() / ->assertFinished() / ->assertResult()
  ->assertContext() / ->assertContextMatches() / ->assertContextHas() / ->assertContextMissing() / ->assertContextIncludes()
  ->assertTransition() / ->assertGuarded() / ->assertValidationFailed()
  ->assertHistoryContains() / ->assertHistoryOrder()
  ->assertPath()
  ->assertRegionState()
  ->assertBehaviorRan() / ->assertBehaviorNotRan()
  ->faking() / ->withoutPersistence() / ->withoutParallelDispatch()
  ->machine() / ->state() / ->context()
  ->resetFakes()

InvokableBehavior (2 invocation methods)
  ::run(mixed ...$args)               ← Pre-v5.1: Raw invocation, no engine injection
  ::runWithState(State, ?Event, ?args) ← v5.1: Engine-identical parameter injection

Fakeable trait (16 methods)            ← Pre-v5.1, enhanced in v5.1
  ::fake() / ::spy()
  ::shouldRun() / ::shouldNotRun() / ::shouldReturn() / ::mayReturn() / ::allowToRun()
  ::assertRan() / ::assertNotRan() / ::assertRanWith() / ::assertRanTimes()
  ::isFaked() / ::getFake() / ::resetFakes() / ::resetAllFakes()

ResolvesBehaviors trait (5 methods)    ← Pre-v5.1: Extract behaviors from definition
  ::getBehavior(path) / ::getGuard(name) / ::getAction(name) / ::getCalculator(name) / ::getEvent(name)

State::forTesting()                    ← v5.1: Lightweight State without MachineDefinition
EventBehavior::forTesting()            ← v5.1: Test event factory
Machine::test()                        ← v5.1: TestMachine shortcut
```

### Pattern Migration Guide

| Pre-v5.1 Pattern | v5.1+ Ideal Pattern | Why Migrate |
|-------------------|---------------------|-------------|
| `MyGuard::run($context)` | `MyGuard::runWithState(State::forTesting([...]))` | `run()` doesn't use engine's parameter injection; doesn't pass eventQueue; raised events are lost |
| `(new MyAction())($ctx, $event)` | `MyAction::runWithState($state, $event)` | Direct instantiation bypasses container — no DI, fakes ignored |
| `Machine::create() → send() → assertEquals(...)` | `Machine::test() → send() → assertState()` | Fluent API; built-in assertions with contextual failure messages; `withoutPersistence()` support |
| `$guard = Machine::getGuard('name'); $guard($ctx)` | `MyGuard::runWithState($state)` (for class behaviors) | `runWithState()` goes through container (DI + fakes work) |
| `$machine->state->context->get('x') === $expected` | `->assertContext('x', $expected)` or `->assertContextMatches(...)` | Better failure messages; fluent chain |

### Ideal Testing Levels

#### Level 1: Isolated Behavior (Unit Test)

Test a single guard, action, or calculator in isolation from the machine.

```php
// IDEAL — v5.1 pattern
$state = State::forTesting(['count' => 5, 'order' => $order]);
$result = IsCountPositiveGuard::runWithState($state);
expect($result)->toBeTrue();

// With event and arguments
$state = State::forTesting(['completed_at' => now()->subDays(10)]);
$result = CheckDaysGuard::runWithState(
    state: $state,
    eventBehavior: MyEvent::forTesting(['key' => 'val']),
    arguments: ['7'],
);
expect($result)->toBeTrue();

// Action with raised events
$state = State::forTesting(['order_id' => 1]);
$eventQueue = ProcessOrderAction::runWithState($state);
expect($eventQueue)->toHaveCount(1);
expect($eventQueue->first()['type'])->toBe('ORDER_PROCESSED');
```

**Why `runWithState()` over `run()`:**

| Feature | `run()` | `runWithState()` |
|---------|---------|-------------------|
| Container resolution (DI) | ✅ | ✅ |
| Respects fakes | ✅ | ✅ |
| Engine parameter injection | ❌ manual | ✅ automatic |
| eventQueue passed | ❌ empty Collection() | ✅ shared Collection |
| Raised events accessible | ❌ lost | ✅ returned |
| Behavior arguments | ❌ manual | ✅ `$arguments` param |
| State/History injection | ❌ unavailable | ✅ from State object |

#### Level 2: Inline Behavior (Unit Test)

For closures registered in the behavior config — `runWithState()` can't be used because it's a static method on a class.

```php
// ONLY way to test inline behaviors — still valid
$guard = OrderMachine::getGuard('isPositiveGuard');
$state = State::forTesting(['count' => 5]);

$params = InvokableBehavior::injectInvokableBehaviorParameters(
    actionBehavior: $guard,
    state: $state,
);

$result = $guard(...$params);
expect($result)->toBeTrue();
```

> `ResolvesBehaviors::getGuard/getAction/getCalculator` remains the ONLY way to extract and test inline
> closures. No migration needed — this is the ideal pattern for this use case.

#### Level 3: Machine Integration (Integration Test)

Test state flow, transitions, guard routing, and behavior orchestration.

```php
// IDEAL — v5.1 fluent API
TrafficLightsMachine::test(['count' => 0])
    ->withoutPersistence()
    ->send('INCREASE')
    ->assertState('active')
    ->assertContext('count', 1)
    ->send('INCREASE')
    ->assertContext('count', 2);

// Multi-step flow
OrderMachine::test(['order_id' => $order->id])
    ->withoutPersistence()
    ->send('START')
    ->assertState('processing')
    ->send('COMPLETE')
    ->assertState('done')
    ->assertFinished();

// With faking
OrderMachine::test()
    ->withoutPersistence()
    ->faking([SendEmailAction::class])
    ->send('APPROVE')
    ->assertState('approved')
    ->assertBehaviorRan(SendEmailAction::class);
```

**Why `Machine::test()` over `Machine::create()`:**

| Feature | `Machine::create()` | `Machine::test()` |
|---------|---------------------|---------------------|
| Fluent assertions | ❌ manual assertEquals | ✅ 21 assertion methods |
| Failure messages | ❌ generic | ✅ contextual ("Expected state [x], got [y]") |
| Persistence control | ❌ must modify definition | ✅ `->withoutPersistence()` |
| Parallel control | ❌ must modify config | ✅ `->withoutParallelDispatch()` |
| Behavior faking | ❌ manual setup | ✅ `->faking([...])` |
| Path testing | ❌ manual loop | ✅ `->assertPath([...])` |
| History assertions | ❌ manual pluck | ✅ `->assertHistoryContains(...)` |
| Chaining | ❌ send() returns State | ✅ all methods return self |

#### Level 4: Persistence / E2E Testing

When you need real DB interaction — `Machine::test()` still works, just don't disable persistence:

```php
// TestMachine WITH persistence
OrderMachine::test(['order_id' => $order->id])
    ->send('START')
    ->assertState('processing');

// Verify DB state
$this->assertDatabaseHas('machine_events', [
    'type' => 'START',
    'machine_value' => json_encode(['order_machine.processing']),
]);

// Or use Machine::create() when you need the raw Machine instance
$machine = OrderMachine::create();
$machine->send(['type' => 'START']);
$rootEventId = $machine->state->history->first()->root_event_id;
// ... DB assertions on machine_events table
```

> `Machine::create()` is NOT deprecated — it's the production API. Use it when you specifically
> need the raw Machine instance or are testing persistence/restoration logic.

### Deprecated Patterns

These patterns still work but should be migrated:

#### 1. `BehaviorClass::run($context, $event)` → `runWithState()`

```php
// ❌ AVOID — no engine injection, raised events lost
$result = MyGuard::run($context);
MyAction::run($context, $event);

// ✅ PREFERRED
$state = State::forTesting($context->toArray());
$result = MyGuard::runWithState($state);
$eventQueue = MyAction::runWithState($state, $event);
```

**Migration effort:** Change `::run($context)` to `::runWithState(State::forTesting([...]))`. For guards, wrap result in expect. For actions, capture returned eventQueue if testing raised events.

**When `run()` is still acceptable:** In Fakeable trait tests where you're testing the fake/mock mechanism itself, not behavior logic. The `InvokableBehaviorFakeTest.php` uses `run()` because it's testing that fakes intercept calls — the actual parameter injection doesn't matter there.

#### 2. `new BehaviorClass()($context, $event)` → `runWithState()`

```php
// ❌ AVOID — bypasses container, no DI, fakes ignored
$guard = new IsFraudDetectedGuard();
$guard->validateRequiredContext($context);
$result = $guard($context, $event);

// ✅ PREFERRED
$state = State::forTesting($context->toArray());
IsFraudDetectedGuard::validateRequiredContext($state->context); // static, no instance needed
$result = IsFraudDetectedGuard::runWithState($state, $event);
```

#### 3. `Machine::create()` for assertion-heavy tests → `Machine::test()`

```php
// ❌ VERBOSE — manual state comparison
$machine = TrafficLightsMachine::create();
$state = $machine->send(['type' => 'INCREASE']);
assertEquals(1, $state->context->get('count'));
assertTrue($state->matches('traffic_lights_machine.active'));

// ✅ CLEAN
TrafficLightsMachine::test()
    ->send('INCREASE')
    ->assertContext('count', 1)
    ->assertState('active');
```

#### 4. `Machine::getGuard('name')` for class behaviors → `runWithState()`

```php
// ❌ AVOID for class behaviors — unnecessary indirection
$guard = OrderMachine::getGuard('isValidOrderGuard');
$result = $guard($context);

// ✅ PREFERRED — direct, goes through container
$result = IsValidOrderGuard::runWithState(State::forTesting([...]));

// ✅ STILL IDEAL for inline closures — no alternative
$guard = OrderMachine::getGuard('isPositiveGuard');  // fn($ctx) => $ctx->count > 0
```

### Pattern Decision Tree

```
Q: What are you testing?
│
├─ A single behavior class (guard/action/calculator)?
│  ├─ It's a class (extends GuardBehavior etc)?
│  │  └─ Use: BehaviorClass::runWithState(State::forTesting([...]))
│  └─ It's an inline closure (fn($ctx) => ...)?
│     └─ Use: Machine::getGuard('name') + injectInvokableBehaviorParameters()
│
├─ Machine state flow / transitions?
│  ├─ Need DB persistence?
│  │  └─ Use: Machine::test() (don't call withoutPersistence)
│  └─ No DB needed?
│     └─ Use: Machine::test()->withoutPersistence()->send()->assertState()
│
├─ Behavior orchestration (which sub-actions fire)?
│  └─ Use: Machine::test()->faking([...])->send()->assertBehaviorRan()
│
├─ Context validation (requiredContext)?
│  └─ Use: BehaviorClass::validateRequiredContext($context) (static, no instance)
│
└─ Event validation?
   └─ Use: MyEvent::forTesting([...])->selfValidate()
```

### 3. Parametric Guard Testing

### 5. assertGuardedBy()

**Problem:** `assertGuarded()` only tells you the event was blocked, not WHICH guard blocked it. In OrderCancellationMachine, `check.order` has 5 guards — knowing which one fired is essential for debugging.

**Current limitation:**
```php
OrderCancellationMachine::test(['order_id' => $demoOrder->id])
    ->withoutPersistence()
    ->assertGuarded('START');
// We know it was blocked, but was it orderIsNotDemoGuard? isNotCancelledGuard?
```

**Proposed API:**
```php
OrderCancellationMachine::test(['order_id' => $demoOrder->id])
    ->withoutPersistence()
    ->assertGuardedBy('START', 'orderIsNotDemoGuard');
```

**Implementation in `TestMachine.php`:**

```php
/**
 * Assert an event is guarded by a specific guard.
 *
 * Sends the event, confirms the machine didn't transition, then checks
 * history for the guard's fail event.
 *
 * @param  string  $guardName  The behavior key of the guard (e.g. 'orderIsNotDemoGuard').
 */
public function assertGuardedBy(EventBehavior|array|string $event, string $guardName): self
{
    $before = $this->machine->state->value;

    try {
        $this->send($event);
    } catch (NoTransitionDefinitionFoundException) {
        expect(false)->toBeTrue(
            "Event not recognized by machine — cannot verify guard [{$guardName}]"
        );
    }

    // Verify state didn't change
    expect($this->machine->state->value)->toBe($before,
        "Expected event to be guarded by [{$guardName}], but a transition occurred"
    );

    // Internal guard fail events: {machineId}.guard.{guardName}.fail
    $machineId      = $this->machine->state->currentStateDefinition->machine->id;
    $guardFailEvent = "{$machineId}.guard.{$guardName}.fail";

    $guardFailed = $this->machine->state->history
        ->pluck('type')
        ->contains($guardFailEvent);

    expect($guardFailed)->toBeTrue(
        "Expected guard [{$guardName}] to block the event, but it didn't. "
        ."History: [".implode(', ', $this->machine->state->history->pluck('type')->toArray()).']'
    );

    return $this;
}
```

**Test:**
```php
it('asserts event is guarded by specific guard', function (): void {
    AllInvocationPointsMachine::test(['count' => 0])
        ->assertGuardedBy('PROCESS', 'isCountPositiveGuard');
});

it('assertGuardedBy fails when wrong guard name', function (): void {
    expect(fn () => AllInvocationPointsMachine::test(['count' => 0])
        ->assertGuardedBy('PROCESS', 'nonExistentGuard'))
        ->toThrow(\PHPUnit\Framework\ExpectationFailedException::class);
});
```

**Effort:** ~30 lines of code + 2 tests. Low risk.

### 6. withContext() — Pre-Start Context Injection

**Problem:** `Machine::test(['key' => $value])` applies context AFTER `Machine::create()` — entry actions on the initial state see the machine's DEFAULT context, not the test overrides.

For `ApplicationMachine`, the initial state has entry actions that read context — they get `null` instead of the test-provided model.

**Current behavior (broken for entry actions):**
```php
// Entry action runs BEFORE context override:
ApplicationMachine::test(['application' => $application])
    ->send('START');
// Entry action sees context->application as null
```

**Proposed API:**
```php
// Context applied BEFORE machine start:
ApplicationMachine::withContext(['application' => $application])
    ->withoutPersistence()
    ->send('START')
    ->assertState('score.checking');
// Entry action sees $application correctly
```

**Implementation approach:**

The key insight: context must be set on the ContextManager BEFORE `getInitialState()` triggers entry actions.

```php
/**
 * Create a TestMachine with context applied BEFORE machine initialization.
 *
 * Unlike test() which applies context after start, this method merges
 * context into the definition's initial context before entry actions run.
 */
public static function withContext(string $machineClass, array $context): self
{
    $definition                = $machineClass::definition();
    $definition->shouldPersist = false;
    $definition->machineClass  = $machineClass;

    // Merge test context into definition's initial context config
    // so that getInitialState() -> ContextManager includes these values
    $definition->config['context'] = array_merge(
        $definition->config['context'] ?? [],
        $context,
    );

    // Now start — entry actions will see merged context
    $machine        = Machine::withDefinition($definition);
    $machine->state = $definition->getInitialState();

    return new self($machine);
}
```

Convenience method on `Machine`:
```php
public static function withContext(array $context): TestMachine
{
    return TestMachine::withContext(static::class, $context);
}
```

> **Investigation needed:** Verify that `MachineDefinition::getInitialState()` reads `config['context']`
> to populate the ContextManager. If context is initialized differently (e.g., from behavior config),
> the merge point may need adjustment.

**Effort:** ~25 lines + investigation into MachineDefinition context lifecycle. Medium risk.

### 7. tap() — Fluent Chain for Custom Assertions

**Problem:** After a machine transition, backend tests need to verify DB state or notifications. Currently requires breaking the fluent chain.

**Current pattern (chain-breaking):**
```php
$test = OrderCancellationMachine::test(['order_id' => $order->id])
    ->withoutPersistence()
    ->send('CANCEL');

// Must break chain:
$this->assertDatabaseHas('orders', ['id' => $order->id, 'status' => 'cancelled']);
$test->assertState('done.cancelled');
```

**Proposed API:**
```php
OrderCancellationMachine::test(['order_id' => $order->id])
    ->withoutPersistence()
    ->send('CANCEL')
    ->assertState('done.cancelled')
    ->tap(fn () => expect($order->fresh()->status)->toBe('cancelled'));
```

**Implementation:**

```php
/**
 * Execute a callback within the fluent chain for custom assertions.
 *
 * Useful for asserting side effects (DB state, notifications, etc.)
 * without breaking the fluent chain.
 *
 * @param  callable(self): void  $callback
 */
public function tap(callable $callback): self
{
    $callback($this);

    return $this;
}
```

**Usage examples:**
```php
// DB assertion
OrderCancellationMachine::test(['order_id' => $order->id])
    ->withoutPersistence()
    ->send('CANCEL')
    ->assertState('done.cancelled')
    ->tap(fn () => expect($order->fresh()->status)->toBe('cancelled'))
    ->tap(fn () => expect($order->fresh()->cancelled_at)->not->toBeNull());

// Notification assertion
Notification::fake();
OrderCancellationMachine::test(['order_id' => $order->id])
    ->withoutPersistence()
    ->send('CANCEL')
    ->tap(fn () => Notification::assertSentTo($order->user, CancellationNotification::class));

// Access TestMachine inside tap
OrderCancellationMachine::test(['order_id' => $order->id])
    ->withoutPersistence()
    ->send('CANCEL')
    ->tap(function (TestMachine $tm) {
        expect($tm->context()->get('cancelled_at'))->not->toBeNull();
    });
```

**Why `tap()` instead of `assertDatabaseHas()`?**
- Generic — supports any assertion type (DB, notifications, mail, queues)
- No framework coupling — TestMachine stays independent of Laravel test traits
- Follows Laravel Collection's `tap()` convention

**Effort:** 8 lines + 1 test. Very low risk.

## P2: Medium API Additions

### 8. withScenario()

**Problem:** `ApplicationMachine` has 9 scenarios that override state transitions. No test verifies scenario routing. Scenarios are activated via `payload['scenarioType']` on the triggering event.

**How scenarios work internally:**
1. Machine config has `'scenarios_enabled' => true`
2. `MachineDefinition::$scenarios` maps scenario names to state overrides
3. `EventBehavior::getScenario()` reads `payload['scenarioType']`
4. Engine calls `getScenarioStateIfAvailable()` — replaces state definition with scenario-specific one

**Proposed API:**
```php
ApplicationMachine::test()
    ->withoutPersistence()
    ->withScenario('ml_score_approved_scenario')
    ->send(ApplicationStartedEvent::forTesting(['direct' => true]))
    ->assertState('approved.idle');
```

**Implementation (SIMPLIFIED — no send() modification needed):**

The engine's `getScenarioStateIfAvailable()` reads `scenarioType` from `$state->context`. Setting context is sufficient — no need to inject into event payloads.

```php
/**
 * Set the scenario for subsequent events.
 *
 * Sets scenarioType in context so the engine uses scenario-specific
 * state definitions. Works with all event types (string, array, EventBehavior).
 */
public function withScenario(string $scenarioName): self
{
    $this->machine->state->context->set('scenarioType', $scenarioName);

    return $this;
}
```

> **No send() modification.** The engine reads scenario from context after the first event sets it.
> If an EventBehavior's payload contains a different `scenarioType`, it overrides withScenario() —
> this is correct behavior (event-level takes precedence).

**Usage examples:**
```php
// ML-approved scenario
ApplicationMachine::test()
    ->withoutPersistence()
    ->withScenario('ml_score_approved_scenario')
    ->send('START')
    ->assertState('approved.idle');

// Rejection scenario
ApplicationMachine::test()
    ->withoutPersistence()
    ->withScenario('ml_score_rejected_scenario')
    ->send('START')
    ->assertState('rejected');

// Verify scenario-specific transitions
ApplicationMachine::test()
    ->withoutPersistence()
    ->withScenario('manual_review_scenario')
    ->send('START')
    ->assertState('review.pending')
    ->send('APPROVE')
    ->assertState('approved.manual');
```

### 9. assertTransitionedThrough() — @always Testing

**Problem:** States with `@always` transitions (e.g., `idleStateResolver`, `findeks.reportSaved`) route immediately based on guards. The machine never "rests" there — they're transient. No tests verify which path is taken.

**How @always works:** When a machine enters an `@always` state, it immediately evaluates guards and transitions to the first matching target.

**Existing API already handles the output state:**
```php
// This works today — you test the final resting state:
ApplicationMachine::test(['application_type' => 'farmer'])
    ->withoutPersistence()
    ->send('START')
    ->assertState('farmer.idle');  // @always routed here
```

**New assertion — verify the transient state was visited:**

```php
/**
 * Assert the machine transitioned through a specific state.
 *
 * Checks history's machine_value arrays for the given state ID.
 * Useful for verifying @always (transient) states were visited.
 */
public function assertTransitionedThrough(string $stateId): self
{
    $machineId = $this->machine->state->currentStateDefinition->machine->id;

    if (!str_starts_with($stateId, $machineId)) {
        $stateId = $machineId.'.'.$stateId;
    }

    $visited = $this->machine->state->history
        ->pluck('machine_value')
        ->flatten()
        ->unique()
        ->contains($stateId);

    expect($visited)->toBeTrue(
        "Expected machine to have transitioned through [{$stateId}]"
    );

    return $this;
}
```

**Usage:**
```php
ApplicationMachine::test(['application_type' => 'farmer'])
    ->withoutPersistence()
    ->send('START')
    ->assertTransitionedThrough('idleStateResolver')  // transient @always state
    ->assertState('farmer.idle');                       // final resting state

ApplicationMachine::test(['findeks_data' => $data])
    ->withoutPersistence()
    ->send('FINDEKS_COMPLETE')
    ->assertTransitionedThrough('findeks.reportSaved')  // @always router
    ->assertState('findeks.validated');
```

**Effort:** ~20 lines + 2 tests. Low risk.

## P3: Large API/Infrastructure Changes

### 11. Parallel @done/@fail Testing

**Problem:** `CarSalesMachine` uses parallel states with 2-level nesting. When all regions reach final states, the parent should emit `@done`. ZERO tests exist.

**New assertions:**

```php
/**
 * Assert all regions of the current parallel state have reached final states.
 */
public function assertAllRegionsCompleted(): self
{
    $currentState = $this->machine->state->currentStateDefinition;

    // Walk up to find the parallel parent
    $parallelParent = $currentState;
    while ($parallelParent !== null && $parallelParent->type !== StateDefinitionType::PARALLEL) {
        $parallelParent = $parallelParent->parent;
    }

    expect($parallelParent)->not->toBeNull('Machine is not in a parallel state');

    // Check each region's active state is a final state
    foreach ($this->machine->state->value as $activeStateId) {
        $activeState = $this->machine->definition->idMap[$activeStateId] ?? null;
        expect($activeState)->not->toBeNull("State [{$activeStateId}] not found in idMap");
        expect($activeState->type)->toBe(
            StateDefinitionType::FINAL,
            "Region state [{$activeStateId}] is not a final state"
        );
    }

    return $this;
}
```

**Usage:**
```php
CarSalesMachine::test()
    ->withoutParallelDispatch()
    ->withoutPersistence()
    ->send('START_COLLECTION')
    ->assertRegionState('documents', 'collected')
    ->assertRegionState('identity', 'verified')
    ->assertAllRegionsCompleted();
```

**Effort:** ~50 lines + 3-4 tests. Medium risk — requires parallel state test stubs.

### 12. Guard Chain Debugging

**Problem:** When a transition has 5 guards, knowing which one failed and their evaluation order is hard. `assertGuardedBy()` (P1) solves single-guard verification, but full chain debugging needs more.

**Proposed API — `debugGuards()` (non-fluent, returns data):**

```php
/**
 * Send an event and return guard evaluation results for debugging.
 *
 * @return array<string, bool> Guard name => passed (true) or failed (false).
 */
public function debugGuards(EventBehavior|array|string $event): array
{
    $historyCountBefore = count($this->machine->state->history);

    try {
        $this->send($event);
    } catch (NoTransitionDefinitionFoundException) {
        return ['_error' => 'Event not recognized'];
    }

    $machineId = $this->machine->state->currentStateDefinition->machine->id;
    $results   = [];

    $newEvents = $this->machine->state->history->slice($historyCountBefore);
    foreach ($newEvents as $historyEvent) {
        if (preg_match("/{$machineId}\.guard\.(.+)\.(pass|fail)/", $historyEvent->type, $matches)) {
            $results[$matches[1]] = $matches[2] === 'pass';
        }
    }

    return $results;
}
```

**Usage:**
```php
$results = OrderCancellationMachine::test(['order_id' => $order->id])
    ->withoutPersistence()
    ->debugGuards('START');

// Returns:
// [
//     'orderIsNotDemoGuard'      => true,
//     'isNotCancelledGuard'      => true,
//     'hasValidStatusGuard'      => false,  <-- this one blocked
// ]
```

> **VERIFIED:** Engine emits BOTH `guard.{name}.pass` AND `guard.{name}.fail` events for ALL guards
> (not just ValidationGuards). No engine change needed.

**Effort:** ~40 lines + 2 tests. Low risk.

## Review Findings & Corrections

These findings are based on three parallel investigations that verified engine internals:

### Finding 1: Guard events — BOTH pass AND fail are recorded ✓

`TransitionDefinition.php` records:
- `{machineId}.guard.{placeholder}.pass` for every guard that returns true
- `{machineId}.guard.{placeholder}.fail` for the first guard that returns false (then breaks)

Placeholder format:
- Inline key: `isEvenGuard` → `machine.guard.isEvenGuard.pass`
- FQCN: `IsCountPositiveGuard::class` → `machine.guard.IsCountPositiveGuard.pass`

**Impact:** `debugGuards()` (#12) risk dropped from HIGH to LOW. No engine change needed.

### Finding 2: @always transient states ARE recorded in history ✓

When a machine enters a transient `@always` state:
- `STATE_EXIT`, `TRANSITION_FINISH` events contain the transient state's ID in `machine_value`
- For initial states with `@always`, `STATE_ENTER` also contains the transient state's ID

**Impact:** `assertTransitionedThrough()` (#9) implementation is correct as designed.

### Finding 3: Context is lazily initialized ✓

`config['context']` is read at `getInitialState()` time, not at construct time. Modifying it after `MachineDefinition::define()` but before `getInitialState()` works.

**Exception:** If `config['context']` is a ContextManager subclass string, `setupContextManager()` moves it to `behavior['context']` and clears `config['context']` to `[]`.

**Impact:** `withContext()` (#6) works for array context. Need to also handle typed ContextManager case.

### Finding 4: withScenario() doesn't need send() modification ✓

`getScenarioStateIfAvailable()` reads scenario from `$state->context->get('scenarioType')`. Just setting context is sufficient. Event payload injection is unnecessary.

**Impact:** `withScenario()` (#8) simplified from 25 LOC to 5 LOC. send() stays untouched.

### Finding 5: assertGuardedBy() needs dual placeholder handling

Guard events use inline key as-is or `classBasename()` for FQCN. The assertion should try both formats.

### Finding 6: Missing fluent wrapper — assertBehaviorRanWith()

`Fakeable::assertRanWith(callable)` exists but had no TestMachine fluent wrapper. Added to #10.

### Finding 7: P0 recipes should use package stubs, not backend names

Documentation recipes should use `TrafficLightsMachine`, `TestabilityMachine`, `TestMachine::define()` — not backend-specific machine names like `OrderCancellationMachine`.

### Finding 8: run() vs runWithState() — clear migration path

| Metric | `::run()` usage | `::runWithState()` usage |
|--------|----------------|--------------------------|
| In package tests | ~45 calls | ~13 calls |
| Primary file | `InvokableBehaviorFakeTest.php` | `IsolatedTestingTest.php` |

`run()` is heavily used in `InvokableBehaviorFakeTest.php` because that file tests the Fakeable MECHANISM itself (not behavior logic). For behavior testing, `runWithState()` should be the default.

### Finding 9: Machine::create() still dominant in tests

~100 occurrences vs ~34 for `Machine::test()`. Most are in older test files (CalculatorTest, TransitionsTest) and E2E tests. Package's own tests should gradually migrate to `Machine::test()`.
