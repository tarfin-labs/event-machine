# @done.{finalState} — Declarative Child Final State Routing

**Status:** Draft v3 (reviewed, fixes applied)
**Author:** Claude
**Date:** 2025-03-18
**Scope:** Add declarative routing based on child machine's final state name

---

## Problem

When a child machine reaches a final state, `@done` fires regardless of _which_ final state it reached. The parent must use guards to inspect child output and route accordingly:

```php
// Parent config — imperative routing via guards
'@done' => [
    ['target' => 'completed', 'guards' => 'isChildSuccessfulGuard'],
    ['target' => 'failed'],
],

// Guard implementation — hidden logic
public function __invoke(ContextManager $context, ChildMachineDoneEvent $event): bool
{
    return $event->output('isSuccessful') === true;
}

// Child — forced output convention
'report_saved' => ['type' => 'final', 'output' => ['isSuccessful' => true]],
'failed'       => ['type' => 'final', 'output' => ['isSuccessful' => false]],
```

### Pain Points

1. **Output convention**: Every child final state must carry a discriminator like `isSuccessful`
2. **Guard boilerplate**: Every child delegation needs a guard to inspect child output
3. **Hidden logic**: "What happens if child fails?" requires reading guard implementation, not config
4. **Imperative**: Guard logic rather than declarative config

### Key Observation

`ChildMachineCompletionJob` and `routeChildDone()` already know the child's final state (`$childMachine->state->currentStateDefinition`). This information is available but not exposed to the parent's routing config.

## Design: Dot Notation on State-Level Keys

### Syntax

```php
'processing' => [
    'machine' => ReportMachine::class,

    '@done.report_saved' => 'completed',
    '@done.failed'       => 'error',
    '@done.expired'      => 'error',

    '@fail' => 'system_error',
],
```

Each `@done.{state}` key matches a specific child final state name (the `key` property, not the full `id` path).

### Why Dot Notation Over Mapping Syntax

The mapping syntax (`'@done' => ['report_saved' => 'completed']`) was considered but rejected:

| Criterion | Dot Notation | Mapping Syntax |
|-----------|-------------|----------------|
| Parse ambiguity | None — distinct config keys | Ambiguous with `['target' => '...', 'actions' => '...']` |
| Per-state guards/actions | Full existing transition syntax | Would need nested format |
| Catch-all fallback | `@done` stays as-is | Needs special `'*'` key |
| Consistency | Matches `@done`/`@fail`/`@timeout` pattern | New nested pattern |

### Full Transition Syntax Per Final State

Each `@done.{state}` supports the exact same formats as `@done`:

```php
// Simple string target
'@done.approved' => 'processing',

// Single config with actions
'@done.approved' => [
    'target'  => 'processing',
    'actions' => 'storeApprovalAction',
],

// Multi-branch with guards
'@done.approved' => [
    ['target' => 'vip_processing', 'guards' => 'isHighValueGuard'],
    ['target' => 'standard_processing'],
],
```

### Resolution Priority

1. **`@done.{childFinalStateKey}`** — specific match for the child's final state key
2. **`@done`** — catch-all fallback (existing behavior)
3. **No match** — see Validation section below

### Guard Fallthrough

If a `@done.approved` guard fails (all branches reject), resolution falls through to `@done` catch-all. This is consistent with how multi-branch guard semantics work — first match wins, then fallback.

### XState Comparison

XState v5 does NOT expose the child's final state name. It relies purely on `output` + guards. This is intentional actor model encapsulation.

EventMachine deliberately makes a different trade-off: it already has `@done`/`@fail` (XState uses `onDone`/`onError`), showing a preference for **declarative config over encapsulation**. `@done.{state}` naturally extends this philosophy — the definition tells you what happens without reading guard code.

## Validation: Final State Coverage

### The Problem Without Validation

If only `@done.approved` and `@done.rejected` are defined but the child reaches `expired`, the parent gets stuck — no transition fires, and the reason is invisible.

### Rule: All Child Final States Must Be Covered

When running `php artisan machine:validate`, if a state has `machine` + any `@done.*` keys:

1. Resolve the child machine class
2. Call `$childClass::definition()` to get child's config
3. Collect all child final states (states with `type: 'final'`)
4. Check that every child final state is covered by either:
   - A specific `@done.{state}` key, OR
   - A catch-all `@done` key

If uncovered final states exist → throw `InvalidArgumentException`:

```
State 'processing' has @done.{state} routing but child machine
'ReportMachine' has uncovered final states: expired.
Add '@done.expired' or a catch-all '@done' to handle all outcomes.
```

### Where To Validate

**`StateConfigValidator::validateMachineConfig()`** — this method already validates `machine` key config. Add the coverage check here.

> **Note:** `StateConfigValidator` is invoked by `php artisan machine:validate`, NOT during `MachineDefinition::define()`. Coverage validation is a static analysis check, not a runtime guard. This is intentional — calling `$childClass::definition()` during parent definition construction could cause side effects or circular dependencies.

The validator must handle edge cases:
- Child machine class doesn't exist yet (skip validation with warning)
- Child machine `definition()` throws (skip validation with warning)
- Mixed usage: some `@done.*` keys + a `@done` catch-all → valid (catch-all covers remaining)
- Only `@done` (no dot notation) → valid, existing behavior
- Only `@done.*` keys, no catch-all → must cover ALL child final states

### Fire-and-Forget Interaction

Current fire-and-forget detection:
```php
$isFireAndForget = isset($stateConfig['queue']) && !isset($stateConfig['@done']);
```

Must be updated to also check for `@done.*` keys:
```php
$hasDoneRouting = isset($stateConfig['@done']) || self::hasDoneDotKeys($stateConfig);
$isFireAndForget = isset($stateConfig['queue']) && !$hasDoneRouting;
```

### Phase 1: Data — Carry `finalState` Through the Pipeline

#### 1.1 `ChildMachineDoneEvent` — add `finalState()` accessor

```php
// src/Behavior/ChildMachineDoneEvent.php

public function finalState(): ?string
{
    return $this->payload['final_state'] ?? null;
}
```

No breaking change — `finalState()` returns `null` for events without the key.

#### 1.2 `MachineDefinition::routeChildDone()` — fix early return + pass final state key

**Critical:** The current early return exits before the event is even built:

```php
// CURRENT (line ~1498) — MUST BE UPDATED:

    return;  // ← Exits when only @done.{state} keys exist (no @done catch-all)!
}

// UPDATED:

    && $stateDefinition->onDoneStateTransitions === []) {
    return;

```

Without this fix, sync delegation with only `@done.{state}` keys (no `@done` catch-all) would silently fail.

Then add `final_state` to the event:

```php
// src/Definition/MachineDefinition.php line ~1505

$doneEvent = ChildMachineDoneEvent::forChild([
    'result'        => $childMachine->result(),
    'output'        => self::resolveChildOutput(...) ?? $childContext,
    'machine_id'    => $childRootEventId,
    'machine_class' => $childMachineClass,
```

Uses `->key` (leaf name like `report_saved`), not `->id` (full path like `report_machine.report_saved`).

### Phase 2: Config — Parse `@done.*` Keys

#### 2.1 `StateDefinition` — new property + parsing

Add property:
```php
// src/Definition/StateDefinition.php

/** @var array<string, TransitionDefinition> Keyed by child final state name */
public array $onDoneStateTransitions = [];
```

In `initializeTransitions()`, after the existing `@done` parsing:
```php
// Parse @done.{finalState} keys
foreach ($this->config as $key => $value) {
    if (str_starts_with($key, '@done.')) {
        $finalStateName = substr($key, 6); // after '@done.'
        $this->onDoneStateTransitions[$finalStateName] = new TransitionDefinition(
            transitionConfig: $value,
            source: $this,
            event: '@done.'.$finalStateName,
        );
    }
}
```

#### 2.2 `StateConfigValidator` — allow `@done.*` keys

`ALLOWED_STATE_KEYS` is a static list. `@done.{state}` keys have dynamic suffixes. Update `validateStateConfig()` to filter out `@done.*` keys before checking `ALLOWED_STATE_KEYS`:

```php
// In validateStateConfig(), before the invalidKeys check:
$configKeysToValidate = array_filter(
    array_keys($stateConfig),
    fn (string $k) => !str_starts_with($k, '@done.'),
);
$invalidKeys = array_diff($configKeysToValidate, self::ALLOWED_STATE_KEYS);
```

Then validate each `@done.*` key — reject empty suffix, validate value as transition config:
```php
foreach ($stateConfig as $key => $value) {
    if (str_starts_with($key, '@done.')) {
        $suffix = substr($key, 6);
        if ($suffix === '') {
            throw new InvalidArgumentException(
                "State '{$path}' has invalid key '@done.' — final state name after the dot cannot be empty."
            );
        }
        self::validateDoneFailConfig($value, $key, $path);
    }
}
```

#### 2.3 `StateConfigValidator` — final state coverage validation

Add new method `validateChildFinalStateCoverage()`, called from `validateMachineConfig()`:

```php
private static function validateChildFinalStateCoverage(array $stateConfig, string $path): void
{
    // Collect @done.{state} keys
    $doneStateKeys = [];
    foreach (array_keys($stateConfig) as $key) {
        if (str_starts_with($key, '@done.')) {
            $doneStateKeys[] = substr($key, 6);
        }
    }

    // No dot notation used → skip (backward compatible)
    if ($doneStateKeys === []) {
        return;
    }

    // Has catch-all @done → all states covered
    if (isset($stateConfig['@done'])) {
        return;
    }

    // Resolve child machine's final states
    $machineClass = $stateConfig['machine'];
    if (!is_string($machineClass) || !class_exists($machineClass)) {
        return; // Can't validate — class not available
    }

    try {
        $childDefinition = $machineClass::definition();
    } catch (\Throwable) {
        return; // Can't validate — definition failed
    }

    $childFinalStates = self::collectFinalStateKeys($childDefinition->config);

    $uncovered = array_diff($childFinalStates, $doneStateKeys);
    if ($uncovered !== []) {
        throw new InvalidArgumentException(
            "State '{$path}' has @done.{state} routing but child machine "
            ."'{$machineClass}' has uncovered final states: "
            .implode(', ', $uncovered)
            .". Add specific '@done.{state}' keys or a catch-all '@done' to handle all outcomes."
        );
    }
}

private static function collectFinalStateKeys(array $config): array
{
    $finals = [];
    foreach ($config['states'] ?? [] as $name => $stateConfig) {
        if (isset($stateConfig['type']) && $stateConfig['type'] === 'final') {
            $finals[] = $name;
        }
    }
    return $finals;
}
```

### Phase 3: Routing — Use Final State for Dispatch

#### 3.1 `MachineDefinition::routeChildDoneEvent()` — specific → fallback resolution

```php
public function routeChildDoneEvent(
    State $state,
    StateDefinition $stateDefinition,
    ChildMachineDoneEvent $doneEvent
): void {
    $finalState = $doneEvent->finalState();

    // 1. Try specific @done.{finalState} first
    if ($finalState !== null && isset($stateDefinition->onDoneStateTransitions[$finalState])) {
        $branch = $this->resolveOnDoneOrFailBranch(
            $stateDefinition->onDoneStateTransitions[$finalState],
            $state,
            $doneEvent,
        );

        if ($branch instanceof TransitionBranch) {
            $this->executeChildTransitionBranch($state, $stateDefinition, $branch, $doneEvent);
            return;
        }
        // Guard failed on specific route → fall through to catch-all
    }

    // 2. Fall back to @done catch-all
    if (!$stateDefinition->onDoneTransition instanceof TransitionDefinition) {
        return;
    }

    $branch = $this->resolveOnDoneOrFailBranch(
        $stateDefinition->onDoneTransition,
        $state,
        $doneEvent,
    );

    if (!$branch instanceof TransitionBranch) {
        return;
    }

    $this->executeChildTransitionBranch($state, $stateDefinition, $branch, $doneEvent);
}
```

```php
// Current:
```

```json

    "src": "reportMachine",
    "onDone": [

      { "guard": "__finalState_failed", "target": "error" },
      { "guard": "__finalState_expired", "target": "error" }
    ]
  }
}
```

If both `@done.*` keys and a `@done` catch-all exist, append the catch-all as the last entry without a guard (default branch).

These synthetic guards are for visualization only — they make Stately Studio show the conditional routing.

### Strengths
- Zero API surface change (no new classes, methods, or config keys beyond `@done.*`)
- `Machine::fake(finalState:)` already exists — the parameter just becomes meaningful
- Validation catches misconfiguration early (uncovered final states)
- Dot notation is consistent with existing `@done`/`@fail`/`@timeout` pattern
- Full backward compatibility — no existing tests should break

### Risks
- **Validation coupling**: `validateChildFinalStateCoverage()` calls `$childClass::definition()` at validation time. If the child has side effects in `definition()` or isn't autoloadable during validation, this could fail. Mitigation: try/catch with graceful skip.
- **Nested final states**: If a child has compound states with nested finals (e.g., `processing.sub_approved`), the `key` would be `sub_approved`, which might not match what the parent expects. Mitigation: document that `@done.{state}` matches the leaf key only. For v1 this is acceptable since final states are almost always leaf states.
- **Serialization**: `ChildMachineCompletionJob` is a queued job. Adding `childFinalState` as a constructor param is safe (nullable, string-only), but verify it serializes correctly in the queue.

### Trade-offs Accepted
- **Encapsulation break**: Parent now "knows" child's final state names. This is a deliberate trade-off — EventMachine favors declarative definitions over actor encapsulation. The `with`/`output`/`@done`/`@fail` system already couples parent and child.
- **No `@fail.{state}` equivalent**: Not implementing `@fail.{state}` in this version. `@fail` is triggered by exceptions, not final states, so per-state routing doesn't apply the same way. Can be added later if needed.
