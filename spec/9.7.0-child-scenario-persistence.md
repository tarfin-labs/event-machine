# Child Scenario Persistence + Forward Endpoint Scenario Awareness

**Status:** Draft
**Date:** 2026-04-02

## Problem

When `executeChildScenario()` pauses a child machine at an interactive state, the child is volatile (in-memory only). Three related bugs prevent child scenarios from working in real QA flows:

### Bug 1: Child Not Persisted on Pause

`executeChildScenario()` runs the child with `shouldPersist=false`. When the child pauses at an interactive state (e.g., `awaiting_pin`), nothing is written to DB:
- No `machine_events` history
- No `machine_current_states` record
- No `machine_children` parent-child link

Forward endpoints require `machine_children.child_root_event_id` to restore the child. Without DB records, QA cannot interact with the paused child.

**TDD test:** `tests/Features/ScenarioPlayerChildScenarioTest.php` — "child scenario pausing at interactive state persists child to DB" (INCOMPLETE)

### Bug 2: Child Context Not Passed

`handleScenarioOutcome()` calls `executeChildScenario()` without passing parent context to the child. The `$input` parameter defaults to `[]`. But the child machine needs parent data (e.g., FindeksMachine needs `tckn`, `farmerId`, `birthDate` from CarSalesContext).

Normal delegation (non-scenario) calls `$invokeDefinition->resolveChildContext($state->context)` to resolve the input. The scenario path skips this entirely.

**TDD test:** "child scenario receives parent context via resolveChildContext" (INCOMPLETE)

### Bug 3: Forward Endpoint Scenario-Blind

`executeForwardedEndpoint()` in MachineController does NOT call `maybeRegisterScenarioOverrides()` before `$machine->send()`. When a forwarded event arrives for a child machine with an active continuation scenario:
1. Child is restored from DB (scenario_class is in machine_current_states)
2. `restoreStateFromRootEventId()` registers overrides from DB ✓
3. BUT `executeForwardedEndpoint()` bypasses continuation detection
4. The event is processed without continuation flow (no `executeContinuation()`)

Compare: regular endpoints go through `executeEndpoint()` → `maybeRegisterScenarioOverrides()` → detects continuation → `executeContinuation()`. Forwarded endpoints skip this entirely.

**TDD test:** "forward endpoint activates child continuation overrides" (INCOMPLETE)

## Audit: Similar Issues at Other Code Paths

| Code Path | Scenario Aware? | Issue |
|-----------|----------------|-------|
| `handleMachineIdBound()` | ✓ Yes | Calls `maybeRegisterScenarioOverrides()` via `handleEndpoint()` |
| `handleModelBound()` | ✓ Yes | Same as above |
| `handleStateless()` | ✓ Yes | Same as above |
| `tryForwardEventToChild()` | ❌ No | **Bug 3** — child restored but no continuation check before `send()` |
| `SendToMachineJob::handle()` | ✓ Yes | Restores from DB, overrides auto-registered |
| `ChildMachineCompletionJob::handle()` | ✓ Yes | Restores parent from DB, overrides auto-registered |
| `ChildMachineJob::handle()` | ❌ No | Fresh start, no scenario check — but this is expected: ChildMachineJob runs real child behavior (non-scenario path) |
| `ParallelRegionJob::handle()` | ✓ Yes | Restores from DB, overrides auto-registered |
| `Machine::create(state:)` | ✓ Yes | Delegates to `restoreStateFromRootEventId()` which reads scenario_class |

**Conclusion:** Only `executeForwardedEndpoint()` is broken. All other restoration paths correctly use `restoreStateFromRootEventId()` which auto-registers overrides from DB.

## Proposed Solution

### Fix 1: Persist Child on Interactive Pause

In `executeChildScenario()`, after the @continue loop, when child is NOT final:

```php
// Child paused at interactive state — persist for forward endpoint access
$childMachine->definition->shouldPersist = true;
$childMachine->persist();

$childRootEventId = $childMachine->state->history->first()?->root_event_id;
```

Create `machine_children` record:

```php
$childRootEventId = $childMachine->state->history->first()?->root_event_id;

if ($childRootEventId === null) {
    return null; // Child failed to start — no events recorded
}

MachineChild::create([
    'parent_root_event_id' => $parentRootEventId,
    'parent_machine_class' => $parentMachineClass,
    'parent_state_id'      => $parentStateId,
    'child_root_event_id'  => $childRootEventId,
    'child_machine_class'  => $childMachineClass,
    'status'               => MachineChild::STATUS_RUNNING,
    'created_at'           => now(),
]);
```

Persist scenario for continuation:

```php
if ($childScenario->hasContinuation()) {
    MachineCurrentState::where('root_event_id', $childRootEventId)
        ->update([
            'scenario_class'  => $childScenarioClass,
            'scenario_params' => null,
        ]);
}
```

### Fix 2: Pass Child Context from Parent

In `handleScenarioOutcome()`, resolve child context before calling `executeChildScenario()`:

```php
$invokeDefinition = $stateDefinition->getMachineInvokeDefinition();
$childContext     = $invokeDefinition->resolveChildContext($state->context);

$childState = ScenarioPlayer::executeChildScenario(
    childScenarioClass: $childScenario,
    childMachineClass: $invokeDefinition->machineClass,
    input: $childContext,
    parentRootEventId: $state->history->first()?->root_event_id,
    parentMachineClass: $this->machineClass,
    parentStateId: $stateDefinition->id,
);
```

### Fix 3: Add Continuation Check to `tryForwardEventToChild()`

The forwarded endpoint path operates on the PARENT machine, but the child's scenario is stored under the CHILD's `root_event_id`. Adding `maybeRegisterScenarioOverrides()` to `executeForwardedEndpoint()` is wrong — it would query the parent's scenario_class (which doesn't exist).

The correct location is `MachineDefinition::tryForwardEventToChild()` (line ~1971), where the child is restored from DB. `Machine::create(state:)` → `restoreStateFromRootEventId()` already auto-registers overrides from the child's `scenario_class`. The only missing piece: check if the child has an active continuation and dispatch via `executeContinuation()` instead of plain `send()`.

Extract a helper method `tryForwardWithContinuation()` that returns `true` if continuation handled the event:

```php
// In tryForwardEventToChild(), after restoring child:
$childMachine = $childMachineClass::create(state: $childRecord->child_root_event_id);

// Try continuation first — if child has active continuation, dispatch via executeContinuation
if (!$this->tryChildContinuation($childMachine, $childRecord->child_root_event_id, $childEventType, $eventBehavior)) {
    // Normal (non-scenario) forwarded event
    $childMachine->send(['type' => $childEventType, 'payload' => $eventBehavior->payload]);
}

// If child reached a final state, dispatch completion...
```

Helper method:

```php
private function tryChildContinuation(
    Machine $childMachine,
    string $childRootEventId,
    string $childEventType,
    EventBehavior $eventBehavior,
): bool {
    if (!(bool) config('machine.scenarios.enabled', false)) {
        return false;
    }

    $record = MachineCurrentState::where('root_event_id', $childRootEventId)
        ->whereNotNull('scenario_class')
        ->first(['scenario_class', 'scenario_params']);

    if ($record === null || !is_string($record->scenario_class) || !class_exists($record->scenario_class)) {
        return false;
    }

    $scenario = new ($record->scenario_class)();

    if (!$scenario->hasContinuation()) {
        return false;
    }

    // hydrateParams MUST precede executeContinuation — hasContinuation() inside
    // the continuation loop depends on hydrated state for re-persistence decisions
    $scenario->hydrateParams($record->scenario_params ?? []);
    $scenario->isContinuation = true;

    $player = new ScenarioPlayer($scenario);

    // executeContinuation handles: override registration, machine->send(),
    // @continue loop, scenario deactivation on final, re-persistence on pause
    $player->executeContinuation(
        machine: $childMachine,
        eventPayload: $eventBehavior->payload ?? [],
        rootEventId: $childRootEventId,
        eventType: $childEventType,
    );

    return true;
}
```

**Key notes:**
- `executeContinuation()` already handles the full scenario lifecycle: registers continuation overrides, sends the event, runs @continue loop, deactivates on final state, re-persists on interactive pause
- No redundant scenario cleanup needed after this method returns
- `$childMachine->state` is updated internally by `executeContinuation()` via `machine->send()` — no explicit assignment needed
- `MachineDefinition.php` needs `use Tarfinlabs\EventMachine\Models\MachineCurrentState;` import (not currently imported)
- `hydrateParams()` MUST be called before `executeContinuation()` — continuation loop depends on hydrated scenario state

## New Parameters for executeChildScenario

```php
public static function executeChildScenario(
    string $childScenarioClass,
    string $childMachineClass,
    array $input = [],
    ?string $parentRootEventId = null,
    ?string $parentMachineClass = null,
    ?string $parentStateId = null,
): ?State
```

Optional parameters preserve backward compatibility. When parent context is provided AND child pauses, persistence activates.

## What Gets Added

| Component | Change |
|-----------|--------|
| `ScenarioPlayer::executeChildScenario()` | Persist child on interactive pause, create machine_children record, persist scenario_class |
| `MachineDefinition::handleScenarioOutcome()` | Pass resolveChildContext() + parent context to executeChildScenario |
| `MachineDefinition::tryForwardEventToChild()` | Check child's scenario_class, dispatch via executeContinuation() if continuation active |

## What Doesn't Change

- `execute()` — unchanged
- `executeContinuation()` — unchanged
- `handleMachineIdBound()` / `handleModelBound()` — already scenario-aware
- `SendToMachineJob` / `ChildMachineCompletionJob` — already scenario-aware via restore
- `Machine::create(state:)` — already reads scenario_class on restore
- `executeForwardedEndpoint()` — unchanged (operates on parent, not child)
- Child scenarios reaching FINAL state — no persistence needed
