# 9.11.0 — Reject `@queue` in non-listener tuples

## Motivation

`@queue` is the only framework-reserved `@`-prefixed tuple key today. It's documented as
"queues this listener action to a worker" and supported by `MachineDefinition::parseListenConfig`
in `listen.entry`, `listen.exit`, and `listen.transition` config.

Until 9.11.0, putting `@queue` anywhere else — for example, in a state's `entry` action list —
was *silently dropped* by `BehaviorTupleParser::parseTuple`, which filtered all `@`-prefixed keys
as "framework metadata". The action would then run synchronously inline, with no exception, no
log line, and no warning. Users would assume their work was queued when it wasn't.

A real-world report from production code looked like:

```php
'approved' => [
    'entry' => [
        TerminateOtherApplicationsAction::class,
        ApproveAction::class,
        CompleteApplicationAction::class,
        [CreateInstallmentPromissoryNoteAction::class, '@queue' => true], // ← silently sync
        SendApprovalNotificationAction::class,
    ],
],
```

The user expected the third action to run async. It didn't, and there was no way to tell from the
config.

## Change

`BehaviorTupleParser::parseTuple` (called for actions, guards, calculators, outputs, and transition
behaviors) now throws `InvalidBehaviorDefinitionException::reservedKeyInTuple()` when it encounters
any `@`-prefixed key. The exception message points at `listen.entry` / `listen.exit` /
`listen.transition` and lists the three async-action options (job actor, queued listener,
inline `dispatch()`).

State `entry` and `exit` actions are now also validated **at definition time**: `StateDefinition`
calls `BehaviorTupleParser::parse()` over each action tuple in `initializeEntryActions()` /
`initializeExitActions()`, so misuse fails fast at machine definition rather than waiting for the
state to be entered at runtime.

Listener parsing (`MachineDefinition::parseListenConfig`) is unchanged — it has its own parser and
never reaches `BehaviorTupleParser`.

## Backward compatibility

This is **technically breaking** for any code that previously placed `@queue` in a non-listener
tuple. Such code was already broken (the `@queue` had no effect), but instead of failing silently
it now throws `InvalidBehaviorDefinitionException` immediately on `MachineDefinition::define()`.

We treat this as a fix for a footgun rather than a feature change. Tagged as `9.11.0` (minor) per
the project's "fix silently-broken behavior" convention. No deprecation cycle: there's no plausible
caller relying on the silent drop.

## Files changed

- `src/Support/BehaviorTupleParser.php` — reject `@`-prefixed keys in `parseTuple`
- `src/Exceptions/InvalidBehaviorDefinitionException.php` — new factory `reservedKeyInTuple()`
- `src/Definition/StateDefinition.php` — eager validation in `initializeEntryActions()` and
  `initializeExitActions()` via new `validateActionTuples()`
- `tests/Features/NamedParamsTest.php` — new `§K — Reserved @-prefix keys` (Tests 78–83)
- `docs/building/defining-states.md` — explicit warning callout + "Async Work in Entry Actions"
  decision table with three rewrite examples
- `docs/behaviors/introduction.md` — `@` Prefix Convention section now states the scope rule and
  shows the rejected case
- `skills/event-machine/SKILL.md` — new gotcha "@queue works only in `listen` config" with the
  three async-action options
