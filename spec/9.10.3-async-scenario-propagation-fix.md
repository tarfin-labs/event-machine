# 9.10.3 — Async Scenario Propagation Fix

## Problem

Backend agent reported in 9.10.1 (still present in 9.10.2): scenarios with a child-scenario reference at a delegation state with `'queue' => '...'` configuration do not propagate to the dispatched child machine. The async ChildMachineJob worker boots the child without scenario context; leaf-state delegation outcomes (e.g., `'phone_resolution.checking_single_phone' => ['outcome' => '@done', 'output' => [...]]`) never fire; the child runs full I/O.

Example incident: 162 real Findeks API calls from a single QA session that was supposed to skip them.

## Root Cause Analysis

`ScenarioPlayer::play()` performs three coordinated steps to make scenario interception work (lines 90, 93, 97):

1. **`self::$isActive = true`** — gates the interception in `MachineDefinition::handleMachineInvoke` (line 1479)
2. **`registerOverrides($scenario)`** — populates static behavior overrides (Guard/Action class fakes)
3. **`classifyPlanValues()`** — populates `self::$outcomes` and `self::$childScenarios` (the source of truth for `getOutcome()`/`getChildScenario()`)

`MachineDefinition::handleScenarioOutcome()` (line 1815) needs all three to function:
- Reads `getOutcome($stateRoute)` and `getChildScenario($stateRoute)` (depends on #3)
- Only called when `isActive` (depends on #1)
- Behavior overrides from #2 fire via `InvokableBehaviorFake` independently

Three places restore scenarios for an already-running machine:

- **Machine.php §9 Async Propagation (line 600)** — calls only `registerOverrides`. Misses #1, #3.
- **ScenarioPlayer::executeChildScenario (line 392)** — calls `registerOverrides` and inlines the classify logic at line 396-421. Has #2, #3 but only inside the player's process.
- **ChildMachineJob::handle()** — does NOTHING scenario-related. Misses all three.

The user's bug exhibits when the parent's scenario has a child-scenario reference at an `'queue' => '...'` state AND `handleScenarioOutcome` doesn't intercept (e.g., parent transitioned naturally outside the @continue loop). The real ChildMachineJob dispatches → worker boots child fresh → no scenario context → I/O fires.

## Fix

Surgical, non-breaking, no migration:

### Change 1: `ScenarioPlayer` — public activation helpers

Add two public static methods to encapsulate the three-step activation:

```php
public static function activateForAsyncBoot(MachineScenario $scenario): void
public static function deactivate(): void
```

Internally:
- `activateForAsyncBoot` — clears `$outcomes`/`$childScenarios`, classifies the scenario plan, registers behavior overrides, sets `$isActive = true`
- `deactivate` — calls `cleanupOverrides`, clears `$outcomes`/`$childScenarios`, sets `$isActive = false`

Refactor private `classifyPlanValues` body into a static helper that takes a plan array, so both `play()` and `activateForAsyncBoot` reuse it.

### Change 2: `ChildMachineJob` — carry scenarioClass payload

Constructor adds `?string $scenarioClass = null`. `handle()`:

```php
if ($this->scenarioClass !== null && is_subclass_of($this->scenarioClass, MachineScenario::class)) {
    ScenarioPlayer::activateForAsyncBoot(new $this->scenarioClass());
}

try {
    // ... existing child machine create + start + persist ...
    
    // After persist, write scenario_class to current_states row for subsequent restores:
    if ($this->scenarioClass !== null && $childRootEventId !== null) {
        MachineCurrentState::where('root_event_id', $childRootEventId)
            ->update(['scenario_class' => $this->scenarioClass]);
    }
} finally {
    if ($this->scenarioClass !== null) {
        ScenarioPlayer::deactivate();
    }
}
```

### Change 3: `MachineDefinition::handleAsyncMachineInvoke` — thread scenarioClass through dispatch

At line 1689, before constructing `ChildMachineJob`:

```php
$scenarioClass = ScenarioPlayer::isActive()
    ? ScenarioPlayer::getChildScenario($stateDefinition->id)
    : null;

$job = new ChildMachineJob(
    // ... existing params ...
    scenarioClass: $scenarioClass,
);
```

### Change 4: `ScenarioPlayer::executeChildScenario` line 516 — relax `hasContinuation()`

Always persist `scenario_class` on the child row when the player paused with a child running, regardless of whether the child scenario declares `@continue`. Outcome-only overrides need this for subsequent restores.

### Change 5: `Machine::restoreStateFromRootEventId` §9 — populate outcomes

Replace the existing `registerOverrides`-only call with `activateForAsyncBoot`-style activation when restoring a row that has `scenario_class`. (Subsequent restorations during a job's life — e.g., SendToMachineJob — need full activation, not just behavior overrides.)

Be careful: §9 has a `!isActive` guard to avoid double-registering during the player's run. Keep that guard. Activation only happens for fresh worker processes where isActive starts false.

## Acceptance Criteria

1. Scenario with child-scenario reference at `'queue' => '...'` parent state: child machine dispatched, worker boots, leaf-state delegation outcome fires, no real I/O.
2. `MachineCurrentState.scenario_class` row populated on the child after async dispatch.
3. Subsequent SendToMachineJob events on the async child apply scenario overrides (outcomes + behaviors).
4. No-scenario flow unchanged (regression check).
5. Sync child path unchanged (regression check).
6. Quality gate green; phpstan clean; type coverage 100%.

## Test Plan

`tests/Features/AsyncScenarioPropagationTest.php`:

- **Happy path:** parent scenario with child reference + `queue` → ChildMachineJob dispatched with scenarioClass set; worker handle() activates ScenarioPlayer; child's leaf delegation outcome fires; no real child run.
- **Regression — no scenario:** ChildMachineJob without scenarioClass behaves exactly as before.
- **Restore propagation:** after async boot, scenario_class on row enables subsequent SendToMachineJob to apply overrides.
- **hasContinuation relaxation:** child paused at interactive state with outcome-only scenario → row populated, restore works.

## Out of Scope

- Synthetic completion primitive (`'outcome' => '@done.X', 'output' => [...]'` at parent level) — already implemented in 9.10.x at MachineDefinition.php:1869-1921. Backend agent unaware. Documentation/skill update should follow-up but no code change needed.
- Outbox pattern for SIGTERM-resistant propagation — separate v10 epic.
- @fail / @timeout simulation symmetry — already implemented at MachineDefinition.php:1876-1903.
