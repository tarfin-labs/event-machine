## Architecture Audit (VERIFIED)

### Recursive Dispatch Points — `$this->transition()` Calls

8 points total, all within the same PHP call stack (same macrostep):

| # | Line | Method | Trigger | Status |
|---|------|--------|---------|--------|
| 1 | 387 | `getInitialState()` | `@always` on initial state | ✅ propagates depth |
| 2 | 403 | `getInitialState()` | Event queue after init | ✅ propagates depth |
| 3 | 2179 | `transitionParallelState()` | `@always` on region after parallel processing | ✅ propagates depth |
| 4 | 2201 | `transitionParallelState()` | Event queue after parallel processing | ✅ propagates depth |
| 5 | 2342 | `transition()` | `@always` on active region after entering parallel target | ⚠️ **MISSING — must fix** |
| 6 | 2359 | `transition()` | Event queue after entering parallel target | ⚠️ **MISSING — must fix** |
| 7 | 2374 | `transition()` | `@always` on new state (normal non-parallel) | ✅ propagates depth |
| 8 | 2391 | `transition()` | Event queue after normal transition | ✅ propagates depth |

**Critical: Points #5 and #6 do NOT pass `recursionDepth`.** An @always loop through a parallel state entry path will bypass the depth guard entirely. Must fix before writing tests.

### v7 Methods That Do NOT Call `$this->transition()` (No Propagation Needed)

| Method | Mechanism Used | Why Safe |
|--------|---------------|----------|
| `routeChildDoneEvent()` | `executeChildTransitionBranch()` | Bypasses `transition()` entirely |
| `routeChildFailEvent()` | `executeChildTransitionBranch()` | Same bypass |
| `routeChildTimeoutEvent()` | `executeChildTransitionBranch()` | Same bypass |
| `processCompoundOnDone()` | Self-recursion (not `transition()`) | Own recursion pattern |
| `processParallelOnDone()` | `exitParallelStateAndTransitionToTarget()` | Different code path |
| `processParallelOnFail()` | `exitParallelStateAndTransitionToTarget()` | Different code path |
| `processNestedParallelCompletion()` | `exitParallelStateAndTransitionToTarget()` | Different code path |
| `handleMachineInvoke()` | Creates child `MachineDefinition` | Different machine object |
| `sendTo()` / `dispatchTo()` | Operates on target machine | Different machine object |
| `tryForwardEventToChild()` | Restores child from DB | Different machine object |

### Known Limitation (Separate Issue — Document, Don't Fix Here)

`executeChildTransitionBranch()` and `exitParallelStateAndTransitionToTarget()` set a new state but do NOT evaluate `@always` transitions on that new state. This means:
- After @done/@fail routes to a new state with `@always`, the `@always` won't fire
- This is a correctness gap (not a loop protection issue)
- Track as separate issue
