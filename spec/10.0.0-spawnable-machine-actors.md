# Spawnable Machine Actors

## Status
Draft

## Problem

EventMachine's current delegation model (`machine` key) uses **state-scoped lifetime**: a child machine is created when the parent enters a delegating state and its completion is routed via that state's `@done`/`@fail`. This is the SCXML `<invoke>` pattern — deterministic and clean, but insufficient for three real-world patterns:

1. **Resumable delegation** — A parent needs to interact with the same child machine from multiple states across its lifecycle. Example: CarSalesMachine starts FindeksMachine early for phone resolution (Phase 1), then resumes the same instance later for report retrieval (Phase 2). Today this requires duplicating logic in a standalone job + creating a fresh child that redoes Phase 1 work.

2. **Dynamic fan-out** — A parent needs to spawn N child machines at runtime based on data (e.g., process 10 items in parallel). The current `machine` key only supports static, config-time declaration of a single child per state.

3. **Multi-phase child interaction** — A child reaches a "checkpoint" state (not final) and the parent needs to know about it, continue its own flow, then send additional data to the child later. Current model only signals on `@done`/`@fail` (final states).

### Research Foundation

Five actor systems were analyzed. All converge on two delegation models:

| Model | Systems | Lifetime | EventMachine |
|-------|---------|----------|-------------|
| **Invoke** (state-scoped) | SCXML `<invoke>`, XState `invoke` | Tied to parent state | Current `machine` key |
| **Spawn** (independent) | XState `spawn()`, Erlang `spawn_monitor`, Temporal `startChild(ABANDON)`, Akka `actorOf`, Orleans `GetGrain` | Independent of parent state | **This spec** |

All systems that support both models (XState v5, Akka, Erlang) treat them as complementary — invoke for simple deterministic cases, spawn for dynamic/multi-phase cases. This spec adds spawn while preserving invoke.

## Solution

Add three new primitives to EventMachine:

1. **`MachineRef`** — A serializable, first-class reference to a machine instance
2. **`spawn()`** — An `InvokableBehavior` method that creates independent-lifetime child machines
3. **Actor Registry** — A per-parent persistent registry of spawned actor IDs and their MachineRefs

### Design Principles

- **Spawn complements invoke, doesn't replace it.** The existing `machine` key delegation continues unchanged.
- **`InvokableBehavior` is the communication hub.** `spawn()` lives alongside `raise()`, `sendTo()`, `dispatchTo()` — not on ContextManager (which is a data container, not a side-effect producer).
- **Actor IDs are user-assigned aliases.** Every spawned machine gets a human-readable `actorId` scoped to its parent, persisted in the actor registry.
- **Parent identity is engine-managed.** Spawned children know their parent automatically — behaviors don't need to pass `$context` for parent communication.
- **Refs are persistent.** The actor registry ensures refs survive across parent state transitions, restores, and even archival.

## Specification

### 1. MachineRef Value Object

`MachineRef` is an immutable, serializable reference to a machine instance.

```php
namespace Tarfinlabs\EventMachine\Actor;

class MachineRef implements \JsonSerializable, \Stringable
{
    public function __construct(
        public readonly string $machineClass,   // FQCN of the Machine subclass
        public readonly string $rootEventId,    // machine instance identity
        public readonly string $actorId,        // user-assigned alias (unique per parent)
    ) {}

    // ── Communication ──────────────────────────────────────

    /** Send an event synchronously (blocking, same process). */
    public function send(string|EventBehavior|array $event): void;

    /** Dispatch an event asynchronously (via queue job). */
    public function dispatch(string|EventBehavior|array $event): void;

    // ── Query (read-only, no lock acquired) ────────────────

    /** Get a snapshot of the machine's current state. */
    public function snapshot(): MachineSnapshot;

    /** Get the machine's output (same as Machine::output()). */
    public function output(): mixed;

    /** Get the current state value array. */
    public function state(): array;

    /** Check if machine is in a specific state. */
    public function isInState(string $state): bool;

    // ── Lifecycle ──────────────────────────────────────────

    /** Machine exists and has not reached a final state. */
    public function isRunning(): bool;

    /** Machine has reached a final state. */
    public function isCompleted(): bool;

    /** Machine has reached a @fail final state. */
    public function isFailed(): bool;

    // ── Serialization ──────────────────────────────────────

    /** @return array{machineClass: string, rootEventId: string, actorId: string} */
    public function jsonSerialize(): array;

    /** Reconstruct from serialized form (for context persistence). */
    public static function fromArray(array $data): self;

    public function __toString(): string;
}
```

#### MachineSnapshot

Read-only snapshot returned by `MachineRef::snapshot()`:

```php
enum ActorStatus: string
{
    case Running   = 'running';
    case Completed = 'completed';
    case Failed    = 'failed';
}

class MachineSnapshot
{
    public function __construct(
        public readonly array $state,        // state value array
        public readonly array $context,      // toResponseArray() output
        public readonly ActorStatus $status, // lifecycle status
        public readonly mixed $output,       // output() result if completed, null otherwise
    ) {}
}
```

#### Query Methods Are Lock-Free

`snapshot()`, `output()`, `state()`, `isInState()`, `isRunning()`, `isCompleted()`, `isFailed()` restore the machine from events but do **not** acquire a lock or persist anything. They are read-only operations safe to call concurrently. This follows Orleans' `[AlwaysInterleave]` pattern for read-only grain methods.

**Re-entrancy note:** `MachineRef::send()` acquires a lock on the **target** machine. Calling `send()` from within a behavior on the parent is safe (different lock). However, if the target machine synchronously calls back to the parent (e.g., via `sendToParent`), the parent's lock is already held — this follows the existing re-entrant lock pattern via `Machine::$heldLockIds`.

### 2. spawn() on InvokableBehavior

`spawn()` is a method on `InvokableBehavior`, alongside existing communication methods:

| Existing | New |
|----------|-----|
| `$this->raise(event)` | `$this->spawn(class, actorId, input)` |
| `$this->sendTo(class, id, event)` | `$this->ref(actorId)` |
| `$this->dispatchTo(class, id, event)` | `$this->refs()` |
| `$this->sendToParent(event)` | — |
| `$this->dispatchToParent(event)` | — |

**Guard purity:** `spawn()` is only available on `ActionBehavior` subclasses. Calling `spawn()` from a `GuardBehavior` or `CalculatorBehavior` throws `SpawnNotAllowedInGuardException`. The `ref()` and `refs()` read-only methods are available in all behaviors (guards may need to check child state).

**Execution context:** `spawn()`, `ref()`, and `refs()` require the current machine's `rootEventId` to query the actor registry. This is provided via the `ActorContext` instance, which is constructed by the engine with the current machine's identity and injected into behaviors. `InvokableBehavior` delegates to its internal `ActorContext` reference (set by the engine before invocation, same pattern as the `eventQueue`).

```php
abstract class InvokableBehavior
{
    // ── Existing methods ───────────────────────────────────
    public function raise(EventBehavior|array $eventBehavior): void;
    public function sendTo(string $machineClass, string $rootEventId, EventBehavior|array $event): void;
    public function dispatchTo(string $machineClass, string $rootEventId, EventBehavior|array $event): void;
    public function sendToParent(ContextManager $context, EventBehavior|array $event): void;
    public function dispatchToParent(ContextManager $context, EventBehavior|array $event): void;

    // ── New methods ────────────────────────────────────────

    /**
     * Spawn an independent-lifetime child machine.
     *
     * Creates the child machine, persists it to the actor registry,
     * and returns a MachineRef. The rootEventId is pre-generated (ULID)
     * before dispatch, so the MachineRef is valid immediately even for
     * async spawns where the child hasn't started yet.
     *
     * When async=true (default), dispatches SpawnedMachineJob to the queue.
     * When async=false, starts the child synchronously in the same process.
     *
     * Throws DuplicateActorIdException if an actor with the same actorId
     * already exists for this parent. Throws SpawnNotAllowedInGuardException
     * if called from a GuardBehavior or CalculatorBehavior.
     *
     * @param  string  $machineClass  FQCN of the Machine subclass to spawn.
     * @param  string  $actorId       User-assigned alias, unique per parent.
     * @param  array<string, mixed>|null  $input  Initial context data for the child.
     * @param  bool  $async  If true (default), child starts via queue job. If false, child starts synchronously.
     */
    public function spawn(
        string $machineClass,
        string $actorId,
        ?array $input = null,
        bool $async = true,
        ?string $completionEvent = null,
        ?string $queue = null,
    ): MachineRef;

    /**
     * Get a MachineRef by actorId from the actor registry.
     *
     * Returns null if no actor with that ID exists for the current parent.
     */
    public function ref(string $actorId): ?MachineRef;

    /**
     * Get all MachineRefs from the actor registry for the current parent.
     *
     * @return array<string, MachineRef>  Keyed by actorId.
     */
    public function refs(): array;
}
```

#### spawn() in Inline Closures

Inline closures receive injected parameters but don't have `$this`. For closures, `spawn()`, `ref()`, and `refs()` are available through a new injectable `ActorContext`:

```php
'entry' => function (ContextManager $ctx, ActorContext $actors) {
    $ref = $actors->spawn(
        machineClass: FindeksMachine::class,
        actorId: 'findeks',
        input: ['tckn' => $ctx->get('tckn')],
    );
},

// Later, in another state:
'entry' => function (ContextManager $ctx, ActorContext $actors) {
    $actors->ref('findeks')->dispatch([
        'type'    => ReportRequestedEvent::class,
        'payload' => ['applicationId' => $ctx->get('applicationId')],
    ]);
},
```

`ActorContext` is a new injectable type that wraps the actor registry for the current machine instance:

```php
class ActorContext
{
    public function __construct(
        private readonly string $parentRootEventId,
        private readonly string $parentMachineClass,
        private readonly ActorRegistry $registry,
    ) {}

    public function spawn(string $machineClass, string $actorId, ?array $input = null, bool $async = true, ?string $completionEvent = null, ?string $queue = null): MachineRef;
    public function ref(string $actorId): ?MachineRef;
    /** @return array<string, MachineRef> */
    public function refs(): array;
}
```

Both `InvokableBehavior` methods and `ActorContext` delegate to the same underlying `ActorRegistry`.

#### ActorContext Injection Mechanism

`ActorContext` is provided to behaviors through two paths:

1. **For `InvokableBehavior` subclasses:** The engine sets a `protected ?ActorContext $actorContext` property on `InvokableBehavior` base class (alongside the existing `protected ?Collection $eventQueue`). This is set **after** construction via a setter method `setActorContext()` called by the engine before invocation — same approach as `eventQueue` which is passed via constructor. No constructor signature change needed; existing subclasses are unaffected. The `spawn()`, `ref()`, `refs()` methods on `InvokableBehavior` delegate to `$this->actorContext`.

2. **For inline closures:** `ActorContext` is added to the type-match list in `injectInvokableBehaviorParameters()`. A new parameter `?ActorContext $actorContext = null` is added to the static method signature. When a closure type-hints `ActorContext`, it receives the instance. The engine constructs `ActorContext` once per macrostep and passes it to `injectInvokableBehaviorParameters()`.

**Guard purity for inline closures:** When `ActorContext` is injected into a guard/calculator closure, the `spawn()` method checks an internal flag `$this->readOnly` (set by the engine based on behavior type). If `readOnly` is true, `spawn()` throws `SpawnNotAllowedInGuardException`. Read-only methods (`ref()`, `refs()`) always work.

#### SpawnedMachineJob

`SpawnedMachineJob` is the queue job dispatched when `spawn(async: true)` is called. It:

1. Receives `machineClass`, `rootEventId` (pre-generated ULID), `input`, `parentRootEventId`, and `parentMachineClass`
2. Creates the child machine: `$machineClass::create(input: $input, rootEventId: $rootEventId)`
3. Sets machine identity on the child's context: `setMachineIdentity($rootEventId, $parentRootEventId, $parentMachineClass)`
4. Updates `machine_children` status from `'pending'` to `'running'`

This is analogous to the existing `ChildMachineJob` but without the `@done`/`@fail` routing — completion is handled by `ActorCompletionJob` via normal event flow.

#### Sync Spawn Timing

When `async=false`, the child machine is started **immediately** during the action (not deferred to after the macrostep like SCXML invoker-05). This is intentional: sync spawn is an explicit imperative action, not a declarative state config. The parent's macrostep continues after the child's `start()` completes. If the child's entry actions call `sendToParent()`, this is handled within the same process (re-entrant lock via `Machine::$heldLockIds`).

#### Spawn and Guard Context Snapshot

EventMachine's guard purity enforcement snapshots context before evaluating guards and rolls back if a guard fails. Since `spawn()` is only allowed in **actions** (not guards), and actions only run after guards pass, there is no guard-rollback scenario for spawn.

However, in multi-branch transitions (multiple `[guard, actions, target]` branches), if Branch A's guard passes and its action calls `spawn()`, but then Branch A's transition is superseded (e.g., by a higher-priority branch), the `machine_children` record and the dispatched `SpawnedMachineJob` cannot be rolled back. This is acceptable — the spawned child runs as a fire-and-forget orphan, same as any side effect in actions. **Users should only call `spawn()` in actions on transitions where the guard outcome is certain.**

For async spawns, if the parent's macrostep ultimately fails (uncaught exception), the `SpawnedMachineJob` is already dispatched. The child will start and become orphaned. This matches the existing behavior of `dispatchTo()` — queue-dispatched side effects are not transactional.

### 3. Actor Registry

A per-parent persistent store of spawned actor relationships. Extends the existing `machine_children` table.

#### Schema Changes

```sql
ALTER TABLE machine_children ADD COLUMN actor_id VARCHAR(255) NULLABLE;
ALTER TABLE machine_children ADD COLUMN lifetime VARCHAR(20) NOT NULL DEFAULT 'scoped';
ALTER TABLE machine_children ADD COLUMN completion_event VARCHAR(255) NULLABLE;

-- Actor ID is unique per parent (partial index — excludes NULLs for scoped children)
CREATE UNIQUE INDEX idx_parent_actor ON machine_children (parent_root_event_id, actor_id)
    WHERE actor_id IS NOT NULL;
```

**SQLite compatibility:** SQLite supports partial indexes with `WHERE` clauses. For MySQL (which does not support partial indexes), the unique constraint is enforced at the application level in `ActorRegistry::register()` before insert. The migration uses a standard index (not unique) on MySQL, with a pre-insert query to check for duplicates.

| Column | Existing | New | Description |
|--------|----------|-----|-------------|
| `id` | Yes | — | ULID primary key |
| `parent_root_event_id` | Yes | — | Parent machine identity |
| `parent_state_id` | Yes | — | State that created the child (for independent: state where `spawn()` was called) |
| `parent_machine_class` | Yes | — | Parent FQCN |
| `child_machine_class` | Yes | — | Child FQCN |
| `child_root_event_id` | Yes | — | Child machine identity (pre-generated ULID for async spawns) |
| `status` | Yes | — | pending/running/completed/failed |
| `actor_id` | — | Yes | User-assigned alias (unique per parent when non-null) |
| `lifetime` | — | Yes | `scoped` (existing behavior) or `independent` (new) |
| `completion_event` | — | Yes | Custom completion event type override (nullable) |

Existing `machine` key delegation writes `lifetime = 'scoped'` and `actor_id = null` (backward compatible). `spawn()` writes `lifetime = 'independent'` and the user-provided `actor_id`.

#### Registry Resolution

`ref('findeks')` queries `machine_children` by `parent_root_event_id + actor_id` and constructs a `MachineRef`. This is a database lookup, not an in-memory cache — consistent with EventMachine's stateless model.

### 4. Parent Identity — Engine-Managed

Currently, `dispatchToParent()` requires passing `$context` to extract transient parent identity. This is fragile (identity is lost after restore) and leaks engine concerns into behavior code.

#### Change: Parent Identity Persisted in Actor Registry

When a child is spawned, the actor registry record already contains `parent_root_event_id` and `parent_machine_class`. The child machine can look up its parent from the registry rather than relying on transient context properties.

#### Simplified API

Both `sendToParent()` and `dispatchToParent()` are simplified:

```php
// BEFORE (current): requires ContextManager, fragile
$this->sendToParent($context, ['type' => 'PHONES_RESOLVED', 'payload' => [...]]);
$this->dispatchToParent($context, ['type' => 'PHONES_RESOLVED', 'payload' => [...]]);

// AFTER: engine resolves parent from actor registry
$this->sendToParent(['type' => 'PHONES_RESOLVED', 'payload' => [...]]);
$this->dispatchToParent(['type' => 'PHONES_RESOLVED', 'payload' => [...]]);
```

The `InvokableBehavior` base class resolves the parent by querying `machine_children` for the current machine's `rootEventId` as `child_root_event_id`. This works for both `scoped` (existing) and `independent` (new) children.

**Resolution order:** The engine first checks the `ActorContext` for parent identity (populated from `machine_children` at behavior invocation time). The query uses `child_root_event_id` without filtering by status — a completed/failed parent record still contains valid parent identity. If no record is found (e.g., standalone machine), `NoParentMachineException` is thrown — same as today. If multiple records exist (a machine can be a child in multiple contexts, e.g., re-spawned), the most recent record is used.

**Backward compatibility:** The PHP signatures change to accept both old and new calling conventions via argument count:

```php
// New unified signature — detects old vs new by argument count
public function sendToParent(ContextManager|EventBehavior|array $eventOrContext, EventBehavior|array|null $event = null): void
{
    if ($event !== null) {
        // Old path: sendToParent($context, $event) — 2 args, deprecated
        @trigger_error('Passing ContextManager to sendToParent() is deprecated. Use sendToParent($event) instead.', E_USER_DEPRECATED);
        // resolve parent from $eventOrContext (ContextManager)
    } else {
        // New path: sendToParent($event) — 1 arg
        // resolve parent from ActorContext (engine-managed)
    }
}
```

Same pattern for `dispatchToParent()`. The old 2-argument form is deprecated and will be removed in v11. This approach avoids PHP's lack of method overloading and is PHPStan-compatible (union type on first param).

### 5. Child-to-Parent Communication

When a spawned child needs to signal its parent (e.g., "Phase 1 done"), it uses existing communication primitives with the simplified parent API:

```php
// Inside FindeksMachine — phones resolved, signal parent
class NotifyPhonesResolvedAction extends ActionBehavior
{
    public function __invoke(ContextManager $ctx): void
    {
        $this->dispatchToParent([
            'type' => 'FINDEKS_PHONES_RESOLVED',
            'payload' => ['phones' => $ctx->get('resolvedPhones')],
        ]);
    }
}
```

The parent receives this as a normal event and transitions accordingly. No new "signal" or "checkpoint" concept needed — events are sufficient.

### 6. Completion Propagation

When an independent-lifetime child reaches a **final state**, the engine dispatches a completion event to the parent. **This uses a fundamentally different routing path than scoped children:**

| Aspect | Scoped (`machine` key) | Independent (`spawn()`) |
|--------|------------------------|------------------------|
| Completion job | `ChildMachineCompletionJob` | `ActorCompletionJob` (new) |
| Routing mechanism | `routeChildDoneEvent` / `routeChildFailEvent` (reads `@done`/`@fail` config) | `Machine::send()` (normal event dispatch) |
| Idempotency check | `parentStateId` must match current state | None — event is delivered to whatever state the parent is in |
| Event type | `done.invoke.{childId}` (internal) | `{actorId}.done` / `{actorId}.fail` (user-facing, in `on` config) |

**Why a separate job?** The existing `ChildMachineCompletionJob` is tightly coupled to scoped semantics: it checks that the parent is still in the invoking state (`parentStateId` idempotency guard), then calls `routeChildDoneEvent` which reads `@done`/`@fail` from the `StateDefinition`. Independent actors survive parent state transitions by design, and their completion events are normal transitions in `on` config — not `@done`/`@fail` config.

`ActorCompletionJob` is simpler: it calls `Machine::send()` on the parent with the completion event. It catches two exceptions silently: `NoTransitionDefinitionFoundException` (parent state doesn't handle this event) and `MachineAlreadyRunningException` (parent is being concurrently processed — the job retries via standard queue retry).

**Completion event payload:**

```php
// Done event payload
['type' => 'findeks.done', 'payload' => [
    'actorId'  => 'findeks',           // which actor completed
    'output'   => [...],               // child machine output()
    'childRootEventId' => '01HX...',   // child identity for direct queries
]]

// Fail event payload
['type' => 'findeks.fail', 'payload' => [
    'actorId'  => 'findeks',
    'error'    => '...',               // failure info
    'childRootEventId' => '01HX...',
]]
```

The `actorId` in the payload is critical for the fan-out pattern — when multiple actors share the same `completionEvent`, the parent can identify which child completed.

**Dispatch trigger:** `ActorCompletionJob` is dispatched from `Machine::persist()` (or the final-state detection path). After persisting a machine that has reached a final state, the engine checks `machine_children` for the current machine's `rootEventId` as `child_root_event_id` with `lifetime = 'independent'`. If found, it dispatches `ActorCompletionJob` to the parent. This mirrors the existing scoped child completion path that dispatches `ChildMachineCompletionJob`.

**Chain propagation:** When `ActorCompletionJob` delivers a completion event via `Machine::send()` and this causes the parent to reach a final state, the parent's own `persist()` triggers its own completion check. If the parent is itself a scoped child, `ChildMachineCompletionJob` is dispatched to the grandparent. If the parent is itself an independent child, another `ActorCompletionJob` is dispatched. This naturally chains through mixed scoped/independent hierarchies.

**`machine_children` status updates:** `ActorCompletionJob` updates the `machine_children` record's status to `completed` or `failed` after dispatching the event to the parent. This happens regardless of whether the parent handled the event.

**Event type format:** The event type uses dot-notation `{actorId}.done` / `{actorId}.fail` (consistent with EventMachine's internal event naming, e.g., `{machine}.parallel.{region}.region.timeout`). The event is dispatched as a plain array `['type' => 'findeks.done', 'payload' => ['output' => [...]]]` — no special event class needed.

**`completionEvent` override:** When specified, the custom event type replaces both done and fail defaults. Done uses `{completionEvent}` as-is. Fail appends `.fail` → `{completionEvent}.fail`. The `completionEvent` value is persisted in the `machine_children` table (new column).

**Deep chains:** If Machine A spawns Machine B, and Machine B spawns Machine C, completion propagates one level at a time (C→B via `ActorCompletionJob`, then B→A via another `ActorCompletionJob` if B reaches final). Each hop is a separate job.

**Orphaned children:** When a parent reaches a final state while independent children are still running, the children continue to completion. `ActorCompletionJob` attempts `Machine::send()` on the parent, catches `NoTransitionDefinitionFoundException` (parent in final state, no transitions), and silently drops the event. The `machine_children` record's status is updated to `completed`/`failed` by the child's own persist flow.

The parent handles completion events like any event in its `on` config:

```php
'on' => [
    'findeks.done' => [
        'actions' => function (ActorContext $actors) {
            $output = $actors->ref('findeks')->output();
            // use output...
        },
        'target' => 'findeks_complete',
    ],
    'findeks.fail' => 'findeks_failed',
],
```

**Opt-out:** If the parent doesn't define handlers for `{actorId}.done` / `{actorId}.fail`, completion events are silently dropped (fire-and-forget pattern).

### 7. Lifecycle Rules

| Aspect | Scoped (`machine` key) | Independent (`spawn()`) |
|--------|------------------------|------------------------|
| Created when | Parent enters delegating state | `spawn()` called in action |
| Cancelled when | Parent exits delegating state | Never (runs to completion) |
| Parent knows about it via | `@done` / `@fail` on state config | `{actorId}.done` / `{actorId}.fail` events |
| Multiple from same state | One per `machine` key | Unlimited |
| Dynamic count | No | Yes |
| Survives parent state exit | No | **Yes** |
| Survives parent completion | No | Yes (orphaned, runs to completion) |
| Addressable by alias | No | Yes (`ref('actorId')`) |
| Can receive events mid-life | Via `forward` key | Via `ref('actorId')->dispatch(event)` |

### 8. Per-Machine Supervisor (Queue & Worker Topology)

Machines declare their queue and worker requirements as part of their definition. The package auto-registers Horizon supervisors transparently — same DX as endpoint registration. Whether Horizon is installed or not makes **zero difference** to the developer.

#### Machine Definition: `supervisor` Key

`supervisor` is a new top-level parameter on `MachineDefinition::define()`, alongside `config`, `behavior`, `endpoints`, and `schedules`:

```php
class FindeksReportMachine extends Machine
{
    public static function definition(): MachineDefinition
    {
        return MachineDefinition::define(
            config: [
                'id'      => 'findeks_report',
                'initial' => 'idle',
                'states'  => [...],
            ],
            behavior: [...],
            supervisor: [
                'queue'      => 'findeks',        // queue name (required)
                'connection' => 'redis',           // queue connection
                'balance'    => 'auto',            // auto | simple | false
                'minWorkers' => 2,                 // minimum worker processes
                'maxWorkers' => 8,                 // maximum worker processes
                'tries'      => 3,                 // max job attempts
                'timeout'    => 120,               // job timeout (seconds)
                'backoff'    => [30, 60],           // retry backoff (seconds)
                'memory'     => 256,               // memory limit (MB)
                'maxJobs'    => 1000,              // restart worker after N jobs
                'maxTime'    => 3600,              // restart worker after N seconds
                'nice'       => 0,                 // process priority
            ],
        );
    }
}
```

Only `queue` is required. All other fields have sensible defaults (inherited from Horizon's defaults).

#### Auto-Registration Flow

`MachineServiceProvider::boot()` handles everything:

```
MachineServiceProvider::boot()
    │
    ├─ 1. Discover machines
    │     ├─ Production: read from bootstrap/cache/machines.php (~1ms)
    │     └─ Development: live filesystem scan via MachineDiscovery (~50-200ms)
    │
    ├─ 2. MachineQueueRegistry::register($supervisors)
    │     → Static map: machineClass → queue name
    │     → ALL jobs use this for queue resolution (Horizon-agnostic)
    │
    └─ 3. if (class_exists(Horizon::class))
          │
          ├─ Shared queue dedup + merge (max-wins strategy)
          ├─ config()->set("horizon.environments.{$env}", merged)
          └─ Supervisors appear in Horizon dashboard automatically
```

This mirrors the endpoint registration pattern:
- **Endpoints:** Machine defines → `MachineRouter::register()` → Laravel routes
- **Supervisors:** Machine defines → `MachineServiceProvider::boot()` → Horizon config

No manual `config/horizon.php` editing. No conditional logic in application code.

#### MachineQueueRegistry

Horizon-agnostic static registry. All job dispatch points use this:

```php
class MachineQueueRegistry
{
    /** @var array<class-string, string> machineClass → queue name */
    private static array $queues = [];

    public static function register(array $supervisors): void;
    public static function queueFor(string $machineClass): ?string;
    public static function reset(): void; // for testing
}
```

Job dispatch integration:

```php
// In SendToMachineJob dispatch:
$queue = MachineQueueRegistry::queueFor($targetMachineClass);
if ($queue !== null) {
    $job->onQueue($queue);
}
```

Works identically with or without Horizon — the queue name is standard Laravel.

#### Queue Resolution Priority

Each job type resolves its queue with specificity ordering (first match wins):

| Job | Resolution Order |
|-----|-----------------|
| `ChildMachineJob` | State `queue` key → Child `supervisor.queue` → default |
| `SendToMachineJob` | Target machine's `supervisor.queue` → default |
| `ParallelRegionJob` | Machine's `supervisor.queue` → `config('machine.parallel_dispatch.queue')` → default |
| `ActorCompletionJob` | Parent machine's `supervisor.queue` → default |
| `SpawnedMachineJob` | spawn() `queue` override → Spawned machine's `supervisor.queue` → default |
| `ChildMachineCompletionJob` | Parent machine's `supervisor.queue` → default |
| `ListenerJob` | Machine's `supervisor.queue` → default |

**Principle:** Each job goes to its **target** machine's queue. Parent dispatching a child → child's queue. Completion flowing back → parent's queue.

#### Spawn Queue Override

Fan-out scenarios may need per-spawn queue control:

```php
$this->spawn(
    machine: WorkerMachine::class,
    actorId: "worker-{$i}",
    input: ['chunk' => $chunk],
    queue: 'worker-burst',  // ← override WorkerMachine's default supervisor.queue
);
```

This override applies only to the `SpawnedMachineJob` dispatch, not to the spawned machine's own internal jobs (those still use the machine's `supervisor.queue`).

#### Shared Queue Merge

When multiple machines share a queue name, configs are merged with **max-wins** strategy:

```php
// FindeksMachine:  supervisor: ['queue' => 'external-api', 'maxWorkers' => 8, 'timeout' => 120]
// TurmobMachine:   supervisor: ['queue' => 'external-api', 'maxWorkers' => 4, 'timeout' => 60]

// Merged result:
'machine-external-api' => [
    'queue'        => ['external-api'],
    'maxProcesses' => 8,    // max(8, 4)
    'timeout'      => 120,  // max(120, 60)
]
```

Merge rules:
- `minWorkers`, `maxWorkers`, `tries`, `timeout`, `memory`, `maxJobs`, `maxTime` → **max** value
- `connection`, `balance` → Must match. Conflicting values throw `LogicException` at boot:

```
EventMachine: Queue 'external-api' has conflicting 'connection' values:
  FindeksMachine: redis
  TurmobMachine: sqs
Define the same connection or use separate queues.
```

## Migration Guide

### For Existing Users

**No breaking changes to existing functionality.** The existing `machine` key delegation is unaffected. The `machine_children` table gets two new columns (`actor_id` nullable, `lifetime` with default `'scoped'`) that preserve existing behavior. The old `sendToParent(ContextManager, event)` and `dispatchToParent(ContextManager, event)` signatures are deprecated but continue to work.

### New Capability Adoption

Users who want independent-lifetime children:

1. Run the migration to add new columns to `machine_children`
2. Use `$this->spawn()` in behavior classes or `$actors->spawn()` in closures
3. Use `$this->ref()` / `$actors->ref()` to access spawned machines
4. Handle `{actorId}.done` / `{actorId}.fail` events in parent transitions

### Example 2: Dynamic Worker Fan-out

```php
class BatchProcessorMachine extends Machine
{
    public static function definition(): MachineDefinition
    {
        return MachineDefinition::define(
            config: [
                'id'      => 'batch_processor',
                'initial' => 'idle',
                'context' => ['items' => [], 'results' => [], 'completedCount' => 0],
                'states'  => [

                    'idle' => [
                        'on' => ['PROCESS' => 'spawning_workers'],
                    ],

                    'spawning_workers' => [
                        'entry' => 'spawnWorkersAction',
                        'on'    => [
                            '@always' => 'awaiting_results',
                        ],
                    ],

                    'awaiting_results' => [
                        'on' => [
                            // Workers spawn with completionEvent: 'WORKER_COMPLETED'
                            // Success: WORKER_COMPLETED, Fail: WORKER_COMPLETED.fail
                            'WORKER_COMPLETED' => [
                                [
                                    'guard'   => 'allWorkersCompleteGuard',
                                    'actions' => 'trackCompletionAction',
                                    'target'  => 'aggregating',
                                ],
                                [
                                    'actions' => 'trackCompletionAction',
                                ],
                            ],
                            // Handle worker failures
                            'WORKER_COMPLETED.fail' => [
                                'actions' => 'trackFailureAction',
                                'target'  => 'failed',
                            ],
                        ],
                    ],

                    'aggregating' => [
                        'entry' => 'aggregateResultsAction',
                        'on'    => ['@always' => 'completed'],
                    ],

                    'completed' => ['type' => 'final'],
                    'failed'    => ['type' => 'final'],
                ],
            ],
            behavior: [
                'actions' => [
                    'spawnWorkersAction'     => SpawnWorkersAction::class,
                    'trackCompletionAction'  => TrackCompletionAction::class,
                    'trackFailureAction'     => TrackFailureAction::class,
                    'aggregateResultsAction' => AggregateResultsAction::class,
                ],
                'guards' => [
                    'allWorkersCompleteGuard' => AllWorkersCompleteGuard::class,
                ],
            ],
            supervisor: [
                'queue'      => 'batch',
                'minWorkers' => 1,
                'maxWorkers' => 4,
            ],
        );
    }
}

class SpawnWorkersAction extends ActionBehavior
{
    public function __invoke(ContextManager $ctx): void
    {
        $items = $ctx->get('items');
        foreach ($items as $item) {
            $this->spawn(
                machineClass: ItemProcessorMachine::class,
                actorId: "worker_{$item['id']}",
                input: ['item' => $item],
                completionEvent: 'WORKER_COMPLETED',
                queue: 'worker-burst',  // override child's default queue for this fan-out
            );
        }
    }
}

class AggregateResultsAction extends ActionBehavior
{
    public function __invoke(ContextManager $ctx): void
    {
        $results = [];
        foreach ($this->refs() as $actorId => $ref) {
            if ($ref->isCompleted()) {
                $results[$actorId] = $ref->output();
            }
        }
        $ctx->set('results', $results);
    }
}
```

### Dynamic actorId Event Routing

For the fan-out pattern with dynamic actorIds (`worker_1`, `worker_2`, ...), the parent cannot statically list all `worker_N.done` events in its `on` config. Two approaches:

**Approach A: Consolidation via `completionEvent`.** Each child's completion triggers an entry in `machine_children` with `status = 'completed'`. The parent defines a generic `WORKER_COMPLETED` event and `ActorCompletionJob` dispatches this consolidated event type (configurable per spawn) instead of `{actorId}.done`. The `actorId` is included in the payload so the parent can identify which child completed:

```php
$this->spawn(
    machineClass: ItemProcessorMachine::class,
    actorId: "worker_{$item['id']}",
    input: ['item' => $item],
    completionEvent: 'WORKER_COMPLETED', // override default {actorId}.done
);
```

**Approach B: Wildcard event matching (future).** Not in scope for v10 — evaluate based on real-world usage.

## Non-Goals

- **Supervision/restart strategies** — `@fail` routing is sufficient. Stateless restore makes "restart" = event replay.
- **Observable/subscription pattern** — Queue-based model doesn't suit reactive streams.
- **Virtual actors (auto-activation)** — Auto-restore from archive exists; generalizing is a separate concern.
- **Cluster sharding** — Laravel queue distribution is sufficient.
- **Snapshots for fast restore** — Good idea but separate scope.
- **Replacing `machine` key** — Invoke semantics remain the right choice for simple, deterministic delegation.
