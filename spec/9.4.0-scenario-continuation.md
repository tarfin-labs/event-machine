# Scenario Continuation — Multi-Request Scenario Flows

**Status:** Draft
**Date:** 2026-04-01

## Problem

Scenarios are one-shot: activate → reach target → cleanup. But real QA flows need scenarios that span multiple HTTP requests across interactive states.

### The Findeks Example

```
AtAwaitingPinScenario:
  source: awaiting_report_request
  event:  ReportRequestedEvent
  target: awaiting_pin

Phase 1 (Request 1 — scenario activated):
  awaiting_report_request → requesting (job, mocked) → polling (job, mocked) → awaiting_pin ✓
  → ScenarioPlayer::execute() finishes, cleanupOverrides() runs
  → DB: scenario_class = AtAwaitingPinScenario (persisted)

Phase 2 (Request 2 — QA sends PinConfirmedEvent):
  → MachineController: scenarioSlug === null → deactivateScenario()
  → machine->send(PinConfirmedEvent)
  → awaiting_pin → confirming_pin (job state)
  → ConfirmFindeksPinJob runs FOR REAL → hits Findeks API → no session → CRASH
```

QA has to know about a second scenario (`AtReportSavedWithPinScenario`) and manually activate it before sending the pin. This is poor UX — the scenario that got the machine to `awaiting_pin` should know what happens after QA acts.

### Root Cause

`ScenarioPlayer::execute()` cleans up all overrides in its `finally` block (line 168). The DB retains `scenario_class`, but `maybeRegisterScenarioOverrides()` deactivates it on the next request without a scenario slug (line 615).

There's no mechanism for a scenario to declare: "after reaching target, these overrides should apply to subsequent requests."

## Proposed Solution

Add a `continuation()` method to `MachineScenario`. When present, the scenario stays active after reaching target — subsequent requests auto-restore and apply continuation overrides.

### API

```php
class AtAwaitingPinScenario extends MachineScenario
{
    protected string $source      = 'awaiting_report_request';
    protected string $event       = ReportRequestedEvent::class;
    protected string $target      = 'awaiting_pin';
    protected string $description = 'Findeks — PIN girişi gerekli';

    protected function plan(): array
    {
        return [
            // Phase 1: source → target (existing behavior)
            'awaiting_report_request' => [
                MatchPhoneAndStoreQueryIdAction::class => ['maskedPhone' => '0555***4567', 'queryId' => 100],
            ],
            'requesting' => [
                'outcome'                      => '@done',
                IsBirthDateMismatchGuard::class => false,
                StoreRequestIdAction::class     => ['findeksRequestId' => 1],
            ],
            'polling' => [
                'outcome'                => '@done',
                IsPinRequiredGuard::class => true,
            ],
        ];
    }

    protected function continuation(): array
    {
        return [
            // Phase 2: after target — applies on subsequent requests
            'confirming_pin' => '@done',
            'polling'        => [
                'outcome'                => '@done',
                IsPinRequiredGuard::class => false,
            ],
            'saving_report' => '@done',
        ];
    }
}
```

### Key Design Decisions

**Why `continuation()` instead of `@after` in plan()?**

`polling` appears in both phases with different overrides:
- Phase 1: `IsPinRequiredGuard => true` (PIN required → go to `awaiting_pin`)
- Phase 2: `IsPinRequiredGuard => false` (PIN done → go to `saving_report`)

PHP arrays can't have duplicate keys. A separate method cleanly separates the phases.

**Why not just "don't clean up after target"?**

Phase 1 overrides (e.g., `IsPinRequiredGuard => true`) would leak into Phase 2 where they're wrong. Each phase needs its own fresh overrides.

**Why not chain two scenarios automatically?**

That's the current workaround (AtAwaitingPinScenario + AtReportSavedWithPinScenario). The problem: QA must manually select the second scenario, know it exists, and time the switch correctly. Continuation is a single scenario with a complete plan.

## Execution Flow

### Request 1: Initial Activation

```
POST /endpoint { scenario: "at-awaiting-pin", type: "REPORT_REQUESTED" }

1. MachineController → maybeRegisterScenarioOverrides() → returns scenario
2. ScenarioPlayer::execute()
   a. registerOverrides(plan())     → Phase 1 overrides active
   b. machine->send(REPORT_REQUESTED)
   c. @continue loop: requesting(@done) → polling(@done) → awaiting_pin
   d. validateTarget(awaiting_pin)  → ✓
   e. finally: cleanupOverrides()   → Phase 1 overrides removed
3. DB: scenario_class + scenario_params persisted (existing behavior)
4. Response: { state: awaiting_pin, ... }
```

### Request 2: Continuation

```
POST /endpoint { type: "PIN_CONFIRMED", payload: { pin: "123456" } }

1. MachineController → maybeRegisterScenarioOverrides()
   a. scenarioSlug === null
   b. DB: scenario_class = AtAwaitingPinScenario → restore instance
   c. scenario->hasContinuation() → true
   d. return scenario (flagged as continuation)
2. ScenarioPlayer::executeContinuation()
   a. registerOverrides(continuation())  → Phase 2 overrides active
   b. machine->send(PIN_CONFIRMED)
   c. @continue loop: confirming_pin(@done) → polling(@done) → saving_report(@done) → report_saved
   d. report_saved is FINAL → deactivateScenario()
   e. finally: cleanupOverrides()
3. DB: scenario_class cleared (final state)
4. Response: { state: report_saved, ... }
```

### Request 2 Alternative: Interactive Pause in Continuation

If the continuation hits another interactive state (no `@continue` for it), the machine pauses there. The scenario stays active for a third request:

```
Request 2 → machine reaches another interactive state
  → no @continue → loop stops
  → scenario stays in DB → next request uses continuation again
Request 3 → same continuation overrides → machine advances further
...
Request N → final state → deactivate
```

## Implementation

### MachineScenario — New Method

```php
// Default: no continuation (existing behavior preserved)
protected function continuation(): array
{
    return [];
}

public function hasContinuation(): bool
{
    return $this->continuation() !== [];
}

public function resolvedContinuation(): array
{
    return $this->continuation();
}
```

### ScenarioPlayer — New Method

`executeContinuation()` reuses the core @continue loop logic from `execute()` but:
- Uses `continuation()` instead of `plan()` for overrides
- No source/target validation
- Accepts any event (not just the scenario's declared event)
- Deactivates on final state
- Keeps scenario active on interactive state

```php
public function executeContinuation(
    Machine $machine,
    array $eventPayload,
    string $rootEventId,
    string $eventType,
): State {
    self::$isActive = true;

    try {
        // Register Phase 2 overrides
        self::registerOverrides($this->scenario, useContinuation: true);
        $classified = $this->classifyPlanValues(useContinuation: true);

        // Send the event (whatever QA sent — not the scenario's declared event)
        $state = $machine->send([
            'type'    => $eventType,
            'payload' => $eventPayload,
        ]);

        // @continue loop — same logic as execute()
        $continueCount = 0;
        $maxDepth      = config('machine.max_transition_depth', 100);

        while ($continueCount < $maxDepth) {
            $currentRoute = $this->resolveCurrentRoute($state);
            $continue     = $this->findContinueForState($currentRoute, $classified['overrides']);

            if ($continue === null) {
                break;
            }

            $continueCount++;
            [$eventClass, $payload] = $this->parseContinueValue($continue);

            $state = $machine->send([
                'type'    => $this->resolveEventType($eventClass),
                'payload' => $payload,
            ]);
        }

        // Deactivate if final state reached
        if ($state->currentStateDefinition->type === StateDefinitionType::FINAL) {
            self::deactivateScenario($rootEventId);
        }

        return $state;
    } finally {
        self::cleanupOverrides();
        self::$isActive = false;
    }
}
```

### MachineController — Modified Detection

```php
// In maybeRegisterScenarioOverrides():
if ($scenarioSlug === null) {
    $rootEventId = $machine->state->history?->first()?->root_event_id;
    if ($rootEventId !== null) {
        $currentState = MachineCurrentState::where('root_event_id', $rootEventId)
            ->whereNotNull('scenario_class')
            ->first(['scenario_class', 'scenario_params']);

        if ($currentState !== null) {
            $scenarioClass = $currentState->scenario_class;

            if (class_exists($scenarioClass)) {
                $scenario = new $scenarioClass();

                if ($scenario->hasContinuation()) {
                    $scenario->hydrateParams(
                        json_decode($currentState->scenario_params ?? '[]', true) ?? []
                    );
                    return ['scenario' => $scenario, 'isContinuation' => true];
                }
            }

            // No continuation — deactivate
            ScenarioPlayer::deactivateScenario($rootEventId);
        }
    }
    return null;
}
```

### MachineController — Dispatch to Correct Method

```php
if ($scenarioResult !== null) {
    $scenario        = $scenarioResult['scenario'];
    $isContinuation  = $scenarioResult['isContinuation'] ?? false;
    $rootEventId     = $machine->state->history?->first()?->root_event_id;
    $player          = new ScenarioPlayer($scenario);

    if ($isContinuation) {
        $state = $player->executeContinuation(
            machine: $machine,
            eventPayload: $event->payload ?? [],
            rootEventId: $rootEventId,
            eventType: $event->type,
        );
    } else {
        $state = $player->execute(
            machine: $machine,
            eventPayload: $event->payload ?? [],
            rootEventId: $rootEventId,
        );
    }
}
```

## Deactivation Rules

| Condition | Action |
|-----------|--------|
| Final state reached (in execute or executeContinuation) | Auto-deactivate |
| Request with `scenario: null` or empty | Deactivate (explicit opt-out) |
| Request with different scenario slug | New scenario replaces old (deactivate + activate) |
| Interactive state in continuation | Keep active — next request continues |

## What Doesn't Change

- `plan()` behavior unchanged — Phase 1 works identically
- `execute()` unchanged — same source/target/validation/cleanup
- Scenarios without `continuation()` work exactly as before (`continuation()` returns `[]`)
- DB schema unchanged — `scenario_class` + `scenario_params` columns already exist
- `machine:scenario-validate` — Level 1 + 2 validation unchanged for `plan()`

## What Gets Added

| Component | Change |
|-----------|--------|
| `MachineScenario::continuation()` | New method, returns `[]` by default |
| `MachineScenario::hasContinuation()` | New method, checks `continuation() !== []` |
| `MachineScenario::resolvedContinuation()` | New method, returns `continuation()` |
| `ScenarioPlayer::executeContinuation()` | New method — Phase 2 execution |
| `ScenarioPlayer::registerOverrides()` | Add `useContinuation` param to select plan vs continuation |
| `ScenarioPlayer::classifyPlanValues()` | Add `useContinuation` param |
| `MachineController::maybeRegisterScenarioOverrides()` | Detect continuation from DB |
| `MachineController` main flow | Dispatch to `executeContinuation()` for continuations |
| `ScenarioValidator` | Validate `continuation()` state routes and overrides |
| `machine:scenario` scaffold | Generate `continuation()` method when target is interactive |

## Breaking Change?

**No.** `continuation()` returns `[]` by default. Existing scenarios are unaffected.
