# Time-Based Events: `after` & `every` on Transitions

## Status
**Phase:** Design finalized, ready for implementation
**Version:** v7 (ships with inter-machine communication)
**Date:** 2026-03-15
**Branch:** Create from `feature/inter-machine-communication` (not from `main`)

## Problem Statement

State machines often need time-based behavior:
- "Cancel order if not paid within 7 days"
- "Send reminder email every 24 hours while awaiting payment"
- "Escalate ticket if not responded within 4 hours"
- "Retry failed payment every 6 hours, max 3 times"

Currently, users must implement this outside EventMachine: write a Laravel Job,
register it in Kernel.php, query machines manually. This creates boilerplate,
scatters behavior across files, and makes the machine definition incomplete.

## Theoretical Foundation

Time events are a **first-class concept** in the statechart specification:
- **Harel (1987):** `tm(duration)` timeout events defined in original paper
- **W3C SCXML:** `<send delay="5s">` and `<cancel sendid="...">` in official spec
- **UML State Machines:** `after(duration)` as standard time event trigger
- **XState v5:** `after: { 5000: 'target' }` — syntactic sugar for send() + cancel()

Every major state machine framework has time-based features. No PHP state machine
library does. EventMachine fills this gap.

### Core Insight: Time Is Just Another Event Source

XState's `after` compiles to:
1. **State entry:** `send('xstate.after.5000')` with 5000ms delay
2. **State exit:** `cancel('xstate.after.5000')`
3. **Transition:** `on: { 'xstate.after.5000': 'targetState' }`

A timer is NOT a separate mechanism — it is a PROPERTY of a transition.
"ORDER_EXPIRED fires after 7 days" is a complete transition definition:
event name, behavior, and timing together. Splitting this across `on` and
a separate `@after` section creates artificial duplication.

### Design Principle: Timer as Transition Property

Instead of separate `@after`/`@every` state-level keys, timer configuration
lives directly on the `on` transition — using `after` and `every` keys.
This is consistent with the endpoint analogy: endpoints don't have a separate
section either; they map directly to event transitions.

## Config API

### Timer Value Object (Duration Only)

```php
use Tarfinlabs\EventMachine\Support\Timer;

Timer::seconds(30)    // 30 seconds
Timer::minutes(5)     // 5 minutes
Timer::hours(6)       // 6 hours
Timer::days(7)        // 7 days
Timer::weeks(2)       // 2 weeks
```

Timer is a **duration-only** value object. It holds a number of seconds.
All behavior config (`target`, `guards`, `actions`) stays in the standard
transition format — Timer adds only timing.

```php
namespace Tarfinlabs\EventMachine\Support;

class Timer
{
    private int $totalSeconds;

    public static function seconds(int $value): self;
    public static function minutes(int $value): self;
    public static function hours(int $value): self;
    public static function days(int $value): self;
    public static function weeks(int $value): self;

    public function inSeconds(): int;
}
```

### Why NOT ManagesFrequencies Wrapper

| ManagesFrequencies | Timer |
|-------------------|-------|
| Schedule-based: "when to run" | Duration-based: "how long to wait" |
| Produces cron expressions | Holds duration in seconds |
| Absolute (wall clock) | Relative (from state entry) |
| Infrastructure concern (sweep frequency) | Business logic concern (event timing) |

ManagesFrequencies determines sweep command frequency (infrastructure).
Timer determines event timing (business logic). Different layers.

### `after` — One-Shot Timer on Transition

"Auto-trigger this event after N time in this state."

The `after` key is added to any `on` transition config:

```php
'awaiting_payment' => [
    'on' => [
        'PAY'              => 'processing',
        'ORDER_EXPIRED'    => ['target' => 'cancelled', 'after' => Timer::days(7)],
        'PAYMENT_REMINDER' => ['actions' => 'sendReminderAction', 'after' => Timer::days(1)],
    ],
],
```

Reading this: "PAY is external only. ORDER_EXPIRED auto-fires after 7 days
and transitions to cancelled. PAYMENT_REMINDER auto-fires after 1 day and
runs sendReminderAction."

One-shot: fires once per state entry. Tracked via `machine_timer_fires` (status='fired').
The event can ALSO be triggered externally — `after` adds auto-trigger, doesn't prevent manual.

### `every` — Recurring Timer on Transition

"Auto-trigger this event every N time while in this state."

```php
'active_subscription' => [
    'on' => [
        'BILLING' => ['actions' => 'processBillingAction', 'every' => Timer::days(30)],
        'CANCEL'  => 'cancelled',
    ],
],
```

### `every` with `max` and `then`

```php
'retrying_payment' => [
    'on' => [
        'RETRY_PAYMENT'  => ['actions' => 'retryPaymentAction', 'every' => Timer::hours(6), 'max' => 3, 'then' => 'MAX_RETRIES'],
        'MAX_RETRIES'    => 'payment_failed',
        'PAYMENT_SUCCESS' => 'paid',
    ],
],
```

After 3 fires, `MAX_RETRIES` event is sent exactly once (tracked via status='exhausted').

`then` accepts the same formats as event types throughout EventMachine:

```php
'then' => 'MAX_RETRIES',                  // string event type
'then' => MaxRetriesEvent::class,          // EventBehavior class FQCN
```

### Combined Example

```php
'awaiting_payment' => [
    'on' => [
        'PAY'              => 'processing',
        'PAYMENT_REMINDER' => ['actions' => 'sendReminderAction', 'every' => Timer::days(1)],
        'ORDER_EXPIRED'    => ['target' => 'cancelled', 'guards' => 'isNotPaidGuard', 'after' => Timer::days(7)],
    ],
],
```

Single source of truth: each event's timing, behavior, and guards in one place.

### Guarded Timer Transition

Guards work exactly like standard guarded transitions:

```php
'on' => [
    'ORDER_EXPIRED' => [
        'target' => 'cancelled',
        'guards' => 'isNotPaidGuard',
        'after'  => Timer::days(7),
    ],
],
```

`after` fires once (status='fired') regardless of guard result. If guard rejects,
event is consumed but no transition happens. User's responsibility to design guards
appropriately. For complex branching, use an action:

```php
'on' => [
    'ORDER_EXPIRED' => ['actions' => 'handleExpiryAction', 'after' => Timer::days(7)],
],

// Action handles branching logic:
// if (!$context->get('is_paid')) { $this->raise(['type' => 'CANCEL_ORDER']); }
```

### `on` Transition Key Reference (Extended)

Existing keys unchanged. New keys added for timer support:

| Key | Type | Context | Description |
|-----|------|---------|-------------|
| `target` | `string` | Existing | Target state |
| `guards` | `string\|array` | Existing | Guard(s) |
| `actions` | `string\|array` | Existing | Action(s) |
| `calculators` | `string\|array` | Existing | Calculator(s) |
| `description` | `string` | Existing | Transition description |
| `after` | `Timer` | **New** | Auto-trigger after duration (one-shot) |
| `every` | `Timer` | **New** | Auto-trigger at interval (recurring) |
| `max` | `int` | **New** | Max fire count (requires `every`) |
| `then` | `string` | **New** | Event type or EventBehavior FQCN to send after max reached (requires `max`) |

### No Separate `@after`/`@every` State Keys

Timer config lives on transitions, NOT as state-level keys. This eliminates:
- Duplication between `on` handlers and `@after`/`@every` sections
- The need for a separate `event` key (the event name IS the `on` key)
- The need for a separate `delay`/`interval` key (it's `after`/`every` on the transition)

## @timeout Coexistence

`@timeout` is a child machine delegation deadline (delayed job mechanism).
`after`/`every` on transitions are general-purpose state timers (sweep mechanism).
They serve different purposes and coexist naturally:

```php
'processing' => [
    'machine'  => PaymentMachine::class,
    '@done'    => 'completed',
    '@timeout' => ['target' => 'timed_out', 'after' => Timer::minutes(5)],
    'on' => [
        'ALERT' => ['actions' => 'sendAlertAction', 'after' => Timer::days(1)],
    ],
],
```

## Architecture: PHP-Native Sweep

### Why NOT Delayed Jobs

| Problem | Impact |
|---------|--------|
| Redis flush/restart | All delayed jobs lost |
| AWS SQS max 15min delay | 7-day delays impossible |
| Queue worker restart | Delayed jobs may be lost |
| No visibility | Can't query "which instances have pending timers" |

### Why Sweep (Cron-Based)

| Advantage | Description |
|-----------|-------------|
| Survives restarts | Cron is independent of queue |
| No delay limit | Works for any duration |
| Self-healing | Missed timers caught on next sweep |
| Deployment-friendly | New timer configs automatically apply to existing instances |
| Visibility | Can query pending timers from DB |
| PHP-native | Cron is PHP's natural timer mechanism |

### Implicit Cancel Mechanism

Timers have NO explicit cancel action. Cancellation is implicit:

```
Instance in awaiting_payment → sweep finds it → deadline passed → fires event
Instance transitions to processing → sweep queries awaiting_payment → NOT FOUND → implicit cancel
```

The `machine_timer_fires` record stays in DB but is never queried again because
the instance is no longer in the matching state. No cleanup needed.

### Sweep Architecture

```
Machine Definition (blueprint)
  'ORDER_EXPIRED' => ['target' => 'cancelled', 'after' => Timer::days(7)]
  'BILLING'       => ['actions' => 'processBillingAction', 'every' => Timer::days(30)]
        |  (parsed at boot — transitions with after/every keys extracted)
        v
MachineServiceProvider::boot()
  -> Auto-discovers Machine subclasses (via MachineClassVisitor)
  -> Registers machine:process-timers per class with Laravel Scheduler
  -> User adds NOTHING to Kernel.php
        |  (cron triggers via configured resolution)
        v
ProcessTimersCommand (withoutOverlapping, per machine class)
  1. Load definition, find transitions with after/every keys
  2. Query machine_current_states for instances in target states
  3. Check timing (state_entered_at + after, last_fired_at + every)
  4. Check dedup (machine_timer_fires status)
  5. Batch dispatch SendToMachineJob for eligible instances
        |  (queue jobs via Bus::batch)
        v
SendToMachineJob (existing)
  -> Restore machine -> send(event)
  -> Double-guard: still in expected state?
  -> Persist -> machine_current_states updated via diff
```

### MachineServiceProvider Auto-Registration

```php
// MachineServiceProvider::boot()
public function boot(): void
{
    parent::boot(); // Spatie PackageServiceProvider handles migrations, commands, config

    // Safe scheduler access pattern for package providers
    $this->callAfterResolving(Schedule::class, function (Schedule $schedule): void {
        $machineClasses = $this->discoverTimerMachines(); // via MachineClassVisitor
        $resolution     = config('machine.timer_resolution', TimerResolution::EVERY_MINUTE);

        foreach ($machineClasses as $class) {
            $schedule->command("machine:process-timers --class={$class}")
                ->{$resolution->value}()
                ->withoutOverlapping()
                ->runInBackground();
        }
    });
}
```

User does nothing. Auto-discovery finds Machine subclasses with timer-configured
transitions. Each class gets its own sweep cycle, its own lock, runs in parallel.

### Machine Class Discovery & Cache

Auto-discovery only. No config listing. Reuse existing `MachineClassVisitor`
(PhpParser-based, `src/Commands/MachineClassVisitor.php`) already used by
`MachineConfigValidatorCommand`.

#### Discovery Cache (like `route:cache` / `config:cache`)

MachineClassVisitor scans PHP files on every boot — expensive for large apps.
Cache the discovery result similar to Laravel's route/config cache:

```bash
php artisan machine:cache    # Scan, extract timer configs, write bootstrap/cache/machines.php
php artisan machine:clear    # Delete cache, fall back to runtime discovery
```

Cache file stores:
- Machine class list (FQCNs)
- Per-class timer-configured states and transition keys
- Pre-computed timer definitions (delay/interval in seconds)

```php
// MachineServiceProvider::boot()
$this->callAfterResolving(Schedule::class, function (Schedule $schedule): void {
    // Use cache if available, otherwise discover at runtime
    $timerMachines = file_exists($cachePath = $this->app->getCachedMachinesPath())
        ? require $cachePath
        : $this->discoverTimerMachines();

    // ... register per-class commands
});
```

In production: `machine:cache` runs during deploy (like `route:cache`).
In development: no cache, runtime discovery on every boot.

## Database Schema

### machine_current_states (prerequisite, general-purpose)

Normalized table — one row per active state per instance.
Supports both simple and parallel state machines.

```sql
CREATE TABLE machine_current_states (
    root_event_id    VARCHAR(26) NOT NULL,
    machine_class    VARCHAR(255) NOT NULL,
    state_id         VARCHAR(255) NOT NULL,
    state_entered_at TIMESTAMP NOT NULL,

    PRIMARY KEY (root_event_id, state_id),
    INDEX idx_sweep (machine_class, state_id, state_entered_at)
);
```

| Machine Type | State Value | Rows |
|-------------|-------------|------|
| Simple | `['order.awaiting_payment']` | 1 |
| Parallel (2 regions) | `['order.r1.s1', 'order.r2.s2']` | 2 |
| Parallel (3 regions) | `['order.r1.s1', 'order.r2.s2', 'order.r3.s3']` | 3 |

Example rows for a simple OrderMachine instance:

```
| root_event_id | machine_class              | state_id                  | state_entered_at    |
|---------------|----------------------------|---------------------------|---------------------|
| 01JFABC123    | App\Machines\OrderMachine   | order.awaiting_payment    | 2026-03-08 10:00:00 |
```

Example rows for a parallel ProcessingMachine instance (2 regions):

```
| root_event_id | machine_class                    | state_id                    | state_entered_at    |
|---------------|----------------------------------|-----------------------------|---------------------|
| 01JFXYZ789    | App\Machines\ProcessingMachine    | processing.payment.charging | 2026-03-14 14:30:00 |
| 01JFXYZ789    | App\Machines\ProcessingMachine    | processing.inventory.reserving | 2026-03-14 14:30:00 |
```

After the payment region transitions to `charged`:

```
| root_event_id | machine_class                    | state_id                       | state_entered_at    |
|---------------|----------------------------------|--------------------------------|---------------------|
| 01JFXYZ789    | App\Machines\ProcessingMachine    | processing.payment.charged     | 2026-03-14 15:00:00 |  ← updated
| 01JFXYZ789    | App\Machines\ProcessingMachine    | processing.inventory.reserving | 2026-03-14 14:30:00 |  ← unchanged
```

#### Sync Logic in Machine::persist()

```php
$newStates = $this->state->value;
$existing  = MachineCurrentState::where('root_event_id', $id)->pluck('state_id')->toArray();

$added   = array_diff($newStates, $existing);
$removed = array_diff($existing, $newStates);

// Remove states no longer active
MachineCurrentState::where('root_event_id', $id)->whereIn('state_id', $removed)->delete();

// Add newly entered states (state_entered_at = now)
foreach ($added as $stateId) {
    MachineCurrentState::create([
        'root_event_id'    => $id,
        'machine_class'    => $this->definition->machineClass,
        'state_id'         => $stateId,
        'state_entered_at' => now(),
    ]);
}

// UNCHANGED states are NOT touched → state_entered_at preserved
// Self-loop (same state): $added and $removed are both empty → no update
```

### machine_timer_fires (after dedup + every tracking)

```sql
CREATE TABLE machine_timer_fires (
    root_event_id  VARCHAR(26) NOT NULL,
    timer_key      VARCHAR(255) NOT NULL,
    last_fired_at  TIMESTAMP NOT NULL,
    fire_count     INT DEFAULT 0,
    status         VARCHAR(20) DEFAULT 'active',

    PRIMARY KEY (root_event_id, timer_key)
);
```

| Status | Meaning | Used By |
|--------|---------|---------|
| `active` | Timer is active, recurring | `every` |
| `fired` | One-shot has been fired | `after` (prevents re-firing) |
| `exhausted` | Max reached, `then` event sent | `every` max/then (prevents re-sending) |

### Timer Key Format

`{state_id}:{event_type}:{total_seconds}`

Examples:
- `order.awaiting_payment:ORDER_EXPIRED:604800`
- `subscription.active:BILLING:2592000`
- `order.retrying:RETRY_PAYMENT:21600`

Timer keys are internal-only — users never write them. They are auto-generated
from the transition config and used in `machine_timer_fires` for dedup/tracking.

## Sweep Performance Architecture

### Performance Principles (from Temporal, Solid Queue, db-scheduler research)

1. **Filter in DB, not PHP** — Never load all instances then filter in PHP
2. **FOR UPDATE SKIP LOCKED** — Concurrent-safe polling without blocking
3. **Batch dispatch** — Bus::batch instead of individual dispatch calls
4. **Machine class sharding** — Parallel sweep per machine class (automatic)
5. **Backpressure** — Skip sweep cycle if queue is saturated
6. **Bounded execution** — Always cap batch size

### `after` Query (DB-level filtering)

```sql
SELECT mcs.root_event_id
FROM machine_current_states mcs
LEFT JOIN machine_timer_fires mtf
    ON mtf.root_event_id = mcs.root_event_id
    AND mtf.timer_key = :timer_key
    AND mtf.status = 'fired'
WHERE mcs.machine_class = :class
    AND mcs.state_id = :state
    AND mcs.state_entered_at <= NOW() - INTERVAL :seconds SECOND
    AND mtf.root_event_id IS NULL
LIMIT :batch_size
FOR UPDATE OF mcs SKIP LOCKED
```

Returns ONLY instances that: (a) are in the target state, (b) past deadline, (c) not yet fired.

### `every` Query (DB-level filtering)

```sql
SELECT mcs.root_event_id, COALESCE(mtf.fire_count, 0) as fire_count
FROM machine_current_states mcs
LEFT JOIN machine_timer_fires mtf
    ON mtf.root_event_id = mcs.root_event_id
    AND mtf.timer_key = :timer_key
WHERE mcs.machine_class = :class
    AND mcs.state_id = :state
    AND (mtf.status IS NULL OR mtf.status = 'active')
    AND COALESCE(mtf.last_fired_at, mcs.state_entered_at) <= NOW() - INTERVAL :seconds SECOND
LIMIT :batch_size
FOR UPDATE OF mcs SKIP LOCKED
```

### `every` max/then — Separate Query

```sql
SELECT mcs.root_event_id
FROM machine_current_states mcs
INNER JOIN machine_timer_fires mtf
    ON mtf.root_event_id = mcs.root_event_id
    AND mtf.timer_key = :timer_key
WHERE mcs.machine_class = :class
    AND mcs.state_id = :state
    AND mtf.status = 'active'
    AND mtf.fire_count >= :max_fires
LIMIT :batch_size
FOR UPDATE OF mcs SKIP LOCKED
```

For instances at max: send `then` event, set status='exhausted'.

### Batch Dispatch

```php
// Resolve event array from transition config: ['type' => 'ORDER_EXPIRED']
$eventArray = ['type' => $eventName];

$jobs = collect($instances)->map(fn ($i) => new SendToMachineJob(
    machineClass: $machineClass,
    rootEventId: $i->root_event_id,
    event: $eventArray,
));

if ($jobs->isNotEmpty()) {
    Bus::batch($jobs->toArray())
        ->name("timer:{$machineClass}:{$stateId}:{$eventName}")
        ->allowFailures()
        ->dispatch();
}
```

### Index Strategy

```sql
-- Primary sweep index
CREATE INDEX idx_sweep ON machine_current_states
    (machine_class, state_id, state_entered_at);

-- Timer lookup (composite for upsert and status check)
CREATE INDEX idx_timer_lookup ON machine_timer_fires
    (root_event_id, timer_key, status);
```

### Performance Expectations

| Scale | Instances | Due per sweep | Query time | Total sweep |
|-------|-----------|--------------|------------|-------------|
| Small | 1K | ~10 | <1ms | <100ms |
| Medium | 100K | ~100 | <10ms | <500ms |
| Large | 1M | ~1000 | <50ms | <2s |
| XL | 10M | ~5000 | <200ms | <5s |

## Guard Handling

Guards on timer transitions work exactly like standard guarded transitions.
The `after`/`every` key just adds auto-trigger timing — guard evaluation
happens at event processing time (in SendToMachineJob), not at sweep time.

### Single-branch guarded timer

```php
'on' => [
    'ORDER_EXPIRED' => [
        'target' => 'cancelled',
        'guards' => 'isNotPaidGuard',
        'after'  => Timer::days(7),
    ],
],
```

### Multi-branch guarded timer (standard format, mixed array)

```php
'on' => [
    'ORDER_EXPIRED' => [
        ['target' => 'cancelled', 'guards' => 'isNotPaidGuard'],
        ['target' => 'late_payment'],  // fallback if guard fails
        'after' => Timer::days(7),
    ],
],
```

PHP mixed array: numeric keys = branches, string keys = timer config.
Parser extracts `after`/`every`/`max`/`then` from string keys, processes
numeric entries as standard multi-branch transitions.

`after` fires once (status='fired') regardless of guard result. Guard
determines WHICH branch executes, not WHETHER the timer fires.

## Inter-Machine Integration

Timer transitions work naturally with inter-machine communication. All scenarios verified:

### Case 1: Parent timer while delegating to child

```php
'awaiting_child' => [
    'machine' => PaymentMachine::class,
    '@done'   => 'completed',
    'on' => [
        'REMIND_CHILD' => ['actions' => 'nudgeChildAction', 'every' => Timer::hours(6)],
        'FORCE_CANCEL' => ['target' => 'timed_out', 'after' => Timer::days(7)],
    ],
],
```

- Child completes within 7 days → @done fires → parent leaves state → timers implicit cancel
- Every 6 hours: nudge action uses dispatchTo() to send EXPEDITE to child
- After 7 days: parent transitions to timed_out, cleanupActiveChildren cancels child

### Case 2: Child machine has its own timer transitions

```php
// Child machine definition:
'waiting_for_webhook' => [
    'on' => [
        'WEBHOOK_RECEIVED' => 'processing',
        'WEBHOOK_TIMEOUT'  => ['target' => 'timed_out', 'after' => Timer::hours(4)],
    ],
],
'timed_out' => ['type' => 'final'],  // triggers @fail on parent
```

- Sweep monitors child independently (separate machine class, separate sweep)
- Webhook arrives → child transitions → timer implicit cancel
- No webhook in 4 hours → child transitions to timed_out (final) → triggers @fail on parent

### Case 3: Parallel state with timer on one region

```php
'processing' => [
    'type' => 'parallel',
    'states' => [
        'payment' => [
            'initial' => 'charging',
            'states' => [
                'charging' => [
                    'machine' => PaymentMachine::class,
                    '@done' => 'charged',
                    'on' => [
                        'PAYMENT_SLOW' => ['actions' => 'alertSlowPaymentAction', 'after' => Timer::hours(1)],
                    ],
                ],
                'charged' => ['type' => 'final'],
            ],
        ],
        'inventory' => [
            'initial' => 'reserving',
            'states' => [
                'reserving' => [
                    'machine' => InventoryMachine::class,
                    '@done' => 'reserved',
                ],
                'reserved' => ['type' => 'final'],
            ],
        ],
    ],
],
```

- machine_current_states: 2 rows (charging + reserving)
- Sweep finds instance in 'charging' → PAYMENT_SLOW timer applies
- Inventory region unaffected

### Case 4: Timer event arrives after state change (race condition)

- Sweep finds instance in 'awaiting_payment' at time T1
- Between T1 and job execution, instance transitions to 'processing'
- SendToMachineJob runs, restores machine (now in 'processing')
- Sends 'ORDER_EXPIRED' → no transition in 'processing' for this event
- Event dropped gracefully (NoTransitionDefinitionFoundException caught by SendToMachineJob)
- Double-guard pattern ensures safety

### Case 5: @timeout + timer coexist on delegation state

```php
'processing' => [
    'machine'  => PaymentMachine::class,
    '@done'    => 'completed',
    '@timeout' => ['target' => 'timed_out', 'after' => Timer::minutes(5)],
    'on' => [
        'ALERT' => ['actions' => 'sendAlertAction', 'after' => Timer::days(1)],
    ],
],
```

- @timeout: child deadline (delayed job, short duration)
- after on transition: general state timer (sweep, long duration)
- Independent mechanisms, both work

## Resolved Design Questions

1. **Timer resolution:** Default `TimerResolution::EVERY_MINUTE`. Configurable via `config/machine.php`.

2. **Machine class discovery:** Auto-discovery via `MachineClassVisitor`. No config listing.

3. **Concurrency:** `withoutOverlapping()` per machine class on ProcessTimersCommand.

4. **Guards:** Standard guarded transitions. Timer fires once regardless. For complex branching, use actions.

5. **after + every on same state:** Yes, on different transitions. Independent mechanisms.

6. **@timeout stays:** Different semantics (delegation deadline). Different mechanism (delayed job). Coexists with after/every.

7. **Self-loop:** state_entered_at NOT updated when state doesn't change. Diff-based sync handles this.

8. **Implicit cancel:** Sweep filters by current state → instances that left state are not found → natural cancellation.

9. **every max/then dedup:** `status='exhausted'` prevents re-sending `then` event.

10. **Parallel states:** Normalized table (one row per active state per instance). Sweep query unchanged.

11. **No separate @after/@every keys:** Timer config lives on `on` transitions as `after`/`every` keys. Eliminates duplication.

12. **Multi-branch guarded timers:** Use standard multi-branch format with `after`/`every` as sibling string keys (PHP mixed array). No special `branches` key needed.

13. **Event naming mandatory:** Every timer needs an explicit event name in `on`. Better for logging, testing, debugging.
