## Problem

v7 added major features (delegation, timers, schedules, parallel states, forward endpoints, available_events, @done.{state}) but testing documentation and infrastructure didn't fully keep up:

1. **Wrong API in docs** — `best-practices/testing-strategy.md` uses `Machine::fakeResult()` which doesn't exist, `advanceTimers(days: 7)` wrong signature
2. **Missing method docs** — 5 available_events assertion methods exist in code but aren't documented in canonical `test-machine.md`
3. **Missing recipes** — no recipes for forward endpoints, available_events
4. **TestMachine.resetFakes() gap** — doesn't clear Machine::fake() child fakes
5. **Persistence docs gap** — 3 of 5 DB tables have zero testing examples
6. **TestMachine lacks v7 integration** — child delegation, async simulation, cross-machine communication, forward endpoints all require boilerplate outside the fluent chain

## Phase 5B: Real Infrastructure Testing Guidance

### Problem

Most EventMachine testing works fine with fakes (`Queue::fake()`, `Machine::fake()`, `advanceTimers()`). But some async scenarios can ONLY be verified with real infrastructure. Users need to know:
- **When** fakes are sufficient vs when they're not
- **What** to verify in a real environment
- **Which patterns** to use (without prescribing a specific stack)

> **Not in scope:** Package-internal LocalQA setup. That's for event-machine contributors, not users.

### 5B.1 Add "When Fakes Aren't Enough" section to overview.md

**File:** `docs/testing/overview.md`

Add after the decision tree (Phase 5.1). Content:

**The Fake Boundary:**

| What fakes verify | What fakes DON'T verify |
|-------------------|------------------------|
| Job dispatch (`Queue::assertPushed`) | Job execution → child runs → completion routes back to parent |
| Timer registration (`assertHasTimer`) | `machine:process-timers` command reads DB and fires correctly |
| Schedule definition (`assertHasSchedule`) | `machine:process-scheduled` command runs via Laravel scheduler |
| Child invocation (`assertChildInvoked`) | Real child machine starts, persists, reaches final state |
| Lock exception thrown | Two concurrent requests actually block each other |
| Forward event accepted | Full HTTP → parent → child → response pipeline |

**Rule of thumb:** If your test only checks "was the right thing dispatched/registered?" → fakes are sufficient. If you need "does the full pipeline complete end-to-end?" → real infrastructure.

**When to invest in real infrastructure tests:**
- Your machine uses async delegation (`queue`) and the parent's behavior after `@done` is business-critical
- You use timers (`after`/`every`) and need to verify the sweep command picks them up
- You have concurrent access patterns and need to verify lock behavior under contention
- You use forward endpoints and need to verify the full HTTP → child → response chain

### 5B.2 Add "Real Infrastructure Patterns" section to recipes.md

**File:** `docs/testing/recipes.md`

#### Recipe: Async Delegation Pipeline Verification

Pattern for verifying the full async cycle without package-internal tooling:

```markdown
## Recipe: Full Async Delegation Pipeline

When `Queue::fake()` isn't enough — verify the complete cycle:
parent dispatches → child runs → child completes → parent routes via @done.

Requirements: Real database + Redis + queue worker (Horizon or `queue:work`).
```

Content:
- Create parent, send event, persist
- Assert parent is in delegating state (DB query)
- Start a queue worker (or use Horizon in test environment)
- Poll DB for parent state change (waitFor pattern):
  ```php
  $completed = retry(30, function () use ($rootEventId) {
      $cs = MachineCurrentState::where('root_event_id', $rootEventId)->first();
      return $cs && str_contains($cs->state_id, 'completed') ? true : throw new \Exception('waiting');
  }, sleepMilliseconds: 1000);
  ```
- Assert final parent state

Key gotchas:
- `.env.testing` must have real DB + Redis config (not sqlite/sync)
- Redis prefix must match between test process and queue worker
- Queue worker must be started BEFORE the test runs
- Always clean tables between tests (truncate, not `RefreshDatabase` which rolls back)

#### Recipe: Timer Sweep Verification

Pattern for verifying `machine:process-timers` picks up and fires timers:

```markdown
## Recipe: Timer Sweep in Real Environment
```

Content:
- Create machine, transition to state with `after` timer
- Persist (timer is registered in `machine_timer_fires`)
- Fast-forward time: `$this->travel(8)->days()` (Laravel time helpers)
- Run `Artisan::call('machine:process-timers')`
- Assert machine transitioned

#### Recipe: Concurrent Lock Verification

Pattern for verifying machine locks prevent race conditions:

```markdown
## Recipe: Testing Concurrent Access
```

Content:
- Create and persist a machine
- Dispatch two simultaneous `send()` calls (via async jobs or parallel processes)
- Assert one succeeds and one throws `MachineLockTimeoutException`
- Verify final state is consistent (not corrupted)

## Phase 6: TestMachine v2 API Extensions

### Overview

Extend `TestMachine` with 15 new methods across 4 categories to make v7 features first-class testable through the fluent chain. Currently, testing these features requires static `Machine::fake()` calls, raw `Queue::fake()` assertions, or manual DB record creation.

**Success criteria:** A delegation test reads as a single fluent chain:
```php
OrderMachine::test()
    ->fakingChild(PaymentMachine::class, result: ['id' => 'pay_1'], finalState: 'approved')
    ->send('START_PAYMENT')
    ->assertState('completed')
    ->assertChildInvoked(PaymentMachine::class)
    ->assertRoutedViaDoneState('approved');
```

### Category 1: Child Delegation Integration (6 methods)

#### 6.1 `fakingChild(string $childClass, ...): self`

```php
public function fakingChild(
    string $childClass,
    ?array $result = null,
    bool $fail = false,
    ?string $error = null,
    ?string $finalState = null,
): self
```

Registers `Machine::fake()` for the child class within the fluent chain. Tracks in `$fakedChildMachines` for selective cleanup via `Machine::resetMachineFake($class)` in `resetFakes()`.

**Complexity:** Trivial. **New storage:** `TestMachine::$fakedChildMachines` array.

#### 6.2-6.5 `assertChildInvoked/NotInvoked/InvokedTimes/InvokedWith`

```php
public function assertChildInvoked(string $childClass): self
public function assertChildNotInvoked(string $childClass): self
public function assertChildInvokedTimes(string $childClass, int $times): self
public function assertChildInvokedWith(string $childClass, array $expectedContext): self
```

Thin wrappers over existing `Machine::getMachineInvocations()`. Return `self` for chaining + clear failure messages.

**Complexity:** Trivial. **New storage:** None (reads `Machine::$machineFakes`).

#### 6.6 `assertRoutedViaDoneState(string $expectedFinalState): self`

Asserts the most recent child completion used the `@done.{state}` branch for the specified final state.

**Requires new infrastructure:** `State::$lastChildDoneRoute` property (nullable string, transient — not persisted). Set in `MachineDefinition::routeChildDoneEvent()` when a specific `@done.{state}` branch wins, null when catch-all `@done` fires.

```php
// State.php
public ?string $lastChildDoneRoute = null;

// MachineDefinition::routeChildDoneEvent() — after specific branch match:
$state->lastChildDoneRoute = $finalState;
// After catch-all match:
$state->lastChildDoneRoute = null;
```

**Complexity:** Simple. **New storage:** `State::$lastChildDoneRoute` (1 property + 2 lines in routeChildDoneEvent).

### Category 2: Async Simulation (3 methods)

#### 6.7 `simulateChildDone(string $childClass, array $result = [], ?string $finalState = null): self`

Simulates async child completion without real queues. Validates current state delegates to `$childClass`, builds `ChildMachineDoneEvent`, calls `routeChildDoneEvent()` directly.

Mirrors `ChildMachineCompletionJob::handle()` without DB restore, locks, or queue infrastructure.

**Complexity:** Moderate. **New storage:** None.

#### 6.8 `simulateChildFail(string $childClass, string $errorMessage = 'Simulated failure'): self`

Same pattern as `simulateChildDone` but builds `ChildMachineFailEvent` and calls `routeChildFailEvent()`.

**Complexity:** Simple. **New storage:** None.

#### 6.9 `simulateChildTimeout(string $childClass): self`

Calls `routeChildTimeoutEvent()` to fire the `@timeout` branch.

**Complexity:** Simple. **New storage:** None.

### Category 3: Cross-Machine Communication (5 methods + 1 new class)

#### New class: `src/Testing/CommunicationRecorder.php`

Lightweight static recorder capturing `sendTo()`/`raise()` calls during behavior execution. Opt-in via `recordingCommunication()`.

**Key design decision:** When recording `sendTo()`, skip the actual send (avoids requiring target machine to exist). When recording `raise()`, still push to event queue (engine needs it) but also record.

#### 6.10 `recordingCommunication(): self`

Enables `CommunicationRecorder`. Cleanup added to `resetFakes()`.

**Complexity:** Trivial.

#### 6.11-6.12 `assertSentTo/assertNotSentTo(string $machineClass, ?string $eventType = null): self`

Assert `sendTo()` was/wasn't called targeting the given machine class. Reads from `CommunicationRecorder`.

**Complexity:** Simple.

#### 6.13 `assertDispatchedTo(string $machineClass, ?string $eventType = null): self`

Wraps `Queue::assertPushed(SendToMachineJob::class)` with a fluent API. Validates `Queue::fake()` is active.

**Complexity:** Simple. **New storage:** None (uses Laravel's queue fake).

#### 6.14 `assertRaisedEvent(string $eventType): self`

Checks machine history for the processed raised event. No new infrastructure — uses existing history.

**Complexity:** Trivial. **New storage:** None.

### Category 4: Forward Endpoint Helper (1 method)

#### 6.15 `withRunningChild(string $childClass): self`

Sets up DB scaffolding for forward endpoint tests: creates `MachineChild` record (status: running), persists child machine instance. Replaces ~15 lines of manual setup.

Requires persistence enabled. v1 starts child in initial state only.

**Complexity:** Moderate. **New storage:** DB records (MachineChild, MachineEvent, MachineCurrentState).

## Open Questions (resolved)

1. **Should `fakingChild()` support array of multiple children in one call?**
   **Decision: No.** Chain multiple `fakingChild()` calls instead. Simpler API, same result. Add array variant later if demand arises.

2. **Should `simulateChildDone()` accept `output` separately from `result`?**
   **Decision: No.** `result` populates both `output()` and `result()` accessors — matching `Machine::fake()` behavior. Separate `output` param would create "hangisini ne zaman kullanayım?" confusion. %99 use case tek parametre ile çözülür. Gerçek ayrım gerekirse kullanıcı `routeChildDoneEvent()` ile manual event build edebilir.

3. **Should `assertRaisedEvent()` catch guarded-but-not-transitioned raises via CommunicationRecorder?**
   **Decision: No.** History-based approach is simpler and asserts the stronger claim (event was processed). To test raise calls that may be guarded, test the action in isolation with `State::forTesting()`.

4. **Should `withRunningChild()` accept a closure for child setup?**
   **Decision: No.** Start child in initial state only. Users needing a specific child state can create the child separately with `TestMachine::for()`. Closure variant eklenebilir ileride if needed.
