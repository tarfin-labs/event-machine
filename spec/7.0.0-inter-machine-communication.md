# Inter-Machine Communication & Machine Systems

> Machine-to-machine communication, machines working together, and systems composed of multiple machines.

**Date:** 2026-03-12 (created), 2026-03-13 (reviewed & finalized)
**Status:** Design Phase (All Questions Decided — Ready for Implementation)

---

## 1. Vision & Philosophy

### What EventMachine Is — and Isn't

EventMachine is NOT an XState port. It is a Laravel package **inspired by** XState but with its own identity. The defining traits of this identity:

- **Config/Behavior split** — Topology (structure) and implementation (code) are always separate
- **`@`-prefixed lifecycle hooks** — `@done`, `@fail`, `@always` — not XState's `onDone`/`onError`
- **Context-based data flow** — Data always flows through context; no dedicated `input`/`output` channels
- **Laravel-native** — Queue, Eloquent, Service Container are used naturally
- **Declarative config, imperative behavior** — Config defines *what*, behavior defines *how*
- **Readability** — Config reads like a specification: "This state is a parallel state; when @done, go to completed"
- **Endpoints** — Machines can expose HTTP endpoints for external interaction

### Design Principles

1. **Extend existing patterns** — Stretch `@done`/`@fail`, context, queue, and endpoints to child machines instead of inventing new abstractions
2. **Machines in config, code in behavior** — Child machine references go in config; data transformations live in actions/calculators
3. **Orchestrator machine = system definition** — The orchestrator machine's `definition()` declares the full topology; the Eloquent model is just Laravel glue
4. **Sync by default, async by choice** — Zero-config for simple cases, queue for complex ones
5. **Gradual adoption** — Existing machines don't change; new features only activate when used

## 2. Core Concept: Machine Delegation (`machine` Key)

### The Idea

A state can **delegate** to another machine. When the state is entered, the child machine starts. When the child machine completes (reaches a final state), the parent's `@done` transition fires.

This follows the exact same logic as `type: 'parallel'`:
- `type: 'parallel'` → "This state contains parallel regions; when all reach final, `@done` fires"
- `machine` → "This state delegates to another machine; when child reaches final, `@done` fires"

### Basic Example

```php
class OrderWorkflowMachine extends Machine
{
    public static function definition(): MachineDefinition
    {
        return MachineDefinition::define(
            config: [
                'id'      => 'order_workflow',
                'initial' => 'validating',
                'context' => [
                    'order_id'       => null,
                    'total_amount'   => 0,
                    'payment_result' => null,
                ],
                'states' => [
                    'validating' => [
                        'machine' => ValidationMachine::class,
                        'with'    => ['order_id'],
                        '@done'   => [
                            'target'  => 'processing_payment',
                            'actions' => 'storeValidationResultAction',
                        ],
                        '@fail' => 'validation_failed',
                    ],
                    'processing_payment' => [
                        'machine' => PaymentMachine::class,
                        'with'    => ['order_id', 'total_amount'],
                        '@done'   => [
                            'target'  => 'shipping',
                            'actions' => 'storePaymentResultAction',
                        ],
                        '@fail' => 'payment_failed',
                    ],
                    'shipping' => [
                        'machine' => ShippingMachine::class,
                        'with'    => ['order_id'],
                        '@done'   => 'completed',
                        '@fail'   => 'shipping_failed',
                    ],
                    'completed'         => ['type' => 'final'],
                    'validation_failed' => ['type' => 'final'],
                    'payment_failed'    => ['type' => 'final'],
                    'shipping_failed'   => ['type' => 'final'],
                ],
            ],
            behavior: [
                'actions' => [
                    'storeValidationResultAction' => StoreValidationResultAction::class,
                    'storePaymentResultAction'    => StorePaymentResultAction::class,
                ],
            ],
        );
    }
}
```

**Reads as:** "`validating` state delegates to `ValidationMachine`. Passes `order_id` to the child. When child completes, stores the validation result and transitions to `processing_payment`."

### Machine Delegation Inside Parallel States

The `machine` key can be used at **any** state level — including within parallel regions:

```php
'processing' => [
    'type'   => 'parallel',
    '@done'  => 'shipping',
    '@fail'  => 'compensating',
    'states' => [
        'payment' => [
            'initial' => 'charging',
            'states'  => [
                'charging' => [
                    'machine' => PaymentMachine::class,
                    'with'    => ['order_id', 'total_amount'],
                    '@done'   => 'charged',
                    '@fail'   => 'charge_failed',
                ],
                'charged'       => ['type' => 'final'],
                'charge_failed' => ['type' => 'final'],
            ],
        ],
        'inventory' => [
            'initial' => 'reserving',
            'states'  => [
                'reserving' => [
                    'machine' => InventoryMachine::class,
                    'with'    => ['order_id', 'items'],
                    '@done'   => 'reserved',
                    '@fail'   => 'reserve_failed',
                ],
                'reserved'       => ['type' => 'final'],
                'reserve_failed' => ['type' => 'final'],
            ],
        ],
    ],
],
```

**Composition:** Parallel state + machine delegation compose naturally. Each region runs its own child machine; the region's `@done` fires when the child completes; the parallel state's `@done` fires when all regions reach final.

## 3. Config API Design

### State-Level `machine` Key

| Key | Type | Required | Description |
|-----|------|----------|-------------|
| `machine` | `string` (FQCN) | Yes | Child machine class. Must extend `Machine`. |
| `with` | `array\|Closure` | No | Data to pass from parent context to child. |
| `@done` | `string\|array` | No | Fires when child reaches a final state. Uses existing transition format. |
| `@fail` | `string\|array` | No | Fires when child fails. Uses existing transition format. |
| `@timeout` | `array` | No | Fires when child doesn't complete within the given time. |
| `queue` | `bool\|string\|array` | No | Run child on a queue. |
| `forward` | `array` | No | Event types to forward from parent to the running child machine. |
| `on` | `array` | No | Additional events the parent can receive while child is running. |
| `entry` | `string\|array` | No | Actions to run before child starts. |
| `exit` | `string\|array` | No | Actions to run when exiting the state (cleanup). |
| `meta` | `array` | No | Metadata. |

### `with` — Parent → Child Context Transfer

```php
// Format 1: Pass same-named keys
'with' => ['order_id', 'total_amount'],
// Child context: { order_id: ..., total_amount: ... }

// Format 2: Rename keys (child_key => parent_key)
'with' => [
    'id'     => 'order_id',        // child sees 'id', parent has 'order_id'
    'amount' => 'total_amount',    // child sees 'amount', parent has 'total_amount'
],

// Format 3: Dynamic (closure)
'with' => fn (ContextManager $ctx) => [
    'order_id' => $ctx->get('order_id'),
    'amount'   => $ctx->get('total_amount') * 100, // transformations allowed
],
```

**Without `with`:** Child starts with its own `context` defaults. No parent data is transferred automatically.

### `@done` — Child Completion

`@done` retains its existing format exactly. When the child machine reaches a final state, the parent's `@done` transition fires.

```php
// String shorthand
'@done' => 'next_state',

// Single branch with actions
'@done' => [
    'target'      => 'next_state',
    'actions'     => 'handleResultAction',
    'calculators' => 'mapResultCalculator',
],

// Multi-branch guarded fork
'@done' => [
    ['target' => 'approved', 'guards' => 'isApprovedGuard'],
    ['target' => 'review',   'actions' => 'requestReviewAction'],
],
```

**Accessing child result data:**
When `@done` fires, the event is a `ChildMachineDoneEvent` with typed accessors (see Section 10):

```php
class StorePaymentResultAction extends ActionBehavior
{
    // Type-hint ChildMachineDoneEvent for IDE autocomplete
    public function __invoke(ContextManager $context, ChildMachineDoneEvent $event): void
    {
        // Typed accessors — no payload digging
        $context->set('payment_id', $event->result('payment_id'));
        $context->set('receipt', $event->childContext('receipt_url'));
        // Also available: $event->childMachineId(), $event->childMachineClass()
    }
}
```

### `@fail` — Child Failure

Fires when the child machine reaches a failure final state or throws an exception.

```php
'@fail' => 'error_state',

// Or with guards
'@fail' => [
    ['target' => 'retrying', 'guards' => 'canRetryGuard', 'actions' => 'incrementRetryAction'],
    ['target' => 'failed',   'actions' => 'logFailureAction'],
],
```

### `@timeout` — Child Timeout (New)

```php
'processing_payment' => [
    'machine'  => PaymentMachine::class,
    '@done'    => 'shipping',
    '@fail'    => 'payment_failed',
    '@timeout' => [
        'after'   => 300,                // seconds
        'target'  => 'payment_timed_out',
        'actions' => 'logTimeoutAction', // optional
    ],
],
```

`@timeout` is only meaningful in async (`queue`) mode. Follows the existing `ParallelRegionTimeoutJob` pattern.

### `queue` — Async Execution

```php
// Run on default queue
'queue' => true,

// Run on a specific queue
'queue' => 'payments',

// Detailed configuration
'queue' => [
    'connection' => 'redis',
    'queue'      => 'payments',
    'retry'      => 3,
],

// Without queue key (default): child runs synchronously
```

### `forward` — Event Forwarding to Child Machine

When the parent is in a state with `machine` key and receives an event, the parent normally handles it via `on`. But sometimes an event should be **forwarded** to the running child machine instead.

```php
// Forward listed event types to the child machine
'forward' => ['APPROVE_PAYMENT', 'UPDATE_AMOUNT'],

// Forward with event type mapping (parent event → child event)
'forward' => [
    'APPROVE_PAYMENT',                         // forward as-is
    'UPDATE_SHIPPING_INFO' => 'UPDATE_INFO',   // rename for child
],
```

**Without `forward`:** Events not matched by `on` are unhandled. The child only receives events via its own endpoints or direct `sendTo()` calls.

**`forward` requires `queue`:** The `forward` key is only valid on states that also have `queue` (async mode). In sync mode, the child runs to completion inline — there is no intermediate state to forward events to. `StateConfigValidator` rejects `forward` without `queue`.

**Event routing priority:**
1. Parent checks its `on` map first
2. If matched → parent handles it (may transition away, cancelling child)
3. If NOT matched AND event is in `forward` list → forward to child
4. If neither → event is unhandled

**How forwarding works internally (child restoration):**
```
Parent receives 'APPROVE_PAYMENT' event
  → Parent is in 'processing_payment' state (has machine key)
  → 'APPROVE_PAYMENT' not in parent's 'on' map
  → 'APPROVE_PAYMENT' IS in 'forward' list
  → Look up child's root_event_id:
      - From machine_children table: WHERE parent_root_event_id = ? AND status = 'running'
      - Or from parent's State.activeChildren map
  → Restore child: ChildMachine::restoreStateFromRootEventId($childRootEventId)
  → $childMachine->send($event)
  → If child reaches final state → fire parent's @done
  → If child doesn't reach final → child persists in new state, parent stays waiting
```

The child restoration uses the existing `Machine::restoreStateFromRootEventId()` infrastructure — the same mechanism that endpoints already use to restore machines from HTTP requests.

### `machine` + Other State Properties

The `machine` key can coexist with other state properties:

```php
'processing_payment' => [
    'machine' => PaymentMachine::class,
    'with'    => ['order_id', 'total_amount'],
    '@done'   => 'shipping',
    '@fail'   => 'payment_failed',
    '@timeout' => ['after' => 300, 'target' => 'timed_out'],
    'queue'   => 'payments',

    // Events forwarded to the child machine
    'forward' => ['APPROVE_PAYMENT', 'RETRY_CHARGE'],

    // Events handled by parent (parent transitions)
    'on' => [
        'CANCEL_ORDER' => 'cancelled',
    ],

    // Before/after child
    'entry' => 'logPaymentStartAction',
    'exit'  => 'logPaymentEndAction',

    'meta' => ['description' => 'Payment processing'],
],
```

## 4. Endpoints Integration

### How Endpoints Work Today

Endpoints expose machine events via HTTP:

```php
MachineDefinition::define(
    config: [...],
    behavior: [...],
    endpoints: [
        'START'    => null,                     // POST /start
        'CANCEL'   => '/custom-cancel',         // POST /custom-cancel
        'COMPLETE' => [
            'result'     => 'myResult',
            'status'     => 201,
            'action'     => CustomEndpointAction::class,
            'middleware'  => ['auth'],
        ],
    ],
);
```

The endpoint lifecycle: HTTP Request → EventBehavior validation → `action.before()` → `Machine::send()` → `action.after()` → Response.

### Endpoints for Child Machines

Child machines can define their own endpoints, enabling external systems (webhooks, payment gateways, etc.) to send events directly to a running child machine.

**Important clarifications:**
- Endpoint URIs are **fully custom** — you define the entire path.
- `{machineId}` is a **route parameter** (like any Laravel route parameter). When the HTTP request arrives, `MachineController` uses this parameter to find and restore the correct machine instance via `restoreStateFromRootEventId()`.
- The webhook pattern shown below is **one use case** — endpoints work for any HTTP-triggered event, not just webhooks.

```php
class PaymentMachine extends Machine
{
    public static function definition(): MachineDefinition
    {
        return MachineDefinition::define(
            config: [
                'id'      => 'payment',
                'initial' => 'awaiting_charge',
                'context' => ['order_id' => null, 'amount' => 0, 'payment_id' => null],
                'states'  => [
                    'awaiting_charge' => [
                        'entry' => InitiateChargeAction::class,
                        'on'    => [
                            'CHARGE_SUCCEEDED' => 'charged',
                            'CHARGE_FAILED'    => 'failed',
                        ],
                    ],
                    'charged' => [
                        'type'   => 'final',
                        'result' => PaymentResultBehavior::class,
                    ],
                    'failed' => ['type' => 'final'],
                ],
            ],
            // Endpoints are fully custom URIs — {machineId} is a route parameter
            // resolved by MachineController to restore the correct machine instance
            endpoints: [
                'CHARGE_SUCCEEDED' => [
                    'uri'    => '/webhooks/payment/{machineId}/success',
                    'method' => 'POST',
                    'action' => PaymentWebhookEndpointAction::class,
                ],
                'CHARGE_FAILED' => [
                    'uri'    => '/webhooks/payment/{machineId}/failure',
                    'method' => 'POST',
                ],
            ],
        );
    }
}
```

**Flow:** Parent invokes PaymentMachine → PaymentMachine starts, calls payment gateway (passing its `_machine_id` in the webhook URL) → Gateway sends webhook to child's endpoint → `MachineController` restores child from `{machineId}` route parameter → Child transitions to `charged` → Parent's `@done` fires.

### Endpoint-Mediated Communication Pattern

For complex integrations where child machines need external interaction:

```
Parent Machine
  │
  └── 'processing_payment' state
        │
        ├── machine: PaymentMachine (async, queue)
        │     │
        │     ├── awaiting_charge (entry: call Stripe API)
        │     │     │
        │     │     └── Stripe sends webhook ──→ PaymentMachine endpoint
        │     │                                   POST /webhooks/payment/{machineId}/success
        │     │                                   │
        │     ├── charged (final) ◄───────────────┘
        │     │
        │     └── (parent receives @done with payment result)
        │
        └── @done → shipping
```

### Parent-Child Endpoint Coordination

The auto-injected `_machine_id` context key (see Section 5) makes endpoint URL construction trivial:

```php
class InitiateChargeAction extends ActionBehavior
{
    public function __invoke(ContextManager $context): void
    {
        // IDE-friendly method — no magic string
        $machineId = $context->machineId();

        PaymentGateway::charge([
            'amount'      => $context->get('amount'),
            'webhook_url' => route('payment.webhook.success', ['machineId' => $machineId]),
        ]);
    }
}
```

No manual wiring needed — the child machine always knows its own identity via `$context->machineId()`.

## 5. Data Flow

### Current Model (Unchanged)

```
Event.payload → Action (writes to context) → Calculator → Guard → Transition → Next State
```

### Data Flow with Child Machines

```
Parent Context
    │
    ├── 'with' filters/maps ──→ Child Context (initial)
    │                                │
    │                                ├── Child lives its own lifecycle
    │                                │   (entry → events → transitions → actions)
    │                                │
    │                                └── Child reaches final state
    │                                      │
    │                                      ├── ResultBehavior output (if defined)
    │                                      └── Child's final context
    │
    ├── Available in @done event payload ◄──┘
    │     {
    │       'result':        <ResultBehavior output>,
    │       'child_context': <child's final context>,
    │       'machine_id':    <child's root_event_id>,
    │       'machine_class': <child's FQCN>,
    │     }
    │
    └── @done actions write to parent context
```

### Auto-Injected Context Keys

When a child machine is created via delegation, EventMachine automatically injects special context keys (prefixed with `_`) alongside the `with` data:

| Key | Value | Purpose |
|-----|-------|---------|
| `_machine_id` | Child's own `root_event_id` | Self-identification (e.g., for constructing endpoint URLs) |
| `_parent_root_event_id` | Parent's `root_event_id` | Enables child → parent communication via `sendTo()` |
| `_parent_state_id` | Parent state that invoked the child | Debugging, correlation |

```php
// Inside a child machine action:
class InitiateChargeAction extends ActionBehavior
{
    public function __invoke(ContextManager $context): void
    {
        // IDE-friendly methods (see Section 10 for API details)
        $machineId = $context->machineId();          // own root_event_id
        $parentId  = $context->parentMachineId();    // parent's root_event_id

        PaymentGateway::charge([
            'amount'      => $context->get('amount'),
            'webhook_url' => route('payment.webhook.success', ['machineId' => $machineId]),
        ]);
    }
}
```

**Note:** `_machine_id` is also injected for non-child machines (every machine gets it). This makes `root_event_id` easily accessible from any behavior without the clunky `$state->history->first()->root_event_id` path. The raw `$context->get('_machine_id')` string access remains as a fallback.

### Important: Context Isolation

Child machine context is **completely isolated** from the parent. There is no automatic merge. This differs from parallel regions (where context IS merged because they share the same machine instance).

Why:
- Child has a different machine definition with different context keys
- Automatic merge could pollute parent context
- Explicit mapping (via actions) is safer and more readable
- Consistent with EventMachine's "data flows through actions" principle

### Important: State Isolation

Parent's `State::$value` does **NOT** include child machine states. When a parent is in `processing_payment` (which delegates to `PaymentMachine`), the parent's state value is simply:

```php
$state->value; // ['order_workflow.processing_payment']
```

NOT a nested structure like `['processing_payment' => ['payment' => 'awaiting_charge']]`.

**Why:**
- `State::$value` currently returns `array` of strings (single for normal, multiple for parallel). Child states would break this contract.
- Context isolation principle extends to state isolation — the child's internal state is the child's business.
- Parallel regions are different: they're part of the SAME machine, so their states appear in `$value`. Child machines are SEPARATE machines.

**To query a child's state (when needed for observability/debugging):**
```php
// Restore the child and read its state directly
$childMachine = Machine::restoreStateFromRootEventId($context->machineId());
$childState = $childMachine->state->value;
```

This is a deliberate read, not automatic exposure. For runtime logic, the parent should rely on `@done`/`@fail`/`forward` — not poll the child's state.

## 6. Lifecycle & Queue Integration

### Sync Mode (Default)

```
Parent.send(EVENT)
  → Parent transitions → enters 'processing_payment' state
  → Runs entry actions
  → Detects 'machine' key
  → Resolves child context via 'with'
  → ChildMachine::create(context: $childContext)
  → Child completes its full lifecycle inline
  │
  ├── Child reaches final state → fire parent's @done transition
  │   → @done actions run (child result → parent context)
  │   → Parent moves to next state
  │
  └── Child fails → fire parent's @fail transition
      → @fail actions run
      → Parent moves to error state
```

### Async Mode (Queue)

```
Parent.send(EVENT)
  → Parent transitions → enters 'processing_payment' state
  → Runs entry actions
  → Detects 'machine' + 'queue'
  → Dispatches ChildMachineJob
  → Parent STAYS in 'processing_payment' state (waiting)
  → Parent.persist() → save to DB
  → Release lock

         ┌──── Queue ────┐
         │                │
         ▼                │
  ChildMachineJob.handle()│
  → ChildMachine::create()│
  → Run child lifecycle   │
  │                       │
  ├── Child reaches final │
  │   → ChildMachineCompletionJob dispatch
  │     → Restore parent from DB
  │     → Send @done event to parent (via Machine::send)
  │     → Parent transitions
  │
  └── Child fails
      → ChildMachineCompletionJob dispatch (with error payload)
      → Send @fail event to parent
      → Parent transitions to error state
```

### Multi-Step Async Children (Webhook Pattern)

Some child machines need external input mid-lifecycle (e.g., payment webhooks). For these:

```
Parent enters state with machine + queue
  → ChildMachineJob dispatches
  → Child starts, reaches 'awaiting_charge' state
  → Child's entry action calls external API (e.g., Stripe)
  → Child persists in 'awaiting_charge' state
  → ChildMachineJob completes (child is NOT yet final)
  │
  │  ... time passes ...
  │
  → External webhook hits child's endpoint
  → MachineController restores child from DB
  → Child processes CHARGE_SUCCEEDED event → transitions to 'charged' (final)
  → ChildMachineCompletionJob dispatches
  → Parent receives @done
```

This is where endpoints + machine delegation compose powerfully. The child machine exposes its own HTTP endpoints for external interaction, and the completion flows back to the parent automatically.

## 7. Cross-Machine Communication: `sendTo()`

### Motivation

The primary communication pattern is orchestration via `machine` key (see Section 8). But two scenarios need `sendTo()`:

1. **Child → parent progress reporting** (the main use case → use `sendToParent()`)
2. **Escape hatch** for rare edge cases where orchestration doesn't fit

### `sendTo()` and `sendToParent()` — New Methods on InvokableBehavior

`raise()` sends events to the same machine. `sendTo()` sends events to another machine. `sendToParent()` is a shorthand for the most common case:

```php
// PRIMARY USE CASE: Child → parent progress reporting
class ReportProgressAction extends ActionBehavior
{
    public function __invoke(ContextManager $context): void
    {
        // sendToParent() — IDE-friendly, uses $context->parentMachineId() internally
        $this->sendToParent(
            event: ['type' => 'CHILD_PROGRESS', 'payload' => ['percent' => 50]],
        );
    }
}

// ESCAPE HATCH: Direct machine-to-machine (use sparingly — prefer orchestrator pattern)
class NotifyExternalMachineAction extends ActionBehavior
{
    public function __invoke(ContextManager $context): void
    {
        // Via root_event_id string
        $this->sendTo(
            machine: $context->get('target_machine_id'),
            event: ['type' => 'NOTIFICATION', 'payload' => ['message' => 'done']],
        );

        // Async (queue)
        $this->sendTo(
            machine: $context->get('target_machine_id'),
            event: ['type' => 'NOTIFICATION'],
            async: true,
        );
    }
}
```

> **Design rule:** Prefer the orchestrator pattern (`machine` key) for all inter-machine workflows. Use `sendTo()` only when orchestration genuinely doesn't fit (e.g., fire-and-forget notifications, cross-model communication).

### raise() vs sendTo() vs sendToParent()

| Aspect | `raise()` | `sendTo()` | `sendToParent()` |
|--------|-----------|------------|------------------|
| Target | Same machine | Any machine (by ID or instance) | Parent machine |
| Mechanism | In-memory eventQueue | `send()` call or Job dispatch | Same as `sendTo()` |
| Timing | Same transition cycle | Immediate (sync) or queued (async) | Same as `sendTo()` |
| Data flow | Same context | Isolated contexts | Isolated contexts |
| Lock | Same lock | Different locks | Different locks |
| Use case | Internal events | Sibling/peer messaging | Progress reporting, child→parent |

### Progress Reporting: Child → Parent Status Updates

The combination of `sendTo()` + auto-injected `_parent_root_event_id` enables **bidirectional communication**. A child can report progress to its parent:

```php
// Child machine action — reports progress to parent
class ProcessBatchItemAction extends ActionBehavior
{
    public function __invoke(ContextManager $context): void
    {
        $processed = $context->get('processed_count') + 1;
        $total     = $context->get('total_items');
        $context->set('processed_count', $processed);

        // Report progress to parent — IDE-friendly, no magic strings
        $this->sendToParent(
            event: ['type' => 'CHILD_PROGRESS', 'payload' => [
                'percent'   => round(($processed / $total) * 100),
                'processed' => $processed,
                'total'     => $total,
            ]],
        );
    }
}
```

```php
// Parent machine config — receives progress updates
'batch_processing' => [
    'machine' => BatchProcessMachine::class,
    'with'    => ['items'],
    '@done'   => 'completed',
    '@fail'   => 'failed',

    // Progress events from child (parent stays in this state)
    'on' => [
        'CHILD_PROGRESS' => ['actions' => 'updateProgressAction'],
        'CANCEL'         => 'cancelled',
    ],
],
```

```php
class UpdateProgressAction extends ActionBehavior
{
    public function __invoke(ContextManager $context, EventBehavior $event): void
    {
        $context->set('progress_percent', $event->payload['percent']);

        // Could also notify UI, broadcast to websocket, etc.
        broadcast(new OrderProgressEvent(
            orderId: $context->get('order_id'),
            percent: $event->payload['percent'],
        ));
    }
}
```

**Key insight:** Progress events arrive as regular events on the parent's `on` map. The parent stays in the same state (no target transition) — it just runs actions. This composes naturally with existing EventMachine patterns.

## 8. Machine Systems — Orchestrator Machine IS the System Definition

### Philosophy

All inter-machine relationships are declared **inside machine definitions** — not on the Eloquent model, not in a separate `MachineSystem` class, not in config files.

The orchestrator machine's `definition()` IS the system definition. Reading it tells you:
- Which machines exist in the system
- How they relate to each other
- What data flows between them
- What happens when each completes or fails

```php
// OrderWorkflowMachine::definition() — THIS IS THE SYSTEM DEFINITION
'states' => [
    'validating'         => ['machine' => ValidationMachine::class, 'with' => ['order_id'], ...],
    'processing_payment' => ['machine' => PaymentMachine::class, 'with' => ['order_id', 'total_amount'], ...],
    'shipping'           => ['machine' => ShippingMachine::class, 'with' => ['order_id'], ...],
],
```

### The Orchestrator IS One of the Machines

```
OrderWorkflowMachine (orchestrator — defines the full system topology)
    ├── invokes ValidationMachine     via `machine` key
    ├── invokes PaymentMachine        via `machine` key
    └── invokes ShippingMachine       via `machine` key
```

There is NO separate `MachineSystem` class because the orchestrator machine already declares everything. A `MachineSystem` would duplicate what the orchestrator definition already provides.

### What About `$casts`?

The model's `$casts` is **NOT the system definition** — it's just Laravel integration glue:

```php
class Order extends Model
{
    use HasMachines;

    protected $casts = [
        'workflow' => OrderWorkflowMachine::class.':order', // Laravel attr casting
    ];
}
```

`$casts` tells Laravel "cast this attribute to this type." The actual system topology is inside `OrderWorkflowMachine::definition()`.

### Why No `MachineSystem` Class

| Concern | Solved By |
|---------|-----------|
| "Where are all the machines defined?" | Orchestrator machine's `definition()` — `machine` key references |
| "How do machines communicate?" | `@done`/`@fail` (parent-child) + `sendToParent()` (child→parent) |
| "Who coordinates the flow?" | The orchestrator machine |
| "How do sibling machines talk?" | They don't talk directly. Flow goes through the orchestrator. |

### Design Rule: Always Use an Orchestrator

Sibling machines should NOT communicate directly. If `PaymentMachine` needs to trigger `ShippingMachine`, don't use `sendTo()` between them — instead, let the orchestrator handle the flow:

```php
// WRONG: PaymentMachine directly tells ShippingMachine to start
// (scattered across action classes, not visible in any definition)
class NotifyShippingAction extends ActionBehavior {
    public function __invoke(ContextManager $context): void {
        $this->sendTo(machine: $order->shipping, event: [...]);
    }
}

// RIGHT: Orchestrator manages the flow (visible in definition)
'processing_payment' => [
    'machine' => PaymentMachine::class,
    '@done'   => 'shipping',              // orchestrator decides what's next
],
'shipping' => [
    'machine' => ShippingMachine::class,  // orchestrator invokes next child
],
```

**`sendTo()` is an escape hatch**, not the primary communication pattern. Its main use case is `sendToParent()` for progress reporting from child to parent.

### Communication Patterns

| Pattern | Mechanism | Best For |
|---------|-----------|----------|
| **Orchestration** | `machine` key (parent-child) | All inter-machine workflows — **the primary and recommended pattern** |
| **Progress reporting** | `sendToParent()` | Child → parent status updates |
| **External interaction** | Endpoints (webhooks) | Third-party system callbacks |
| **Loose coupling** | Laravel Events + Listeners | Cross-model, fire-and-forget notifications |
| **Escape hatch** | `sendTo()` | Edge cases that don't fit orchestration |

## 9. Edge Cases & Tricky Parts

> Informed by analysis of XState (140+ tests), laravel-workflow, Temporal PHP, broadway-saga, and spatie/laravel-event-sourcing test suites.

### 9.1 Lifecycle Edge Cases

| # | Edge Case | Expected Behavior | Source |
|---|-----------|-------------------|--------|
| 1 | **`@always` transition skips machine delegation** | If a state has both `machine` and `@always`, and `@always` fires immediately, the child machine must NEVER be started. | XState: invoke.test.ts |
| 2 | **State exits while async child is running** | Parent receives `CANCEL` event and transitions away. The child must be cancelled — its completion should be silently ignored. | XState: invoke.test.ts |
| 3 | **Child completes but parent already left the state** | `ChildMachineCompletionJob` must check if parent is still in the invoking state. If not, discard the result. Idempotent by design. | laravel-workflow |
| 4 | **Re-entering the same state with machine key** | If parent transitions away and back to the same state, a NEW child machine must be created. The old child must be cancelled. | XState: system.test.ts |
| 5 | **Sync-completing child during parent initialization** | A child machine that immediately reaches final state (no events needed) must not crash the parent's initialization. | XState: actor.test.ts |
| 6 | **Nested delegation (child invokes grandchild)** | Must work. Grandchild completes → child's @done fires → child reaches final → parent's @done fires. No depth limit. | XState: invoke.test.ts |
| 7 | **Stopped child's events are silently dropped** | If a child machine is cancelled/stopped and its actions try to send events back, those events must be silently ignored. | XState: actor.test.ts |

### 9.2 Parallel + Machine Delegation Edge Cases

| # | Edge Case | Expected Behavior | Source |
|---|-----------|-------------------|--------|
| 8 | **Cross-parallel @fail isolation** | If one parallel region's child machine fails, only that region's `@fail` fires. Other regions continue unaffected until the parallel state's `@fail` fires. | XState: invoke.test.ts |
| 9 | **Same machine class in multiple parallel regions** | Two regions both delegating to `PaymentMachine::class` must create two independent child instances with unique root_event_ids. Events must not leak between them. | XState: invoke.test.ts |
| 10 | **One region's child completes while another's is still running** | Completed region transitions to its final state. The parallel state does NOT fire `@done` until ALL regions are final. | Existing parallel behavior |
| 11 | **Parallel + async: race condition between child completions** | Two async children complete simultaneously. Both `ChildMachineCompletionJob`s try to send events to parent. Lock ensures sequential processing. Only one can acquire the lock at a time. | ParallelRegionJob pattern |

### 9.3 Queue/Async Edge Cases

| # | Edge Case | Expected Behavior | Source |
|---|-----------|-------------------|--------|
| 12 | **ChildMachineJob fails (exception)** | `failed()` method dispatches `ChildMachineCompletionJob` with error payload. Parent receives `@fail`. | laravel-workflow |
| 13 | **ChildMachineCompletionJob runs but parent machine is gone** | Job must handle gracefully: if parent's root_event_id doesn't exist in DB, log and discard. No exception. | broadway-saga |
| 14 | **Duplicate completion (idempotency)** | If `ChildMachineCompletionJob` runs twice (queue retry), the second run must be a no-op. Check: is parent still in the invoking state? | laravel-workflow |
| 15 | **Transaction race: parent persists but completion job runs before commit** | Use `afterCommit` on ChildMachineJob dispatch. Ensures parent state is committed to DB before child can read it. | laravel-workflow |
| 16 | **@timeout fires after child already completed** | Timeout job must check child status before firing. If child already completed, the timeout is a no-op. | Temporal |
| 17 | **@timeout fires while @done CompletionJob is in queue** | Parent must handle only the first event (either @done or @timeout). The second arrival finds parent in a different state → discard. | Temporal |
| 18 | **Child machine needs external events mid-lifecycle (webhook)** | Child starts, calls external API, persists in intermediate state. External webhook hits child's endpoint → child transitions → child reaches final → parent gets @done. | Endpoints feature |

### 9.4 Data Flow Edge Cases

| # | Edge Case | Expected Behavior | Source |
|---|-----------|-------------------|--------|
| 19 | **`with` references a context key that doesn't exist** | Should throw a clear exception during child creation, not a silent null. | Defensive design |
| 20 | **Child modifies context keys that match parent's keys** | No effect on parent. Contexts are fully isolated. Child's `order_id` has zero connection to parent's `order_id`. | Core principle |
| 21 | **@done action tries to access child_context but child had no ResultBehavior** | `event.payload['result']` is null. `event.payload['child_context']` still contains the child's final context. | Graceful fallback |
| 22 | **Multiple @done events from the same child** | Must not happen. A child machine can only reach ONE final state. The first `@done` is definitive. | State machine semantics |

### 9.5 Event Forwarding Edge Cases

| # | Edge Case | Expected Behavior | Source |
|---|-----------|-------------------|--------|
| 23 | **Forwarded event but child already completed** | Child is in final state, forwarded event is unhandled on child. Parent should receive an error or the event is silently dropped. | Defensive design |
| 24 | **Forwarded event arrives while child is restoring** | Lock on child machine prevents concurrent processing. Forwarded event waits for lock. | Existing lock pattern |
| 25 | **`forward` without `queue` in config** | Caught at boot-time by `StateConfigValidator`. Not a runtime edge case — throws `MachineDefinitionException` during definition parsing. | Validation rule |
| 26 | **Forwarded event causes child to reach final** | After forwarding, check if child reached final → fire parent's `@done`. Same flow as webhook completion. | Endpoint pattern |
| 27 | **Progress event from child via `sendTo()` while parent is transitioning** | Parent's lock prevents race condition. Progress event waits for lock, then is processed against parent's `on` map. | Existing lock pattern |

### 9.6 Error Propagation Edge Cases

| # | Edge Case | Expected Behavior | Source |
|---|-----------|-------------------|--------|
| 28 | **Child's action throws exception (sync mode)** | Exception propagates to parent's `@fail` transition. If no `@fail` defined, exception bubbles up to the caller of parent's `send()`. | XState: invoke.test.ts |
| 29 | **Child's guard fails** | Guard failure is NOT an error. The child continues in its current state (no transition). This is normal operation. | Existing guard behavior |
| 30 | **Child's validation guard fails** | `MachineValidationException` in child should propagate as `@fail` to parent with error details. | Existing validation behavior |
| 31 | **Compensation: parent enters compensating state which itself invokes a machine** | Must work. The compensating machine is just another child machine. If the compensating machine fails, its @fail fires on the compensating state. | broadway-saga |

## 10. Internal Architecture & New Classes

### New Classes

```
src/
  Definition/
    MachineInvokeDefinition.php     # Parses machine + with + forward + queue + @timeout config
  Behavior/
    ChildMachineDoneEvent.php       # Typed @done event with result/childContext accessors
    ChildMachineFailEvent.php       # Typed @fail event with error details
  Jobs/
    ChildMachineJob.php             # Runs child machine on queue
    ChildMachineCompletionJob.php   # Sends @done/@fail back to parent
    ChildMachineTimeoutJob.php      # @timeout handler (like ParallelRegionTimeoutJob)
```

### ChildMachineDoneEvent — Typed Child Result Access

Instead of digging into raw payload arrays, `@done` actions receive a typed event:

```php
class ChildMachineDoneEvent extends EventBehavior
{
    public static function getType(): string
    {
        return 'CHILD_MACHINE_DONE';
    }

    /**
     * Get the child's ResultBehavior output.
     */
    public function result(?string $key = null): mixed
    {
        $result = $this->payload['result'] ?? null;

        return $key !== null ? data_get($result, $key) : $result;
    }

    /**
     * Get the child's final context (as array).
     */
    public function childContext(?string $key = null): mixed
    {
        $context = $this->payload['child_context'] ?? [];

        return $key !== null ? data_get($context, $key) : $context;
    }

    /**
     * Get the child machine's root_event_id.
     */
    public function childMachineId(): string
    {
        return $this->payload['machine_id'];
    }

    /**
     * Get the child machine's FQCN.
     */
    public function childMachineClass(): string
    {
        return $this->payload['machine_class'];
    }
}
```

**Usage in @done actions (type-hint injection works automatically):**

```php
class StorePaymentResultAction extends ActionBehavior
{
    // Type-hint ChildMachineDoneEvent instead of EventBehavior
    public function __invoke(ContextManager $context, ChildMachineDoneEvent $event): void
    {
        // IDE autocomplete works — no magic strings for payload keys
        $context->set('payment_id', $event->result('payment_id'));
        $context->set('receipt_url', $event->childContext('receipt_url'));
    }
}
```

**Before vs After:**
```php
// Before (raw payload):
$event->payload['result']['payment_id']        // no IDE help, typo-prone
$event->payload['child_context']['receipt_url'] // deeply nested

// After (typed event):
$event->result('payment_id')                    // IDE autocomplete, discoverable
$event->childContext('receipt_url')              // clean, typed
```

| Class | Change |
|-------|--------|
| `StateDefinition` | New `machine`, `forward` properties, `getMachineInvokeDefinition()`, `hasMachineInvoke(): bool` |
| `MachineDefinition` | Machine invoke start logic in `enterState()`, cleanup in `exitState()`, forward routing in `transition()` |
| `InvokableBehavior` | New `sendTo()` and `sendToParent()` methods |
| `InternalEvent` | New cases: `CHILD_MACHINE_DONE`, `CHILD_MACHINE_FAIL`, `CHILD_MACHINE_TIMEOUT` |

| `ContextManager` | Auto-inject `_machine_id` on initialization (every machine) |
| `StateDefinitionType` | No new type. A state with `machine` key remains `ATOMIC` — delegation is a property, not a state type (same as how `entry`/`exit` don't change the type) |

### Root Event ID Accessibility

Currently, accessing `root_event_id` from a behavior requires the clunky path:
```php
$state->history->first()->root_event_id
```

With this plan, `_machine_id` is auto-injected into context for **every** machine:
```php
// Before (current):
public function __invoke(State $state): void {
    $id = $state->history->first()->root_event_id; // ugly
}

// After (with this plan):
public function __invoke(ContextManager $context): void {
    $id = $context->get('_machine_id'); // clean
}
```

This is injected in `Machine::create()` / `Machine::start()` right after context initialization, before any entry actions run.

```php
        return $this->get('_machine_id');
```

### IDE-Friendly API: InvokableBehavior::sendToParent()

Shorthand for the common pattern of sending events back to the parent:

```php
// InvokableBehavior addition:
class InvokableBehavior
{
    /**
     * Send an event to the parent machine that invoked this child.
     * Shorthand for sendTo(parent_root_event_id, event).
     */
    public function sendToParent(array|EventBehavior $event, bool $async = false): void
    {
        $parentId = $this->context->parentMachineId();

        if ($parentId === null) {
            throw new MachineException('Cannot sendToParent: this machine was not invoked by a parent.');
        }

        $this->sendTo(machine: $parentId, event: $event, async: $async);
    }
}
```

**Comparison — Progress reporting before vs after:**

```php
// Before (string-based, error-prone):
$this->sendTo(
    machine: $context->get('_parent_root_event_id'),
    event: ['type' => 'CHILD_PROGRESS', 'payload' => ['percent' => 20]],
);

// After (IDE-friendly, discoverable):
$this->sendToParent(
    event: ['type' => 'CHILD_PROGRESS', 'payload' => ['percent' => 20]],
);
```

Both `$context->machineId()` and `$this->sendToParent()` will have proper PHPDoc and return types for full IDE support.

### MachineInvokeDefinition

```php
class MachineInvokeDefinition
{
    public function __construct(
        public readonly string $machineClass,
        public readonly array|Closure|null $with,
        public readonly array $forward,        // event types to forward to child
        public readonly bool $async,
        public readonly ?string $queue,
        public readonly ?string $connection,
        public readonly ?int $timeout,
        public readonly ?int $retry,
    ) {}

    public function resolveChildContext(ContextManager $parentContext): array
    {
        if ($this->with === null) {
            return [];
        }

        if ($this->with instanceof Closure) {
            return ($this->with)($parentContext);
        }

        $childContext = [];
        foreach ($this->with as $key => $value) {
            if (is_int($key)) {
                // Format 1: ['order_id'] → same name
                $childContext[$value] = $parentContext->get($value);
            } else {
                // Format 2: ['amount' => 'total_amount'] → rename
                $childContext[$key] = $parentContext->get($value);
            }
        }

        return $childContext;
    }
}
```

### InternalEvent Additions

```php
enum InternalEvent: string
{
    // ... existing 26 events (MACHINE_*, STATE_*, TRANSITION_*, ACTION_*, GUARD_*, CALCULATOR_*, EVENT_*, PARALLEL_*) ...

    // New: Child Machine lifecycle events
    case CHILD_MACHINE_START     = '{machine}.child.{placeholder}.start';     // child created & started
    case CHILD_MACHINE_DONE      = '{machine}.child.{placeholder}.done';      // child reached final state
    case CHILD_MACHINE_FAIL      = '{machine}.child.{placeholder}.fail';      // child failed (exception or error final)
    case CHILD_MACHINE_TIMEOUT   = '{machine}.child.{placeholder}.timeout';   // @timeout fired
    case CHILD_MACHINE_CANCELLED = '{machine}.child.{placeholder}.cancelled'; // parent exited state, child stopped
}
```

**Rationale for 5 events:** Mirrors the existing PARALLEL_* pattern (PARALLEL_REGION_ENTER, PARALLEL_DONE, PARALLEL_FAIL, PARALLEL_REGION_TIMEOUT, PARALLEL_REGION_STALLED). The START/CANCELLED pair enables observability for debugging long-running async children.

## 11. Database Schema Changes

### New Table: `machine_children` (async mode only)

```sql
CREATE TABLE machine_children (
    id                    CHAR(26) PRIMARY KEY,       -- ULID
    parent_root_event_id  CHAR(26) NOT NULL,          -- Parent machine
    parent_state_id       VARCHAR(255) NOT NULL,       -- Which parent state invoked this
    child_machine_class   VARCHAR(255) NOT NULL,       -- Child FQCN
    child_root_event_id   CHAR(26) NULL,              -- Filled after child starts
    status                VARCHAR(20) NOT NULL,        -- pending|running|completed|failed|cancelled|timed_out
    created_at            TIMESTAMP NOT NULL,
    completed_at          TIMESTAMP NULL,

    INDEX idx_parent (parent_root_event_id, parent_state_id),
    INDEX idx_child (child_root_event_id),
    INDEX idx_status (status)
);
```

Sync mode doesn't need this table — the child runs inline.

### machine_events Table: No Changes

Child machine events are stored under the child's own `root_event_id`. The parent's context stores the child's `root_event_id` for reference.

### No `machine_systems` Table

Earlier drafts considered a `machine_systems` table. This is NOT needed because:
- The Eloquent model IS the system — machines are grouped via `HasMachines` trait
- Parent-child relationships are tracked in `machine_children`
- Sibling relationships are implicit (same model instance)
- No new table needed for `sendTo()` — it's a direct `Machine::send()` call

## 12. Configuration — Definition-Level, Not Global

### Design Decision: No `config/machine.php` Additions

EventMachine has never had per-machine configuration in `config/machine.php`. That file contains only **infrastructure-level** settings (parallel dispatch, archival). Adding child machine defaults there would break this pattern.

Instead, all child machine settings live **inside the machine definition** — exactly where they belong:

```php
// Queue, timeout, retry are all per-state (in the config array)
'processing_payment' => [
    'machine'  => PaymentMachine::class,
    'with'     => ['order_id', 'total_amount'],
    'queue'    => ['connection' => 'redis', 'queue' => 'payments', 'retry' => 3],
    '@timeout' => ['after' => 300, 'target' => 'timed_out'],
    '@done'    => 'shipping',
    '@fail'    => 'payment_failed',
],
```

### Why This Is Better

1. **Consistency:** `should_persist`, `context`, `scenarios_enabled` are all per-definition. Child machine settings should be too.
2. **Self-documenting:** Reading a machine definition tells you everything about how it behaves — no need to check a separate config file.
3. **Independent:** Different states can delegate to different children with different queue/timeout settings.
4. **No global coupling:** One team's PaymentMachine doesn't affect another team's ShippingMachine.

### Shared Defaults via Base Class (If Needed)

If a team wants consistent defaults across machines, they can use a base machine class:

```php
abstract class BaseAsyncMachine extends Machine
{
    protected static function defaultQueueConfig(): array
    {
        return ['connection' => 'redis', 'queue' => 'machine-children', 'retry' => 3];
    }
}
```

This is the Laravel way — composition through inheritance, not global config.

## 16. Full Example: E-Commerce Order System

```php
class ECommerceOrderMachine extends Machine
{
    public static function definition(): MachineDefinition
    {
        return MachineDefinition::define(
            config: [
                'id'      => 'ecommerce_order',
                'initial' => 'validating',
                'context' => [
                    'order_id'       => null,
                    'customer_id'    => null,
                    'items'          => [],
                    'total_amount'   => 0,
                    'payment_result' => null,
                    'tracking_code'  => null,
                ],
                'states' => [
                    // Phase 1: Order validation (sync child)
                    'validating' => [
                        'machine' => OrderValidationMachine::class,
                        'with'    => ['order_id', 'items', 'total_amount'],
                        '@done'   => 'fulfilling',
                        '@fail'   => 'rejected',
                    ],

                    // Phase 2: Parallel fulfillment (async children)
                    'fulfilling' => [
                        'type'   => 'parallel',
                        '@done'  => 'shipping',
                        '@fail'  => 'compensating',
                        'states' => [
                            'payment' => [
                                'initial' => 'charging',
                                'states'  => [
                                    'charging' => [
                                        'machine'  => PaymentMachine::class,
                                        'with'     => ['order_id', 'total_amount'],
                                        'queue'    => 'payments',
                                        '@done'    => [
                                            'target'  => 'charged',
                                            'actions' => 'storePaymentResultAction',
                                        ],
                                        '@fail'    => 'charge_failed',
                                        '@timeout' => ['after' => 120, 'target' => 'charge_failed'],
                                    ],
                                    'charged'       => ['type' => 'final'],
                                    'charge_failed' => ['type' => 'final'],
                                ],
                            ],
                            'inventory' => [
                                'initial' => 'reserving',
                                'states'  => [
                                    'reserving' => [
                                        'machine'  => InventoryReservationMachine::class,
                                        'with'     => ['order_id', 'items'],
                                        'queue'    => 'inventory',
                                        '@done'    => 'reserved',
                                        '@fail'    => 'reserve_failed',
                                        '@timeout' => ['after' => 60, 'target' => 'reserve_failed'],
                                    ],
                                    'reserved'       => ['type' => 'final'],
                                    'reserve_failed' => ['type' => 'final'],
                                ],
                            ],
                        ],
                    ],

                    // Phase 3: Shipping (async child with webhook)
                    'shipping' => [
                        'machine'  => ShippingMachine::class,
                        'with'     => ['order_id', 'customer_id'],
                        'queue'    => 'shipping',
                        '@done'    => [
                            'target'  => 'completed',
                            'actions' => 'storeTrackingCodeAction',
                        ],
                        '@fail' => 'shipping_failed',
                    ],

                    // Compensation (saga rollback)
                    'compensating' => [
                        'machine' => OrderCompensationMachine::class,
                        'with'    => ['order_id', 'payment_result'],
                        '@done'   => 'cancelled',
                    ],

                    // Terminal states
                    'completed'       => ['type' => 'final'],
                    'rejected'        => ['type' => 'final'],
                    'cancelled'       => ['type' => 'final'],
                    'shipping_failed' => ['type' => 'final'],
                ],
            ],
            behavior: [
                'actions' => [
                    'storePaymentResultAction' => StorePaymentResultAction::class,
                    'storeTrackingCodeAction'  => StoreTrackingCodeAction::class,
                ],
            ],
        );
    }
}
```

## 17. Open Questions

### Decided

1. **~~Eloquent model injection into child~~** → **Leaning towards YES via `with`**. The `with` key should be the mechanism — if you want the child to have the model, pass it explicitly. Auto-injection would break context isolation.

2. **~~Multi-step async children (webhook pattern)~~** → **Option B**. The parent simply doesn't receive `@done` until the child actually reaches a final state. The `machine_children` table tracks status (`running`) but the parent doesn't poll — it just waits. The completion event (`@done`/`@fail`) is the only signal.

3. **~~Nested async depth limit~~** → **Skip for now.** Cycle detection and depth limiting will be a separate PR. Not part of the initial implementation.

4. **~~Child machine auto-context key~~** → **YES**. Auto-inject `_machine_id` (every machine) and `_parent_root_event_id` (child machines). This enables `sendTo()` for progress reporting and easy endpoint URL construction. See Section 5 "Auto-Injected Context Keys".

5. **~~`machine` + `type: 'parallel'` mutual exclusivity~~** → **YES, mutually exclusive**. Validate at boot in `StateConfigValidator`, throw `MachineDefinitionException` (same pattern as other config validation errors).

6. **~~`forward` + sync mode~~** → **Invalid combination**. Validate at boot. In sync mode, the child runs inline to completion — there is no intermediate state to forward events to. `forward` is only meaningful for async children.

7. **~~`_machine_id` breaking change~~** → **Not a concern**. Inter-machine communication will ship as a major version. Adding `_machine_id` to all machine contexts is acceptable.

8. **~~IDE-friendly API for special context keys~~** → **YES**. Add `$context->machineId()`, `$context->parentMachineId()` to `ContextManager`, and `$this->sendToParent()` to `InvokableBehavior`. String-based `$context->get('_machine_id')` remains as fallback. See Section 10.

9. **~~Child result data access~~** → **YES**. New `ChildMachineDoneEvent` class with typed accessors: `->result()`, `->childContext()`, `->childMachineId()`. See Section 10.

10. **~~Parent state includes child states?~~** → **NO**. Parent's `State::$value` only shows parent states. Child state is encapsulated. Query child separately if needed. See Section 5.

11. **~~`machine_systems` table needed?~~** → **NO**. Only `machine_children` table. The Eloquent model IS the system — no separate tracking table needed.

12. **~~Declarative sibling routing / MachineSystem class~~** → **NO separate class needed**. The orchestrator machine's `definition()` IS the system definition. All inter-machine relationships are declared via `machine` key inside the orchestrator. Sibling machines don't communicate directly — flow goes through the orchestrator. `sendTo()` is an escape hatch (mainly for `sendToParent()` progress reporting). No `MachineSystem` class, no `$machineRouting` on models.

### All Questions Decided

No open questions remain. All decisions are captured in the "Decided" section above.
