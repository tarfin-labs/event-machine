# Fakeable Machine::create() + Auto Teardown

**Status:** Draft v3 (final review fixes applied)
**Author:** Claude
**Date:** 2025-03-18
**Scope:** Make Machine::create() fakeable for controller testing + auto-reset fakes after each test

---

## Problem 1: Machine::create() Can't Be Faked

Controller code:
```php
$machine = CarSalesMachine::create(state: $consentLink->machine_root_event_id);
$machine->send(['type' => ConsentGrantedEvent::getType()]);
$machine->persist();
```

`Machine::create()` is static â€” it doesn't go through Laravel's container. In controller tests, the user wants to test DB logic without running the machine pipeline. But `create(state:)` triggers full DB restore + `start()`, and `send()` throws `NoTransitionDefinitionFoundException` if the machine isn't in the right state.

**The user shouldn't have to simulate the entire machine flow just to test a controller's DB operations.**

---

## Problem 2: resetMachineFakes() is Manual

Every test that uses `Machine::fake()` must manually call `Machine::resetMachineFakes()`.

### Audit: Current Usage

| File | How reset is called | Auto-teardown safe? |
|------|--------------------|--------------------|
| `MachineFakingTest.php` | `afterEach` at line 385 + redundant manual calls in test bodies | Yes â€” `afterEach` already handles it |

| E2E tests | Not used â€” no `Machine::fake()` calls | N/A |

**Critical finding:** No test relies on fakes persisting across test boundaries. No `beforeAll` or shared `beforeEach` sets up fakes for multiple tests. Every `fake()` call is inside an `it()` block.

**Conclusion:** Auto-teardown via trait is safe â€” breaks zero existing tests.

## Design

### Part 1: Fakeable Machine::create()

When `Machine::fake()` is active for a class, `Machine::create()` returns a lightweight stub. The stub:
- Does NOT restore from DB, does NOT call `start()`
- Has a null-safe State with fake result as context
- Records that `create()` was called (for assertions)
- `send()` records the event but doesn't transition (no-op)
- `persist()` is a no-op

**Intercept at the top of `create()`:**

```php
public static function create(...): self
{
    if (static::isMachineFaked()) {
        return static::createFaked();
    }
    // ... existing logic
}
```

**`createFaked()` implementation:**

```php
protected static function createFaked(): self
{
    $fake = self::getMachineFake(static::class);

    $machine = self::withDefinition(static::definition());
    $machine->definition->machineClass = static::class;
    $machine->definition->shouldPersist = false; // defense-in-depth

    // Initialize with minimal state (no DB, no start)
    $machine->state = new State(
        context: new ContextManager(data: $fake['result'] ?? []),
        currentStateDefinition: null,
    );

    // Mark as faked instance (instance-level, not class-level)
    $machine->isFakedInstance = true;

    // Record create() call â€” use static::class here (we're still in the static context)
    self::$machineFakes[static::class]['creations'][] = [];

    return $machine;
}
```

**`send()` no-op â€” instance-level guard:**

```php
public function send(...): State
{
    if ($this->isFakedInstance) {
        // Record the event type for assertSent()
        // Use $this->definition->machineClass (NOT static::class â€” see note below)
        $eventType = match (true) {
            is_array($event)                => $event['type'] ?? null,
            $event instanceof EventBehavior => $event->type,
            is_string($event)               => $event,
            default                         => null,
        };

        self::$machineFakes[$this->definition->machineClass]['sends'][] = [
            'type' => $eventType,
        ];

        return $this->state;
    }
    // ... existing logic
}
```

> **Why `$this->definition->machineClass` not `static::class`?** `createFaked()` uses `self::withDefinition()` which creates a `Machine` base class instance. Inside `send()`, `static::class` would resolve to `Machine`, but the fake is stored under `CarSalesMachine::class`. Using `$this->definition->machineClass` (already set to `static::class` in `createFaked()`) correctly resolves to the faked class.

> **Why `$this->isFakedInstance` not `static::isMachineFaked()`?** A `withDefinition()`-created instance of the same class (used in child delegation) must NOT be affected by the fake. Only instances from `createFaked()` should be no-ops.

**`persist()` no-op:**

```php
public function persist(...): void
{
    if ($this->isFakedInstance) {
        return;
    }
    // ... existing logic
}
```

**`$isFakedInstance` property:**

```php
protected bool $isFakedInstance = false;
```

> `protected` not `public` â€” nothing outside Machine should read this.

### Tracking: Separate Arrays (not mixed into `invocations`)

The existing `invocations` array stores child delegation context (domain data from `with` config). Mixing `_source` metadata would break `assertInvokedWith()` subset matching.

**Updated `$machineFakes` structure:**

```php
self::$machineFakes[static::class] = [
    'result'      => $result,
    'fail'        => $fail,
    'error'       => $error,
    'finalState'  => $finalState,
    'invocations' => [],   // child delegation context (existing, unchanged)
    'creations'   => [],   // create() calls (new)
    'sends'       => [],   // send() calls on faked instances (new)
];
```

This keeps concerns separated:
- `assertInvoked()` / `assertInvokedWith()` â†’ reads `invocations` (unchanged)
- `assertCreated()` â†’ reads `creations`
- `assertSent()` â†’ reads `sends`

### Assertion Methods (complete family)

```php
// create() assertions
CarSalesMachine::assertCreated();           // at least once
CarSalesMachine::assertCreatedTimes(int $n); // exactly N times
CarSalesMachine::assertNotCreated();        // never

// send() assertions
CarSalesMachine::assertSent(string $type);       // event type sent at least once
CarSalesMachine::assertNotSent(string $type);     // event type never sent
CarSalesMachine::assertSentTimes(string $type, int $n); // exact count
```

Follows the same pattern as existing `assertInvoked/NotInvoked/InvokedTimes/InvokedWith` family.

### What Machine::fake() now means (unified semantics):

| Context | Behavior |
|---------|----------|
| Child delegation (`machine` key) | Parent short-circuits child creation, routes @done/@fail |
| Standalone `create()` | Returns stub, send/persist no-ops |
| Both in same test | Same `$machineFakes` entry, both work independently |

### Controller test example:

```php
it('approves consent link', function () {
    CarSalesMachine::fake();

    $consentLink = ConsentLink::factory()->create([
        'machine_root_event_id' => 'evt_123',
    ]);

    $this->postJson("/consent/{$consentLink->hash}/approve")
        ->assertOk()
        ->assertJson(['status' => 'approved']);

    expect($consentLink->fresh()->status)->toBe(ConsentLinkStatus::APPROVED);

    CarSalesMachine::assertCreated();
    CarSalesMachine::assertSent(ConsentGrantedEvent::getType());
    // No resetMachineFakes() needed â€” InteractsWithMachines handles it
});
```

### Part 2: Auto Teardown

**Trait using Laravel's `tearDown{TraitName}` convention:**

```php
// src/Testing/InteractsWithMachines.php
namespace Tarfinlabs\EventMachine\Testing;

use Tarfinlabs\EventMachine\Actor\Machine;
use Tarfinlabs\EventMachine\Behavior\InlineBehaviorFake;

trait InteractsWithMachines
{
    protected function tearDownInteractsWithMachines(): void
    {
        Machine::resetMachineFakes();
        CommunicationRecorder::reset();
        InlineBehaviorFake::resetAll();
    }
}
```

> `tearDown{TraitName}` is verified to work in both Laravel TestCase and Orchestra Testbench. It's registered via `beforeApplicationDestroyed()` in `InteractsWithTestCaseLifecycle::setUpTraits()`.

> **Prerequisite:** TestCase must extend Laravel's or Testbench's TestCase. Pure PHPUnit TestCase without these traits won't auto-call tearDown. Document this.

**User setup:**
```php
// tests/Pest.php
uses(
    Tests\TestCase::class,
    Illuminate\Foundation\Testing\RefreshDatabase::class,
    Tarfinlabs\EventMachine\Testing\InteractsWithMachines::class,
)->in('Feature');
```

**What the trait resets:**

| Static State | Method | Was Previously |
|-------------|--------|----------------|
| `Machine::$machineFakes` | `Machine::resetMachineFakes()` | Manual in afterEach |
| `CommunicationRecorder` | `CommunicationRecorder::reset()` | Manual or via TestMachine::resetFakes() |
| `InlineBehaviorFake` | `InlineBehaviorFake::resetAll()` | Manual or via TestMachine::resetFakes() |

**Backward compatibility:** Existing tests that manually call `resetMachineFakes()` â†’ still work (double-reset is idempotent, verified in audit).

## Edge Cases

### `create(definition: [...])` on a faked subclass
Returns the stub â€” inline definition is ignored. Faking overrides all instantiation paths. Documented.

### Queue jobs that call `create()`
`Machine::fake()` affects all `create()` calls in the same PHP process. `Queue::fake()` already prevents job execution. Document: use `Queue::fake()` alongside `Machine::fake()` in controller tests.

### `Machine::test()` IS affected by fake

`Machine::test()` â†’ `TestMachine::create()` â†’ `$machineClass::create()` â†’ fake intercept fires â†’ stub returned.

This is semantically correct: `Machine::fake()` means "don't run this machine." If you then try to `Machine::test()` the same class, you're contradicting yourself â€” fake says skip, test says exercise.

**Decision: Don't fix, document.**

`::: warning` in docs: "Don't combine `Machine::fake()` with `Machine::test()` for the same class in the same test. `fake()` is for skipping the machine, `test()` is for exercising it. If you need both, use separate tests."

Test F27 verifies this behavior: `Machine::test()` on a faked class returns the faked stub, confirming the expected behavior rather than treating it as a bug.
