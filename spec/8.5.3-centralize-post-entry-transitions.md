# Centralize processPostEntryTransitions into enterState()

**Status:** Draft
**Author:** Claude
**Date:** 2026-03-22
**Scope:** Eliminate the recurring "forgot to call processPostEntryTransitions" bug class by moving it inside enterState()

---

## Problem

The same bug appeared **7 times** across 5 releases (8.2.4, 8.5.1, 8.5.2, 8.5.3):

| Release | Missing Location | What Was Forgotten |
|---------|-----------------|-------------------|
| 8.2.4 | `executeChildTransitionBranch` | @always + event queue after child @done |
| 8.5.1 | `processCompoundOnDone` | @always + event queue after compound @done |
| 8.5.1 | `processNestedParallelCompletion` | @always + event queue after nested parallel @done |
| 8.5.1 | `exitParallelStateAndTransitionToTarget` | @always + event queue after parallel escape |
| 8.5.2 | `handleJobInvoke` fire-and-forget | @always + event queue after fire-and-forget target |
| 8.5.2 | `handleAsyncMachineInvoke` fire-and-forget | Same |
| 8.5.2 | `handleFakedMachineInvoke` fire-and-forget | Same |
| 8.5.3 | `exitParallelStateAndTransitionToTarget` null path | @always + event queue in dispatch mode (null eventBehavior) |

**Root cause:** `processPostEntryTransitions()` is a SEPARATE method that each `enterState()` caller must remember to call. Every new code path that enters a state risks forgetting it.

---

## Solution: Move processPostEntryTransitions INTO enterState()

### Why It's Safe (No Double Processing)

`transition()` and `getInitialState()` have their own @always/event queue checks AFTER calling `enterState()`. If `enterState()` also runs them internally, won't they double-fire?

**No.** Because:

1. **@always:** If `processPostEntryTransitions` finds an @always, it calls `$this->transition()` which changes the state. When control returns to the outer `transition()`'s @always check, the state has already changed â€” the @always condition no longer matches.

2. **Event queue:** If `processPostEntryTransitions` shifts an event and calls `$this->transition()`, the queue is empty. When control returns to the outer `transition()`'s queue check, it finds an empty queue â€” no-op.

3. **Both empty:** If neither @always nor event queue applies, `processPostEntryTransitions` is a no-op. The outer check also finds nothing. No change.

### What Changes

**Before:**
```php
// enterState() â€” only does entry protocol
protected function enterState(State $state, StateDefinition $target, ?EventBehavior $eventBehavior, ...): void
{
    // resolve compound, record STATE_ENTER, run entry actions, listeners, handleMachineInvoke, processCompoundOnDone
    // BUT NOT: @always, event queue
}

// Every caller must do:
$this->enterState($state, $target, $eventBehavior);
$this->processPostEntryTransitions($state, $eventBehavior);  // EASY TO FORGET
```

**After:**
```php
// enterState() â€” complete entry protocol INCLUDING post-entry transitions
protected function enterState(State $state, StateDefinition $target, ?EventBehavior $eventBehavior, ...): void
{
    // resolve compound, record STATE_ENTER, run entry actions, listeners, handleMachineInvoke, processCompoundOnDone
    // AND: @always, event queue (processPostEntryTransitions)
}

// Callers just call enterState â€” nothing else needed
$this->enterState($state, $target, $eventBehavior);
// processPostEntryTransitions is handled internally â€” CANNOT be forgotten
```

### Callers to Update

Remove explicit `processPostEntryTransitions()` calls from:

| Caller | Current | After |
|--------|---------|-------|
| `executeChildTransitionBranch` | `enterState()` + `processPostEntryTransitions()` | `enterState()` only |
| `processCompoundOnDone` | `enterState()` + `processPostEntryTransitions()` | `enterState()` only |
| `processNestedParallelCompletion` | `enterState()` + `processPostEntryTransitions()` | `enterState()` only |
| `exitParallelStateAndTransitionToTarget` | `enterState()` + `processPostEntryTransitions()` (conditional) | `enterState()` only |
| `transitionToFireAndForgetTarget` | `enterState()` + `processPostEntryTransitions()` | `enterState()` only |

### Callers That Already Have Their Own @always/Event Queue Handling

These callers will now get `processPostEntryTransitions` inside `enterState()`, but their own post-entry checks become no-ops:

| Caller | Own @always Check | Own Event Queue Check | Impact |
|--------|-------------------|----------------------|--------|
| `transition()` | Lines ~2940-2960 | Lines ~2962-2970 | Both become no-ops (already processed) |
| `getInitialState()` | Lines ~475-490 | Lines ~492-500 | Both become no-ops (already processed) |

**These checks can be left in place as defensive no-ops, or removed.** Leaving them is safer â€” they verify the invariant but never fire.

### Parallel Context Guard

`processCompoundOnDone` currently has a parallel context guard:
```php
if (count($state->value) <= 1) {
    $this->processPostEntryTransitions($state, $eventBehavior);
}
```

This guard must be preserved inside `enterState()` â€” @always transitions in parallel regions are handled by `transitionParallelState`, not by `enterState()`.

## Implementation

### Step 1: Add processPostEntryTransitions to enterState()

```php
protected function enterState(
    State $state,
    StateDefinition $target,
    ?EventBehavior $eventBehavior,
    bool $fireTransitionListeners = false,
): void {
    $resolvedTarget = $target->findInitialStateDefinition() ?? $target;

    // ... existing entry protocol (STATE_ENTER, entry actions, listeners, invoke, onDone) ...

    // Process @always and raised events (only in non-parallel context)
    if (count($state->value) <= 1) {
        $this->processPostEntryTransitions($state, $eventBehavior);
    }
}
```

### Step 2: Remove processPostEntryTransitions calls from all callers

Remove from:
- `executeChildTransitionBranch()`
- `processCompoundOnDone()`
- `processNestedParallelCompletion()`
- `exitParallelStateAndTransitionToTarget()`
- `transitionToFireAndForgetTarget()`

### Step 3: Remove or keep redundant checks in transition()/getInitialState()

**Recommended:** Keep as defensive no-ops with a comment explaining they're now handled by `enterState()`.

### Step 4: Delete processPostEntryTransitions method

Once all callers are updated, the method can be made private or deleted entirely since it's only called from `enterState()`. Actually, keep it as a private method called from `enterState()` â€” separation of concerns.

## Edge Cases

| Case | Handled By |
|------|-----------|
| Non-parallel state entry | `enterState()` â†’ `processPostEntryTransitions()` runs |
| Parallel state entry (enterParallelState) | `enterStateInParallelRegion()` â€” separate method, not affected |
| Parallel @done via sync | `exitParallelStateAndTransitionToTarget()` â†’ `enterState()` â†’ `processPostEntryTransitions()` |
| Parallel @done via dispatch (null eventBehavior) | Same â€” `processPostEntryTransitions` now nullable |
| Fire-and-forget target | `transitionToFireAndForgetTarget()` â†’ `enterState()` â†’ `processPostEntryTransitions()` |
| transition() caller | @always/event queue handled by `enterState()`, outer checks become no-ops |
| getInitialState() caller | Same as transition() |
