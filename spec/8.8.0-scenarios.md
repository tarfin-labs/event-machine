## 1. Problem

Complex machines like `CarSalesMachine` have deep state hierarchies with parallel child delegations, API integrations, and multi-step guard chains. To test a new feature that affects behavior at state `checking_protocol`, a human must:

1. Trigger `START_APPLICATION` with valid retailer/farmer models
2. Grant consent → pass eligibility check
3. Wait for `FindeksMachine` to complete (6+ steps, external API calls)
4. Wait for `TurmobMachine` to complete (external API call)
5. Both parallel regions must finish → guard check → exit to `checking_protocol`

This requires 10+ manual steps, 2 child machines, multiple API calls, and deep domain knowledge. Product teams and QA cannot do this in staging without developer assistance.

**Scenarios solve this:** define behavior overrides and child machine outcomes once, activate from existing frontend endpoints via a `scenario` field, arrive at the desired state with a fully functional machine.

## 2. Design Principles

| Principle | Rationale |
|-----------|-----------|
| **Behavior is real** | Machine transitions, event history, context mutations, entry/exit actions — all real. The resulting machine is indistinguishable from one that arrived at that state organically. |
| **Override, don't fake** | External dependencies return predetermined responses via `arrange()`. The behavior still runs in the engine — it just gets scenario-defined data instead of calling real services. |
| **State-keyed overrides** | `arrange()` is keyed by full state route — you see WHERE each override applies. Same behavior can return different values in different states. |
| **Child outcomes, not internals** | Parent scenarios declare child machine outcomes (`@done.report_saved`), not child internal guards/actions. Child machines are black boxes from the parent's perspective. |
| **Identity = Machine × State × Event** | A scenario is defined by which machine, at which state (full route), triggered by which event. This triple determines where the scenario appears in `available_scenarios`. |
| **Endpoint-native** | Scenarios are activated through existing machine endpoints via a `scenario` field in the request body. |
| **Environment-gated** | Disabled by default (`MACHINE_SCENARIOS_ENABLED=true`). Zero overhead in production. |

## 3. Scenario Identity — Machine × State × Event

A scenario is positioned by three coordinates:

| Coordinate | Property | Example | Purpose |
|-----------|----------|---------|---------|
| **Machine** | `$machine` | `CarSalesMachine::class` | Which machine this scenario targets |
| **State** | `$state` | `'awaiting_customer_start'` | Full state route — where the machine is BEFORE the event |
| **Event** | `$event` | `CustomerStartedEvent::class` | Which event triggers this scenario (class or string) |

This triple determines:
- **Discovery** — which scenarios are available at a given state for a given event
- **Validation** — the machine must be at `$state` when the scenario activates, and the event being sent must match `$event`
- **`available_scenarios`** — grouped by event in endpoint responses

### Full State Route Requirement

`$state` and `arrange()` keys must use the **full route** (dot-notation path), not just the leaf key:

```php
// ❌ Ambiguous — could exist in multiple parallel regions
protected string $state = 'awaiting_vehicle_and_pricing';

// ✅ Unambiguous — full path from machine root
protected string $state = 'data_collection.retailer.awaiting_vehicle_and_pricing';
```

This matches how `$state->value` returns state paths in endpoint responses. For root-level states (e.g., `routing`, `eligibility_check`), the leaf key IS the full route.

## 4. MachineScenario Class

```php
abstract class MachineScenario
{
    /** Which machine this scenario targets (class-string). */
    protected string $machine;

    /** Full state route — the state the machine must be in BEFORE the event. */
    protected string $state;

    /** The event that triggers this scenario (class-string or event type string). */
    protected string $event;

    /** Human-readable description shown in artisan --list and endpoint responses. */
    protected string $description;

    /**
     * State-keyed behavior overrides and child machine outcomes.
     *
     * Keys: full state routes (string) or child machine classes (class-string<Machine>)
     * Values: behavior overrides or outcome declarations
     */
    protected function arrange(): array { return []; }
}
```

**4 properties** (`$machine`, `$state`, `$event`, `$description`).
**1 optional method** (`arrange()`).

Base class validates all properties are set at instantiation.

### Example: Minimal Scenario (no arrange)

```php
class AtPreventedScenario extends MachineScenario
{
    protected string $machine     = CarSalesMachine::class;
    protected string $state       = 'awaiting_customer_start';
    protected string $event       = CustomerStartedEvent::class;
    protected string $description = 'Farmer prevented — not eligible';
}
```

### Example: Scenario with arrange

```php
class AtVerificationScenario extends MachineScenario
{
    protected string $machine     = CarSalesMachine::class;
    protected string $state       = 'awaiting_customer_start';
    protected string $event       = CustomerStartedEvent::class;
    protected string $description = 'At verification — farmer eligible';

    protected function arrange(): array
    {
        return [
            'eligibility_check' => [
                IsFarmerNotEligibleGuard::class => false,
            ],
        ];
    }
}
```

### Naming Convention

| Type | Base class | Concrete example | Suffix |
|------|-----------|-----------------|--------|
| Guard scenario | `GuardScenarioBehavior` | `HasConsentGuardScenario` | `{OriginalName}Scenario` |
| Action scenario | `ActionScenarioBehavior` | `StoreApplicationActionScenario` | `{OriginalName}Scenario` |
| Machine scenario | `MachineScenario` | `AtVerificationScenario` | Descriptive + `Scenario` suffix |

**Guard/Action scenario names** mirror the original behavior name with `Scenario` suffix. **Search advantage:** `HasConsent` finds both `HasConsentGuard` and `HasConsentGuardScenario` side by side.

**Machine scenario names** are descriptive — typically the target state. When multiple scenarios target the same state via different paths, disambiguate:
- `AtVerificationScenario` — unique target, sufficient
- `AtVerificationViaConsentScenario` — disambiguated: same target, different state/event
- ❌ `AtVerificationCarSalesMachineScenario` — machine name is redundant

## 5. arrange() — State-Keyed Behavior Scenarios

`arrange()` keys must be **full state routes** — same requirement as `$state` (see [§3](#3-scenario-identity--machine--state--event)).

```php
protected function arrange(): array
{
    return [
        // Root-level state — leaf key = full route
        'eligibility_check' => [
            IsFarmerNotEligibleGuard::class => false,
        ],
        // Compound state — full route required
        'data_collection.retailer.awaiting_vehicle_and_pricing' => [
            IsVehicleVinMismatchGuard::class => false,
        ],
    ];
}
```

Reading this, you immediately understand WHERE each override applies.

### Three Value Forms — Mirrors Behavior Definition Pattern

**Guards:**

```php
'eligibility_check' => [
    // 1. Bool shorthand
    IsFarmerNotEligibleGuard::class => false,

    // 2. Closure — inline with DI
    HasConsentGuard::class => function (ContextManager $ctx): bool {
        return $ctx->get('farmer')->hasValidConsent();
    },

    // 3. Class — reusable scenario behavior
    IsCustomerInfoCompleteGuard::class => IsCustomerInfoCompleteGuardScenario::class,
],
```

**Actions:**

```php
'data_collection.retailer.calculating_prices' => [
    // 1. Array shorthand — key-value pairs written to context
    CheckProtocolAction::class => ['protocolEligible' => true],

    // 2. Closure — inline with DI
    StoreApplicationAction::class => function (ContextManager $ctx) {
        $ctx->set('applicationId', 'APP-'.Str::random());
    },

    // 3. Class — reusable scenario behavior
    CalculatePricesAction::class => CalculatePricesActionScenario::class,
],
```

### Scenario Behavior Base Classes

Class-based overrides extend `GuardScenarioBehavior` or `ActionScenarioBehavior`:

```php
abstract class GuardScenarioBehavior extends GuardBehavior {}
abstract class ActionScenarioBehavior extends ActionBehavior {}
```

DI is free (inherited from `InvokableBehavior`), type-compatible (can replace the original in the container).

```php
class CalculatePricesActionScenario extends ActionScenarioBehavior
{
    public function __invoke(ContextManager $ctx): void
    {
        $ctx->set('totalCashPrice', Money::of(150000, 'TRY'));
        $ctx->set('monthlyInstallment', Money::of(12500, 'TRY'));
    }
}
```

### Same Behavior, Different Values Per State

When the same behavior appears under multiple states with **different values**, ScenarioPlayer creates a **state-aware override** that checks `$state->currentStateDefinition->route` at invocation time:

```php
'routing' => [
    HasConsentGuard::class => true,
],
'data_collection.customer_info.info_checking' => [
    HasConsentGuard::class => false,
],
```

| Situation | What ScenarioPlayer does |
|-----------|-------------------------|
| Behavior in one state only | Simple bind (state key is documentary) |
| Same value across states | Simple bind (state key is documentary) |
| Different values across states | State-aware bind (checks `->route`) |
| Behavior runs in un-overridden state | Original behavior executes |

## 6. Child Machine Outcomes in arrange()

Parent scenarios declare child machine **outcomes**, not internal overrides. From the parent's perspective, child machines are black boxes — the parent only knows about `@done`, `@done.{finalState}`, `@fail`, and `@timeout`.

### Declaring Outcomes

Child machine outcomes use the child machine class as key in `arrange()`:

```php
protected function arrange(): array
{
    return [
        // Parent state overrides (string key → state route)
        'eligibility_check' => [
            IsFarmerNotEligibleGuard::class => false,
        ],

        // Child machine outcomes (Machine class key → outcome)
        FindeksMachine::class => '@done.report_saved',
        TurmobMachine::class => '@done',
    ];
}
```

### Outcome with Output Data

When the parent's `@done` action needs data from the child (accessed via `$event->output()`):

```php
FindeksMachine::class => [
    'outcome' => '@done.report_saved',
    'output'  => ['reportId' => 'RPT-001', 'score' => 1400],
],
```

### Outcome Formats

| Format | Example | When to use |
|--------|---------|-------------|
| Simple string | `'@done.report_saved'` | Parent only cares about final state routing |
| With output | `['outcome' => '@done.report_saved', 'output' => [...]]` | Parent's `@done` action reads child output |
| Failure | `'@fail'` | Test failure path |
| Timeout | `'@timeout'` | Test timeout path |

### How It Works

ScenarioPlayer detects Machine class keys in `arrange()`:

1. Intercepts child machine dispatch
2. Does NOT run the child machine
3. Simulates the child completion by sending the declared outcome to the parent
4. Parent's `@done`/`@fail`/`@timeout` transition fires with the declared output

### Child Machine's Own Scenarios

To test a child machine's internal flow, define scenarios on the child machine itself:

```php
class AtReportSavedScenario extends MachineScenario
{
    protected string $machine = FindeksMachine::class;
    protected string $state   = 'idle';
    protected string $event   = CheckExistingReportEvent::class;
    // ...
}
```

These are used when QA interacts with FindeksMachine directly via its own endpoints — completely independent from parent scenarios.

## 7. Parameterized Behaviors in Scenarios

EventMachine supports parameterized behaviors via colon syntax: `'guards' => 'hasRoleGuard:admin'`. Closures and class-based scenario behaviors receive these parameters via DI:

```php
// Machine: 'guards' => ['hasRoleGuard:admin', 'hasRoleGuard:editor']
'state_a' => [
    HasRoleGuard::class => function (ContextManager $ctx, string $role): bool {
        return match ($role) {
            'admin'  => true,
            'editor' => false,
        };
    },
],
```

| Override form | Parameterized behavior support |
|---------------|-------------------------------|
| `bool` | Returns same value for ALL calls regardless of parameters |
| `array` | Writes same context for ALL calls regardless of parameters |
| `Closure` | Receives parameters via DI — can differentiate |
| `class-string` | Receives parameters via `__invoke` — can differentiate |

## 8. Endpoint Integration

### QA Workflow

1. QA opens frontend, sees scenario selector (staging only)
2. Endpoint response includes `available_events` and `available_scenarios` (grouped by event)
3. QA selects a scenario from the dropdown for a specific event
4. QA triggers the event — frontend sends `{ type: "...", scenario: "..." }`
5. Machine processes event with scenario overrides active
6. Endpoint returns final state — QA continues manually

### `available_scenarios` — Grouped by Event

After any successful event transition, the response includes `available_scenarios` keyed by event type. Only events that have scenarios are included:

```json
{
    "data": {
        "machine_id": "evt_01HXYZ...",
        "value": ["car_sales.checking_protocol"],
        "context": { "..." },
        "available_events": [
            { "type": "ProtocolPassedEvent", "source": "parent" },
            { "type": "ProtocolRejectedEvent", "source": "parent" },
            { "type": "ExpiredEvent", "source": "parent" }
        ],
        "available_scenarios": {
            "ProtocolPassedEvent": [
                { "slug": "at-approved-scenario", "description": "Fast-forward to approved" },
                { "slug": "at-data-collection-scenario", "description": "Arrive at data collection" }
            ],
            "ProtocolRejectedEvent": [
                { "slug": "at-rejected-scenario", "description": "Rejection with specific reason" }
            ]
        }
    }
}
```

### Example: CarSalesMachine Scenario

```php
class AtCheckingProtocolScenario extends MachineScenario
{
    protected string $machine     = CarSalesMachine::class;
    protected string $state       = 'awaiting_customer_start';
    protected string $event       = CustomerStartedEvent::class;
    protected string $description = 'At checking protocol — children completed';

    protected function arrange(): array
    {
        return [
            'eligibility_check' => [
                IsFarmerNotEligibleGuard::class => false,
            ],
            FindeksMachine::class => [
                'outcome' => '@done.report_saved',
                'output'  => ['reportId' => 'RPT-001', 'score' => 1400],
            ],
            TurmobMachine::class => '@done',
        ];
    }
}
```

**Usage:**

```http
POST /api/car-sales/{applicationId}/customer-started
{
    "type": "CustomerStartedEvent",
    "scenario": "at-checking-protocol-scenario"
}
```

**What happens:**
1. Controller detects `scenario`, validates `$state` = current state, `$event` = event being sent
2. Registers behavior overrides, prepares child outcome simulations
3. `$machine->send(CustomerStartedEvent)` — transitions to `eligibility_check`
4. `IsFarmerNotEligibleGuard` → scenario returns `false` → `verification` (parallel)
5. Child machines intercepted → `FindeksMachine` simulated as `@done.report_saved`, `TurmobMachine` as `@done`
6. Both children done → `@done` guard passes → `checking_protocol` ✅
7. Cleanup, response with final state + `available_scenarios`

### Endpoint Flow

Overrides are registered BEFORE `$machine->send()`:

```
MachineController::executeEndpoint()
│
├─ 1. maybeRegisterScenarioOverrides(Request, machineClass)
│     ├─ Read 'scenario' slug from request
│     ├─ If null or disabled → skip
│     ├─ Resolve scenario class (via ScenarioDiscovery)
│     ├─ Validate $state matches current state
│     ├─ Validate $event matches event being sent
│     ├─ Register state-keyed overrides in container
│     ├─ Prepare child outcome simulations
│     └─ Return scenario instance
│
├─ 2. $state = $machine->send(event)
│     ├─ @always guards use scenario behaviors ✅
│     └─ Child machines intercepted → outcomes simulated ✅
│
├─ 3. finally: cleanup — unbind scenario behaviors
│
└─ 4. buildResponse() — final state, available_events, available_scenarios
```

**Synchronous guarantee:** The endpoint does NOT return until the scenario fully completes.

### Per-Machine Scenario Routes

Registered by `MachineRouter::register()` when enabled and machine has scenarios:

```
GET    {prefix}/scenarios                    → List this machine's scenarios
GET    {prefix}/scenarios/{slug}/describe    → Scenario details
```

## 9. ScenarioPlayer — Runtime Engine

```
execute(scenario, eventPayload, machine)
│
├─ 1. Validate environment (scenarios enabled?)
├─ 2. Parse arrange()
│     ├─ Separate state-keyed overrides (string keys)
│     └─ Separate child outcomes (Machine class keys)
├─ 3. Register state-keyed overrides in container (state-aware bindings)
├─ 4. Prepare child outcome simulations
│     └─ Intercept child dispatch, simulate @done/@fail/@timeout
│
├─ 5. Send trigger event
│     └─ $event with payload from endpoint request or artisan
│
├─ 6. Build ScenarioResult
│
└─ 7. Cleanup (finally)
      └─ Unbind overrides from container
```

## 10. Environment Gating

**Config in `config/machine.php`:**

```php
'scenarios' => [
    'enabled' => env('MACHINE_SCENARIOS_ENABLED', false),
],
```

| Point | Behavior when disabled |
|-------|----------------------|
| `ScenarioPlayer` | Throws `ScenariosDisabledException` |
| `machine:scenario` artisan | Prints error, exits with code 1 |
| `MachineRouter::register()` | Scenario routes not registered |
| `MachineController::buildResponse()` | `available_scenarios` omitted |
| `MachineController::executeEndpoint()` | `scenario` field silently ignored |

## 11. Artisan Command

```bash
# List all scenarios
php artisan machine:scenario --list

# List scenarios for a specific machine
php artisan machine:scenario --list --machine=CarSalesMachine

# Play a scenario on an existing machine
php artisan machine:scenario AtCheckingProtocolScenario --machine-id=evt_01HXYZ...

# Play with event payload (JSON)
php artisan machine:scenario AtCheckingProtocolScenario --machine-id=evt_01HXYZ... --payload='{"farmerId":42}'
```

The artisan command performs the same flow as the endpoint: register overrides → send `$event` with payload → cleanup.

## 12. File Organization & Discovery

### Machine structure mirroring

Each machine's scenarios live under its own `Scenarios/` directory:

**Simple scenarios** — single file:

```
app/Machines/CarSales/
├── CarSalesMachine.php
├── Guards/
│   ├── HasConsentGuard.php
│   └── IsFarmerNotEligibleGuard.php
├── Actions/
│   └── StoreApplicationAction.php
│
└── Scenarios/
    └── AtVerificationScenario.php
```

**Scenarios with class-based scenario behaviors** — own directory, mirroring Guards/Actions:

```
app/Machines/CarSales/
└── Scenarios/
    ├── AtVerificationScenario.php
    │
    └── AtCheckingProtocolScenario/
        ├── AtCheckingProtocolScenario.php
        └── Guards/
            └── IsFarmerNotEligibleGuardScenario.php
```

### Discovery

`ScenarioDiscovery` resolves each machine's `Scenarios/` directory from the machine class file location:

```php
$machineFile = (new ReflectionClass($machineClass))->getFileName();
$scenarioDir = dirname($machineFile).'/Scenarios';
```

Machine-local discovery. `machine:scenario-cache` builds a combined scenario class map from all discovered machines.

## 13. Error Handling & requiredContext Hints

### Exceptions

| Exception | When |
|-----------|------|
| `ScenariosDisabledException` | `MACHINE_SCENARIOS_ENABLED=false` |
| `ScenarioFailedException` | Guard rejection, invalid event, `$state` mismatch, `$event` mismatch. Includes `eventType`, `currentState`, `rejectionReason`. |
| `ScenarioConfigurationException` | Invalid child outcome format. Unknown child machine class in `arrange()`. Missing required properties. |

### requiredContext Hints

When a `MissingMachineContextException` is thrown during scenario replay, the error is enriched with `requiredContext` hints:

```
`farmer` is missing in context.

Required context for behaviors at this state:
  → IsFarmerNotEligibleGuard: farmer (App\Models\Farmer), retailer (App\Models\Retailer)
```
