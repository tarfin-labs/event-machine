## 1. Problem

Complex machines like `CarSalesMachine` have deep state hierarchies with parallel child delegations, API integrations, and multi-step guard chains. To test a new feature that affects behavior at state `checking_protocol`, a human must:

1. Trigger `START_APPLICATION` with valid retailer/farmer models
2. Grant consent â†’ pass eligibility check
3. Wait for `FindeksMachine` to complete (6+ steps, external API calls)
4. Wait for `TurmobMachine` to complete (external API call)
5. Both parallel regions must finish â†’ guard check â†’ exit to `checking_protocol`

This requires 10+ manual steps, 2 child machines, multiple API calls, and deep domain knowledge. Product teams and QA cannot do this in staging without developer assistance.

**Scenarios solve this:** define behavior overrides and optional event steps once, activate from the existing frontend endpoints via a single `scenario` field, arrive at the desired state with a fully functional machine.

## 2. Design Principles

| Principle | Rationale |
|-----------|-----------|
| **Behavior is real** | Machine transitions, event history, context mutations, entry/exit actions â€” all real. The resulting machine is indistinguishable from one that arrived at that state organically. |
| **Override, don't fake** | External dependencies return predetermined responses via `arrange()`. The behavior still runs in the engine â€” it just gets scenario-defined data instead of calling real services. |
| **State-keyed overrides** | `arrange()` is keyed by full state route â€” you see WHERE each override applies. Same behavior can return different values in different states. |
| **Identity = Machine Ă— State Ă— Event** | A scenario is defined by which machine, at which state (full route), triggered by which event. This triple determines where the scenario appears in `available_scenarios`. |
| **Endpoint-native** | Scenarios are activated through existing machine endpoints via a `scenario` field in the request body. No separate scenario endpoints needed for QA usage. |
| **Composable** | Scenarios can extend other scenarios via `parent()`. |
| **Child-aware** | Parent scenarios can trigger child machine scenarios via `$this->child()`. |
| **Environment-gated** | Disabled by default (`MACHINE_SCENARIOS_ENABLED=true`). Zero overhead in production. |

## 3. Scenario Identity â€” Machine Ă— State Ă— Event

A scenario is positioned by three coordinates:

| Coordinate | Method | Example | Purpose |
|-----------|--------|---------|---------|
| **Machine** | `machine()` | `CarSalesMachine::class` | Which machine this scenario targets |
| **State** | `from()` | `'awaiting_customer_start'` | Full state route â€” where the machine is BEFORE the event |
| **Event** | `event()` | `'CustomerStartedEvent'` | Which event triggers this scenario |

This triple determines:
- **Discovery** â€” which scenarios are available at a given state for a given event
- **Validation** â€” the machine must be at `from()` state when the scenario activates
- **`available_scenarios`** â€” grouped by event in endpoint responses

### Full State Route Requirement

States must use the **full route** (dot-notation path), not just the leaf key:

```php
// âťŚ Ambiguous â€” 'awaiting_vehicle_and_pricing' could exist in multiple regions
protected function from(): string { return 'awaiting_vehicle_and_pricing'; }

// âś… Unambiguous â€” full path from machine root
protected function from(): string { return 'data_collection.retailer.awaiting_vehicle_and_pricing'; }
```

This matches how `$state->value` returns state paths in endpoint responses (e.g., `["car_sales.data_collection.retailer.awaiting_vehicle_and_pricing"]`).

### No Separate play()/playOn()

Since every scenario is tied to a specific state + event, execution is always the same:

1. Register `arrange()` overrides in container
2. Send `event()` with payload (from endpoint request or artisan)
3. Play `steps()` if any
4. Cleanup overrides

Whether the machine is new or existing doesn't matter â€” the scenario is activated through the normal event flow. No separate `play()`/`playOn()` API needed.

## 5. arrange() â€” State-Keyed Behavior Scenarios

`arrange()` is keyed by **full state route**. This serves two purposes:
1. **Readability** â€” you see WHERE in the machine each override applies
2. **Functionality** â€” same behavior can return different values in different states

```php
protected function arrange(): array
{
    return [
        'routing' => [
            HasConsentGuard::class => true,
        ],
        'eligibility_check' => [
            IsFarmerNotEligibleGuard::class => false,
            StoreApplicationAction::class => ['applicationId' => 'APP-SCENARIO'],
        ],
    ];
}
```

Reading this, you immediately understand the **story**: at `routing`, consent is true; at `eligibility_check`, farmer is eligible.

### Three Value Forms â€” Mirrors Behavior Definition Pattern

**Guards:**

```php
'eligibility_check' => [
    // 1. Bool shorthand â€” simplest
    IsFarmerNotEligibleGuard::class => false,

    // 2. Closure â€” inline with DI
    HasConsentGuard::class => function (ContextManager $ctx): bool {
        return $ctx->get('farmer')->hasValidConsent();
    },

    // 3. Class â€” reusable scenario behavior
    IsCustomerInfoCompleteGuard::class => IsCustomerInfoCompleteGuardScenario::class,
],
```

**Actions:**

```php
'checking_protocol' => [
    // 1. Array shorthand â€” key-value pairs written to context
    CheckProtocolAction::class => ['protocolEligible' => true],

    // 2. Closure â€” inline with DI
    StoreApplicationAction::class => function (ContextManager $ctx) {
        $ctx->set('applicationId', 'APP-'.Str::random());
    },

    // 3. Class â€” reusable scenario behavior
    CalculatePricesAction::class => CalculatePricesActionScenario::class,
],
```

### Same Behavior, Different Values Per State

When the same behavior appears under multiple states with **different values**, ScenarioPlayer creates a **state-aware override** automatically:

```php
protected function arrange(): array
{
    return [
        'routing' => [
            HasConsentGuard::class => true,
        ],
        'data_collection' => [
            HasConsentGuard::class => false,
        ],
    ];
}
```

ScenarioPlayer detects the difference and creates a state-aware container binding that checks `$state->currentStateDefinition->key` at invocation time:

| Situation | What ScenarioPlayer does |
|-----------|-------------------------|
| Behavior in one state only | Simple bind (state key is documentary) |
| Same value across states | Simple bind (state key is documentary) |
| Different values across states | State-aware bind (state key is functional) |
| Behavior runs in un-overridden state | Original behavior executes |

## 6. Scenario Behavior Classes

### Base Classes


Extending `GuardBehavior` / `ActionBehavior` means:

## 7. Parameterized Behaviors in Scenarios

EventMachine supports parameterized behaviors via colon syntax: `'guards' => 'hasRoleGuard:admin'`. The same behavior class can be called with different parameters.

Closures and class-based scenario behaviors receive these parameters via DI:

```php
// Machine: 'guards' => ['hasRoleGuard:admin', 'hasRoleGuard:editor']
// Scenario override â€” closure receives the parameter:
'state_a' => [
    HasRoleGuard::class => function (ContextManager $ctx, string $role): bool {
        return match ($role) {
            'admin'  => true,
            'editor' => false,
        };
    },
],
```

| Override form | Parameterized behavior support |
|---------------|-------------------------------|
| `bool` | Returns same value for ALL calls regardless of parameters |
| `array` | Writes same context for ALL calls regardless of parameters |
| `Closure` | Receives parameters via DI â€” can differentiate |
| `class-string` | Receives parameters via `__invoke` â€” can differentiate |

## 8. Endpoint Integration

### QA Workflow

1. QA opens frontend, sees scenario selector (staging only)
2. Endpoint response includes `available_events` and `available_scenarios` (grouped by event)
3. QA selects a scenario from the dropdown for a specific event
4. QA triggers the event â€” frontend sends `{ type: "...", scenario: "..." }`
5. Machine processes event with scenario overrides active
6. Endpoint returns final state â€” QA continues manually

### `available_scenarios` in Response â€” Grouped by Event

After any successful event transition, the response includes `available_scenarios` keyed by event type:

### Endpoint Flow â€” Override Timing

Scenario overrides must be registered BEFORE `$machine->send()` so `@always` guards use them:

```
MachineController::executeEndpoint()
â”‚
â”śâ”€ 1. maybeRegisterScenarioOverrides(Request, machineClass)
â”‚     â”śâ”€ Read 'scenario' slug from request body
â”‚     â”śâ”€ If null or MACHINE_SCENARIOS_ENABLED=false â†’ return null
â”‚     â”śâ”€ Resolve scenario class, validate from() matches current state
â”‚     â”śâ”€ Validate event() matches the event being sent
â”‚     â”śâ”€ Instantiate, register arrange() in container
â”‚     â””â”€ Return scenario instance
â”‚
â”śâ”€ 2. $state = $machine->send(event)
â”‚     â””â”€ @always guards now use scenario behaviors âś…
â”‚
â”śâ”€ 3. maybePlayScenarioSteps($scenario, $machine, $state)
â”‚     â”śâ”€ If scenario->getSteps() is empty â†’ skip
â”‚     â””â”€ For each step: $machine->send(step)
â”‚
â”śâ”€ 4. finally: cleanup â€” unbind scenario behaviors from container
â”‚
â””â”€ 5. buildResponse() â€” final state, available_events, available_scenarios
```

**Synchronous guarantee:** The endpoint does NOT return until the scenario fully completes.

### Per-Machine Scenario Routes

Registered by `MachineRouter::register()` when `MACHINE_SCENARIOS_ENABLED=true` and machine has scenarios:

```
GET    {prefix}/scenarios                    â†’ List this machine's scenarios
GET    {prefix}/scenarios/{slug}/describe    â†’ Scenario details
```

No play/playOn routes â€” scenarios are triggered through normal event endpoints via the `scenario` field.

## 9. Composition â€” Extending Scenarios

Scenarios can build on top of each other via `parent()`. The child inherits the parent's `from()` and `event()` â€” execution starts from the parent's trigger point.

```php
class AtDataCollectionScenario extends MachineScenario
{
    protected function machine(): string { return CarSalesMachine::class; }
    protected function description(): string { return 'At data collection'; }

    // Inherits from() and event() from parent
    protected function from(): string { return 'awaiting_customer_start'; }
    protected function event(): string { return 'CustomerStartedEvent'; }

    protected function parent(): string
    {
        return AtVerificationScenario::class;
    }

    protected function arrange(): array
    {
        return [
            'checking_protocol' => [
                CheckProtocolAction::class => ['protocolEligible' => true],
            ],
        ];
    }

    protected function steps(): array
    {
        return [
            $this->send('ProtocolUndecidedEvent'),
        ];
    }
}
```

**Execution order:**
1. Register BOTH parent + child `arrange()` overrides (child overrides parent for same state+behavior)
2. Send trigger event (from parent's `event()`)
3. Play parent's `steps()`
4. Play child's `steps()`

**Merge rules:**

| Aspect | Behavior |
|--------|----------|
| `machine()` | Must match across chain |
| `from()` + `event()` | Must match across chain (same trigger point) |
| `arrange()` | Deep merged â€” child state keys override parent for same behavior class |
| `steps()` | Sequential â€” parent steps first, then child steps |

## 10. Child Machine Scenarios

When a parent machine delegates to a child machine, use `$this->child()` in `steps()`:

```php
protected function steps(): array
{
    return [
        $this->child(FindeksMachine::class)
            ->scenario(FindeksCompletedScenario::class),
        $this->child(TurmobMachine::class)
            ->scenario(TurmobCompletedScenario::class),
    ];
}
```

**How it works:**
1. `Bus::fake([ChildMachineJob::class])` intercepts async child dispatch
2. ScenarioPlayer finds child machine in `machine_children` table
3. If `child_root_event_id` is null (Bus::fake intercepted), creates child machine and updates record
4. Replays child scenario against the child machine instance
5. Child reaches final state â†’ parent's `@done` transition fires naturally

## 11. Async Dispatch & Timers

### Child Machine Dispatch

`Bus::fake([ChildMachineJob::class, ChildJobJob::class])` before replay starts. Only these specific jobs are intercepted â€” all other jobs dispatch normally.

### Timer Handling

Timers are lazy in EventMachine â€” `MachineTimerFire` records are created by `ProcessTimersCommand` during scheduled sweeps, not during state entry. Since scenario replay is synchronous (milliseconds), the sweep command doesn't run during replay. Intermediate state timers never fire.

## 17. Error Handling & requiredContext Hints

### Exceptions

| Exception | When |
|-----------|------|
| `ScenariosDisabledException` | `MACHINE_SCENARIOS_ENABLED=false` |
| `ScenarioFailedException` | Step fails (guard rejection, invalid event, `from()` mismatch, `event()` mismatch). Includes `stepIndex`, `eventType`, `currentState`, `rejectionReason`. |
| `ScenarioConfigurationException` | `machine()` mismatch in parent chain. `from()`/`event()` mismatch in parent chain. |

### requiredContext Hints

When a `MissingMachineContextException` is thrown during scenario replay, the error message is enriched with `requiredContext` hints:

```

  â†’ IsFarmerNotEligibleGuard: farmer (App\Models\Farmer), retailer (App\Models\Retailer)
```

The old scenario system (`scenarios_enabled`, `$scenarios` param, `getScenarioStateIfAvailable`, `EventBehavior::getScenario`, `TestMachine::withScenario`) is preserved with `@deprecated` markers. Will be removed in v9.

**Coexistence â€” no conflicts:**

| Aspect | Old System | New System |
|--------|-----------|------------|
| Config | `scenarios_enabled` (machine config) | `machine.scenarios.enabled` (Laravel config) |
| Payload | `scenarioType` (event payload) | `scenario` (HTTP request body) |
| Mechanism | State definition swap (`idMap`) | Container binding + event replay |

## 19. What NOT to Include

| Feature | Why excluded |
|---------|-------------|
| `models()` / model factories | Staging QA has models in DB. Parameters come from frontend event payload. |
| `defaults()` / `param()` | Parameters come from event payload via frontend. |
| `ScenarioStubContract` / `applyStub()` | Override logic belongs in `arrange()`, not on the action class. |
| `play()` / `playOn()` static methods | Execution always through ScenarioPlayer â€” endpoint or artisan triggers event with overrides. |
| Automatic rollback / cleanup | Scenarios create persistent state intentionally. |
| Parallel step execution | Steps must be sequential. |
| Conditional steps | Create different scenarios instead. |
| UI / dashboard | Artisan + endpoint responses are sufficient. |
| Global `/machine/scenarios/` routes | Each machine owns its routes under its prefix. |
| Separate scenario play endpoint | Scenarios are activated through normal event endpoints via `scenario` field. |
