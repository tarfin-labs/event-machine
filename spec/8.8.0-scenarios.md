## 1. Problem

Complex machines like `CarSalesMachine` have deep state hierarchies with parallel child delegations, API integrations, and multi-step guard chains. To test a new feature that affects behavior at state `checking_protocol`, a human must:

1. Trigger `START_APPLICATION` with valid retailer/farmer models
2. Grant consent → pass eligibility check
3. Wait for `FindeksMachine` to complete (6+ steps, external API calls)
4. Wait for `TurmobMachine` to complete (external API call)
5. Both parallel regions must finish → guard check → exit to `checking_protocol`

This requires 10+ manual steps, 2 child machines, multiple API calls, and deep domain knowledge. Product teams and QA cannot do this in staging without developer assistance.

**Scenarios solve this:** define behavior overrides and delegation outcomes once, activate from existing frontend endpoints via a `scenario` field, arrive at the desired state with a fully functional machine.

## 2. Design Principles

| Principle | Rationale |
|-----------|-----------|
| **Behavior is real** | Machine transitions, event history, context mutations, entry/exit actions — all real. The resulting machine is indistinguishable from one that arrived at that state organically. |
| **Override, don't fake** | External dependencies return predetermined responses via `plan()`. The behavior still runs in the engine — it just gets scenario-defined data instead of calling real services. |
| **State-route-keyed plan** | Every `plan()` key is a full state route — you see WHERE each override or delegation outcome applies. No mixed key types. |
| **Child delegation as outcomes or scenarios** | Parent declares child outcomes (`@done.report_saved`) for fast-forward, or references a child scenario for interactive states (e.g., OTP entry). Child internals are never directly overridden from the parent. |
| **Identity = Machine × State × Event × Slug** | A scenario is defined by which machine, at which state (full route), triggered by which event, identified by slug. Multiple scenarios can share the same trigger point with different plans. |
| **Endpoint-native** | Scenarios are activated through existing machine endpoints via a `scenario` field in the request body. |
| **Async-safe** | Scenario info is persisted to the database, so async queue jobs (parallel regions, listeners, child completion) restore overrides automatically. |
| **Environment-gated** | Disabled by default (`MACHINE_SCENARIOS_ENABLED=true`). Zero overhead in production. |

## 3. Scenario Identity — Machine × State × Event × Slug

A scenario is positioned by four coordinates:

| Coordinate | Property | Example | Purpose |
|-----------|----------|---------|---------|
| **Machine** | `$machine` | `CarSalesMachine::class` | Which machine this scenario targets |
| **State** | `$state` | `'awaiting_customer_start'` | Full state route — where the machine is BEFORE the event |
| **Event** | `$event` | `CustomerStartedEvent::class` | Which event triggers this scenario (class or string) |
| **Slug** | (from class name) | `'at-verification-scenario'` | Unique identifier — multiple scenarios can share the same state+event |

The first three determine **where** the scenario can be activated. The slug determines **which** scenario is selected. Multiple scenarios can share the same state+event (e.g., `AtVerificationScenario` and `AtCheckingProtocolScenario` both activate from `awaiting_customer_start` + `CustomerStartedEvent` — different plans, same trigger point).

This quadruple determines:
- **Discovery** — which scenarios are available at a given state for a given event
- **Validation** — the machine must be at `$state` when the scenario activates, and the event being sent must match `$event`
- **`available_scenarios`** — grouped by event in endpoint responses

A machine can be running normally without any scenario and then at **any state**, the user can activate a scenario — as long as a scenario exists for that state + event combination.

### Full State Route Requirement

`$state` and `plan()` keys must use the **full route** (dot-notation path), not just the leaf key:

```php
// ❌ Ambiguous — could exist in multiple parallel regions
protected string $state = 'awaiting_vehicle_and_pricing';

// ✅ Unambiguous — full path from machine root
protected string $state = 'data_collection.retailer.awaiting_vehicle_and_pricing';
```

This matches how `$state->value` returns state paths in endpoint responses. For root-level states (e.g., `routing`, `eligibility_check`), the leaf key IS the full route.

## 4. MachineScenario Class

```php
abstract class MachineScenario
{
    /** Which machine this scenario targets (class-string). */
    protected string $machine;

    /** Full state route — the state the machine must be in BEFORE the event. */
    protected string $state;

    /** The event that triggers this scenario (class-string or event type string). */
    protected string $event;

    /** Human-readable description shown in endpoint responses. */
    protected string $description;

    /**
     * The scenario's plan — every key is a full state route.
     * Value type determines meaning: array → behavior overrides,
     * string starting with @ → delegation outcome,
     * MachineScenario class → child scenario reference.
     */
    protected function plan(): array { return []; }

    /**
     * Scenario parameter validation rules — same pattern as EventBehavior.
     * Laravel validation rules. Validated before plan() is called.
     */
    protected function params(): array { return []; }

    /**
     * Access a validated scenario parameter.
     * Values come from frontend's scenarioParams, validated against params() rules.
     */
    protected function param(string $key, mixed $default = null): mixed
    {
        return $this->resolvedParams[$key] ?? $default;
    }
}
```

**4 properties** (`$machine`, `$state`, `$event`, `$description`) — set as simple property assignments.
**2 optional methods** (`plan()`, `params()`).

Base class validates all properties are set at instantiation.

### Properties vs Methods

Properties are the default for identity fields. Methods can be used when computed values are needed:

```php
// Default — property assignment (recommended)
protected string $machine = CarSalesMachine::class;
protected string $state   = 'awaiting_customer_start';

// Alternative — method override (when computation is needed)
protected function machine(): string
{
    return config('machines.car_sales.class');
}
```

The base class checks: method first, property fallback.

### Example

```php
class AtVerificationScenario extends MachineScenario
{
    protected string $machine     = CarSalesMachine::class;
    protected string $state       = 'awaiting_customer_start';
    protected string $event       = CustomerStartedEvent::class;
    protected string $description = 'At verification — farmer eligible';

    protected function plan(): array
    {
        return [
            'eligibility_check' => [
                IsFarmerNotEligibleGuard::class => false,
            ],
        ];
    }
}
```

### Example with Parameters

```php
class AtRejectedScenario extends MachineScenario
{
    protected string $machine     = CarSalesMachine::class;
    protected string $state       = 'checking_protocol';
    protected string $event       = ProtocolRejectedEvent::class;
    protected string $description = 'Application rejected with specific reason';

    protected function params(): array
    {
        return [
            'reason'       => ['required', Rule::enum(RejectionReason::class)],
            'findeksScore' => ['integer', 'min:0', 'max:1900'],
        ];
    }

    protected function plan(): array
    {
        return [
            'allocation' => [
                RejectAction::class => [
                    'rejectionReason' => $this->param('reason'),
                    'findeksScore'    => $this->param('findeksScore', 750),
                ],
            ],
        ];
    }
}
```

### Naming Convention

| Type | Base class | Concrete example | Suffix |
|------|-----------|-----------------|--------|
| Guard scenario | `GuardScenarioBehavior` | `HasConsentGuardScenario` | `{OriginalName}Scenario` |
| Action scenario | `ActionScenarioBehavior` | `StoreApplicationActionScenario` | `{OriginalName}Scenario` |
| Calculator scenario | `CalculatorScenarioBehavior` | `CustomerContextCalculatorScenario` | `{OriginalName}Scenario` |
| Result scenario | `ResultScenarioBehavior` | `OrderSummaryResultScenario` | `{OriginalName}Scenario` |
| Machine scenario | `MachineScenario` | `AtVerificationScenario` | Descriptive + `Scenario` suffix |

**Behavior scenario names** mirror the original behavior name with `Scenario` suffix. **Search advantage:** `HasConsent` finds both `HasConsentGuard` and `HasConsentGuardScenario` side by side.

**Machine scenario names** are descriptive — typically the target state. When multiple scenarios target the same state via different paths, disambiguate:
- `AtVerificationScenario` — unique target, sufficient
- `AtVerificationViaConsentScenario` — disambiguated: same target, different state/event
- ❌ `AtVerificationCarSalesMachineScenario` — machine name is redundant

## 5. plan() — Unified State-Route-Keyed Plan

Every `plan()` key is a **full state route**. The **value type** determines what happens at that state:

| Value type | Detection | Meaning |
|-----------|-----------|---------|
| `array` without `'outcome'` key | `is_array($value) && !isset($value['outcome'])` | Behavior overrides — guard/action/calculator/result overrides |
| `string` starting with `@` | `str_starts_with($value, '@')` | Delegation outcome — simulate child completion/failure/timeout |
| `array` with `'outcome'` key | `isset($value['outcome'])` | Delegation outcome with optional output AND/OR behavior overrides for `@done`/`@fail` transition guards/actions |
| `class-string<MachineScenario>` | `is_subclass_of($value, MachineScenario::class)` | Child scenario reference — run child with its own scenario |

ScenarioPlayer validates each key against the machine definition:
- State exists? If not → `ScenarioConfigurationException`
- State has no delegation? Value must be behavior override array (no `'outcome'` key)
- State has delegation? Value must be outcome string, child scenario, or `'outcome'`-keyed array

### Behavior Overrides

For states without delegation — override guards, actions, calculators, and results:

```php
protected function plan(): array
{
    return [
        'routing' => [
            // Calculator override — runs BEFORE guards, prepares data
            CustomerContextCalculator::class => ['farmer' => $mockFarmer],

            // Guard override
            HasConsentGuard::class => true,
        ],
        'eligibility_check' => [
            IsFarmerNotEligibleGuard::class => false,
        ],
    ];
}
```

All behavior types (`GuardBehavior`, `ActionBehavior`, `CalculatorBehavior`, `ResultBehavior`) that extend `InvokableBehavior` can be overridden. The container binding mechanism is identical for all.

**Override value forms by behavior type:**

| Behavior | Bool | Array (context write) | Closure | Class |
|----------|------|----------------------|---------|-------|
| **Guard** | ✅ return value | ❌ | ✅ must return bool | ✅ `GuardScenarioBehavior` |
| **Action** | ❌ | ✅ key-value to context | ✅ void | ✅ `ActionScenarioBehavior` |
| **Calculator** | ❌ | ✅ key-value to context | ✅ void | ✅ `CalculatorScenarioBehavior` |
| **Result** | ❌ | ✅ return as result | ✅ must return mixed | ✅ `ResultScenarioBehavior` |

**Guards:**

```php
'eligibility_check' => [
    // 1. Bool shorthand
    IsFarmerNotEligibleGuard::class => false,

    // 2. Closure — inline with DI
    HasConsentGuard::class => function (ContextManager $ctx): bool {
        return $ctx->get('farmer')->hasValidConsent();
    },

    // 3. Class — reusable scenario behavior
    IsCustomerInfoCompleteGuard::class => IsCustomerInfoCompleteGuardScenario::class,
],
```

**Actions:**

```php
'data_collection.retailer.calculating_prices' => [
    // 1. Array shorthand — key-value pairs written to context
    CheckProtocolAction::class => ['protocolEligible' => true],

    // 2. Closure — inline with DI
    StoreApplicationAction::class => function (ContextManager $ctx) {
        $ctx->set('applicationId', 'APP-'.Str::random());
    },

    // 3. Class — reusable scenario behavior
    CalculatePricesAction::class => CalculatePricesActionScenario::class,
],
```

**Calculators:**

```php
'routing' => [
    // 1. Array shorthand — pre-set context values that calculator would compute
    CustomerContextCalculator::class => [
        'farmer'   => $mockFarmer,
        'retailer' => $mockRetailer,
    ],

    // 2. Closure — inline with DI
    RetailerContextCalculator::class => function (ContextManager $ctx) {
        $ctx->set('retailer', Retailer::find(7));
    },

    // 3. Class — reusable scenario behavior (alternative to array shorthand above)
    // CustomerContextCalculator::class => CustomerContextCalculatorScenario::class,
],
```

**Results:**

```php
'approved' => [
    // 1. Array shorthand — return as result data
    OrderSummaryResult::class => ['orderId' => 'ORD-001', 'status' => 'approved'],

    // 2. Closure — inline with DI (alternative to above)
    // OrderSummaryResult::class => function (ContextManager $ctx): array {
    //     return ['orderId' => $ctx->get('orderId'), 'status' => 'approved'];
    // },

    // 3. Class — reusable scenario behavior (alternative to above)
    // OrderSummaryResult::class => OrderSummaryResultScenario::class,
],
```

### Scenario Behavior Base Classes

All behavior types have a corresponding scenario base class:

```php
abstract class GuardScenarioBehavior extends GuardBehavior {}
abstract class ActionScenarioBehavior extends ActionBehavior {}
abstract class CalculatorScenarioBehavior extends CalculatorBehavior {}
abstract class ResultScenarioBehavior extends ResultBehavior {}
```

DI is free (inherited from `InvokableBehavior`), type-compatible (can replace the original in the container).

```php
class CustomerContextCalculatorScenario extends CalculatorScenarioBehavior
{
    public function __invoke(ContextManager $ctx): void
    {
        $ctx->set('farmer', Farmer::find(42));
        $ctx->set('retailer', Retailer::find(7));
    }
}
```

### Same Behavior, Different Values Per State

When the same behavior appears under multiple states with **different values**, ScenarioPlayer creates a **state-aware override** that checks `$state->currentStateDefinition->route` at invocation time:

```php
'routing' => [
    HasConsentGuard::class => true,
],
'data_collection.customer_info.info_checking' => [
    HasConsentGuard::class => false,
],
```

| Situation | What ScenarioPlayer does |
|-----------|-------------------------|
| Behavior in one state only | Simple bind (state key is documentary) |
| Same value across states | Simple bind (state key is documentary) |
| Different values across states | State-aware bind (checks `->route`) |
| Behavior runs in un-overridden state | Original behavior executes |

### Delegation Outcomes

For states with `machine` or `job` delegation — declare what the child/job produces:

```php
protected function plan(): array
{
    return [
        // Child machine outcome — simple string
        'verification.findeks.running' => '@done.report_saved',
        'verification.turmob.verifying' => '@done',

        // Job actor outcome
        'querying_phones' => '@done',
    ];
}

// Alternative — outcome with output data:
'verification.findeks.running' => [
    'outcome' => '@done.report_saved',
    'output'  => ['reportId' => 'RPT-001', 'score' => 1400],
],

// Alternative — outcome with output AND behavior overrides for @done guards:
'polling' => [
    'outcome' => '@done',
    IsPinRequiredGuard::class => true,  // override @done transition guard
],
```

**Outcome formats:**

| Format | Example | When to use |
|--------|---------|-------------|
| Simple string | `'@done.report_saved'` | Parent only cares about routing, no guard overrides needed |
| With output | `['outcome' => '@done.report_saved', 'output' => [...]]` | Parent's `@done` action reads output/result |
| With guard override | `['outcome' => '@done', GuardClass::class => true]` | `@done`/`@fail` transition has guards that need overriding |
| Failure | `'@fail'` | Test failure path |
| Failure with context | `['outcome' => '@fail', 'output' => ['errorCode' => 'INVALID']]` | Parent's `@fail` action reads error context |
| Timeout | `'@timeout'` | Test timeout path |

**How it works:**
1. ScenarioPlayer intercepts delegation dispatch (`ChildMachineJob` / `ChildJobJob`)
2. Does NOT run the delegated machine/job
3. Simulates the completion by sending the declared outcome to the parent
4. Parent's `@done`/`@fail`/`@timeout` transition fires with the declared output

### Child Machine Scenario Reference

Instead of an outcome, reference a **child machine's own scenario** — the child runs and may pause at an interactive state:

```php
protected function plan(): array
{
    return [
        'eligibility_check' => [
            IsFarmerNotEligibleGuard::class => false,
        ],
        'verification.turmob.verifying' => '@done',
        'verification.findeks.running'  => AtAwaitingPinScenario::class,
    ];
}
```

The child's own scenario:

```php
class AtAwaitingPinScenario extends MachineScenario
{
    protected string $machine     = FindeksMachine::class;
    protected string $state       = 'idle';
    protected string $event       = 'MACHINE_START';
    protected string $description = 'FindeksMachine at awaiting_pin';

    protected function plan(): array
    {
        return [
            'checking_existing_report' => [
                HasExistingReportGuard::class => false,
            ],
            'querying_phones' => '@done',
            'requesting'      => '@done',
            'polling' => [
                'outcome' => '@done',
                IsPinRequiredGuard::class => true,  // force awaiting_pin branch
            ],
        ];
    }
}
```

**What happens:**
1. ScenarioPlayer intercepts FindeksMachine dispatch
2. Creates FindeksMachine, applies child scenario's `plan()` overrides
3. Engine processes automatic transitions with overrides
4. FindeksMachine reaches `awaiting_pin` — interactive state, waits for `PinConfirmedEvent`
5. Child **pauses** — parent stays at `verification.findeks.running`
6. Forward endpoints become active — QA can send `PinConfirmedEvent` via `/findeks/pin-confirmed`

**Child completion:**
- Child reaches a **final state** during replay → ScenarioPlayer simulates `@done` to parent
- Child reaches an **interactive state** → child stays running, parent stays in delegating state, forward endpoints active

**`$state` and `$event` in parent-referenced context:**
When referenced from a parent's `plan()`, the child always starts from its **initial state**. The child scenario's `$state` and `$event` are for **independent usage** — when QA interacts with the child directly.

### Parallel States

For parallel states, specify each region's delegation separately. Only regions you mention are controlled — unmentioned delegations run normally:

```php
protected function plan(): array
{
    return [
        'eligibility_check' => [
            IsFarmerNotEligibleGuard::class => false,
        ],
        // Findeks: complete with report
        'verification.findeks.running' => '@done.report_saved',
        // Turmob: complete
        'verification.turmob.verifying' => '@done',
        // Parallel @done guard
        'verification' => [
            'isFindeksRegionCompletedGuard' => true,
        ],
    ];
}
```

**Partial control — mix outcomes and scenarios:**

```php
// Findeks pauses at OTP, Turmob completes
'verification.findeks.running'  => AtAwaitingPinScenario::class,
'verification.turmob.verifying' => '@done',

// Findeks completes, Turmob runs normally (no plan entry)
'verification.findeks.running' => '@done.report_saved',
// verification.turmob.verifying → NOT in plan → real behavior

// Findeks completes, Turmob fails → test @fail path
'verification.findeks.running'  => '@done.report_saved',
'verification.turmob.verifying' => '@fail',
```

### Fire-and-Forget Delegation

Fire-and-forget delegation (machine with `queue` + no `@done`, or job with `target`) does NOT need entries in `plan()`. The parent transitions immediately — the child/job is a background side effect. In scenario mode, fire-and-forget dispatches are **suppressed**.

### Forward Events and Scenario Interplay

| plan() value | Child state | Forward endpoints |
|-------------|-------------|-------------------|
| `'@done.report_saved'` | Completed (simulated) | **Not active** — child didn't run |
| `AtAwaitingPinScenario::class` | Running, paused at interactive state | **Active** — child is real, waiting for input |

### Child Machine's Own Scenarios

Child machines have their own scenarios, used in two contexts:

1. **Referenced from parent** — `'verification.findeks.running' => AtAwaitingPinScenario::class`. Child starts from initial state with scenario overrides.

2. **Used independently** — QA interacts with the child directly via its own endpoints. `$state` and `$event` determine availability.

The same scenario class works in both contexts.

## 6. Parameterized Behaviors in Scenarios

EventMachine supports parameterized behaviors via colon syntax: `'guards' => 'hasRoleGuard:admin'`. Closures and class-based scenario behaviors receive these parameters via DI:

```php
// Machine: 'guards' => ['hasRoleGuard:admin', 'hasRoleGuard:editor']
'state_a' => [
    HasRoleGuard::class => function (ContextManager $ctx, string $role): bool {
        return match ($role) {
            'admin'  => true,
            'editor' => false,
        };
    },
],
```

| Override form | Parameterized behavior support |
|---------------|-------------------------------|
| `bool` | Returns same value for ALL calls regardless of parameters |
| `array` | Writes same context for ALL calls regardless of parameters |
| `Closure` | Receives parameters via DI — can differentiate |
| `class-string` | Receives parameters via `__invoke` — can differentiate |

## 7. Engine Feature Interactions

### Timers (after/every)

Timers are lazy in EventMachine — `MachineTimerFire` records are created by `ProcessTimersCommand` during scheduled sweeps, not during state entry. Since scenario replay is synchronous (milliseconds), the sweep command doesn't run during replay. Intermediate state timers never fire. No special handling needed.

### Scheduled Events

Scheduled events are processed by `machine:process-scheduled` on cron. Since scenario replay is synchronous, scheduled events don't fire during replay. No special handling needed.

### Queued Listeners

Queued listeners (`ListenerJob`) dispatch to the queue during scenario replay. They restore the machine via `Machine::create()` — scenario overrides propagate via `machine_current_states.scenario_class` (§9). Listener behavior respects scenario overrides automatically.

### ValidationGuardBehavior

`ValidationGuardBehavior` throws `MachineValidationException` on failure (returns 422 via endpoints). Scenario overrides work identically to regular guards — the override return value determines whether the guard passes. If the scenario overrides a validation guard to return `true`, no exception is thrown. If overridden to return `false`, the exception fires normally.

### Machine Locking

Scenario activation doesn't change lock semantics. Locks are acquired/released per existing rules during `Machine::send()`. Async jobs (parallel regions, listeners) acquire their own locks when restored. No special handling needed.

### Machine::fake() Compatibility

`Machine::fake()` stubs machine execution entirely — no real transitions, no event history. Scenarios require **real execution** with overridden behaviors. These are incompatible: scenarios should NOT be activated on faked machines. `ScenarioPlayer` should throw if the machine is faked.

### Event History

Scenario replay creates **real event history**. All transitions, entry/exit actions, @always chains, listeners — recorded as normal `MachineEvent` records. The resulting machine is indistinguishable from one that arrived at the state organically. Full restoration from any point is supported.

### Event Archival

Event archival (`ArchiveService`) is transparent to scenarios. Archived machines are auto-restored when new events arrive. No special handling needed.

### Machine Persistence (should_persist)

Scenario respects the machine's `should_persist` config. If `should_persist=false`, scenario-generated events are not persisted (in-memory only). The `machine_current_states.scenario_class` column is only written when `should_persist=true`.

## 8. Endpoint Integration

### QA Workflow

1. QA opens frontend, sees scenario selector (staging only)
2. Endpoint response includes `available_events` and `available_scenarios` (grouped by event)
3. QA selects a scenario from the dropdown for a specific event
4. QA triggers the event — frontend sends `{ type: "...", scenario: "..." }`
5. Machine processes event with scenario overrides active
6. Endpoint returns final state — QA continues manually

A machine can be running normally without any scenario. At **any state**, QA can decide to activate a scenario for the next event — as long as a matching scenario exists for that state + event.

### `available_scenarios` — Grouped by Event

After any successful event transition, the response includes `available_scenarios` keyed by event type. Only events that have scenarios are included:

```json
{
    "data": {
        "machine_id": "evt_01HXYZ...",
        "value": ["car_sales.checking_protocol"],
        "context": { "..." },
        "available_events": [
            { "type": "ProtocolPassedEvent", "source": "parent" },
            { "type": "ProtocolRejectedEvent", "source": "parent" },
            { "type": "ExpiredEvent", "source": "parent" }
        ],
        "available_scenarios": {
            "ProtocolPassedEvent": [
                { "slug": "at-approved-scenario", "description": "Fast-forward to approved", "params": {} },
                { "slug": "at-data-collection-scenario", "description": "Arrive at data collection", "params": {} }
            ],
            "ProtocolRejectedEvent": [
                {
                    "slug": "at-rejected-scenario",
                    "description": "Rejection with specific reason",
                    "params": {
                        "reason": {
                            "rules": ["required", "enum:RejectionReason"],
                            "values": ["GENERAL", "INSUFFICIENT_INCOME", "CREDIT_SCORE_LOW"],
                            "required": true
                        },
                        "findeksScore": {
                            "rules": ["integer", "min:0", "max:1900"],
                            "required": false
                        }
                    }
                }
            ]
        }
    }
}
```

                'isFindeksRegionCompletedGuard' => true,

**What happens:**
1. Controller detects `scenario`, validates `$state` = current state, `$event` = event being sent
2. Persists scenario to `machine_current_states.scenario_class`
3. Registers behavior overrides, prepares delegation handling
4. `$machine->send(CustomerStartedEvent)` — transitions to `eligibility_check`
5. `IsFarmerNotEligibleGuard` → scenario returns `false` → `verification` (parallel)
6. `FindeksMachine` intercepted → simulated as `@done.report_saved`
7. `TurmobMachine` intercepted → simulated as `@done`
8. `isFindeksRegionCompletedGuard` → scenario returns `true` → `checking_protocol` ✅
9. Cleanup, response with final state + `available_scenarios`

### Endpoint Flow

```
MachineController::executeEndpoint()
│
├─ 1. maybeRegisterScenarioOverrides(Request, machineClass)
│     ├─ Read 'scenario' slug and 'scenarioParams' from request
│     ├─ If null → clear scenario_class from machine_current_states, skip
│     ├─ If disabled → skip
│     ├─ Resolve scenario class (via ScenarioDiscovery)
│     ├─ Validate $state matches current state
│     ├─ Validate $event matches event being sent
│     ├─ Validate scenarioParams against params() rules
│     ├─ Hydrate scenario with validated params
│     ├─ Persist scenario_class to machine_current_states
│     ├─ Register state-keyed overrides in container
│     ├─ Prepare delegation handling (outcomes + child scenario references)
│     └─ Return scenario instance
│
├─ 2. $state = $machine->send(event)
│     ├─ @always guards use scenario behaviors ✅
│     ├─ Delegation intercepted → outcomes simulated or child scenarios run ✅
│     └─ Async jobs carry scenario via DB (see §9)
│
├─ 3. finally: cleanup — unbind scenario behaviors from container
│
└─ 4. buildResponse() — final state, available_events, available_scenarios
```

### Per-Machine Scenario Routes

Registered by `MachineRouter::register()` when enabled and machine has scenarios:

```
GET    {prefix}/scenarios                    → List this machine's scenarios
GET    {prefix}/scenarios/{slug}/describe    → Scenario details
```

## 9. Async Propagation

Scenario overrides live in the Laravel container — process-scoped. Queue jobs run in separate processes with their own containers. The scenario must propagate to async jobs.

### The Problem

During `$machine->send()`, the engine may dispatch async jobs:
- `ParallelRegionJob` — parallel regions processed by queue workers
- `ListenerJob` — queued listeners
- `ChildMachineCompletionJob` — child completion routing on parent

These jobs run in separate processes without scenario overrides.

### The Solution: `machine_current_states.scenario_class`

The `machine_current_states` table already has one row per machine instance. A nullable `scenario_class` column stores the active scenario's class name.

**Migration:**

```php
Schema::table('machine_current_states', function (Blueprint $table) {
    $table->string('scenario_class')->nullable()->after('machine_value');
});
```

**Activation** — when QA sends an event with `scenario` field:

```php
MachineCurrentState::where('root_event_id', $rootEventId)
    ->update(['scenario_class' => AtVerificationScenario::class]);
```

**Restoration** — when any job restores a machine via `Machine::create()`:

```php
if (config('machine.scenarios.enabled')) {
    $scenarioClass = MachineCurrentState::where('root_event_id', $rootEventId)
        ->value('scenario_class');
    if ($scenarioClass) {
        $scenario = new $scenarioClass();
        ScenarioPlayer::registerOverrides($scenario);
    }
}
```

**Deactivation** — when QA sends the next event WITHOUT a `scenario` field:

```php
MachineCurrentState::where('root_event_id', $rootEventId)
    ->update(['scenario_class' => null]);
```

### Write Always, Read on Restore

`scenario_class` is **always written** during scenario activation — regardless of whether the machine runs sync or async. Writing is nearly free (one extra column in an existing UPDATE).

**Reading** happens on every `Machine::create()` restoration when `scenarios.enabled=true`. The machine doesn't distinguish sync from async — it always checks. In sync context, overrides may already be registered (idempotent re-registration is harmless). In async context, this is how overrides propagate to queue workers. In production, `scenarios.enabled=false` prevents the query entirely.

### Scenario Lifecycle

```
1. Machine running normally (no scenario)
   → machine_current_states.scenario_class = null

2. QA sends event with scenario field
   → scenario_class = 'AtVerificationScenario'
   → overrides active for this event + any async jobs

3. Async jobs pick up work
   → restore machine → find scenario_class → register overrides

4. QA sends next event WITHOUT scenario field
   → scenario_class = null
   → machine continues with real behavior

5. QA sends next event with DIFFERENT scenario
   → scenario_class = 'AtApprovedScenario'
   → new overrides replace old
```

## 10. ScenarioPlayer — Runtime Engine

```
execute(scenario, eventPayload, machine)
│
├─ 1. Validate environment (scenarios enabled?)
├─ 2. Validate and hydrate params
│     ├─ Validate scenarioParams against params() rules
│     └─ Store validated values → $this->param('key') ready
├─ 3. Parse plan()
│     ├─ Validate all state routes against machine definition
│     ├─ Separate behavior overrides (array values on non-delegation states)
│     ├─ Separate delegation outcomes (@ strings on delegation states)
│     └─ Separate child scenario references (MachineScenario classes on delegation states)
├─ 4. Persist scenario_class to machine_current_states
├─ 5. Register behavior overrides in container (state-aware bindings)
├─ 6. Prepare delegation handling
│     ├─ Outcomes → intercept dispatch, simulate @done/@fail/@timeout
│     └─ Child scenarios → intercept dispatch, create child, apply scenario, run
│
├─ 7. Send trigger event
│     └─ $event with payload from endpoint request
│
├─ 8. Build ScenarioResult
│
└─ 9. Cleanup (finally)
      └─ Unbind overrides from container (DB record stays for async jobs)
```

## 12. File Organization & Discovery

### Machine structure mirroring

Each machine's scenarios live under its own `Scenarios/` directory:

**Simple scenarios** — single file:

```
app/Machines/CarSales/
├── CarSalesMachine.php
├── Guards/
│   ├── HasConsentGuard.php
│   └── IsFarmerNotEligibleGuard.php
├── Actions/
│   └── StoreApplicationAction.php
│
└── Scenarios/
    └── AtVerificationScenario.php
```

**Scenarios with class-based scenario behaviors** — own directory, mirroring Guards/Actions:

```
app/Machines/CarSales/
└── Scenarios/
    ├── AtVerificationScenario.php
    │
    └── AtCheckingProtocolScenario/
        ├── AtCheckingProtocolScenario.php
        └── Guards/
            └── IsFarmerNotEligibleGuardScenario.php
```

### Discovery

`ScenarioDiscovery` resolves each machine's `Scenarios/` directory from the machine class file location:

```php
$machineFile = (new ReflectionClass($machineClass))->getFileName();
$scenarioDir = dirname($machineFile).'/Scenarios';
```

Machine-local discovery. `machine:scenario-cache` builds a combined scenario class map from all discovered machines.

## 13. Error Handling & requiredContext Hints

### Exceptions

| Exception | When |
|-----------|------|
| `ScenariosDisabledException` | `MACHINE_SCENARIOS_ENABLED=false` |
| `ScenarioFailedException` | Guard rejection, invalid event, `$state` mismatch, `$event` mismatch. Includes `eventType`, `currentState`, `rejectionReason`. |
| `ScenarioConfigurationException` | Invalid state route in `plan()`. Delegation outcome on non-delegation state. Behavior override on delegation state. Child scenario `$machine` doesn't match delegation target. Missing required properties. Invalid scenario params (validation failure). Missing required scenario param. |

### requiredContext Hints

When a `MissingMachineContextException` is thrown during scenario replay, the error is enriched with `requiredContext` hints:

```
`farmer` is missing in context.

Required context for behaviors at this state:
  → IsFarmerNotEligibleGuard: farmer (App\Models\Farmer), retailer (App\Models\Retailer)
```
