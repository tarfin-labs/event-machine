## 1. Problem

Complex machines like `CarSalesMachine` have deep state hierarchies with parallel child delegations, API integrations, and multi-step guard chains. To test a new feature that affects behavior at state `checking_protocol`, a human must:

1. Trigger `START_APPLICATION` with valid retailer/farmer models
2. Grant consent → pass eligibility check
3. Wait for `FindeksMachine` to complete (6+ steps, external API calls)
4. Wait for `TurmobMachine` to complete (external API call)
5. Both parallel regions must finish → guard check → exit to `checking_protocol`

This requires 10+ manual steps, 2 child machines, multiple API calls, and deep domain knowledge. Product teams and QA cannot do this in staging without developer assistance.

**Scenarios solve this:** define behavior overrides and optional event steps once, activate from existing frontend endpoints via a `scenario` field, arrive at the desired state with a fully functional machine.

## 2. Design Principles

| Principle | Rationale |
|-----------|-----------|
| **Behavior is real** | Machine transitions, event history, context mutations, entry/exit actions — all real. The resulting machine is indistinguishable from one that arrived at that state organically. |
| **Override, don't fake** | External dependencies return predetermined responses via `arrange()`. The behavior still runs in the engine — it just gets scenario-defined data instead of calling real services. |
| **State-keyed overrides** | `arrange()` is keyed by full state route — you see WHERE each override applies. Same behavior can return different values in different states. |
| **Identity = Machine × State × Event** | A scenario is defined by which machine, at which state (full route), triggered by which event. This triple determines where the scenario appears in `available_scenarios`. |
| **Endpoint-native** | Scenarios are activated through existing machine endpoints via a `scenario` field in the request body. No separate scenario play endpoints needed. |
| **Composable** | Scenarios can extend other scenarios via `parent()`. |
| **Child-aware** | Parent scenarios can trigger child machine scenarios via `$this->child()`. |
| **Environment-gated** | Disabled by default (`MACHINE_SCENARIOS_ENABLED=true`). Zero overhead in production. |

## 3. Scenario Identity — Machine × State × Event

A scenario is positioned by three coordinates:

| Coordinate | Method | Example | Purpose |
|-----------|--------|---------|---------|
| **Machine** | `machine()` | `CarSalesMachine::class` | Which machine this scenario targets |
| **State** | `from()` | `'awaiting_customer_start'` | Full state route — where the machine is BEFORE the event |
| **Event** | `event()` | `'CustomerStartedEvent'` | Which event triggers this scenario |

This triple determines:
- **Discovery** — which scenarios are available at a given state for a given event
- **Validation** — the machine must be at `from()` state when the scenario activates, and the event being sent must match `event()`
- **`available_scenarios`** — grouped by event in endpoint responses

### Full State Route Requirement

States must use the **full route** (dot-notation path), not just the leaf key:

```php
// ❌ Ambiguous — could exist in multiple parallel regions
protected function from(): string { return 'awaiting_vehicle_and_pricing'; }

// ✅ Unambiguous — full path from machine root
protected function from(): string { return 'data_collection.retailer.awaiting_vehicle_and_pricing'; }
```

This matches how `$state->value` returns state paths in endpoint responses.

## 4. MachineScenario Class

```php
abstract class MachineScenario
{
    /** Which machine this scenario targets. */
    abstract protected function machine(): string;

    /** Human-readable description shown in artisan --list and endpoint responses. */
    abstract protected function description(): string;

    /** Full state route — the state the machine must be in BEFORE the event. */
    abstract protected function from(): string;

    /** The event that triggers this scenario. */
    abstract protected function event(): string;

    /**
     * State-keyed behavior scenario definitions.
     * Keys are full state routes, values are arrays of behavior overrides.
     *
     * Supports: bool, Closure, array, class-string<GuardScenarioBehavior|ActionScenarioBehavior>
     */
    protected function arrange(): array { return []; }

    /** Additional events to replay AFTER the trigger event (optional). */
    protected function steps(): array { return []; }

    /** Parent scenario for composition. */
    protected function parent(): ?string { return null; }

    // ─── Step Builders ───────────────────────────────────────
    protected function send(string $eventType, array $payload = []): ScenarioStep { ... }
    protected function child(string $machineClass): ChildScenarioStep { ... }

    // ─── Introspection ───────────────────────────────────────
    public function getMachine(): string { ... }
    public function getDescription(): string { ... }
    public function getFrom(): string { ... }
    public function getEvent(): string { ... }
    public function getArrange(): array { ... }
    public function getSteps(): array { ... }
    public function getParent(): ?string { ... }
}
```

**4 abstract methods** (`machine`, `description`, `from`, `event`).
**3 optional methods** (`arrange`, `steps`, `parent`).

### Naming Convention

| Type | Base class | Concrete example | Suffix pattern |
|------|-----------|-----------------|---------------|
| Guard scenario | `GuardScenarioBehavior` | `HasConsentGuardScenario` | `{OriginalName}Scenario` |
| Action scenario | `ActionScenarioBehavior` | `StoreApplicationActionScenario` | `{OriginalName}Scenario` |
| Machine scenario | `MachineScenario` | `AtVerificationScenario` | `{TargetState}Scenario` |

**Search advantage:** `HasConsent` finds both `HasConsentGuard` and `HasConsentGuardScenario` side by side.

**Machine scenario names** describe the target state, NOT the machine name:
- ✅ `AtVerificationScenario`
- ❌ `AtVerificationCarSalesMachineScenario`

## 5. arrange() — State-Keyed Behavior Scenarios

`arrange()` is keyed by **full state route**. This serves two purposes:
1. **Readability** — you see WHERE in the machine each override applies
2. **Functionality** — same behavior can return different values in different states

```php
protected function arrange(): array
{
    return [
        'routing' => [
            HasConsentGuard::class => true,
        ],
        'eligibility_check' => [
            IsFarmerNotEligibleGuard::class => false,
            StoreApplicationAction::class => ['applicationId' => 'APP-SCENARIO'],
        ],
    ];
}
```

Reading this, you immediately understand the **story**: at `routing`, consent is true; at `eligibility_check`, farmer is eligible.

### Three Value Forms — Mirrors Behavior Definition Pattern

**Guards:**

```php
'eligibility_check' => [
    // 1. Bool shorthand — simplest
    IsFarmerNotEligibleGuard::class => false,

    // 2. Closure — inline with DI
    HasConsentGuard::class => function (ContextManager $ctx): bool {
        return $ctx->get('farmer')->hasValidConsent();
    },

    // 3. Class — reusable scenario behavior
    IsCustomerInfoCompleteGuard::class => IsCustomerInfoCompleteGuardScenario::class,
],
```

**Actions:**

```php
'checking_protocol' => [
    // 1. Array shorthand — key-value pairs written to context
    CheckProtocolAction::class => ['protocolEligible' => true],

    // 2. Closure — inline with DI
    StoreApplicationAction::class => function (ContextManager $ctx) {
        $ctx->set('applicationId', 'APP-'.Str::random());
    },

    // 3. Class — reusable scenario behavior
    CalculatePricesAction::class => CalculatePricesActionScenario::class,
],
```

### Same Behavior, Different Values Per State

When the same behavior appears under multiple states with **different values**, ScenarioPlayer creates a **state-aware override** automatically:

```php
protected function arrange(): array
{
    return [
        'routing' => [
            HasConsentGuard::class => true,
        ],
        'data_collection' => [
            HasConsentGuard::class => false,
        ],
    ];
}
```

ScenarioPlayer detects the difference and creates a state-aware container binding that checks `$state->currentStateDefinition->route` at invocation time:

| Situation | What ScenarioPlayer does |
|-----------|-------------------------|
| Behavior in one state only | Simple bind (state key is documentary) |
| Same value across states | Simple bind (state key is documentary) |
| Different values across states | State-aware bind (checks `->route`) |
| Behavior runs in un-overridden state | Original behavior executes |

## 6. Scenario Behavior Classes

```php
// src/Behavior/GuardScenarioBehavior.php
abstract class GuardScenarioBehavior extends GuardBehavior {}

// src/Behavior/ActionScenarioBehavior.php
abstract class ActionScenarioBehavior extends ActionBehavior {}
```

Extending `GuardBehavior` / `ActionBehavior` means:
- **DI is free** — `InvokableBehavior` parameter injection works automatically
- **Type-compatible** — can replace the original in the container

**Example:**

```php
class IsCustomerInfoCompleteGuardScenario extends GuardScenarioBehavior
{
    public function __invoke(): bool
    {
        return true;
    }
}

class CalculatePricesActionScenario extends ActionScenarioBehavior
{
    public function __invoke(ContextManager $ctx): void
    {
        $ctx->set('totalCashPrice', Money::of(150000, 'TRY'));
        $ctx->set('monthlyInstallment', Money::of(12500, 'TRY'));
    }
}
```

## 7. Parameterized Behaviors in Scenarios

EventMachine supports parameterized behaviors via colon syntax: `'guards' => 'hasRoleGuard:admin'`. Closures and class-based scenario behaviors receive these parameters via DI:

```php
// Machine: 'guards' => ['hasRoleGuard:admin', 'hasRoleGuard:editor']
'state_a' => [
    HasRoleGuard::class => function (ContextManager $ctx, string $role): bool {
        return match ($role) {
            'admin'  => true,
            'editor' => false,
        };
    },
],
```

| Override form | Parameterized behavior support |
|---------------|-------------------------------|
| `bool` | Returns same value for ALL calls regardless of parameters |
| `array` | Writes same context for ALL calls regardless of parameters |
| `Closure` | Receives parameters via DI — can differentiate |
| `class-string` | Receives parameters via `__invoke` — can differentiate |

## 8. Endpoint Integration

### QA Workflow

1. QA opens frontend, sees scenario selector (staging only)
2. Endpoint response includes `available_events` and `available_scenarios` (grouped by event)
3. QA selects a scenario from the dropdown for a specific event
4. QA triggers the event — frontend sends `{ type: "...", scenario: "..." }`
5. Machine processes event with scenario overrides active
6. Endpoint returns final state — QA continues manually

### `available_scenarios` — Grouped by Event

After any successful event transition, the response includes `available_scenarios` keyed by event type. Only events that have scenarios are included:

```json
{
    "data": {
        "machine_id": "evt_01HXYZ...",
        "value": ["car_sales.checking_protocol"],
        "context": { "..." },
        "available_events": [
            { "type": "ProtocolPassedEvent", "source": "parent" },
            { "type": "ProtocolRejectedEvent", "source": "parent" },
            { "type": "ExpiredEvent", "source": "parent" }
        ],
        "available_scenarios": {
            "ProtocolPassedEvent": [
                { "slug": "at-approved-scenario", "description": "Fast-forward to approved" },
                { "slug": "at-data-collection-scenario", "description": "Arrive at data collection" }
            ],
            "ProtocolRejectedEvent": [
                { "slug": "at-rejected-scenario", "description": "Rejection with specific reason" }
            ]
        }
    }
}
```

When `MACHINE_SCENARIOS_ENABLED=false`, `available_scenarios` is omitted entirely.

### Scenario Activation via `scenario` Field

```http
POST /api/car-sales/{applicationId}/protocol-passed
{
    "type": "ProtocolPassedEvent",
    "scenario": "at-approved-scenario"
}
```

### Endpoint Flow

Scenario overrides are registered BEFORE `$machine->send()` so `@always` guards use them:

```
MachineController::executeEndpoint()
│
├─ 1. maybeRegisterScenarioOverrides(Request, machineClass)
│     ├─ Read 'scenario' slug from request
│     ├─ If null or disabled → skip
│     ├─ Resolve scenario class
│     ├─ Validate from() matches current machine state
│     ├─ Validate event() matches the event being sent
│     ├─ Register arrange() in container
│     └─ Return scenario instance
│
├─ 2. $state = $machine->send(event)
│     └─ @always guards use scenario behaviors ✅
│
├─ 3. maybePlayScenarioSteps($scenario, $machine, $state)
│
├─ 4. finally: cleanup — unbind scenario behaviors
│
└─ 5. buildResponse() — final state, available_events, available_scenarios
```

**Synchronous guarantee:** The endpoint does NOT return until the scenario fully completes.

### Per-Machine Scenario Routes

Registered by `MachineRouter::register()` when enabled and machine has scenarios:

```
GET    {prefix}/scenarios                    → List this machine's scenarios
GET    {prefix}/scenarios/{slug}/describe    → Scenario details
```

## 9. Composition — Extending Scenarios

Scenarios can build on top of each other via `parent()`. The child must declare the same `from()` and `event()` as the parent (PHP abstract requirement — values must match across chain).

```php
class AtDataCollectionScenario extends MachineScenario
{
    protected function machine(): string { return CarSalesMachine::class; }
    protected function description(): string { return 'At data collection'; }
    protected function from(): string { return 'awaiting_customer_start'; }
    protected function event(): string { return 'CustomerStartedEvent'; }

    protected function parent(): string
    {
        return AtVerificationScenario::class;
    }

    protected function arrange(): array
    {
        return [
            'checking_protocol' => [
                CheckProtocolAction::class => ['protocolEligible' => true],
            ],
        ];
    }

    protected function steps(): array
    {
        return [
            $this->send('ProtocolUndecidedEvent'),
        ];
    }
}
```

**Execution order:**
1. Register BOTH parent + child `arrange()` (child overrides parent for same state+behavior)
2. Send trigger event
3. Play parent's `steps()`
4. Play child's `steps()`

**Merge rules:**

| Aspect | Behavior |
|--------|----------|
| `machine()` | Must match across chain |
| `from()` + `event()` | Must match across chain |
| `arrange()` | Deep merged — child overrides parent for same state+behavior |
| `steps()` | Sequential — parent first, then child |

## 10. Child Machine Scenarios

When a parent machine delegates to a child machine, use `$this->child()` in `steps()`:

```php
protected function steps(): array
{
    return [
        $this->child(FindeksMachine::class)
            ->scenario(FindeksCompletedScenario::class),
        $this->child(TurmobMachine::class)
            ->scenario(TurmobCompletedScenario::class),
    ];
}
```

**How it works:**
1. `Bus::fake([ChildMachineJob::class])` intercepts async child dispatch
2. ScenarioPlayer finds child in `machine_children` table
3. If `child_root_event_id` is null (intercepted), creates child machine and updates record
4. Replays child scenario against child machine
5. Child reaches final state → parent's `@done` fires naturally

## 11. ScenarioPlayer — Runtime Engine

```
execute(scenario, eventPayload, machine)
│
├─ 1. Validate environment (scenarios enabled?)
├─ 2. Resolve parent chain (if parent() set)
│     ├─ Validate machine() match
│     └─ Validate from() + event() match
├─ 3. Merge arrange() from parent chain (child overrides parent)
├─ 4. Register merged overrides in container (state-aware bindings)
├─ 5. Intercept async dispatch
│     └─ Bus::fake([ChildMachineJob::class, ChildJobJob::class])
│
├─ 6. Send trigger event
│     └─ event() with payload from endpoint request or artisan
│
├─ 7. Play steps (parent first, then child)
│     ├─ ScenarioStep → Machine::send(eventType, payload)
│     └─ ChildScenarioStep → find child, play child scenario
│
├─ 8. Build ScenarioResult
│
└─ 9. Cleanup (finally)
      ├─ Unbind overrides from container
      └─ Restore original Bus dispatcher
```

## 12. Environment Gating

**Config in `config/machine.php`:**

```php
'scenarios' => [
    'enabled' => env('MACHINE_SCENARIOS_ENABLED', false),
],
```

| Point | Behavior when disabled |
|-------|----------------------|
| `ScenarioPlayer` | Throws `ScenariosDisabledException` |
| `machine:scenario` artisan | Prints error, exits with code 1 |
| `MachineRouter::register()` | Scenario routes not registered |
| `MachineController::buildResponse()` | `available_scenarios` omitted |
| `MachineController::executeEndpoint()` | `scenario` field silently ignored |

## 13. Artisan Command

```bash
# List all scenarios
php artisan machine:scenario --list

# List scenarios for a specific machine
php artisan machine:scenario --list --machine=CarSalesMachine

# Play a scenario on an existing machine (sends trigger event)
php artisan machine:scenario AtVerificationScenario --machine-id=evt_01HXYZ...
```

The artisan command performs the same flow as the endpoint: register overrides → send `event()` → play `steps()` → cleanup.

## 14. File Organization & Discovery

### Machine structure mirroring

Each machine's scenarios live under its own `Scenarios/` directory:

**Simple scenarios** — single file:

```
app/Machines/CarSales/
├── CarSalesMachine.php
├── Guards/
│   ├── HasConsentGuard.php
│   └── IsFarmerNotEligibleGuard.php
├── Actions/
│   └── StoreApplicationAction.php
│
└── Scenarios/
    └── SimpleSkipScenario.php
```

**Scenarios with class-based scenario behaviors** — own directory, mirroring Guards/Actions:

```
app/Machines/CarSales/
└── Scenarios/
    ├── SimpleSkipScenario.php
    │
    ├── AtVerificationScenario/
    │   ├── AtVerificationScenario.php
    │   └── Guards/
    │       └── IsFarmerNotEligibleGuardScenario.php
    │
    └── ApprovedScenario/
        ├── ApprovedScenario.php
        ├── Guards/
        │   └── IsCustomerInfoCompleteGuardScenario.php
        └── Actions/
            └── CalculatePricesActionScenario.php
```

### Discovery

`ScenarioDiscovery` resolves each machine's `Scenarios/` directory from the machine class file location:

```php
$machineFile = (new ReflectionClass($machineClass))->getFileName();
$scenarioDir = dirname($machineFile).'/Scenarios';
```

Machine-local discovery. `machine:scenario-cache` builds a combined scenario class map from all discovered machines.

## 15. Error Handling & requiredContext Hints

### Exceptions

| Exception | When |
|-----------|------|
| `ScenariosDisabledException` | `MACHINE_SCENARIOS_ENABLED=false` |
| `ScenarioFailedException` | Step fails (guard rejection, invalid event, `from()` mismatch, `event()` mismatch). Includes `stepIndex`, `eventType`, `currentState`, `rejectionReason`. |
| `ScenarioConfigurationException` | `machine()` mismatch in parent chain. `from()`/`event()` mismatch in parent chain. |

### requiredContext Hints

When a `MissingMachineContextException` is thrown during scenario replay, the error is enriched with `requiredContext` hints:

```
`farmer` is missing in context.

Required context for behaviors at this state:
  → IsFarmerNotEligibleGuard: farmer (App\Models\Farmer), retailer (App\Models\Retailer)
```

## 16. Old Scenario System — @deprecated

The old scenario system (`scenarios_enabled`, `$scenarios` param, `getScenarioStateIfAvailable`, `EventBehavior::getScenario`, `TestMachine::withScenario`) is preserved with `@deprecated` markers. Will be removed in v9.

**Coexistence — no conflicts:**

| Aspect | Old System | New System |
|--------|-----------|------------|
| Config | `scenarios_enabled` (machine config) | `machine.scenarios.enabled` (Laravel config) |
| Payload | `scenarioType` (event payload) | `scenario` (HTTP request body) |
| Mechanism | State definition swap (`idMap`) | Container binding + event replay |

## 17. What NOT to Include

| Feature | Why excluded |
|---------|-------------|
| `models()` / model factories | Staging QA has models in DB. |
| `defaults()` / `param()` | Parameters come from event payload via frontend. |
| `ScenarioStubContract` / `applyStub()` | Override logic belongs in `arrange()`, not on the action class. |
| `play()` / `playOn()` static methods | Execution always through ScenarioPlayer. Endpoint or artisan triggers event with overrides. |
| Separate scenario play endpoints | Scenarios are activated through normal event endpoints via `scenario` field. |
| Automatic rollback / cleanup | Scenarios create persistent state intentionally. |
| Parallel step execution | Steps must be sequential. |
| Conditional steps | Create different scenarios instead. |
| Global `/machine/scenarios/` routes | Each machine owns its routes under its prefix. |
