## 1. Problem

Complex machines like `CarSalesMachine` have deep state hierarchies with parallel child delegations, API integrations, and multi-step guard chains. To test a new feature that affects behavior at state `checking_protocol`, a human must:

1. Trigger `START_APPLICATION` with valid retailer/farmer models
2. Grant consent → pass eligibility check
3. Wait for `FindeksMachine` to complete (6+ steps, external API calls)
4. Wait for `TurmobMachine` to complete (external API call)
5. Both parallel regions must finish → guard check → exit to `checking_protocol`

This requires 10+ manual steps, 2 child machines, multiple API calls, and deep domain knowledge. Product teams and QA cannot do this in staging without developer assistance.

**Scenarios solve this:** define behavior overrides and optional event steps once, activate from the existing frontend endpoints, arrive at the desired state with a fully functional machine.

## 2. Design Principles

| Principle | Rationale |
|-----------|-----------|
| **Behavior is real** | Machine transitions, event history, context mutations, entry/exit actions — all real. The resulting machine is indistinguishable from one that arrived at that state organically. Infrastructure concerns (queue dispatch, timers) may be adjusted. |
| **Override, don't fake** | External dependencies (APIs, services) return predetermined responses via `arrange()`. The behavior still runs in the engine — it just gets scenario-defined data instead of calling real services. |
| **State-keyed overrides** | `arrange()` is keyed by state name — you see WHERE in the machine each override applies. Same behavior can return different values in different states. |
| **Timers suspended** | `after`/`every` timers are lazy (artisan sweep), so they don't fire during synchronous scenario replay. |
| **Composable** | Scenarios can extend other scenarios via `parent()`. |
| **Child-aware** | Parent scenarios can trigger child machine scenarios. |
| **Environment-gated** | Scenarios are disabled by default. Only active when `MACHINE_SCENARIOS_ENABLED=true`. Zero overhead in production. |
| **Endpoint-integrated** | Scenarios are activated through existing machine endpoints via a `scenario` field in the request. No separate scenario endpoints needed for QA usage. |

## 3. MachineScenario Class

```php
abstract class MachineScenario
{
    /** Which machine this scenario targets. */
    abstract protected function machine(): string;

    /** Human-readable description shown in artisan --list and endpoint responses. */
    abstract protected function description(): string;

    /** Parent scenario for composition. */
    protected function parent(): ?string { return null; }

    /** Expected starting state for mid-flight scenarios. */
    protected function from(): ?string { return null; }

    /**
     * State-keyed behavior scenario definitions.
     * Keys are state names, values are arrays of behavior overrides.
     */
    protected function arrange(): array { return []; }

    /** Ordered sequence of events to replay (optional). */
    protected function steps(): array { return []; }

    // ─── Step Builders ───────────────────────────────────────
    protected function send(string $eventType, array $payload = []): ScenarioStep { ... }
    protected function child(string $machineClass): ChildScenarioStep { ... }

    // ─── Execution ───────────────────────────────────────────
    public static function play(): ScenarioResult { ... }
    public static function playOn(string $machineId): ScenarioResult { ... }

    // ─── Introspection ───────────────────────────────────────
    public function getMachine(): string { ... }
    public function getDescription(): string { ... }
    public function getParent(): ?string { ... }
    public function getFrom(): ?string { ... }
    public function getArrange(): array { ... }
    public function getSteps(): array { ... }
}
```

**2 abstract methods** (`machine`, `description`).
**4 optional methods** (`parent`, `from`, `arrange`, `steps`).

## 4. arrange() — State-Keyed Behavior Scenarios

`arrange()` is keyed by **state name**. This serves two purposes:
1. **Readability** — you see WHERE in the machine each override applies
2. **Functionality** — same behavior can return different values in different states

```php
protected function arrange(): array
{
    return [
        'routing' => [
            HasConsentGuard::class => true,  // → registered customer path
        ],
        'eligibility_check' => [
            IsFarmerNotEligibleGuard::class => false,  // → farmer eligible
            StoreApplicationAction::class => ['applicationId' => 'APP-SCENARIO'],
        ],
    ];
}
```

Reading this, you immediately understand the **story**: at `routing`, consent is true; at `eligibility_check`, farmer is eligible.

### Three Value Forms — Mirrors Behavior Definition Pattern

**Guards:**

```php
'eligibility_check' => [
    // 1. Bool shorthand — simplest
    IsFarmerNotEligibleGuard::class => false,

    // 2. Closure — inline with DI
    HasConsentGuard::class => function (ContextManager $ctx): bool {
        return $ctx->get('farmer')->hasValidConsent();
    },

    // 3. Class — reusable scenario behavior
    IsCustomerInfoCompleteGuard::class => IsCustomerInfoCompleteGuardScenario::class,
],
```

**Actions:**

```php
'checking_protocol' => [
    // 1. Array shorthand — key-value pairs written to context
    CheckProtocolAction::class => ['protocolEligible' => true],

    // 2. Closure — inline with DI
    StoreApplicationAction::class => function (ContextManager $ctx) {
        $ctx->set('applicationId', 'APP-'.Str::random());
    },

    // 3. Class — reusable scenario behavior
    CalculatePricesAction::class => CalculatePricesActionScenario::class,
],
```

### Same Behavior, Different Values Per State

When the same behavior appears under multiple states with **different values**, ScenarioPlayer creates a **state-aware override** automatically:

```php
protected function arrange(): array
{
    return [
        'routing' => [
            HasConsentGuard::class => true,   // at routing: true
        ],
        'data_collection' => [
            HasConsentGuard::class => false,  // at data_collection: false
        ],
    ];
}
```

ScenarioPlayer detects `HasConsentGuard` under multiple states with different values. It creates a state-aware container binding that checks `$state->currentStateDefinition->key` at invocation time:

| Situation | What ScenarioPlayer does |
|-----------|-------------------------|
| Behavior in one state only | Simple bind (state key is documentary) |
| Same value across states | Simple bind (state key is documentary) |
| Different values across states | State-aware bind (state key is functional) |
| Behavior runs in un-overridden state | Original behavior executes |

## 5. Scenario Behavior Classes

### Base Classes

```php
// src/Behavior/GuardScenarioBehavior.php
abstract class GuardScenarioBehavior extends GuardBehavior {}

// src/Behavior/ActionScenarioBehavior.php
abstract class ActionScenarioBehavior extends ActionBehavior {}
```

Extending `GuardBehavior` / `ActionBehavior` means:
- **DI is free** — `InvokableBehavior` parameter injection works automatically
- **Type-compatible** — can replace the original in the container
- **Semantic clarity** — "this is the scenario version of a behavior"

### Reusable Scenario Behavior Examples

```php
class IsCustomerInfoCompleteGuardScenario extends GuardScenarioBehavior
{
    public function __invoke(): bool
    {
        return true;
    }
}

class CalculatePricesActionScenario extends ActionScenarioBehavior
{
    public function __invoke(ContextManager $ctx): void
    {
        $ctx->set('totalCashPrice', Money::of(150000, 'TRY'));
        $ctx->set('monthlyInstallment', Money::of(12500, 'TRY'));
    }
}
```

## 6. Parameterized Behaviors in Scenarios

EventMachine supports parameterized behaviors via colon syntax: `'guards' => 'hasRoleGuard:admin'`. The same behavior class can be called with different parameters in the same state.

Closures and class-based scenario behaviors receive these parameters via DI — the same way regular behaviors do:

```php
// Machine definition:
// 'guards' => ['hasRoleGuard:admin', 'hasRoleGuard:editor']
//
// Scenario override — closure receives the parameter as argument:
'state_a' => [
    HasRoleGuard::class => function (ContextManager $ctx, string $role): bool {
        return match ($role) {
            'admin'  => true,
            'editor' => false,
        };
    },
],
```

The same closure is called twice with different `$role` values. Class-based scenario behaviors work the same way — `__invoke()` receives the arguments.

| Override form | Parameterized behavior support |
|---------------|-------------------------------|
| `bool` | Returns same value for ALL calls regardless of parameters |
| `array` | Writes same context for ALL calls regardless of parameters |
| `Closure` | Receives parameters via DI — can differentiate |
| `class-string` | Receives parameters via `__invoke` — can differentiate |

## 7. Mid-Flight Scenarios

Scenarios can continue from an existing machine instead of creating a new one.

### The `from()` Method

Declares which state the machine must be in before the scenario proceeds:

```php
class AtApprovedScenario extends MachineScenario
{
    protected function from(): ?string
    {
        return 'checking_protocol';
    }
    // ...
}
```

### `play()` vs `playOn()`

| Method | Machine | `from()` | `parent()` |
|--------|---------|----------|------------|
| `play()` | Creates new | Ignored | Resolved |
| `playOn(machineId)` | Restores existing | Validated | Ignored |

When `playOn()` is called:
- Machine is restored from the given `machineId`
- If `from()` is defined, current state is validated — throws `ScenarioFailedException` on mismatch
- `parent()` is not used (existing machine IS the starting point)

## 8. Composition — Extending Scenarios

Scenarios can build on top of each other via `parent()`:

```php
class AtDataCollectionScenario extends MachineScenario
{
    protected function parent(): string
    {
        return AtVerificationScenario::class;
    }

    protected function arrange(): array
    {
        return [
            'checking_protocol' => [
                CheckProtocolAction::class => ['protocolEligible' => true],
            ],
        ];
    }

    protected function steps(): array
    {
        return [
            $this->send('ProtocolUndecidedEvent'),
        ];
    }
}
```

**Merge rules:**

| Aspect | Behavior |
|--------|----------|
| `machine()` | Must match across chain — validated by ScenarioPlayer |
| `arrange()` | Deep merged — child state keys override parent for same behavior class |
| `steps()` | Sequential — parent steps play first, then child steps |

## 9. Child Machine Scenarios

When a parent machine delegates to a child machine (via `machine` key on a state), use `$this->child()`:

```php
protected function steps(): array
{
    return [
        $this->send('ConsentGrantedEvent'),
        // Child machines spawned by the send() above
        $this->child(FindeksMachine::class)
            ->scenario(FindeksCompletedScenario::class),
        $this->child(TurmobMachine::class)
            ->scenario(TurmobCompletedScenario::class),
    ];
}
```

**How it works:**
1. `Bus::fake([ChildMachineJob::class])` intercepts async child dispatch
2. ScenarioPlayer finds child machine in `machine_children` table
3. If `child_root_event_id` is null (Bus::fake intercepted), creates child machine and updates tracking record
4. Replays child scenario's steps against the child machine instance
5. Child reaches its final state → parent's `@done` transition fires naturally

## 10. Endpoint Integration

### `available_scenarios` in Response

After any successful event transition, the response includes `available_scenarios` — scenarios that can be played from the machine's current state (post-transition):

```json
{
    "data": {
        "machine_id": "evt_01HXYZ...",
        "value": ["car_sales.checking_protocol"],
        "context": { "..." },
        "available_events": [...],
        "available_scenarios": [
            {
                "slug": "at-approved-scenario",
                "description": "Fast-forward to approved",
                "from": "checking_protocol"
            }
        ]
    }
}
```

When `MACHINE_SCENARIOS_ENABLED=false`, the `available_scenarios` key is omitted entirely.

### Scenario Continuation via `scenario` Field

QA sends a normal event and attaches a `scenario` field to fast-forward:

```http
POST /api/car-sales/{applicationId}/customer-started
{
    "type": "CustomerStartedEvent",
    "scenario": "at-verification-scenario"
}
```

### Endpoint Flow — Override Timing

Scenario behaviors must be registered BEFORE `$machine->send()` so `@always` guards use overrides:

```
MachineController::executeEndpoint()
│
├─ 1. maybeRegisterScenarioOverrides(Request, machineClass)
│     ├─ Read 'scenario' slug from request body
│     ├─ If null or MACHINE_SCENARIOS_ENABLED=false → return null
│     ├─ Resolve scenario class, instantiate, register arrange() in container
│     └─ Return scenario instance
│
├─ 2. $state = $machine->send(event)
│     └─ @always guards now use scenario behaviors ✅
│
├─ 3. Validate from() (if scenario defines it)
│     └─ Mismatch → 422
│
├─ 4. maybePlayScenarioSteps($scenario, $machine, $state)
│
├─ 5. finally: cleanup — unbind scenario behaviors from container
│
└─ 6. buildResponse() — final state, available_events, available_scenarios
```

**Synchronous guarantee:** The endpoint does NOT return until the scenario fully completes.

### Per-Machine Scenario Routes

Registered by `MachineRouter::register()` when `MACHINE_SCENARIOS_ENABLED=true` and machine has scenarios:

```
GET    {prefix}/scenarios                           → List this machine's scenarios
POST   {prefix}/scenarios/{slug}                    → Play scenario (new machine)
POST   {prefix}/scenarios/{slug}/{machineId}        → Play on existing machine
GET    {prefix}/scenarios/{slug}/describe            → Scenario details
```

No global routes — each machine owns its scenario endpoints under its own prefix.

## 11. Async Dispatch & Timers

### Child Machine Dispatch

`Bus::fake([ChildMachineJob::class, ChildJobJob::class])` before replay starts. Only these specific jobs are intercepted — all other jobs dispatch normally. The child machine record is created by the engine, the scenario player finds and drives it synchronously.

### Timer Handling

Timers are lazy in EventMachine — `MachineTimerFire` records are created by `ProcessTimersCommand` during scheduled sweeps, not during state entry. Since scenario replay is synchronous (milliseconds), the sweep command doesn't run during replay. Intermediate state timers never get a chance to fire.

## 12. ScenarioPlayer — Runtime Engine

```
play(scenario, machineId?)
│
├─ 1. Validate environment (scenarios enabled?)
├─ 2. Validate mid-flight constraints (playOn + parent = error)
├─ 3. Resolve parent chain (if parent() set, skip in mid-flight)
├─ 4. Register arrange() overrides in container (state-aware bindings)
├─ 5. Intercept async dispatch (Bus::fake)
│
├─ 6. Replay steps (or restore machine for mid-flight)
│     ├─ Mid-flight: restore machine, validate from()
│     ├─ Parent result: continue from parent's machine
│     ├─ New: create machine on first send()
│     ├─ ScenarioStep → Machine::send(eventType, payload)
│     └─ ChildScenarioStep → find child, play child scenario
│
├─ 7. Build ScenarioResult
│
└─ 8. Cleanup (finally)
      ├─ Unbind overrides from container
      └─ Restore original Bus dispatcher
```

## 13. Environment Gating

**Config in `config/machine.php`:**

```php
'scenarios' => [
    'enabled' => env('MACHINE_SCENARIOS_ENABLED', false),
],
```

**Enforcement points:**

| Point | Behavior when disabled |
|-------|----------------------|
| `ScenarioPlayer::play()` | Throws `ScenariosDisabledException` |
| `machine:scenario` artisan command | Prints error, exits with code 1 |
| `MachineRouter::register()` | Scenario routes not registered |
| `MachineController::buildResponse()` | `available_scenarios` key omitted |
| `MachineController::executeEndpoint()` | `scenario` field in request silently ignored |

## 15. File Organization & Discovery

### Machine structure mirroring

Each machine's scenarios live under its own `Scenarios/` directory:

**Simple scenarios** — single file:

```
app/Machines/CarSales/
├── CarSalesMachine.php
├── Guards/
│   ├── HasConsentGuard.php
│   └── IsFarmerNotEligibleGuard.php
├── Actions/
│   └── StoreApplicationAction.php
│
└── Scenarios/
    └── SimpleSkipScenario.php
```

**Scenarios with class-based scenario behaviors** — own directory, mirroring Guards/Actions:

```
app/Machines/CarSales/
└── Scenarios/
    ├── SimpleSkipScenario.php
    │
    ├── AtVerificationScenario/
    │   ├── AtVerificationScenario.php
    │   └── Guards/
    │       └── IsFarmerNotEligibleGuardScenario.php
    │
    └── ApprovedScenario/
        ├── ApprovedScenario.php
        ├── Guards/
        │   └── IsCustomerInfoCompleteGuardScenario.php
        └── Actions/
            └── CalculatePricesActionScenario.php
```

### Discovery

`ScenarioDiscovery` resolves each machine's `Scenarios/` directory from the machine class file location:

```php
$machineFile = (new ReflectionClass($machineClass))->getFileName();
$scenarioDir = dirname($machineFile).'/Scenarios';
```

Machine-local discovery. `machine:scenario-cache` iterates discovered machine classes and builds a combined scenario class map.

## 16. Naming Convention

| Type | Base class | Concrete example | Suffix pattern |
|------|-----------|-----------------|---------------|
| Guard | `GuardBehavior` | `HasConsentGuard` | `{Name}Guard` |
| Action | `ActionBehavior` | `StoreApplicationAction` | `{Name}Action` |
| Guard scenario | `GuardScenarioBehavior` | `HasConsentGuardScenario` | `{OriginalName}Scenario` |
| Action scenario | `ActionScenarioBehavior` | `StoreApplicationActionScenario` | `{OriginalName}Scenario` |
| Machine scenario | `MachineScenario` | `AtVerificationScenario` | `{TargetState}Scenario` |

**Search advantage:** `HasConsent` finds both `HasConsentGuard` and `HasConsentGuardScenario` side by side.

**Machine scenario names** describe the target state, NOT the machine name:
- ✅ `AtVerificationScenario`
- ❌ `AtVerificationCarSalesMachineScenario`

## 17. Error Handling & requiredContext Hints

### Exceptions

| Exception | When |
|-----------|------|
| `ScenariosDisabledException` | `MACHINE_SCENARIOS_ENABLED=false` |
| `ScenarioFailedException` | Step fails (guard rejection, invalid event, `from()` mismatch). Includes `stepIndex`, `eventType`, `currentState`, `rejectionReason`. |
| `ScenarioConfigurationException` | `machine()` mismatch in parent chain. `playOn()` with `parent()` defined. |

### requiredContext Hints

When a `MissingMachineContextException` is thrown during scenario replay, the error message is enriched with `requiredContext` hints from behaviors at the current state:

```
`farmer` is missing in context.

Required context for behaviors at this state:
  → IsFarmerNotEligibleGuard: farmer (App\Models\Farmer), retailer (App\Models\Retailer)
  → StoreApplicationAction: application (App\Models\Application)
```

This helps scenario authors understand what context keys need to be available.

## 18. Old Scenario System — @deprecated

The old scenario system (`scenarios_enabled`, `$scenarios` param, `getScenarioStateIfAvailable`, `EventBehavior::getScenario`, `TestMachine::withScenario`) is preserved with `@deprecated` markers. It will be removed in v9.

**Coexistence:** Old and new systems use completely different mechanisms — no conflicts:

| Aspect | Old System | New System |
|--------|-----------|------------|
| Config key | `scenarios_enabled` (machine config) | `machine.scenarios.enabled` (Laravel config) |
| Payload field | `scenarioType` (event payload) | `scenario` (HTTP request body) |
| Mechanism | State definition swap (`idMap`) | Container binding + event replay |
| Endpoints | None | Per-machine `{prefix}/scenarios/` |

## 19. What NOT to Include

| Feature | Why excluded |
|---------|-------------|
| `models()` / model factories | Staging QA has models in DB. Parameters come from frontend. |
| `defaults()` / `param()` | Parameters come from event payload via frontend. |
| `ScenarioStubContract` / `applyStub()` | Override logic belongs in `arrange()`, not on the action class. |
| Automatic rollback / cleanup | Scenarios create persistent state intentionally. |
| Parallel step execution | Steps must be sequential. |
| Conditional steps | Create different scenarios instead. |
| UI / dashboard | Artisan + HTTP endpoints are sufficient. |
| Global `/machine/scenarios/` routes | Each machine owns its scenario endpoints. |
