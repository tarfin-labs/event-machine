# Actor Persistence in machine_events — v8.0.0

## Problem

Actor (who triggered an event) is never persisted. The `machine_events` table records what happened, when, and what the context looked like — but not **who** caused it. After a persist/restore cycle, the actor is gone.

This means:
- Event history can't answer "who triggered this transition?"
- No audit trail for "show me all actions by the score service"
- Worker-side restored machines have no actor context

## Solution

Add `actor_type` and `actor_id` columns to `machine_events` table.

### Migration

```php
// database/migrations/xxxx_add_actor_to_machine_events_table.php
Schema::table('machine_events', function (Blueprint $table) {
    $table->string('actor_type')->nullable()->after('meta');
    $table->unsignedBigInteger('actor_id')->nullable()->after('actor_type');
    $table->index(['actor_type', 'actor_id']);
});
```

### Where to persist

In `State::setCurrentEventBehavior()`, when creating the `MachineEvent` model:

```php
$this->history->push(
    new MachineEvent([
        // ... existing fields ...
        'actor_type' => $this->resolveActorType(),
        'actor_id'   => $this->resolveActorId(),
    ])
);
```

The actor source depends on what's available:
1. `$currentEventBehavior->actor($this->context)` — if the event carries an actor
2. `$this->triggeringEvent->actor($this->context)` — if preserved via triggeringEvent

### Actor resolution

```php
private function resolveActorType(): ?string
{
    $actor = $this->resolveActor();
    if ($actor === null) {
        return null;
    }

    return $actor instanceof Model ? $actor->getMorphClass() : get_class($actor);
}

private function resolveActorId(): ?int
{
    $actor = $this->resolveActor();
    if ($actor === null) {
        return null;
    }

    return $actor instanceof Model ? $actor->getKey() : null;
}

private function resolveActor(): mixed
{
    // Current event's actor takes priority
    $actor = $this->currentEventBehavior?->actor($this->context);
    if ($actor !== null) {
        return $actor;
    }

    // Fall back to triggering event's actor
    return $this->triggeringEvent?->actor($this->context);
}
```

### Worker-side restore

When a machine is restored from DB (ListenerJob, ChildMachineCompletionJob), the actor can be read from the last `MachineEvent` record:

```php
$lastEvent = MachineEvent::where('root_event_id', $rootEventId)
    ->latest('sequence_number')
    ->first();

// Actor is available for observability even after restore
$actorType = $lastEvent->actor_type;
$actorId   = $lastEvent->actor_id;
```

This is read-only/observability — restored machines don't need actor in-memory for most operations.

## Open Questions

1. **Should `actor_id` be string or bigint?** ULIDs need string, auto-increment needs bigint. Most backend models use bigint, but future-proofing may favor string.
2. **Should restored machines populate `$state->triggeringEvent->actor` from DB?** This would make actor available after restore, but adds complexity and a DB query.
3. **Index strategy**: `(actor_type, actor_id)` composite index is sufficient for "all events by actor X" queries.
