# Typed Inter-Machine Communication Contracts

**Status:** Ready for implementation
**Version:** 9.0.0
**Date:** 2026-03-25
**Updated:** 2026-03-30

---

## Problem

Inter-machine data flow in EventMachine relies on untyped string-key arrays at every delegation boundary:

```php
// Parent → Child: blind array key selection
'with' => ['orderId', 'customerName'],

// Child → Parent: string keys, no IDE support
'output' => ['paymentId', 'status'],

// Parent @done action: fragile payload access
function (ContextManager $ctx, ChildMachineDoneEvent $event): void {
    $ctx->set('paymentId', $event->output('paymentId'));  // No type safety, no IDE autocomplete
}
```

**Consequences:**
1. No IDE autocompletion for inter-machine data
2. No compile-time or definition-time validation of data contracts
3. Typos in key names cause silent `null` values, not errors
4. Refactoring a child's context keys silently breaks parent integrations
5. No way to document "what does this machine expect/produce" as code

---

## Vision

Every delegation boundary — machine or job — has a typed signature, like a function:

```
PaymentMachine(PaymentInput) → PaymentOutput | PaymentFailure
ProcessPaymentJob(PaymentInput) → PaymentOutput | PaymentFailure
```

The parent doesn't care whether the child is a machine or a job. The DX is identical:

```php
'delegating' => [
    'machine' => PaymentMachine::class,    // or 'job' => ProcessPaymentJob::class
    'input'   => PaymentInput::class,
    '@done'   => [
        'target'  => 'completed',
        'actions' => function (ContextManager $ctx, PaymentOutput $output): void {
            $ctx->set('paymentId', $output->paymentId);  // IDE autocomplete
        },
    ],
    '@fail'   => [
        'target'  => 'errored',
        'actions' => function (ContextManager $ctx, PaymentFailure $failure): void {
            $ctx->set('errorCode', $failure->errorCode);  // IDE autocomplete
        },
    ],
],
```

---

## How This Extends v9's Unified Output

v9 unified `result`/`contextKeys` into a single `output` keyword with type dispatch:

| `output` value type | Current v9 behavior |
|---------------------|---------------------|
| `string` (OutputBehavior subclass) | Container resolves → `__invoke(ContextManager, ...)` → returns mixed |
| `array` | Filters context keys |
| `Closure` | Inline callable → returns mixed |
| `null` | Fallback to `toResponseArray()` |

This spec adds **one new dispatch case** and **two new config keys**:

| Addition | What it is |
|----------|------------|
| `output => MachineOutput subclass` | New dispatch: auto-resolves from context via `fromContext()` → returns typed DTO |
| `'input' => MachineInput subclass` | New config key: typed parent→child contract (replaces `with`) |
| `'failure' => MachineFailure subclass` | New config key: typed exception→structured error mapping |

The existing `output` infrastructure stays intact. `OutputBehavior`, array filters, closures — all continue to work. `MachineOutput` is a new dispatch target within the same mechanism.

### Type dispatch after this spec

```
output value → behavior registry key?        → resolve from registry (existing v9)
            → class_exists?
              → is_subclass_of(MachineOutput) → fromContext() → typed DTO (NEW)
              → is_subclass_of(OutputBehavior) → container resolve → __invoke() → mixed
              → neither                        → InvalidOutputDefinitionException
            → is_array                         → filter context keys → array
            → is Closure                       → invoke → mixed
```

Behavior registry lookup runs first — inline keys like `'output' => 'paymentStepOutput'` (resolved from `behavior['outputs']`) must work before class hierarchy checks.

Both paths serve both purposes:
- **Delegation:** `resolveChildOutput()` resolves to typed DTO or plain array → passed in `ChildMachineDoneEvent`
- **HTTP:** `$machine->output()` resolves the same way → serialized in response envelope

When `resolveChildOutput()` returns a `MachineOutput` instance, the framework injects it into the parent's `@done` action by type-hint. When it returns an array, the parent accesses it via `$event->output()` as before.

---

## v8-Era Naming Cleanup (v9 Clean Break)

v9 unified everything under `output`, but two job interfaces kept v8 naming. This spec completes the rename:

| v8 (removed in v9) | v9 (this spec) | Reason |
|---------------------|----------------|--------|
| `ReturnsResult` + `result()` | `ReturnsOutput` + `output()` | Aligns with v9 output terminology |
| `ProvidesFailureContext` + `failureContext()` | `ProvidesFailure` + `failure()` | Clean, typed, consistent |
| `with` key | `input` key | Typed contract support; `with` removed |

---

## Terminology: `failure` vs `@fail`

| Scenario | Routing | Typed contract |
|----------|---------|----------------|
| Child reaches a **failed final state** (e.g., `rejected`) | `@done.rejected` | Typed `MachineOutput` subclass (same as any other output) |
| Child **throws an exception** (unhandled error) | `@fail` | Typed `MachineFailure` subclass |

**Key distinction:**
- Failed final state = controlled termination → routed via `@done.{finalState}` with typed output
- Exception = uncontrolled termination → routed via `@fail` with typed failure
- Same `MachineFailure` base class used for both machine and job delegation

## Design

### 1. MachineInput — Parent → Child Contract

A typed DTO that defines what data a child requires from its parent. **Consumed at the delegation boundary** — like a function parameter, not a runtime object. The DTO validates and merges into context, then is gone.

The child machine owns its input class. The parent references it.

```php
class PaymentInput extends MachineInput
{
    public function __construct(
        public readonly string $orderId,
        public readonly int $amount,
        public readonly string $currency = 'TRY',
    ) {}
}
```

**Parent-side usage (delegation config):**

The `input` key replaces `with` on delegation config (both machine and job):

```php
// Machine delegation
'delegating' => [
    'machine' => PaymentMachine::class,
    // Option A: auto-resolve from parent context (constructor param names = context keys)
    'input'   => PaymentInput::class,
    // Option B: closure adapter for name mismatches between parent context and child input
    'input'   => function (ContextManager $ctx): PaymentInput {
        return new PaymentInput(
            orderId: $ctx->get('currentOrderId'),   // Parent has different key name
            amount: $ctx->get('totalAmount'),        // Parent has different key name
        );
    },
],

// Job delegation — same syntax
'processing' => [
    'job'   => ProcessPaymentJob::class,
    'input' => PaymentInput::class,
],
```

**Resolution rules:**
- `PaymentInput::class` → auto-resolve via `MachineInput::fromContext()`: constructor param names match camelCase context keys directly
- `Closure` → parent-side adapter for name mapping, must return the declared input type
- `array` → untyped key mapping, same semantics as the old `with` key (e.g., `['orderId']` or `['amount' => 'totalAmount']`)

**Machine vs job — internal resolution differs:**

The `input` key provides the same DX for both, but the framework handles the resolved data differently:
- **Machine delegation:** `fromContext()` → DTO → `toArray()` → merged into child's initial context
- **Job delegation:** `fromContext()` → DTO → `toArray()` → passed as constructor params via `app()->make($jobClass, $data)`

**Child-side declaration (machine config):**

```php
MachineDefinition::define(
    config: [
        'id'      => 'payment',
        'input'   => PaymentInput::class,   // Declares expected input type
        'initial' => 'processing',
        'context' => [
            'paymentId' => null,
            'status'    => 'pending',
        ],
    ],
);
```

When the child starts, the framework validates the provided input against the declared type. Mismatches throw `MachineInputValidationException` at invocation time.

**Input lifecycle — consumed at boundary:**

1. **Created** in parent context: `ChildMachineJob` (async) or `handleMachineInvoke()` (sync) resolves input (auto, closure, or array) before starting child
2. **Validated** against child's declared `input` type
3. **Merged into context:** Input properties are auto-merged into child's initial context (camelCase keys)
4. **Consumed:** The DTO is not stored on State, not injectable in actions. Data lives in context.

After merge, input data is accessed via `$ctx->get('orderId')` — same as any other context value. The DTO's value was at the boundary: IDE autocomplete for the parent, validation for the child, documentation as code.

**Why not inject the DTO in actions?**

If input were injectable AND merged into context, two truth sources emerge:

```php
// Dangerous: action modifies context, but DTO still has old value
$ctx->set('amount', $input->amount * 100);  // context: 15000
$input->amount;                               // still 150 — which is correct?
```

Single truth source (context) prevents this confusion.

### 2. MachineOutput — Child → Parent Contract (extends v9 output)

A typed DTO that defines what data a machine produces. Plugs into v9's existing `output` type dispatch as a new case. Works on any state (not just final), serves both delegation and HTTP responses.

```php
class PaymentOutput extends MachineOutput
{
    public function __construct(
        public readonly string $paymentId,
        public readonly string $status,
        public readonly ?string $transactionRef = null,
    ) {}
}
```

**Child-side: state config (any state, not just final):**

`MachineOutput` subclasses work anywhere the existing `output` key works:

```php
'completed' => [
    'type'   => 'final',
    // New: MachineOutput class reference (auto-resolve from context)
    'output' => PaymentOutput::class,
    // Existing: OutputBehavior class (still works)
    'output' => PaymentStatusOutput::class,
    // Existing: closure (still works, can return MachineOutput)
    'output' => function (ContextManager $ctx): PaymentOutput {
        return new PaymentOutput(
            paymentId: $ctx->get('paymentId'),
            status: 'success',
            transactionRef: $ctx->get('ref'),
        );
    },
    // Existing: array filter (still works)
    'output' => ['paymentId', 'status'],
],
```

**How type dispatch differentiates:**

When the `output` value is a string:
1. Check behavior registry (`behavior['outputs'][$key]`) → resolve registered behavior (existing v9 inline key support)
2. Check `is_subclass_of($class, MachineOutput::class)` → call `$class::fromContext($context)` → returns typed DTO
3. Check `is_subclass_of($class, OutputBehavior::class)` → container resolve → `__invoke()` → returns mixed
4. Neither → `InvalidOutputDefinitionException`

**Parent-side: @done action receives typed output:**

```php
'@done' => [
    'target'  => 'payment_received',
    'actions' => function (ContextManager $ctx, PaymentOutput $output): void {
        $ctx->set('paymentId', $output->paymentId);
        $ctx->set('transactionRef', $output->transactionRef);
    },
],
```

`MachineOutput` is injected via parameter injection. The framework resolves it from `ChildMachineDoneEvent` payload.

**Composition with OutputBehavior:**

When output needs computation beyond direct context mapping, `OutputBehavior` can return a `MachineOutput` instance. If the framework detects `instanceof MachineOutput` on the return value, typed injection is available in parent @done actions:

```php
// Direct MachineOutput — auto-resolve, no computation
'output' => PaymentOutput::class,

// OutputBehavior returning MachineOutput — computation + typed result
'output' => ComputedPaymentOutput::class,

class ComputedPaymentOutput extends OutputBehavior
{
    public function __invoke(ContextManager $ctx): PaymentOutput
    {
        return new PaymentOutput(
            paymentId: $ctx->get('paymentId'),
            status: $this->computeFinalStatus($ctx),  // computation
        );
    }
}
```

**HTTP serialization:**

When `$machine->output()` resolves to a `MachineOutput` instance, the response envelope calls `$output->toArray()`:

```json
{
    "data": {
        "id": "01JARX...",
        "state": ["completed"],
        "output": { "paymentId": "pay_123", "status": "success", "transactionRef": "ref_456" }
    }
}
```

### 3. MachineFailure — Child → Parent Error Contract

A typed DTO for structured error data when a child throws an exception (uncontrolled termination). Used for **both** machine delegation and job delegation.

```php
class PaymentFailure extends MachineFailure
{
    public function __construct(
        public readonly string $errorCode,
        public readonly string $message,
        public readonly ?string $gatewayResponse = null,
    ) {}
}
```

**Machine-side declaration (machine config):**

```php
MachineDefinition::define(
    config: [
        'id'      => 'payment',
        'input'   => PaymentInput::class,
        'failure' => PaymentFailure::class,
        'initial' => 'processing',
        // ...
    ],
);
```

**Job-side declaration (interface):**

```php
class ProcessPaymentJob implements ReturnsOutput, ProvidesFailure
{
    public static function failure(Throwable $e): PaymentFailure
    {
        return PaymentFailure::fromException($e);
    }
}
```

**Resolution:** When the child throws, `MachineFailure::fromException()` maps the exception to a typed failure object. The parent's `@fail` action receives it via injection — regardless of whether the child was a machine or a job:

```php
'@fail' => [
    'target'  => 'payment_failed',
    'actions' => function (ContextManager $ctx, PaymentFailure $failure): void {
        $ctx->set('errorCode', $failure->errorCode);
        $ctx->set('errorDetail', $failure->gatewayResponse);
    },
],
```

**`fromException` sensible default:**

```php
abstract class MachineFailure
{
    private const THROWABLE_GETTERS = [
        'message'  => 'getMessage',
        'code'     => 'getCode',
        'previous' => 'getPrevious',
    ];

    public static function fromException(Throwable $e): static
    {
        $reflection = new ReflectionClass(static::class);
        $params     = [];

        foreach ($reflection->getConstructor()->getParameters() as $param) {
            $name = $param->getName();

            if (isset(self::THROWABLE_GETTERS[$name])) {
                $params[$name] = $e->{self::THROWABLE_GETTERS[$name]}();
            } elseif ($param->isDefaultValueAvailable()) {
                $params[$name] = $param->getDefaultValue();
            } elseif ($param->getType()?->allowsNull()) {
                $params[$name] = null;
            } else {
                throw new MachineFailureResolutionException(static::class, $name);
            }
        }

        return new static(...$params);
    }
}
```

Simple cases (only `$message`, `$code`) need no override. Domain-specific fields (`$errorCode`, `$gatewayResponse`) that can't be resolved from Throwable throw `MachineFailureResolutionException` — the developer overrides `fromException()`.

**`failure` key is optional.** Machines without a `failure` declaration and jobs that don't implement `ProvidesFailure` behave as before — the parent's `@fail` action receives raw exception data via `ChildMachineFailEvent`/`ChildJobFailEvent`, with no typed injection.

### 4. Per-Final-State Output (Discriminated Outputs)

Different final states can produce different typed outputs:

```php
'states' => [
    'approved' => [
        'type'   => 'final',
        'output' => ApprovalOutput::class,
    ],
    'rejected' => [
        'type'   => 'final',
        'output' => RejectionOutput::class,
    ],
],
```

Parent uses `@done.{finalState}` routing with per-state typed injection:

```php
'@done.approved' => [
    'target'  => 'completed',
    'actions' => function (ContextManager $ctx, ApprovalOutput $output): void {
        $ctx->set('approvalId', $output->approvalId);
    },
],
'@done.rejected' => [
    'target'  => 'under_review',
    'actions' => function (ContextManager $ctx, RejectionOutput $output): void {
        $ctx->set('rejectionReason', $output->reason);
    },
],
```

**`@done` catch-all and typed injection:**

Typed `MachineOutput` injection requires `@done.{finalState}` routing. When using a `@done` catch-all with multiple final states producing different output types, the framework cannot determine which concrete type to inject — the action receives output via `ChildMachineDoneEvent::output()` (untyped) as before.

This is a machine design choice: if you want typed output, use per-state routing. If you want a catch-all, accept untyped output.

For cases where a shared typed interface is desired across final states, users can define a common interface:

```php
interface PaymentResult
{
    public function isApproved(): bool;
}

class ApprovalOutput extends MachineOutput implements PaymentResult { /* ... */ }
class RejectionOutput extends MachineOutput implements PaymentResult { /* ... */ }

// Catch-all with shared interface
'@done' => [
    'actions' => function (ContextManager $ctx, PaymentResult $output): void {
        // Shared interface — works for both output types
    },
],
```

This pattern is user-defined — the framework provides `MachineOutput` as the base, users compose with their own interfaces as needed.

### 5. Job Delegation — Typed Contracts

Job actors gain typed input, output, and failure — same parent DX as machine delegation.

**v9 interface renames (clean break):**

```php
// ReturnsResult → ReturnsOutput
interface ReturnsOutput
{
    public function output(): array|MachineOutput;
}

// ProvidesFailureContext → ProvidesFailure
interface ProvidesFailure
{
    public static function failure(Throwable $exception): MachineFailure;
}
```

**Full typed job example:**

```php
class ProcessPaymentJob implements ReturnsOutput, ProvidesFailure
{
    public function __construct(
        private string $orderId,
        private int $amount,
        private PaymentGateway $gateway,  // Laravel DI
    ) {}

    public function handle(): void
    {
        $this->result = $this->gateway->charge($this->orderId, $this->amount);
    }

    public function output(): PaymentOutput
    {
        return new PaymentOutput(
            paymentId: $this->result->id,
            status: 'success',
        );
    }

    public static function failure(Throwable $e): PaymentFailure
    {
        return PaymentFailure::fromException($e);
    }
}
```

**Untyped still works (simple jobs):**

```php
class SimpleJob implements ReturnsOutput
{
    public function output(): array
    {
        return ['done' => true];
    }
}
```

**Framework resolution in ChildJobJob:**

```php
// ChildJobJob::handle()
$output = $job instanceof ReturnsOutput ? $job->output() : [];

// ChildJobJob::failed()
if (is_a($this->jobClass, ProvidesFailure::class, true)) {
    $failure = $this->jobClass::failure($exception);
}
```

### 7. Forward Endpoint Integration — ForwardContext Removed

`ForwardContext` was a v7-era mechanism for accessing child machine internals (raw context + state) when building forward endpoint HTTP responses. With v9's state-level output on any state and this spec's `MachineOutput`, ForwardContext becomes redundant — it was an encapsulation leak that typed output replaces.

**What changes:**
- `ForwardContext` class removed
- `ForwardContext` injection removed from `InvokableBehavior`
- Forward endpoint responses use child's `$machine->output()` (which resolves the child's current state output — typed `MachineOutput` or existing formats)
- Parent's `OutputBehavior` on forward config can inject child's `MachineOutput` by type-hint
- `forward` keyword itself unchanged (still auto-generates HTTP routes for child events)

**Before (ForwardContext — raw access to child internals):**

```php
'forward' => [
    ProvideCardEvent::class => [
        'output' => PaymentStepOutput::class,
    ],
],

class PaymentStepOutput extends OutputBehavior
{
    public function __invoke(ContextManager $ctx, ForwardContext $fwd): array
    {
        return [
            'orderId'   => $ctx->get('orderId'),
            'cardLast4' => $fwd->childContext->get('cardLast4'),  // Raw child internal
            'childStep' => $fwd->childState->value[0],            // Raw state metadata
        ];
    }
}
```

**After (MachineOutput injection — typed, encapsulated):**

```php
'forward' => [
    ProvideCardEvent::class => [
        'output' => PaymentStepOutput::class,
    ],
],

class PaymentStepOutput extends OutputBehavior
{
    public function __invoke(ContextManager $ctx, VerifyingOutput $childOutput): array
    {
        return [
            'orderId'   => $ctx->get('orderId'),
            'cardLast4' => $childOutput->cardLast4,   // Typed child output
            'step'      => $childOutput->step,         // Child exposes what it wants
        ];
    }
}
```

**Simplest case — no custom output on forward config:**

```php
'forward' => ['PROVIDE_CARD'],
// Response uses child's $machine->output() directly
```

The child controls what it exposes via its state-level `output` definition. The parent no longer reaches into child internals.

**When child's state has no MachineOutput defined:**

If a parent's forward endpoint `OutputBehavior` type-hints a `MachineOutput` subclass but the child's current state doesn't define one, the framework throws `MachineOutputInjectionException`:

```
MachineOutputInjectionException: PaymentStepOutput expects VerifyingOutput
but PaymentMachine in state 'verifying' has no MachineOutput defined.
Define 'output' => VerifyingOutput::class on the 'verifying' state.
```

This is a developer error — a contract mismatch between parent and child. `machine:validate` can catch this at definition time.

For the simplest case (no custom OutputBehavior on forward config), no typed injection occurs — the child's `$machine->output()` fallback (`toResponseArray()`) is used as the response. No exception.

The OutputBehavior + MachineOutput composition pattern (Section 2) applies here too — OutputBehavior can receive child's `MachineOutput` via injection and return a computed `MachineOutput`.

### 8. XState v5 Alignment

| XState v5 | EventMachine (proposed) | Notes |
|-----------|------------------------|-------|
| `invoke.input` | `'input' => PaymentInput::class` | Closure or class |
| `invoke.output` | `'output' => PaymentOutput::class` | On any state (v9), typed via MachineOutput |
| `types.input` | `PaymentInput` class | PHP class = TypeScript type |
| `types.output` | `PaymentOutput` class | PHP class = TypeScript type |
| `actor.provide` | N/A (context injection) | EventMachine uses DI |

**XState export (`machine:xstate`)** should map typed contracts to JSON schema:
- `PaymentInput` → `{ type: 'PaymentInput', properties: { orderId: 'string', amount: 'number', currency: 'string' } }`
- PHP type → JSON schema type mapping: `string` → `string`, `int` → `number`, `float` → `number`, `bool` → `boolean`, `array` → `array`, `?T` → `{ oneOf: [T, 'null'] }`

### 9. Validation

**Definition-time validation (`machine:validate`):**
- Parent `input => FooInput::class` vs child `input => BarInput::class` → type mismatch error
- Parent `input => FooInput::class` (auto-resolve) but parent context keys don't cover FooInput's required constructor params → warning
- `@done` action type-hinting `PaymentOutput` but child's final state producing `DifferentOutput` → type mismatch error
- `output` value that is neither registered behavior key, `MachineOutput` subclass, `OutputBehavior` subclass, array, nor Closure → `InvalidOutputDefinitionException`

**Runtime validation:**
- Input is validated when child is invoked (`ChildMachineJob`): provided input must match child's declared input type
- Output is validated when child reaches final state (`ChildMachineCompletionJob`): if child declares typed output, resolved value must be `MachineOutput` instance
- Mismatches throw typed exceptions:
  ```
  MachineInputValidationException: PaymentMachine input validation failed:
    Missing required field 'amount' — parent context has: [orderId, customerName]
  ```

### 10. Base Classes

```php
abstract class MachineInput
{
    /**
     * Auto-resolve from context: constructor param names match camelCase context keys directly.
     * Required params without matching context key throw MachineInputValidationException.
     * Optional/nullable params fall back to default value or null.
     */
    public static function fromContext(ContextManager $context): static
    {
        $reflection = new ReflectionClass(static::class);
        $params     = [];

        foreach ($reflection->getConstructor()->getParameters() as $param) {
            $key   = $param->getName();
            $value = $context->get($key);

            if ($value !== null) {
                $params[$key] = $value;
            } elseif ($param->isDefaultValueAvailable()) {
                $params[$key] = $param->getDefaultValue();
            } elseif ($param->getType()?->allowsNull()) {
                $params[$key] = null;
            } else {
                throw new MachineInputValidationException(
                    static::class,
                    $key,
                    array_keys($context->toArray()),
                );
            }
        }

        return new static(...$params);
    }

    /** Serialize to array for context merging. */
    public function toArray(): array
    {
        $result     = [];
        $reflection = new ReflectionClass(static::class);

        foreach ($reflection->getConstructor()->getParameters() as $param) {
            $result[$param->getName()] = $this->{$param->getName()};
        }

        return $result;
    }
}

abstract class MachineOutput
{
    /**
     * Resolve from context (same reflection pattern as MachineInput::fromContext).
     * Throws MachineOutputResolutionException if a required param is missing from context.
     */
    public static function fromContext(ContextManager $context): static { /* same pattern, different exception */ }

    /** Serialize for ChildMachineDoneEvent payload and HTTP response envelope. */
    public function toArray(): array { /* same as MachineInput::toArray */ }
}

abstract class MachineFailure
{
    private const THROWABLE_GETTERS = [
        'message'  => 'getMessage',
        'code'     => 'getCode',
        'previous' => 'getPrevious',
    ];

    /**
     * Sensible default: maps constructor params to Throwable properties.
     * $message → getMessage(), $code → getCode(), $previous → getPrevious().
     * Unknown params: nullable → null, has default → default, required → MachineFailureResolutionException.
     * Override for domain-specific exception mapping.
     */
    public static function fromException(Throwable $e): static
    {
        $reflection = new ReflectionClass(static::class);
        $params     = [];

        foreach ($reflection->getConstructor()->getParameters() as $param) {
            $name = $param->getName();

            if (isset(self::THROWABLE_GETTERS[$name])) {
                $params[$name] = $e->{self::THROWABLE_GETTERS[$name]}();
            } elseif ($param->isDefaultValueAvailable()) {
                $params[$name] = $param->getDefaultValue();
            } elseif ($param->getType()?->allowsNull()) {
                $params[$name] = null;
            } else {
                throw new MachineFailureResolutionException(static::class, $name);
            }
        }

        return new static(...$params);
    }

    /** Serialize for ChildMachineFailEvent payload. */
    public function toArray(): array { /* same as MachineInput::toArray */ }
}
```

### 11. Serialization

Typed contracts travel through Laravel queues via `ChildMachineCompletionJob`. The framework serializes DTOs to arrays for the queue and reconstructs them on the parent side:

- **Outbound (child → queue):** `$output->toArray()` + output class FQCN stored in completion job
- **Inbound (queue → parent):** Framework reconstructs from stored class + array data, making the typed instance available for injection in parent @done/@fail actions
- **Fallback:** If output class is not a `MachineOutput` subclass (e.g., untyped job returning array), the array is passed as-is — no reconstruction, no typed injection

### 12. Testing

`Machine::fake()` and `simulateChildDone`/`simulateChildFail`/`simulateChildTimeout` support typed contracts:

```php
// Machine::fake() — typed output
PaymentMachine::fake(output: new PaymentOutput(paymentId: 'pay_123', status: 'success'));

// Machine::fake() — untyped still works
PaymentMachine::fake(output: ['paymentId' => 'pay_123']);

// simulateChildDone — typed output
$machine->simulateChildDone(
    output: new PaymentOutput(paymentId: 'pay_123', status: 'success'),
    finalState: 'completed',
);

// simulateChildFail — typed failure
$machine->simulateChildFail(
    failure: new PaymentFailure(errorCode: 'DECLINED', message: 'Card declined'),
);

// Assertions
PaymentMachine::assertInvokedWithInput(PaymentInput::class);
PaymentMachine::assertInvokedWithInput(new PaymentInput(orderId: 'ORD-1', amount: 150));
```

The `result:` parameter on `Machine::fake()` is renamed to `output:` (v9 terminology alignment).

### Typed Contracts — `with` → `input`, `MachineInput`/`MachineOutput`/`MachineFailure`

v9 introduces typed contracts for delegation boundaries. Machines and jobs can declare what data they expect (input), produce (output), and how their exceptions map to structured errors (failure).

**`with` → `input`:**

| Before (v8) | After (v9) | Effect |
|-------------|------------|--------|
| `'with' => ['orderId', 'amount']` | `'input' => ['orderId', 'amount']` | Untyped key mapping (renamed) |
| `'with' => ['amount' => 'totalAmount']` | `'input' => ['amount' => 'totalAmount']` | Key rename mapping (renamed) |
| N/A | `'input' => PaymentInput::class` | Typed: auto-resolve from parent context |
| N/A | `'input' => fn(ContextManager $ctx) => new PaymentInput(...)` | Typed: closure adapter |

**New machine config keys:**

```php ignore
// BEFORE (v8) — no contract declaration
MachineDefinition::define(config: [
    'id'      => 'payment',
    'initial' => 'processing',
    'context' => ['paymentId' => null],
    // ...
]);

// AFTER (v9) — optional typed contracts
MachineDefinition::define(config: [
    'id'      => 'payment',
    'input'   => PaymentInput::class,    // NEW: declares expected input
    'failure' => PaymentFailure::class,  // NEW: maps exceptions to typed failures
    'initial' => 'processing',
    'context' => ['paymentId' => null],
    // ...
]);
```

**Typed output on states via `MachineOutput`:**

```php ignore
// BEFORE (v8) — untyped array filter
'completed' => [
    'type'   => 'final',
    'output' => ['paymentId', 'status'],
],

// AFTER (v9) — typed DTO (auto-resolved from context)
'completed' => [
    'type'   => 'final',
    'output' => PaymentOutput::class,   // extends MachineOutput
],

// Array and OutputBehavior formats still work — MachineOutput is a new option, not a replacement.
```

**Typed output in parent @done/@fail actions:**

```php ignore
// BEFORE (v8) — untyped event access
'@done' => [
    'target'  => 'shipped',
    'actions' => function (ContextManager $ctx, ChildMachineDoneEvent $event): void {
        $ctx->set('paymentId', $event->output('paymentId'));
    },
],

// AFTER (v9) — typed injection
'@done' => [
    'target'  => 'shipped',
    'actions' => function (ContextManager $ctx, PaymentOutput $output): void {
        $ctx->set('paymentId', $output->paymentId);  // IDE autocomplete
    },
],
```

### Job Interface Renames

| Before (v8) | After (v9) |
|-------------|------------|
| `ReturnsResult` | `ReturnsOutput` |
| `result()` | `output()` |
| `ProvidesFailureContext` | `ProvidesFailure` |
| `failureContext(Throwable): array` | `failure(Throwable): MachineFailure` |

```php ignore
// BEFORE (v8)
class ProcessPaymentJob implements ReturnsResult, ProvidesFailureContext
{
    public function result(): array
    {
        return ['paymentId' => $this->paymentId];
    }

    public static function failureContext(Throwable $e): array
    {
        return ['errorCode' => 'GATEWAY_ERROR', 'message' => $e->getMessage()];
    }
}

// AFTER (v9) — typed
class ProcessPaymentJob implements ReturnsOutput, ProvidesFailure
{
    public function output(): PaymentOutput
    {
        return new PaymentOutput(paymentId: $this->paymentId, status: 'success');
    }

    public static function failure(Throwable $e): PaymentFailure
    {
        return PaymentFailure::fromException($e);
    }
}

// AFTER (v9) — untyped still works for output
class SimpleJob implements ReturnsOutput
{
    public function output(): array
    {
        return ['done' => true];
    }
}
```

### ForwardContext Removed

`ForwardContext` is removed. Forward endpoint `OutputBehavior` classes now inject child's `MachineOutput` by type-hint instead of accessing raw child internals.

```php ignore
// BEFORE (v8) — ForwardContext raw access
class PaymentStepOutput extends OutputBehavior
{
    public function __invoke(ContextManager $ctx, ForwardContext $fwd): array
    {
        return [
            'orderId'   => $ctx->get('orderId'),
            'cardLast4' => $fwd->childContext->get('cardLast4'),
        ];
    }
}

// AFTER (v9) — typed child output injection
class PaymentStepOutput extends OutputBehavior
{
    public function __invoke(ContextManager $ctx, VerifyingOutput $childOutput): array
    {
        return [
            'orderId'   => $ctx->get('orderId'),
            'cardLast4' => $childOutput->cardLast4,
        ];
    }
}
```

Forward endpoints without custom `OutputBehavior` use child's `$machine->output()` directly — no migration needed.

### Machine::fake() Parameter Rename

| Before (v8) | After (v9) |
|-------------|------------|
| `Machine::fake(result: [...])` | `Machine::fake(output: [...])` |
| `Machine::fake(result: new PaymentOutput(...))` | `Machine::fake(output: new PaymentOutput(...))` |

### New Base Classes

| Class | Purpose | Factory |
|-------|---------|---------|
| `MachineInput` | Parent → child data contract | `fromContext(ContextManager): static` |
| `MachineOutput` | Child → parent data contract | `fromContext(ContextManager): static` |
| `MachineFailure` | Exception → structured error | `fromException(Throwable): static` |

All three are abstract classes with `readonly` constructor properties and `toArray()` serialization. Subclass them with your domain-specific fields.

### New Exceptions

| Exception | When |
|-----------|------|
| `MachineInputValidationException` | `MachineInput::fromContext()` can't resolve a required constructor param |
| `MachineOutputResolutionException` | `MachineOutput::fromContext()` can't resolve a required constructor param |
| `MachineOutputInjectionException` | Forward endpoint `OutputBehavior` type-hints `MachineOutput` but child state has none |
| `MachineFailureResolutionException` | `MachineFailure::fromException()` can't resolve a required constructor param |

### Migration Checklist (typed contracts additions)

1. Rename `'with' =>` to `'input' =>` in all delegation configs (array format works as-is)
2. Rename `ReturnsResult` → `ReturnsOutput`, `result()` → `output()` in job actors
3. Rename `ProvidesFailureContext` → `ProvidesFailure`, `failureContext()` → `failure()` in job actors — return type changes from `array` to `MachineFailure`
4. Replace `ForwardContext` type-hints in forward endpoint `OutputBehavior` classes with child's `MachineOutput` type-hint
5. Optionally: define `MachineInput`/`MachineOutput`/`MachineFailure` subclasses for typed delegation contracts
6. Optionally: add `'input' => MyInput::class` and `'failure' => MyFailure::class` to machine configs
7. Optionally: replace array `'output'` on states with `MachineOutput` subclasses
8. Run `composer quality`

## Non-Goals

- Changing how `EventBehavior` works (events are already typed)
- Replacing `ContextManager` (context remains the internal state store)
- Making `sendTo`/`dispatchTo`/`dispatchToParent` typed beyond `EventBehavior` (events ARE the contract)
- Changing `OutputBehavior` or `$machine->output()` (this spec extends the dispatch, not replaces it)
- Renaming `forward` keyword (discussed and explicitly decided against)
- Runtime schema evolution or versioning (future concern)
