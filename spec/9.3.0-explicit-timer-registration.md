# Explicit Timer Registration — Eliminate Boot-Time File Scanning

**Status:** Draft
**Date:** 2026-03-31

## Problem

`MachineServiceProvider::boot()` registers timer sweeps by scanning **every PHP file** in `app/` using PhpParser to find Machine subclasses with timer transitions. In a project with 4104 PHP files, this takes ~3.7 seconds per application boot.

PHPUnit boots the application per test. With 91 Findeks machine tests:

```
91 tests × 3.7s discovery = 5.6 minutes overhead (actual test execution: ~2s)
```

### Root Cause

The discovery exists because timer registration is "auto-magic" — define an `@after` or `@every` timer in your machine config and it "just works." But this magic requires knowing which Machine classes exist at boot time, which requires scanning the entire codebase.

Meanwhile, the two other registration systems are explicit:

| System | Registration | File Scanning |
|--------|-------------|--------------|
| **Endpoints** | `MachineRouter::register(OrderMachine::class, [...])` in `routes/api.php` | None |
| **Schedules** | `MachineScheduler::register(OrderMachine::class, 'CHECK_EXPIRY')` in `routes/console.php` | None |
| **Timers** | Auto-discovered via PhpParser file scan | 4104 files parsed |

Timer registration is the architectural outlier.

### Secondary Problem

`machine:cache` and `machine:clear` commands exist **solely** to mitigate this discovery cost. They cache the list of timer machine classes to `bootstrap/cache/machines.php`. With explicit registration, these commands become unnecessary.

## Proposed Solution

Replace auto-discovery with explicit registration, following the `MachineRouter::register()` pattern.

### New API: `MachineTimer::register()`

Registration in `routes/console.php`, consistent with `MachineScheduler::register()`:

```php
// routes/console.php

use Tarfinlabs\EventMachine\Scheduling\MachineTimer;

// Register timer sweeps — everyMinute by default
MachineTimer::register(OrderMachine::class);
MachineTimer::register(FindeksMachine::class);

// Override resolution per machine — returns Laravel SchedulingEvent for fluent chaining
MachineTimer::register(BillingMachine::class)
    ->everyFiveMinutes()
    ->environments(['production', 'staging']);
```

`register()` applies `->everyMinute()->withoutOverlapping()->runInBackground()` as defaults and returns the `SchedulingEvent`. The caller can override the frequency — last call wins (standard Laravel `SchedulingEvent` behavior).

### Implementation

```php
// src/Scheduling/MachineTimer.php
class MachineTimer
{
    public static function register(string $machineClass): SchedulingEvent
    {
        /** @var Schedule $schedule */
        $schedule = resolve(Schedule::class);

        return $schedule->command("machine:process-timers --class={$machineClass}")
            ->everyMinute()
            ->withoutOverlapping()
            ->runInBackground();
    }
}
```

Default is `everyMinute()` — sensible for most timer use cases. Override by chaining a different frequency on the returned `SchedulingEvent`.

### What Gets Deleted

| Component | Reason |
|-----------|--------|
| `MachineServiceProvider::registerTimerSweeps()` | Replaced by explicit `MachineTimer::register()` |
| `MachineServiceProvider::boot()` override | Only existed for timer registration — `parent::boot()` is called automatically by `PackageServiceProvider` |
| `MachineDiscovery` class | Only consumer was `MachineCacheCommand` + `ServiceProvider` — both removed |
| `MachineCacheCommand` (`machine:cache`) | Existed solely for timer discovery cache |
| `MachineClearCommand` (`machine:clear`) | Existed solely to clear timer discovery cache |
| `config('machine.timers.resolution')` | User sets resolution explicitly via fluent API |
| `TimerResolution` enum | Only consumer was `registerTimerSweeps()` — no longer needed |

### What Stays

| Component | Reason |
|-----------|--------|
| `MachineClassVisitor` | Used by `MachineConfigValidatorCommand` (independent PhpParser scan) |
| `MachineDiscoveryException` | Used by `MachineConfigValidatorCommand` |
| `MachineConfigValidatorCommand` (`machine:validate`) | Independent feature, own file scanning logic |
| `config('machine.timers.batch_size')` | Runtime config for sweep command, unrelated to registration |
| `config('machine.timers.backpressure_threshold')` | Runtime config for sweep command, unrelated to registration |
| PhpParser dependency | Used by `machine:validate` |

### Migration

Before (v9.2):
```php
// Nothing needed — timers auto-discovered at boot
// (but costs 3.7s per boot without cache)

// Production required:
// php artisan machine:cache
```

After (v9.3):
```php
// routes/console.php
MachineTimer::register(OrderMachine::class);
MachineTimer::register(FindeksMachine::class);

// No machine:cache needed. No boot-time scanning. No cache to clear.
```

## Breaking Change

**Yes.** Machines with `@after` or `@every` timers must add `MachineTimer::register()` to `routes/console.php`. Without it, timer sweeps silently stop running.

### Mitigation

Upgrading guide must be very explicit:

1. Search machine definitions for `'after'` and `'every'` keys
2. For each machine with timers, add one `MachineTimer::register()` line
3. Remove `machine:cache` from CI/CD pipelines and deploy scripts
4. Remove `bootstrap/cache/machines.php` from production servers

## Testing Plan

### New Unit Tests — `MachineTimerTest.php`

Following `MachineSchedulerTest.php` pattern:

| # | Test | Assertion |
|---|------|-----------|
| 1 | `register returns a SchedulingEvent` | `->toBeInstanceOf(SchedulingEvent::class)` |
| 2 | `register sets correct command with --class` | Command string contains `machine:process-timers` and `--class='...'` |
| 3 | `register applies everyMinute as default` | Cron expression `* * * * *` |
| 4 | `register applies withoutOverlapping` | `$event->withoutOverlapping` is true |
| 5 | `register applies runInBackground` | `$event->runInBackground` is true |
| 6 | `register supports frequency override` | `->everyFiveMinutes()` → expression `*/5 * * * *` |
| 7 | `register supports environments chaining` | `->environments(['production'])` works |
| 8 | `multiple register calls create separate entries` | 2 machines → 2 scheduler events |
| 9 | `register with timer-less machine does not throw` | No exception — runtime no-op |

**Design decision:** `MachineTimer::register()` does NOT validate whether the machine has timers. It only registers a scheduler entry. Validation is the `machine:process-timers` command's responsibility at runtime. This keeps `register()` lightweight — no definition loading at boot time.
