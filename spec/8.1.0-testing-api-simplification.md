# API Simplification: Machine::test() as Single Entry Point

**Status:** Draft (reviewed)
**Author:** Claude
**Date:** 2026-03-22
**Scope:** Unify class-based testing entry points on Machine, remove TestMachine class-accepting factories

---

## Problem

3 entry points for the same thing:

```php
FindeksMachine::test()                                          // limited — no guards/faking
TestMachine::withContext(FindeksMachine::class, context: [...])  // full params but verbose
TestMachine::startingAt(FindeksMachine::class, stateId: '...')   // full params but verbose
```

Confusion: when to use which? Why does `Machine::test()` lack `guards:`/`faking:` but `TestMachine::withContext()` has them?

---

## Decision: Breaking Change in 8.5.0

No deprecation cycle. This is a testing-only change — no runtime impact, mechanical find-replace migration. Breaking change documented in release notes and upgrade guide.

---

## What Changes

| Before | After |
|--------|-------|
| `TestMachine::create(Class::class, context, guards)` | **@internal** → use `Class::test(context:, guards:)` |
| `TestMachine::withContext(Class::class, context, guards, faking)` | **@internal** → use `Class::test(context:, guards:, faking:)` |
| `TestMachine::startingAt(Class::class, stateId, context, guards, faking)` | **@internal** → use `Class::startingAt(stateId:, context:, guards:, faking:)` |
| `Machine::withContext(context)` | **Removed** → `Machine::test(context:)` |
| `Machine::test()` (no guards/faking) | **Updated** with full params |
| `TestMachine::define(config:, behavior:)` | **Unchanged** — inline machines |
| `TestMachine::for(Machine $machine)` | **Unchanged** — wrap existing instance |
| All TestMachine instance methods | **Unchanged** |

## Implementation

### Machine.php — Update `test()`, add `startingAt()`, remove `withContext()`

```php
/**
 * Create a TestMachine for fluent testing.
 *
 * Context is merged BEFORE initialization — entry actions see it.
 * Guards and faking are applied before getInitialState() runs,
 * solving @always timing issues.
 */
public static function test(
    array $context = [],
    array $guards = [],
    array $faking = [],
): TestMachine {
    return TestMachine::withContext(static::class, $context, $guards, $faking);
}

/**
 * Create a TestMachine at a specific state without running lifecycle.
 *
 * No entry actions, no @always, no job dispatch.
 * Uses the real definition — all transitions, guards, and actions available.
 */
public static function startingAt(
    string $stateId,
    array $context = [],
    array $guards = [],
    array $faking = [],
): TestMachine {
    return TestMachine::startingAt(static::class, $stateId, $context, $guards, $faking);
}

// Machine::withContext() → REMOVED
```

### TestMachine.php — Mark class-accepting factories as @internal

Keep `create()`, `withContext()`, `startingAt()` as `public` (Machine proxies delegate to them) but add `@internal` tag. Documentation only shows Machine entry points for class-based testing.

### `test()` Semantics

Current `Machine::test()` delegates to `TestMachine::create()` (post-init context). New `Machine::test()` delegates to `TestMachine::withContext()` (pre-init context).

**This is a behavior change:** context is now available to entry actions on init. This is what users expect — the old post-init behavior was a footgun.

### TestMachineV2Test.php — Update all class-based tests

Replace all `TestMachine::create(Class::class, ...)` and `TestMachine::withContext(Class::class, ...)` with `Class::test(...)`. Replace all `TestMachine::startingAt(Class::class, ...)` with `Class::startingAt(...)`.

`TestMachine::define(...)` tests stay unchanged.

## Migration Guide (for release notes + upgrade guide doc)

### Breaking Changes in 8.5.0

**Testing entry points simplified.** `Machine::test()` and `Machine::startingAt()` are now the only entry points for class-based machine testing.

| Before | After |
|--------|-------|
| `TestMachine::create(MyMachine::class)` | `MyMachine::test()` |
| `TestMachine::create(MyMachine::class, ['key' => 'val'])` | `MyMachine::test(context: ['key' => 'val'])` |
| `TestMachine::withContext(MyMachine::class, [...])` | `MyMachine::test(context: [...])` |
| `TestMachine::withContext(MyMachine::class, [...], guards: [...])` | `MyMachine::test(context: [...], guards: [...])` |
| `TestMachine::startingAt(MyMachine::class, 'state', [...])` | `MyMachine::startingAt('state', context: [...])` |
| `Machine::withContext([...])` | `Machine::test(context: [...])` |

**Behavior change:** `Machine::test(context: [...])` now merges context **before** initialization (entry actions see it). Previously `Machine::test()` applied context post-init.

**`TestMachine::define()` is unchanged** — use it for inline throwaway machines without a Machine class.

**Find-replace migration:**
```bash
# In your test files:
# TestMachine::create(XMachine::class, ...) → XMachine::test(...)
# TestMachine::withContext(XMachine::class, ...) → XMachine::test(context: ...)
# TestMachine::startingAt(XMachine::class, ...) → XMachine::startingAt(...)
```

### Upgrade Guide Doc

Add section to `docs/getting-started/upgrading.md`:

```markdown
## 8.5.0

### Testing Entry Point Simplification

`Machine::test()` and `Machine::startingAt()` are now the recommended entry points
for class-based machine testing. The `TestMachine` class-accepting static factories
(`create()`, `withContext()`, `startingAt()`) are marked `@internal`.

**Before:**
\```php
TestMachine::withContext(MyMachine::class, context: [...], guards: [...])
TestMachine::startingAt(MyMachine::class, stateId: 'polling', context: [...])
\```

**After:**
\```php
MyMachine::test(context: [...], guards: [...])
MyMachine::startingAt('polling', context: [...])
\```

`Machine::withContext()` has been removed. Use `Machine::test(context: [...])` instead.

`Machine::test()` now merges context before initialization (like the old `withContext()`).
Entry actions on the initial state see the injected context values.
```

## Additional: Gaps Found in Parallel/Delegation/Cross-Machine Testing

Audit of CarSalesMachine (2 parallel states, 2 child machine delegations, 4 forward endpoints) and docs revealed these gaps. None require new fluent methods — existing API covers the scenarios. Gaps are documentation + one minor assertion.

### Code: Add `assertNotDispatchedTo()`

Missing symmetry with `assertNotSentTo()`. One-liner in TestMachine:

```php
public function assertNotDispatchedTo(string $machineClass): self
{
    // Mirror of assertNotSentTo but for async dispatchTo
    Queue::assertNotPushed(SendToMachineJob::class, fn ($job) => $job->machineClass === $machineClass);
    return $this;
}
```

### Documentation: Parallel + Delegation Testing Recipe

`docs/testing/recipes.md` — No example of testing a machine with parallel regions that delegate to child machines. This is the most common real-world pattern (CarSalesMachine). Add recipe:

```markdown
## Recipe: Testing Parallel Regions with Child Delegation

Test a parent machine whose parallel regions delegate to child machines:

\```php
Queue::fake();

CarSalesMachine::test(
    context: [...],
    guards: [HasConsentGuard::class => true, IsFarmerNotEligibleGuard::class => false],
    faking: [StoreApplicationAction::class],
)
->withoutPersistence()
->fakingAllActions()
->fakingChild(FindeksMachine::class, result: [...], finalState: 'report_saved')
->fakingChild(TurmobMachine::class, result: [...])
->assertState('checking_protocol');
\```

Both child machines are faked — they complete immediately when the parent enters the
parallel `verification` state. The parent's `@done` guard fires and transitions
to `checking_protocol`.
```

### Documentation: `withRunningChild()` Persistence Warning

`docs/testing/test-machine.md` and `docs/testing/delegation-testing.md` — `withRunningChild()` silently fails with `withoutPersistence()`. Add explicit warning:

```markdown
::: warning withRunningChild() requires persistence
`withRunningChild()` creates a child record in the database. It does NOT work
with `withoutPersistence()` or `TestMachine::define()`. Use `Machine::test()`
(persistence enabled by default).
:::
```
