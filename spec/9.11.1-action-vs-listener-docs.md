# 9.11.1 — Action-vs-Listener semantics + wrapper Action recipe

## Motivation

After 9.11.0 shipped, a follow-up question surfaced: "if a queued listener can fire-and-forget,
why can't an entry action? Should we add a `'jobs'` slot alongside `'actions'`?"

The honest answer is: a queued *action* is structurally unsafe (sequence collisions, restored
state drift, no rollback on worker throws), while a queued *listener* is safe because it
observes a *committed* transition. EventMachine's listener concept is an **observer**, not a
behavior — and 9.11.0's `@queue` validation enforced the rule, but the docs never explained
*why* the rule has to exist.

This release closes that gap with documentation and skill updates only — no code changes.

## Change

- **`docs/building/defining-states.md`** — the "Listeners" section now opens with an explicit
  Action-vs-Listener semantic table (role, mutation, sequence-sensitivity, failure mode,
  async-safety) and an `::: info` callout that derives `@queue`'s scope from those semantics.
- **`docs/building/defining-states.md` → "Async Work in Entry Actions"** — the third option,
  previously called "Inline `dispatch()`", is now framed as **Wrapper Action** and given its own
  `::: tip` recipe with a `DispatchPromissoryNoteAction` example. The callout explicitly
  addresses "I don't want to add a state just to fire-and-forget a job" and explains why
  EventMachine doesn't expose a `'jobs'` slot.
- **`skills/event-machine/SKILL.md`** — new "Action vs Listener — behavior vs observer" section
  precedes the existing `@queue` gotcha. The Wrapper Action recipe is included verbatim so
  agents stop suggesting `'jobs'` slots or queued entry actions.

## Backward compatibility

None — pure documentation and skill update. No code, no API, no behavior change.

## Why ship as 9.11.1 (not just push docs)

The agent skill is materialized into `plugin-dist` only when a tag triggers
`Build Plugin Distribution`. A skill commit pushed *after* the 9.11.0 tag never reaches agents
until the next tag. Since the skill change is the load-bearing part of this update — agents
were the ones suggesting the wrong patterns — it has to ride a tag. 9.11.1 patch satisfies that
without implying any code change.
