# Scheduled Events: `schedules` Key + `MachineScheduler`

## Status
**Phase:** Design finalized, ready for implementation
**Version:** v7 (ships with time-based events)
**Date:** 2026-03-15
**Branch:** Create from `feature/time-based-events`

## Problem Statement

Backend machines need cron-scheduled batch operations:
- "Every day at 00:10, expire applications older than 7 days"
- "Every day at 11:00, send reminders for approved applications"
- "Every month on the 16th, calculate IPS prices"

Currently these are external jobs in `routes/console.php` that query models,
iterate records, and send events to machines. This scatters machine behavior
across files.

## Core Concept: Instance Scope

| Feature | Targets | How |
|---------|---------|-----|
| `endpoints` | **Single instance** | HTTP request includes root_event_id or model ID |
| `schedules` | **ALL instances in DB** | Cron triggers → resolver queries DB → batch dispatch to all matching instances |
| `after`/`every` | **Per-instance** | Timer fires for ONE instance based on its state_entered_at |

A schedule does NOT target a single machine. It triggers a database query
that finds ALL matching instances and dispatches events to each of them.

## Design Principle: Definition vs Registration

Same pattern as endpoints:

```
endpoints:  Definition declares WHAT  →  MachineRouter in routes/ registers HOW
schedules:  Definition declares WHAT  →  MachineScheduler in routes/console.php registers WHEN + HOW
```

**Machine Definition** — declares WHAT event and WHICH instances:

```php
MachineDefinition::define(
    config: [
        'on' => [
            'CHECK_EXPIRY' => ['target' => 'expired', 'guards' => 'isExpiredGuard'],
        ],
        'states' => [...],
    ],
    schedules: [
        'CHECK_EXPIRY' => ExpiredApplicationsResolver::class,
    ],
)
```

**routes/console.php** — registers WHEN and HOW (full Laravel Scheduler API):

```php
use Tarfinlabs\EventMachine\Scheduling\MachineScheduler;

MachineScheduler::register(ApplicationMachine::class, 'CHECK_EXPIRY')
    ->dailyAt('00:10')
    ->environments(['production', 'staging'])
    ->countries([Country::TR])
    ->sentryMonitor('expire-older-applications')
    ->onOneServer();
```

## Schedule Definition API

### `schedules` Key (Machine Definition)

Event type is the key. Value is the resolver (class, closure, or null).

```php
schedules: [
    // Class-based resolver (recommended)
    'CHECK_EXPIRY' => ExpiredApplicationsResolver::class,

    // Inline resolver (closure)
    'SEND_REMINDER' => fn () => Application::where('approved_at', '<=', now()->subDays(2))
                                            ->pluck('application_mre'),

    // No resolver — auto-detect states from idMap, send to all matching instances
    'DAILY_REPORT' => null,

    // EventBehavior FQCN as key
    CheckExpiryEvent::class => ExpiredApplicationsResolver::class,
],
```

### Schedule Entry Values

| Value | Type | Behavior |
|-------|------|----------|
| `ResolverClass::class` | `string` | Container-resolved, DI-supported, returns `Collection<string>` of root_event_ids |
| `fn () => ...` | `Closure` | Inline, returns `Collection<string>` of root_event_ids |
| `null` | `null` | Auto-detect: scan idMap for states handling event, query machine_current_states |

## Resolver: Instance Resolution

### Contract

A resolver is any callable that returns `Collection<string>` of root_event_ids.
The resolver owns the model knowledge (table, column, conditions).
EventMachine receives only root_event_ids — zero model awareness.

### Class-Based (Recommended)

```php
use Tarfinlabs\EventMachine\Contracts\ScheduleResolver;

class ExpiredApplicationsResolver implements ScheduleResolver
{
    public function __construct(
        private readonly ApplicationConfig $config,  // constructor DI
    ) {}

    public function __invoke(): \Illuminate\Support\Collection
    {
        return Application::query()
            ->where('created_at', '<=', now()->subDays($this->config->expiryDays))
            ->whereIn('status', [
                ApplicationStatus::APPROVED,
                ApplicationStatus::WAITING,
                ApplicationStatus::PENDING_REVIEW,
            ])
            ->pluck('application_mre');  // user knows their column name
    }
}
```

Naming convention: `{Description}Resolver` suffix.
Examples: `ExpiredApplicationsResolver`, `UnpaidOrdersResolver`, `ActiveSubscriptionsResolver`.

### Inline (Closure)

```php
'SEND_REMINDER' => fn () => Application::where('approved_at', '<=', now()->subDays(2))
                                        ->pluck('application_mre'),
```

### No Resolver (Auto-Detect)

```php
'DAILY_REPORT' => null,
```

Auto-detect: scan definition's `idMap` for states with `DAILY_REPORT` in `transitionDefinitions`.
If root-level `on` handler exists, send to ALL instances.
Query: `machine_current_states WHERE machine_class = X AND state_id IN (auto-detected)`.

### Safety Cross-Check

Resolver returns root_event_ids from a model query. Models may map to different
machine classes conditionally (e.g., ApplicationMachine vs ConsumerGoodsApplicationMachine).
The sweep cross-checks with `machine_current_states`:

```php
$rootEventIds = $resolver();

$validIds = MachineCurrentState::whereIn('root_event_id', $rootEventIds)
    ->where('machine_class', $machineClass)
    ->pluck('root_event_id');
```

## MachineScheduler: Registration API

### Basic Registration

```php
// routes/console.php
use Tarfinlabs\EventMachine\Scheduling\MachineScheduler;
```

### Full Laravel Scheduler API

```php
MachineScheduler::register(ApplicationMachine::class, 'CHECK_EXPIRY')
    ->dailyAt('00:10')
    ->environments(['production', 'staging'])   // Laravel native
    ->countries([Country::TR])                   // custom macro
    ->sentryMonitor('expire-older-applications') // 3rd party
    ->onOneServer()                              // distributed lock
    ->withoutOverlapping()                       // prevent concurrent
    ->runInBackground();                         // non-blocking
```


### MachineScheduler Implementation

```php
namespace Tarfinlabs\EventMachine\Scheduling;

use Illuminate\Console\Scheduling\Event as SchedulingEvent;

class MachineScheduler
{
    public static function register(
        string $machineClass,
        string $eventType,
    ): SchedulingEvent {
        $schedule = app(\Illuminate\Console\Scheduling\Schedule::class);

        return $schedule->command('machine:process-scheduled', [
            '--class' => $machineClass,
            '--event' => $eventType,
        ]);
    }
}
```

Returns Laravel's `SchedulingEvent` — ALL fluent methods available:
`->dailyAt()`, `->cron()`, `->environments()`, `->onOneServer()`,
`->withoutOverlapping()`, custom macros, etc.

## Architecture

### How It Works

```
Machine Definition                          routes/console.php
  schedules: [                              MachineScheduler::register(X, 'CHECK_EXPIRY')
    'CHECK_EXPIRY' =>                           ->dailyAt('00:10')
      ExpiredApplicationsResolver::class        ->environments(['production'])
  ]                                             ->onOneServer()
        |                                           |
        |  (defines WHAT + WHICH)                   |  (defines WHEN + HOW)
        v                                           v
ProcessScheduledCommand (--class=X --event=CHECK_EXPIRY)
  1. Load definition, find resolver for CHECK_EXPIRY
  2. Has resolver? → run it → get root_event_ids → cross-check machine_class
     No resolver?  → auto-detect states → query machine_current_states
  3. Dispatch SendToMachineJob for each valid instance (Bus::batch)
        |
        v
SendToMachineJob (existing)
  -> Restore machine -> send(event)
  -> Machine's on-transition handles it (guards, actions, transitions)
  -> Event bubbles up to root if child state doesn't handle it
```

### ProcessScheduledCommand

```php
protected $signature = 'machine:process-scheduled
    {--class= : Machine class FQCN (required)}
    {--event= : Event type to send (required)}';

public function handle(): int
{
    $machineClass = (string) $this->option('class');
    $eventType    = (string) $this->option('event');

    if ($machineClass === '' || $eventType === '') {
        $this->error('Both --class and --event are required.');
        return self::FAILURE;
    }

    try {
        $definition = $machineClass::definition();
    } catch (\Throwable $e) {
        $this->error("Failed: {$e->getMessage()}");
        return self::FAILURE;
    }

    // $definition->schedules is array<string, ScheduleDefinition> keyed by event type
    if (!array_key_exists($eventType, $definition->schedules ?? [])) {
        $this->warn("Event '{$eventType}' not found in schedules for {$machineClass}.");
        return self::FAILURE;
    }

    $scheduleDef = $definition->schedules[$eventType];
    $rootEventIds = $this->resolveInstances($scheduleDef, $definition, $machineClass, $eventType);

    if ($rootEventIds->isEmpty()) {
        return self::SUCCESS;
    }

    $jobs = $rootEventIds->map(fn ($id) => new SendToMachineJob(
        machineClass: $machineClass,
        rootEventId: $id,
        event: ['type' => $eventType],
    ));

    Bus::batch($jobs->toArray())
        ->name("schedule:{$machineClass}:{$eventType}")
        ->allowFailures()
        ->dispatch();

    return self::SUCCESS;
}

protected function resolveInstances(
    ScheduleDefinition $scheduleDef,
    MachineDefinition $definition,
    string $machineClass,
    string $eventType,
): Collection {
    // Priority 1: Resolver (class or closure) — model-level query
    if ($scheduleDef->hasResolver()) {
        $resolver = is_string($scheduleDef->resolver)
            ? app()->make($scheduleDef->resolver)
            : $scheduleDef->resolver;

        $rootEventIds = $resolver();

        // Safety cross-check: only keep IDs for this machine class
        return MachineCurrentState::whereIn('root_event_id', $rootEventIds)
            ->where('machine_class', $machineClass)
            ->pluck('root_event_id');
    }

    // Priority 2: Auto-detect from idMap — machine_current_states query
    $targetStates = $this->detectTargetStates($definition, $eventType);
    $isRootLevel  = isset($definition->root->transitionDefinitions[$eventType]);

    $query = DB::table('machine_current_states')
        ->where('machine_class', $machineClass);

    if (!$isRootLevel && $targetStates !== []) {
        $query->whereIn('state_id', $targetStates);
    }

    return $query->pluck('root_event_id');
}

/**
 * Find states that handle the given event type.
 * Root-level on = empty array (means all states via bubbling).
 */
protected function detectTargetStates(MachineDefinition $definition, string $eventType): array
{
    if (isset($definition->root->transitionDefinitions[$eventType])) {
        return []; // root handles it → all states
    }

    $states = [];
    foreach ($definition->idMap as $stateId => $stateDef) {
        if ($stateDef->transitionDefinitions !== null
            && isset($stateDef->transitionDefinitions[$eventType])) {
            $states[] = $stateId;
        }
    }

    return $states;
}
```

## Comparison with Endpoints

| Aspect | `endpoints` | `schedules` |
|--------|-----------|------------|
| Definition key | `endpoints: [...]` | `schedules: [...]` |
| Registration | `MachineRouter::register()` in routes | `MachineScheduler::register()` in routes/console.php |
| What definition declares | Event mapping | Event + resolver |
| What registration declares | Prefix, middleware, model | Cron, environments, macros |
| Instance scope | Single (by ID) | All matching (by resolver/auto-detect) |

## ScheduleResolver Interface

```php
namespace Tarfinlabs\EventMachine\Contracts;

/**
 * Resolves which machine instances should receive a scheduled event.
 *
 * Returns a Collection of root_event_ids. The resolver owns the model
 * knowledge — EventMachine only receives IDs.
 */
interface ScheduleResolver
{
    public function __invoke(): \Illuminate\Support\Collection;
}
```

Class-based resolvers SHOULD implement this interface for type safety and documentation.
Inline closures work without it (duck typing).

## ScheduleDefinition Value Object

```php
namespace Tarfinlabs\EventMachine\Definition;

class ScheduleDefinition
{
    public function __construct(
        public readonly string $eventType,
        public readonly string|\Closure|null $resolver = null,
    ) {}

    public static function fromConfig(string $key, string|\Closure|null $resolver): self
    {
        return new self(
            eventType: self::resolveEventType($key),
            resolver: $resolver,
        );
    }

    public function hasResolver(): bool { return $this->resolver !== null; }

    /**
     * Resolve event type from key — handles both string and EventBehavior FQCN.
     * Same logic as EndpointDefinition::resolveEventType().
     */
    private static function resolveEventType(string $key): string
    {
        if (is_subclass_of($key, \Tarfinlabs\EventMachine\Behavior\EventBehavior::class)) {
            return $key::getType();
        }

        return $key;
    }
}
```

Simpler than before: just event type + resolver. No cron (that's in registration).

**Important:** `parseSchedules()` normalizes FQCN keys to event type strings.
The `$definition->schedules` property stores `array<string, ScheduleDefinition>`
keyed by resolved event type (e.g., `'CHECK_EXPIRY'`, not `CheckExpiryEvent::class`).
This ensures `ProcessScheduledCommand` can lookup by `--event=CHECK_EXPIRY`.

## Resolved Design Questions

1. **Definition vs Registration.** Definition declares WHAT (event + resolver). Registration declares WHEN + HOW (cron, environments, macros). Symmetric with endpoints.

2. **No auto-registration.** Manual registration in routes/console.php gives full Laravel Scheduler API access (environments, custom macros, sentryMonitor, onOneServer). Auto-registration can't support this.

3. **Resolver, not scope.** `ScheduleResolver` is the behavior name. Suffix: `Resolver`. Not `Scope` (Eloquent concept), not `Guard`/`Action` (state machine concepts).

4. **Resolver returns root_event_ids.** The resolver owns model knowledge (table, column, conditions). EventMachine receives only IDs. Column name (`application_mre`, `order_mre`) is the resolver's responsibility.

5. **Safety cross-check.** Resolver results cross-checked with `machine_current_states.machine_class` to handle conditional machine mapping.

6. **Null resolver = auto-detect.** Scan idMap for states handling the event. Root-level `on` = all instances. No resolver needed for simple broadcast.

7. **No cron in definition.** Cron expression is infrastructure config (WHEN), not machine logic (WHAT). Lives in registration.

8. **No guards in schedules.** Guards stay on `on` transitions. Resolvers filter at DB level. Clean separation.

9. **`olderThan` and `machine_created_at` removed.** Resolver handles all filtering including time-based — queries the model's own `created_at`, `approved_at`, etc. No new columns needed on `machine_current_states`.

## Edge Cases

1. **Schedule registered but not in definition:** ProcessScheduledCommand finds no schedule entry → warning, nothing dispatched.
2. **Schedule in definition but not registered:** Never runs — no cron trigger.
3. **Same event in schedules AND after/every:** Independent, both valid.
4. **Resolver throws exception:** try/catch in command, log error, skip.
5. **Resolver returns IDs for wrong machine class:** Cross-check filters them out.
6. **No instances match:** Empty dispatch, success exit.
