## Page Details

### 1. `index.md` — Overview
- What are "best practices" in statechart design
- Quick reference card: one-line summary per topic with link
- "Do / Don't" visual table (condensed)
- Industry references: Harel, XState, UML, Rhapsody

### 2. `event-bubbling.md` — Event Bubbling
- **How it works:** `findTransitionDefinition()` walks `parent` chain from leaf → compound → root
- **Code walkthrough:** 3-level hierarchy, event handled at each level
- **Root-level `on` handlers:** Global event catchers via bubbling
- **Practical example:** order machine with `CANCEL` handled at root (cancels from any state)
- **Anti-pattern:** Unintended catch — parent catches event meant for a specific child
- **Anti-pattern:** Over-relying on bubbling — makes it unclear which state handles what
- **Best practice:** Be explicit about which level handles each event
- **Best practice:** Use root `on` only for truly global events (CANCEL, ABORT, TIMEOUT)
- **Diagram:** ASCII or mermaid showing event walking up hierarchy
- **Cross-reference:** handling-events.md, defining-states.md

### 3. `state-design.md` — State Design
- **States represent conditions, not actions** — the "is" test (from conventions)
- **State vs context decision:** "Does this value change available transitions?" → State. "Just carries data?" → Context
- **Anti-pattern: State explosion** — 3×4×2 = 24 states for 3 independent booleans
  - **Fix:** Use parallel states or context flags
- **Anti-pattern: Flag states** — `processing_with_priority` vs `processing_without_priority`
  - **Fix:** One `processing` state + `is_priority` context flag
- **Anti-pattern: Mirrored states** — `approved_by_manager` + `approved_by_director`
  - **Fix:** One `approved` state + `approved_by` context key
- **Best practice:** Use hierarchical states to share common transitions
- **Best practice:** Keep leaf states meaningful and minimal
- **Example:** Refactoring 12-state flat machine into 4-state hierarchical
- **Cross-reference:** defining-states.md, conventions.md (state naming)

### 4. `guard-design.md` — Guard Design
- **The purity rule:** Given same (state, event, context) → same result. Always.
- **No side effects:** No DB writes, no API calls, no file I/O, no logging
- **No context mutation:** Guards inspect, never modify (use calculators for that)
- **Anti-pattern: Guard with API call** — network failure changes guard behavior
  - **Fix:** Move API call to an action in a previous state, store result in context
- **Anti-pattern: Guard that writes to context** — order of guard evaluation matters
  - **Fix:** Use calculator behavior (runs before guards)
- **Anti-pattern: Time-dependent guard** — `return now() > $deadline`
  - **Fix:** Use `after` timer instead, or store deadline check result in context
- **Anti-pattern: Random guard** — `return random_int(0, 1)`
  - **Fix:** Never. Guards must be deterministic.
- **Best practice:** Keep guards as simple boolean expressions
- **Best practice:** Combine multiple guards with array syntax for readability
- **Best practice:** Use ValidationGuard for input validation (Laravel rules)
- **Example:** Refactoring impure guard into calculator + pure guard
- **Cross-reference:** guards.md, validation-guards.md, calculators.md

### 5. `action-design.md` — Action Design
- **Idempotency:** Actions should produce the same result if run twice (queue retries!)
- **Entry vs transition vs exit actions:**
  - Entry: runs on every state entry (initialization, notifications)
  - Transition: runs only for specific event (business logic)
  - Exit: runs on every state exit (cleanup, logging)
- **Anti-pattern: Action that throws to prevent transition**
  - **Fix:** Use a guard instead — guards gate, actions execute
- **Anti-pattern: Action with side effect ordering dependency**
  - **Fix:** Split into separate states if order matters
- **Anti-pattern: Massive action that does 5 things**
  - **Fix:** One responsibility per action class
- **`raise()` patterns:**
  - Safe: linear chain with termination
  - Dangerous: circular raise chains (loop protection catches this)
  - Best practice: raise at most 1-2 events, keep chains short
- **v7: `sendTo()` vs `raise()` vs `dispatchTo()`:**
  - `raise()` → same machine, same macrostep, sync
  - `sendTo()` → different machine, blocking sync
  - `dispatchTo()` → different machine, async queue
- **Example:** Order processing with entry/transition/exit actions
- **Cross-reference:** actions.md, raised-events.md, sendto.md

### 6. `context-design.md` — Context Design
- **The decision rule:**
  - Changes available transitions? → Make it a state
  - Just carries data through the flow? → Put it in context
- **Anti-pattern: Boolean flags that should be states**
  - `context.is_approved` + `context.is_paid` → separate states `approved`, `paid`
- **Anti-pattern: Context as a data dump** — 30 unrelated keys
  - **Fix:** Group related data, use typed ContextManager subclass
- **Anti-pattern: Storing derived values** — `context.total = qty * price`
  - **Fix:** Use a calculator that computes it fresh each transition
- **Best practice:** Typed ContextManager subclass for complex machines
- **Best practice:** Separate context keys per parallel region (avoid LWW conflict)
- **Best practice:** Keep context minimal — only what the machine needs to make decisions
- **Example:** Refactoring bloated context into lean context + states
- **Cross-reference:** working-with-context.md, custom-context.md

### 7. `transition-design.md` — Transition Design
- **Self-transitions:** Target = current state → exit + entry actions fire
  - Use case: "reset" or "retry" that re-runs entry actions
- **Targetless transitions:** No target → actions run but no state change
  - Use case: update context without leaving the state
- **`@always` chains:** Must always terminate
  - Safe: linear chain to terminal state
  - Safe: guarded loop with eventually-failing condition
  - Dangerous: unguarded cycle (loop protection catches at 100)
- **Multi-branch transitions:** First matching guard wins
  - Last branch without guard = fallback (always matches)
  - **Anti-pattern:** Relying on definition order for priority → be explicit with guards
- **Anti-pattern: Transition to self with entry action as "do this again"**
  - **Fix:** Consider if a targetless transition with action is cleaner
- **Example:** Multi-branch approval flow with escalation
- **Cross-reference:** writing-transitions.md, always-transitions.md

### 8. `machine-decomposition.md` — Machine Decomposition
- **When to split into child machines:**
  - Sub-flow has its own lifecycle (start, succeed, fail, timeout)
  - Sub-flow is reusable across different parent machines
  - Sub-flow is complex enough to warrant its own test suite
- **When to keep in one machine:**
  - Simple linear flow with no reuse
  - States share a lot of context
  - No independent failure/timeout handling needed
- **Anti-pattern: One mega-machine with 50+ states**
  - **Fix:** Identify sub-flows, extract as child machines
- **Anti-pattern: Too many tiny machines** — 1-2 states each
  - **Fix:** Merge related tiny machines into one coherent machine
- **Anti-pattern: Excessive cross-machine messaging**
  - **Fix:** If two machines constantly message each other, they should be one machine
- **Sync vs async delegation decision:**
  - Sync: child runs inline, simple, no queue needed
  - Async: child runs on queue, parent can timeout, external events can reach child
  - Rule: if child might take > 1 second or needs external input, use async
- **Parent orchestrates, child specializes:**
  - Parent: decides which child to invoke, handles @done/@fail routing
  - Child: does the actual work, returns result
- **Example:** Order fulfillment decomposed into payment + shipping + notification children
- **Cross-reference:** machine-delegation.md, delegation-data-flow.md

### 9. `event-design.md` — Event Design
- **Events describe what happened, not what should happen** (from conventions)
- **Be specific:** `ORDER_PAYMENT_RECEIVED` not `UPDATE` or `CHANGE`
- **Payload carries data, type carries intent**
  - Anti-pattern: encoding state in event name (`APPROVE_AS_MANAGER` → `APPROVE` + payload `{role: 'manager'}`)
- **Event classes vs string types:**
  - Class: when you need validation, typed payload, complex structure
  - String: for simple events with no payload (`CANCEL`, `RETRY`)
- **v7: Raised events vs cross-machine events:**
  - `raise()`: internal, same machine, processed in same macrostep
  - `sendTo()`/`dispatchTo()`: external, different machine, separate lifecycle
  - Don't use `raise()` to communicate between machines (use sendTo)
  - Don't use `sendTo()` within the same machine (use raise)
- **Timer events:** Same naming rules, describe what the timer represents (`ORDER_EXPIRED`, `BILLING_DUE`)
- **Example:** Event taxonomy for e-commerce domain
- **Cross-reference:** conventions.md (event naming), events.md, raised-events.md

### 10. `time-based-patterns.md` — Time-Based Patterns
- **`after` for deadlines:** "If nothing happens in N time, trigger this"
  - Example: `ORDER_EXPIRED` after 7 days in `awaiting_payment`
- **`every` for recurring:** "Do this at regular intervals"
  - Example: `BILLING` every 30 days in `active_subscription`
- **`max` + `then` for escalation:** "Retry N times, then escalate"
  - Example: `RETRY` every 6 hours, max 3, then `MAX_RETRIES`
- **Anti-pattern: Timer as polling mechanism**
  - **Fix:** Use scheduled events for batch queries, timers for per-instance behavior
- **Anti-pattern: Very short intervals** (< 1 minute) without backpressure awareness
  - **Fix:** Check `machine.timers.backpressure_threshold`, increase resolution
- **Anti-pattern: Timer event that's not idempotent**
  - **Fix:** Timer sweep may fire edge cases (crash recovery), actions must handle re-fire
- **Best practice:** Timer events should fail gracefully (machine may have left the state)
- **Best practice:** Use `machine:timer-status` to debug timer issues in production
- **Example:** Payment retry escalation chain (3 retries → human escalation → timeout)
- **Cross-reference:** time-based-events.md, time-based-testing.md

### 11. `parallel-patterns.md` — Parallel Patterns
- **Each region must be truly independent** — no execution order assumption
- **Separate context keys per region** — avoid LWW conflict
- **`@done` fires when ALL regions reach final** — not when the first one does
- **Anti-pattern: Regions that depend on execution order**
  - **Fix:** Use sequential states instead of parallel
- **Anti-pattern: Shared context key mutation** — both regions write to `status`
  - **Fix:** `payment_status` + `shipping_status` (separate keys)
- **Anti-pattern: Using parallel states for sequential phases**
  - **Fix:** If A must complete before B, they're sequential, not parallel
- **Cross-region coordination:** `@always` with guards checking sibling state
  - Example: Region A waits for Region B to finish before proceeding
- **Parallel dispatch mode:** Entry actions run on separate queue workers
  - Implications: no shared in-memory state, DB-level conflict detection
- **Example:** Order fulfillment with payment + shipping + notification regions
- **Cross-reference:** parallel-states/overview.md, parallel-states/event-handling.md

### 12. `testing-strategy.md` — Testing Strategy
- **Test layers:**
  - Unit: individual behaviors via `runWithState()`
  - Integration: state flow via `Machine::test()` fluent API
  - E2E: full pipeline with artisan commands and real DB
  - LocalQA: real MySQL + Redis + Horizon for async features
- **What to test at each layer:**
  - Unit: guard logic, action side effects, calculator computations
  - Integration: transition paths, context mutations, guard gating
  - E2E: timer sweep, scheduled events, persist/restore
  - LocalQA: async delegation, timeout, parallel dispatch, locking
- **Machine::fake() for child delegation** — test parent without running child
- **Timer testing via `advanceTimers()`** — don't use `sleep()`
- **Anti-pattern: Testing internal events** — test observable behavior, not implementation
- **Anti-pattern: Testing transition order** — test final state, not intermediate steps
- **Best practice: One assertion per test** — clear, focused, easy to debug
- **Best practice: Use `assertPath()` for multi-step flows**
- **Example:** Testing order workflow at all 4 layers
- **Cross-reference:** testing/overview.md, test-machine.md, fakeable-behaviors.md

## Writing Guidelines

1. **Every "Do / Don't" has a code example** — both showing EventMachine PHP code
2. **All examples follow conventions.md** — snake_case states, SCREAMING_SNAKE_CASE events, PascalCase classes
3. **Use a consistent example domain** — order/payment workflow throughout (same as conventions.md)
4. **Doctest attributes on all code blocks** — `ignore` for config snippets, `no_run` for class examples
5. **Each page is self-contained** — can be read independently, cross-references at bottom
6. **Tone: practical mentor, not academic lecture** — "Generally..." not "You must always..."
7. **Acknowledge exceptions** — every rule has edge cases, note them
8. **Link to implementation docs** — best practices explain WHY, implementation docs explain HOW
9. **Keep each page under 200 lines** — focused, scannable, not overwhelming
