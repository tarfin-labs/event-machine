## 1. Naming Convention & Removals

### Terminology: "Stub" → "Scenario"

"Stub" is testing terminology. Scenarios are QA state setup, not tests. Guard/action behaviors used in scenarios are themselves "micro scenarios" — they describe how a behavior should act within a given scenario context.

### Class Naming Pattern

Follows EventMachine's existing convention — suffix indicates the type:

| Type | Base class | Concrete example | Suffix pattern |
|------|-----------|-----------------|---------------|
| Guard | `GuardBehavior` | `HasConsentGuard` | `{Name}Guard` |
| Action | `ActionBehavior` | `StoreApplicationAction` | `{Name}Action` |
| Guard scenario | `GuardScenarioBehavior` | `HasConsentGuardScenario` | `{OriginalName}Scenario` |
| Action scenario | `ActionScenarioBehavior` | `StoreApplicationActionScenario` | `{OriginalName}Scenario` |
| Machine scenario | `MachineScenario` | `AtVerificationScenario` | `{TargetState}Scenario` |

**Search advantage:** `HasConsent` finds both `HasConsentGuard` and `HasConsentGuardScenario` side by side.

**Machine scenario names** describe the target state, NOT the machine name:
- ✅ `AtVerificationScenario` — short, describes where the machine ends up
- ❌ `AtVerificationCarSalesMachineScenario` — redundant, `machine()` already declares this
- Directory structure provides machine context: `app/Machines/CarSales/Scenarios/AtVerificationScenario.php`

### Internal Rename Table

| Before | After |
|--------|-------|
| `ScenarioGuardStub` | `ScenarioGuardOverride` (internal — wraps bool/closure for container binding) |
| `ScenarioActionStub` | `ScenarioActionOverride` (internal — wraps array/closure for container binding) |
| `ScenarioStubContract` | **Deleted** — scenario logic belongs in `arrange()`, not on the action class |
| `registerStubs()` | `registerOverrides()` |
| `registerGuardStub()` | `registerGuardOverride()` |
| `registerActionStub()` | `registerActionOverride()` |
| `registerServiceStub()` | `registerServiceOverride()` |
| `cleanupStubs()` | `cleanupOverrides()` |

`ScenarioGuardOverride` / `ScenarioActionOverride` are **internal** classes used by `ScenarioPlayer` to wrap values into container-compatible behaviors. User-facing scenario behavior classes extend `GuardScenarioBehavior` / `ActionScenarioBehavior`.

### Removals

**`models()` system** — Staging QA already has models in DB.
- Remove: `models()`, `model()`, `addModel()`, `getModels()`, `getCreatedModels()`, `$createdModels`
- Remove: `ScenarioPlayer::createModels()`, `ScenarioResult::$models`
- Remove: model-related merge logic in `resolveParentChain()`
- Remove: `models` key from controller/command responses

**`defaults()` / `param()` system** — Parameters come from the frontend via event payload.
- Remove: `defaults()`, `getDefaults()`, `param()`, `$resolvedParams`, `hydrate()`
- Remove: defaults merge logic in `resolveParentChain()`
- Remove: `--param` from `ScenarioCommand`, `params` from `ScenarioController`
- Remove: `defaults` from describe endpoint response

**`ScenarioStubContract` / `applyStub()`** — Scenario logic belongs in `arrange()`, not on the action class.
- Delete: `src/Scenarios/ScenarioStubContract.php`
- Remove: `applyStub()` branch from `ScenarioActionOverride::__invoke()`

## 2. Scenario Behavior Classes — `GuardScenarioBehavior` / `ActionScenarioBehavior`

Scenario behaviors follow the **same pattern** as regular EventMachine behaviors — inline closures or dedicated classes, with full DI support via `InvokableBehavior`.

### Base Classes

```php
// src/Behavior/GuardScenarioBehavior.php
abstract class GuardScenarioBehavior extends GuardBehavior {}

// src/Behavior/ActionScenarioBehavior.php
abstract class ActionScenarioBehavior extends ActionBehavior {}
```

Extending `GuardBehavior` / `ActionBehavior` means:
- **DI is free** — `InvokableBehavior` parameter injection works automatically
- **Type-compatible** — can replace the original in the container
- **Semantic clarity** — "this is the scenario version of a behavior"

### State-Keyed `arrange()` — Readable AND Functional

`arrange()` is keyed by **state name**. This serves two purposes:
1. **Readability** — you see WHERE in the machine each override applies
2. **Functionality** — same behavior can return different values in different states

```php
protected function arrange(): array
{
    return [
        'routing' => [
            HasConsentGuard::class => true,  // → registered customer path
        ],
        'eligibility_check' => [
            IsFarmerNotEligibleGuard::class => false,  // → farmer eligible
            StoreApplicationAction::class => ['applicationId' => 'APP-SCENARIO'],
        ],
    ];
}
```

Reading this, you immediately understand the **story**: at `routing`, consent is true; at `eligibility_check`, farmer is eligible.

### Same Behavior, Different Values Per State

When the same behavior appears under multiple states with **different values**, ScenarioPlayer creates a **state-aware override** automatically:

```php
protected function arrange(): array
{
    return [
        'routing' => [
            HasConsentGuard::class => true,   // at routing: true
        ],
        'data_collection' => [
            HasConsentGuard::class => false,  // at data_collection: false
        ],
    ];
}
```

**How it works internally:**

ScenarioPlayer detects `HasConsentGuard` under multiple states with different values. Instead of a simple global binding, it creates a state-aware override:

```php
App::bind(HasConsentGuard::class, fn($app, $params) => new ScenarioGuardOverride(
    stateValues: ['routing' => true, 'data_collection' => false],
    originalClass: HasConsentGuard::class,
    eventQueue: $params['eventQueue'] ?? null,
));
```

The override checks the current state via DI:

```php
class ScenarioGuardOverride extends GuardBehavior
{
    public function __invoke(State $state): bool
    {
        $currentState = $state->currentStateDefinition->key;

        if (array_key_exists($currentState, $this->stateValues)) {
            return $this->stateValues[$currentState];
        }

        // No override for this state → run original guard
        return (new ($this->originalClass)($this->eventQueue))->__invoke($state);
    }
}
```

| Situation | What ScenarioPlayer does |
|-----------|-------------------------|
| Guard in one state only | Simple global bind (state key is documentary) |
| Guard in multiple states, **same** value | Simple global bind (state key is documentary) |
| Guard in multiple states, **different** values | State-aware bind (state key is functional) |
| Guard runs in a state with **no override** | Original guard executes |

### Three Value Forms Per Override

Each value in `arrange()` supports three forms — mirrors behavior definition pattern:

**Guards:**

```php
'eligibility_check' => [
    // 1. Bool shorthand — simplest
    IsFarmerNotEligibleGuard::class => false,

    // 2. Closure — inline with DI
    HasConsentGuard::class => function (ContextManager $ctx): bool {
        return $ctx->get('farmer')->hasValidConsent();
    },

    // 3. Class — reusable scenario behavior
    IsCustomerInfoCompleteGuard::class => IsCustomerInfoCompleteGuardScenario::class,
],
```

**Actions:**

```php
'checking_protocol' => [
    // 1. Array shorthand — key-value pairs written to context
    CheckProtocolAction::class => ['protocolEligible' => true],

    // 2. Closure — inline with DI
    StoreApplicationAction::class => function (ContextManager $ctx) {
        $ctx->set('applicationId', 'APP-'.Str::random());
    },

    // 3. Class — reusable scenario behavior
    CalculatePricesAction::class => CalculatePricesActionScenario::class,
],
```

### Detection logic in `ScenarioPlayer::registerOverrides()`

| Original class | Value type | Behavior |
|----------------|-----------|----------|
| `extends GuardBehavior` | `bool` | `ScenarioGuardOverride` with state-aware routing |
| `extends GuardBehavior` | `Closure` | `ScenarioGuardOverride` wrapping closure with DI |
| `extends GuardBehavior` | `class-string<GuardScenarioBehavior>` | Bind class in container |
| `extends ActionBehavior` | `array` | `ScenarioActionOverride` writing key-values to context |
| `extends ActionBehavior` | `Closure` | `ScenarioActionOverride` wrapping closure with DI |
| `extends ActionBehavior` | `class-string<ActionScenarioBehavior>` | Bind class in container |
| other (service) | `array<method => return>` | Anonymous class with `__call` |

All override types support state-aware routing when the same behavior appears in multiple states.

### DI Resolution

Closures and scenario behavior classes get the same parameter injection as regular behaviors:
- `ContextManager`, `State`, `EventBehavior`, `EventCollection`

Inherited from `InvokableBehavior` via `GuardBehavior` / `ActionBehavior`.

## 3. File Organization

### Machine structure mirroring

Scenarios mirror the machine's directory structure — just like tests mirror production code. Each machine's scenarios live under its own `Scenarios/` directory.

**Simple scenarios** (only inline closures/shorthands in `arrange()`) — single file:

```
app/Machines/CarSales/
├── CarSalesMachine.php
├── Guards/
│   ├── HasConsentGuard.php
│   └── IsFarmerNotEligibleGuard.php
├── Actions/
│   ├── StoreApplicationAction.php
│   └── CalculatePricesAction.php
│
└── Scenarios/
    └── SimpleSkipScenario.php
```

**Scenarios with class-based scenario behaviors** — own directory, mirroring Guards/Actions:

```
app/Machines/CarSales/
└── Scenarios/
    ├── SimpleSkipScenario.php
    │
    ├── AtVerificationScenario/
    │   ├── AtVerificationScenario.php
    │   └── Guards/
    │       └── IsFarmerNotEligibleGuardScenario.php
    │
    └── ApprovedScenario/
        ├── ApprovedScenario.php
        ├── Guards/
        │   └── IsCustomerInfoCompleteGuardScenario.php
        └── Actions/
            └── CalculatePricesActionScenario.php
```

**Key principles:**
- Co-located — scenario class and its behavior classes in the same directory
- Mirrors machine's Guards/Actions structure
- Self-contained — deleting a scenario directory removes everything
- Simple scenarios don't need a directory

### Discovery

`ScenarioDiscovery` resolves each machine's `Scenarios/` directory from the machine class file location using `ReflectionClass`:

```php
$machineFile = (new ReflectionClass($machineClass))->getFileName();
$scenarioDir = dirname($machineFile).'/Scenarios';
```

Machine-local discovery — no single global config path. `machine:scenario-cache` iterates discovered machine classes and builds a combined scenario class map.

## 4. Simplified `MachineScenario`

```php
abstract class MachineScenario
{
    /** Which machine this scenario targets. */
    abstract protected function machine(): string;

    /** Human-readable description shown in artisan --list and HTTP UI. */
    abstract protected function description(): string;

    /** Parent scenario for composition. */
    protected function parent(): ?string { return null; }

    /** Expected starting state for mid-flight scenarios. */
    protected function from(): ?string { return null; }

    /**
     * State-keyed behavior scenario definitions.
     * Keys are state names, values are arrays of behavior overrides.
     *
     * Supports: bool, Closure, array, class-string<GuardScenarioBehavior|ActionScenarioBehavior>
     */
    protected function arrange(): array { return []; }

    /** Ordered sequence of events to replay (optional — empty for endpoint-triggered scenarios). */
    protected function steps(): array { return []; }

    // ─── Step Builders ───────────────────────────────────────
    protected function send(string $eventType, array $payload = []): ScenarioStep { ... }
    protected function child(string $machineClass): ChildScenarioStep { ... }

    // ─── Execution ───────────────────────────────────────────
    public static function play(): ScenarioResult { ... }
    public static function playOn(string $machineId): ScenarioResult { ... }

    // ─── Introspection ───────────────────────────────────────
    public function getMachine(): string { ... }
    public function getDescription(): string { ... }
    public function getParent(): ?string { ... }
    public function getFrom(): ?string { ... }
    public function getArrange(): array { ... }
    public function getSteps(): array { ... }
}
```

**2 abstract methods** (`machine`, `description`).
**4 optional methods** (`parent`, `from`, `arrange`, `steps`).

## 5. Example: CarSalesMachine Scenario (Simulated)

```php
// app/Machines/CarSales/Scenarios/AtVerificationScenario.php

class AtVerificationScenario extends MachineScenario
{
    protected function machine(): string
    {
        return CarSalesMachine::class;
    }

    protected function description(): string
    {
        return 'At verification — consent given, farmer eligible';
    }

    protected function from(): ?string
    {
        return 'eligibility_check';
    }

    protected function arrange(): array
    {
        return [
            'routing' => [
                HasConsentGuard::class => true,
            ],
            'eligibility_check' => [
                IsFarmerNotEligibleGuard::class => false,
            ],
        ];
    }
}
```

**Usage from frontend:**

```http
POST /api/car-sales/{applicationId}/customer-started
{
    "type": "CustomerStartedEvent",
    "scenario": "at-verification-scenario"
}
```

**What happens:**
1. `MachineController` detects `scenario` field in request
2. Instantiates `AtVerificationScenario`, registers state-keyed `arrange()` behaviors in the container
3. `$machine->send(CustomerStartedEvent)` — engine transitions through states
4. At `routing`: `HasConsentGuard` → scenario returns `true` → `awaiting_customer_start`
5. At `eligibility_check`: `IsFarmerNotEligibleGuard` → scenario returns `false` → `verification` ✅
6. Validates `from()`: post-transition state matches `eligibility_check` ✅
7. No additional steps → cleanup scenario behaviors from container
8. Response includes final state, `available_events`, `available_scenarios`

## 6. Fix Endpoint Scenario Behavior Timing

**Problem:** `MachineController::executeEndpoint()` sends the event BEFORE registering scenario behaviors. `@always` guards run with real behavior.

**Fix:** Restructure `executeEndpoint()`:

```
BEFORE (broken):
  send() → registerOverrides() → playSteps()

AFTER (correct):
  detectScenario() → registerOverrides() → send() → playSteps() → cleanup()
```

**Implementation:**

```
MachineController::executeEndpoint()
│
├─ 1. maybeRegisterScenarioOverrides(Request, machineClass)
│     ├─ Read 'scenario' slug from request body
│     ├─ If null or MACHINE_SCENARIOS_ENABLED=false → return null
│     ├─ Resolve scenario class from slug (filtered by machineClass)
│     ├─ Instantiate scenario
│     ├─ Call ScenarioPlayer::registerOverrides() — binds arrange() behaviors in container
│     └─ Return scenario instance
│
├─ 2. $state = $machine->send(event)
│     └─ @always guards now use scenario behaviors from container ✅
│
├─ 3. Validate from() (if scenario defines it)
│     └─ Assert $state->currentStateDefinition->key === $scenario->getFrom()
│         └─ Mismatch → 422: "Scenario expects state 'X' but machine is at 'Y'"
│
├─ 4. maybePlayScenarioSteps($scenario, $machine, $state)
│     ├─ If scenario->getSteps() is empty → skip
│     └─ For each step: $machine->send(step->eventType, step->payload)
│
├─ 5. finally: maybeCleanupScenarioOverrides($scenario)
│     └─ Unbind scenario behaviors from container
│
└─ 6. buildResponse() — uses final state after all steps complete
```

**Synchronous guarantee:** The endpoint does NOT return until the scenario fully completes.

**`from()` validation timing:** Validated AFTER `send()` but BEFORE additional steps. Ensures the event transitioned to the expected state. Mismatch → 422.
