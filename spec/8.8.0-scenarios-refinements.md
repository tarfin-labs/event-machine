## 1. Rename "Stub" → "Override"

"Stub" is testing terminology. Scenarios are QA state setup, not tests. The mechanism is **overriding** behavior.

| Before | After |
|--------|-------|
| `ScenarioGuardStub` | `ScenarioGuardOverride` |
| `ScenarioActionStub` | `ScenarioActionOverride` |
| `ScenarioStubContract` | **Deleted** |
| `registerStubs()` | `registerOverrides()` |
| `registerGuardStub()` | `registerGuardOverride()` |
| `registerActionStub()` | `registerActionOverride()` |
| `registerServiceStub()` | `registerServiceOverride()` |
| `cleanupStubs()` | `cleanupOverrides()` |
| `$originalBindings` | `$originalBindings` (unchanged) |

## 2. Drop `ScenarioStubContract` and `applyStub()`

**Why:** Same argument as "don't put scenarios inside machine definitions" — override logic belongs in the scenario's `arrange()`, not on the action class. Actions shouldn't know about scenarios.

**Remove:**
- `src/Scenarios/ScenarioStubContract.php` — delete file
- `ScenarioActionStub::__invoke()` — remove `applyStub()` branch
- Any test stubs implementing `ScenarioStubContract`

## 3. Drop `models()`

**Why:** Staging QA already has models in DB. `models()` adds a Laravel factory abstraction layer that's unnecessary for the primary use case (staging QA with existing data).

**Remove:**
- `MachineScenario::models()` method
- `MachineScenario::model()` accessor
- `MachineScenario::addModel()` internal method
- `MachineScenario::getModels()` internal method
- `MachineScenario::getCreatedModels()` internal method
- `MachineScenario::$createdModels` property
- `ScenarioPlayer::createModels()` method
- `ScenarioResult::$models` property
- All model-related merge logic in `resolveParentChain()`
- Update `ScenarioController` response (remove `models` key)
- Update `ScenarioCommand` output

---

## 4. Drop `defaults()` and `param()`

**Why:** Parameters come from the frontend (existing farmer/retailer IDs in event payload). No need for scenario-level defaults — the scenario is triggered from an endpoint with real data.

**Remove:**
- `MachineScenario::defaults()` method
- `MachineScenario::getDefaults()` introspection getter
- `MachineScenario::param()` accessor
- `MachineScenario::$resolvedParams` property
- `MachineScenario::hydrate()` method (simplify — no params to merge)
- Defaults merge logic in `ScenarioPlayer::resolveParentChain()`
- `ScenarioCommand` `--param` option
- `ScenarioController` `params` request field
- Update `ScenarioResult` (remove params-related data)
- Update describe endpoint response (remove `defaults` key)

## 5. Override Behavior Classes — `GuardOverrideBehavior` / `ActionOverrideBehavior`

Override behaviors follow the **same pattern** as regular EventMachine behaviors — inline closures or dedicated classes, with full DI support via `InvokableBehavior`.

### Base Classes

```php
abstract class GuardOverrideBehavior extends GuardBehavior {}
abstract class ActionOverrideBehavior extends ActionBehavior {}
```

Extending `GuardBehavior` / `ActionBehavior` means:
- **DI is free** — `InvokableBehavior` parameter injection works automatically
- **Type-compatible** — can replace the original in the container
- **Semantic clarity** — "this is an override, not a real guard/action"

### Three forms in `arrange()` — mirrors behavior definition pattern

**Guards:**

```php
protected function arrange(): array
{
    return [
        // 1. Bool shorthand — simplest form
        IsFarmerNotEligibleGuard::class => false,

        // 2. Closure — inline override with DI
        HasConsentGuard::class => function (ContextManager $ctx): bool {
            return $ctx->get('farmer')->hasValidConsent();
        },

        // 3. Class — reusable override
        IsCustomerInfoCompleteGuard::class => AlwaysCompleteOverride::class,
    ];
}

// Reusable override class
class AlwaysCompleteOverride extends GuardOverrideBehavior
{
    public function __invoke(): bool
    {
        return true;
    }
}
```

**Actions:**

```php
protected function arrange(): array
{
    return [
        // 1. Array shorthand — key-value pairs written to context
        CheckProtocolAction::class => ['protocolEligible' => true],

        // 2. Closure — inline override with DI
        StoreApplicationAction::class => function (ContextManager $ctx) {
            $ctx->set('applicationId', 'APP-'.Str::random());
            $ctx->set('status', 'pending');
        },

        // 3. Class — reusable override
        CalculatePricesAction::class => ScenarioCalculatePricesOverride::class,
    ];
}

// Reusable override class
class ScenarioCalculatePricesOverride extends ActionOverrideBehavior
{
    public function __invoke(ContextManager $ctx): void
    {
        $ctx->set('totalCashPrice', Money::of(150000, 'TRY'));
        $ctx->set('monthlyInstallment', Money::of(12500, 'TRY'));
    }
}
```

### Detection logic in `ScenarioPlayer::registerOverrides()`

| Original class | Value type | Behavior |
|----------------|-----------|----------|
| `extends GuardBehavior` | `bool` | Wrap in `GuardOverrideBehavior`, return value |
| `extends GuardBehavior` | `Closure` | Wrap in `GuardOverrideBehavior`, invoke closure with DI |
| `extends GuardBehavior` | `class-string<GuardOverrideBehavior>` | Instantiate directly |
| `extends ActionBehavior` | `array` | Wrap in `ActionOverrideBehavior`, write key-values to context |
| `extends ActionBehavior` | `Closure` | Wrap in `ActionOverrideBehavior`, invoke closure with DI |
| `extends ActionBehavior` | `class-string<ActionOverrideBehavior>` | Instantiate directly |
| other (service) | `array<method => return>` | Anonymous class with `__call` |

### DI Resolution

Closures and override classes get the same parameter injection as regular behaviors:
- `ContextManager` — current machine context
- `State` — current machine state
- `EventBehavior` — the event being processed
- `EventCollection` — event history

This is inherited from `InvokableBehavior` via `GuardBehavior` / `ActionBehavior`.

## 6. Fix Endpoint Override Timing

**Problem:** `MachineController::executeEndpoint()` sends the event BEFORE registering scenario overrides. `@always` guards run with real behavior.

**Fix:** Restructure `executeEndpoint()`:

```
BEFORE (broken):
  send() → registerOverrides() → playSteps()

AFTER (correct):
  detectScenario() → registerOverrides() → send() → playSteps() → cleanup()
```

**Implementation:**
- Extract `maybeRegisterScenarioOverrides(Request)` — detects scenario, instantiates it, registers arrange() overrides. Returns scenario instance or null.
- `$machine->send()` runs with overrides active — `@always` guards use overrides.
- `maybePlayScenarioSteps($scenario, $machine, $state)` — plays remaining steps if scenario has any.
- `maybeCleanupScenarioOverrides($scenario)` — runs in `finally` block, unbinds overrides.
