# Scenario Refinements — Pre-Release

> Changes to apply before 8.8.0 release. Merge into `spec/8.8.0-scenarios.md` before tagging.

---

## 1. Naming Convention

### Terminology: "Stub" → "Scenario"

"Stub" is testing terminology. Scenarios are QA state setup, not tests. Guard/action overrides used in scenarios are themselves "micro scenarios" — they describe how a behavior should act in a given scenario.

### Class Naming Pattern

Follows EventMachine's existing convention — suffix indicates the type:

| Type | Base class | Concrete example | Suffix pattern |
|------|-----------|-----------------|---------------|
| Guard | `GuardBehavior` | `HasConsentGuard` | `{Name}Guard` |
| Action | `ActionBehavior` | `StoreApplicationAction` | `{Name}Action` |
| Guard scenario | `GuardScenarioBehavior` | `HasConsentGuardScenario` | `{OriginalName}Scenario` |
| Action scenario | `ActionScenarioBehavior` | `CreateApplicationActionScenario` | `{OriginalName}Scenario` |
| Machine scenario | `MachineScenario` | `AtVerificationScenario` | `{TargetState}Scenario` |

**Search advantage:** `HasConsent` finds both `HasConsentGuard` and `HasConsentGuardScenario` — the original behavior and its scenario version appear side by side.

**Machine scenario names** describe the target state, NOT the machine name:
- ✅ `AtVerificationScenario` — short, describes target state
- ❌ `AtVerificationCarSalesMachineScenario` — redundant, `machine()` method already declares this
- The directory structure provides machine context: `app/Machines/Scenarios/CarSales/AtVerificationScenario.php`

### Rename Table

| Before | After |
|--------|-------|
| `ScenarioGuardStub` | `ScenarioGuardOverride` (internal — wraps bool/closure for container binding) |
| `ScenarioActionStub` | `ScenarioActionOverride` (internal — wraps array/closure for container binding) |
| `ScenarioStubContract` | **Deleted** |
| `registerStubs()` | `registerOverrides()` |
| `registerGuardStub()` | `registerGuardOverride()` |
| `registerActionStub()` | `registerActionOverride()` |
| `registerServiceStub()` | `registerServiceOverride()` |
| `cleanupStubs()` | `cleanupOverrides()` |

**Note:** `ScenarioGuardOverride` / `ScenarioActionOverride` are internal classes used by `ScenarioPlayer` to wrap simple values (bool, array, closure) into container-compatible behaviors. User-facing override classes extend `GuardScenarioBehavior` / `ActionScenarioBehavior`.

## 2. Drop `ScenarioStubContract` and `applyStub()`

**Why:** Same argument as "don't put scenarios inside machine definitions" — override logic belongs in the scenario's `arrange()`, not on the action class. Actions shouldn't know about scenarios.

**Remove:**
- `src/Scenarios/ScenarioStubContract.php` — delete file
- `ScenarioActionStub::__invoke()` — remove `applyStub()` branch
- Any test stubs implementing `ScenarioStubContract`

---

## 3. Drop `models()`

**Why:** Staging QA already has models in DB. `models()` adds a Laravel factory abstraction layer that's unnecessary for the primary use case (staging QA with existing data).

**Remove:**
- `MachineScenario::models()` method
- `MachineScenario::model()` accessor
- `MachineScenario::addModel()` internal method
- `MachineScenario::getModels()` internal method
- `MachineScenario::getCreatedModels()` internal method
- `MachineScenario::$createdModels` property
- `ScenarioPlayer::createModels()` method
- `ScenarioResult::$models` property
- All model-related merge logic in `resolveParentChain()`
- Update `ScenarioController` response (remove `models` key)
- Update `ScenarioCommand` output

---

## 4. Drop `defaults()` and `param()`

**Why:** Parameters come from the frontend (existing farmer/retailer IDs in event payload). No need for scenario-level defaults — the scenario is triggered from an endpoint with real data.

**Remove:**
- `MachineScenario::defaults()` method
- `MachineScenario::getDefaults()` introspection getter
- `MachineScenario::param()` accessor
- `MachineScenario::$resolvedParams` property
- `MachineScenario::hydrate()` method (simplify — no params to merge)
- Defaults merge logic in `ScenarioPlayer::resolveParentChain()`
- `ScenarioCommand` `--param` option
- `ScenarioController` `params` request field
- Update `ScenarioResult` (remove params-related data)
- Update describe endpoint response (remove `defaults` key)

## 5. Scenario Behavior Classes — `GuardScenarioBehavior` / `ActionScenarioBehavior`

Scenario behaviors follow the **same pattern** as regular EventMachine behaviors — inline closures or dedicated classes, with full DI support via `InvokableBehavior`.

### Base Classes

```php
// src/Behavior/GuardScenarioBehavior.php
abstract class GuardScenarioBehavior extends GuardBehavior {}

// src/Behavior/ActionScenarioBehavior.php
abstract class ActionScenarioBehavior extends ActionBehavior {}
```

Extending `GuardBehavior` / `ActionBehavior` means:
- **DI is free** — `InvokableBehavior` parameter injection works automatically
- **Type-compatible** — can replace the original in the container
- **Semantic clarity** — "this is a scenario version of a behavior"

### Three forms in `arrange()` — mirrors behavior definition pattern

Just like machine definitions support inline closures and class-based behaviors, `arrange()` supports the same dual pattern plus a shorthand:

**Guards:**

```php
protected function arrange(): array
{
    return [
        // 1. Bool shorthand — simplest form
        IsFarmerNotEligibleGuard::class => false,

        // 2. Closure — inline scenario with DI
        HasConsentGuard::class => function (ContextManager $ctx): bool {
            return $ctx->get('farmer')->hasValidConsent();
        },

        // 3. Class — reusable scenario behavior
        IsCustomerInfoCompleteGuard::class => IsCustomerInfoCompleteGuardScenario::class,
    ];
}

// Reusable guard scenario — lives alongside the original guard
class IsCustomerInfoCompleteGuardScenario extends GuardScenarioBehavior
{
    public function __invoke(): bool
    {
        return true;
    }
}
```

**Actions:**

```php
protected function arrange(): array
{
    return [
        // 1. Array shorthand — key-value pairs written to context
        CheckProtocolAction::class => ['protocolEligible' => true],

        // 2. Closure — inline scenario with DI
        StoreApplicationAction::class => function (ContextManager $ctx) {
            $ctx->set('applicationId', 'APP-'.Str::random());
            $ctx->set('status', 'pending');
        },

        // 3. Class — reusable scenario behavior
        CalculatePricesAction::class => CalculatePricesActionScenario::class,
    ];
}

// Reusable action scenario — lives alongside the original action
class CalculatePricesActionScenario extends ActionScenarioBehavior
{
    public function __invoke(ContextManager $ctx): void
    {
        $ctx->set('totalCashPrice', Money::of(150000, 'TRY'));
        $ctx->set('monthlyInstallment', Money::of(12500, 'TRY'));
    }
}
```

### Detection logic in `ScenarioPlayer::registerOverrides()`

| Original class | Value type | Behavior |
|----------------|-----------|----------|
| `extends GuardBehavior` | `bool` | Wrap in internal `ScenarioGuardOverride`, return value |
| `extends GuardBehavior` | `Closure` | Wrap in internal `ScenarioGuardOverride`, invoke closure with DI |
| `extends GuardBehavior` | `class-string<GuardScenarioBehavior>` | Bind directly in container |
| `extends ActionBehavior` | `array` | Wrap in internal `ScenarioActionOverride`, write key-values to context |
| `extends ActionBehavior` | `Closure` | Wrap in internal `ScenarioActionOverride`, invoke closure with DI |
| `extends ActionBehavior` | `class-string<ActionScenarioBehavior>` | Bind directly in container |
| other (service) | `array<method => return>` | Anonymous class with `__call` |

### DI Resolution

Closures and scenario behavior classes get the same parameter injection as regular behaviors:
- `ContextManager` — current machine context
- `State` — current machine state
- `EventBehavior` — the event being processed
- `EventCollection` — event history

This is inherited from `InvokableBehavior` via `GuardBehavior` / `ActionBehavior`.

### File Organization for Scenario Behaviors

Scenario behavior classes live **alongside their originals** for discoverability:

```
app/Machines/CarSales/Guards/
├── HasConsentGuard.php
├── HasConsentGuardScenario.php          ← scenario version
├── IsFarmerNotEligibleGuard.php
└── IsFarmerNotEligibleGuardScenario.php ← scenario version

app/Machines/CarSales/Actions/
├── StoreApplicationAction.php
├── StoreApplicationActionScenario.php   ← scenario version
├── CalculatePricesAction.php
└── CalculatePricesActionScenario.php    ← scenario version
```

## 6. Simplified `MachineScenario` After Removals

After dropping `models()`, `defaults()`, `param()`, and `hydrate()`, the abstract class becomes minimal:

```php
abstract class MachineScenario
{
    /** Which machine this scenario targets. */
    abstract protected function machine(): string;

    /** Human-readable description shown in artisan --list and HTTP UI. */
    abstract protected function description(): string;

    /** Ordered sequence of events to replay. */
    abstract protected function steps(): array;

    /** Parent scenario for composition. */
    protected function parent(): ?string { return null; }

    /** Expected starting state for mid-flight scenarios. */
    protected function from(): ?string { return null; }

    /**
     * Behavior overrides — guards return predetermined values,
     * actions run scenario-specific logic instead of real code.
     */
    protected function arrange(): array { return []; }

    // ─── Step Builders ───────────────────────────────────────
    protected function send(string $eventType, array $payload = []): ScenarioStep { ... }
    protected function child(string $machineClass): ChildScenarioStep { ... }

    // ─── Execution ───────────────────────────────────────────
    public static function play(): ScenarioResult { ... }
    public static function playOn(string $machineId): ScenarioResult { ... }

    // ─── Introspection ───────────────────────────────────────
    public function getMachine(): string { ... }
    public function getDescription(): string { ... }
    public function getParent(): ?string { ... }
    public function getFrom(): ?string { ... }
    public function getArrange(): array { ... }
    public function getSteps(): array { ... }
}
```

**6 methods to implement** at most (`machine`, `description`, `steps`; optionally `parent`, `from`, `arrange`).
**No `param()`, no `model()`, no `hydrate()`** — the class is focused on what matters.

## 7. Example: CarSalesMachine Scenario (Simulated)

```php
class AtVerificationScenario extends MachineScenario
{
    protected function machine(): string
    {
        return CarSalesMachine::class;
    }

    protected function description(): string
    {
        return 'At verification — consent given, farmer eligible';
    }

    protected function arrange(): array
    {
        return [
            IsFarmerNotEligibleGuard::class => false,
            HasConsentGuard::class          => true,
        ];
    }

    protected function steps(): array
    {
        return [];  // no steps — this is triggered via endpoint with scenario field
    }

    protected function from(): ?string
    {
        return 'eligibility_check';
    }
}
```

**Usage from frontend:**

```http
POST /api/car-sales/{applicationId}/customer-started
{
    "type": "CustomerStartedEvent",
    "scenario": "at-verification-scenario"
}
```

**What happens:**
1. `MachineController` detects `scenario` field
2. Instantiates `AtVerificationScenario`, registers `arrange()` overrides
3. `$machine->send(CustomerStartedEvent)` — engine transitions to `eligibility_check`
4. `@always` runs: `IsFarmerNotEligibleGuard` → override returns `false` → `verification` ✅
5. Scenario has no additional steps → done
6. Cleanup overrides
7. Response: `{ value: "verification", available_scenarios: [...], available_events: [...] }`

## 8. Fix Endpoint Override Timing

**Problem:** `MachineController::executeEndpoint()` sends the event BEFORE registering scenario overrides. `@always` guards run with real behavior.

**Fix:** Restructure `executeEndpoint()`:

```
BEFORE (broken):
  send() → registerOverrides() → playSteps()

AFTER (correct):
  detectScenario() → registerOverrides() → send() → playSteps() → cleanup()
```

**Implementation:**
- Extract `maybeRegisterScenarioOverrides(Request)` — detects scenario, instantiates it, registers arrange() overrides. Returns scenario instance or null.
- `$machine->send()` runs with overrides active — `@always` guards use overrides.
- `maybePlayScenarioSteps($scenario, $machine, $state)` — plays remaining steps if scenario has any.
- `maybeCleanupScenarioOverrides($scenario)` — runs in `finally` block, unbinds overrides.

**Synchronous guarantee:** The endpoint does NOT return until the scenario fully completes. All `Machine::send()` calls, child machine replays, and step processing finish within the same HTTP request. QA sees the final state in the response.
