# Fix: Machine::result() Parameter Injection — v8.0.0

## Bug

`Machine::result()` invokes `ResultBehavior` with **positional arguments** instead of `InvokableBehavior::injectInvokableBehaviorParameters()`. Every other behavior invocation in the framework uses injection — actions, guards, calculators, and even `MachineController::resolveAndRunResult()` which invokes the **same** ResultBehavior correctly.

### Current Code (Broken)

```php
// Machine::result() — line 1086
return $resultBehavior(
    $this->state->context,              // positional: always 1st
    $this->state->currentEventBehavior, // positional: always 2nd
    $arguments ?? null,                 // positional: always 3rd
);
```

### Correct Code (MachineController)

```php
// MachineController::resolveAndRunResult() — line 236
$params = InvokableBehavior::injectInvokableBehaviorParameters(
    actionBehavior: $resultBehavior,
    state: $state,
    eventBehavior: $state->currentEventBehavior,
    forwardContext: $forwardContext,
);
return $resultBehavior(...$params);
```

## What Breaks

1. **Parameter order matters** — `__invoke(EventBehavior $event, MyContext $context)` receives context as $event (silent type mismatch)
2. **Only 3 types work** — State, EventCollection, ForwardContext type-hints never injected
3. **Container injection broken** — constructor DI on ResultBehavior not resolved via `resolve()`; uses raw `new $resultBehavior()` instead

## Backend Impact

19 classes extend `ResultBehavior` in the backend. All are in `Actions/` directories and use the `ReturnsResult` interface pattern (they're actions that also produce results). Their `__invoke` signatures:

| Pattern | Count | Affected? |
|---------|-------|-----------|
| `__invoke(SpecificContext $context, SpecificEvent $event)` | Most | Works by accident (positional matches injection order) |
| `__invoke(SpecificContext $context)` | Some | Works — extra args ignored |
| `__invoke()` with no params | Rare | Works — no injection needed |

**Risk is low** — most ResultBehaviors follow the `(Context, Event)` order which matches both positional and injection. The fix makes them work correctly regardless of order.

## Fix

### Machine::result()

Replace positional call with injection:

```php
public function result(): mixed
{
    $currentStateDefinition = $this->state->currentStateDefinition;
    $behaviorDefinition     = $this->definition->behavior[BehaviorType::Result->value];

    if ($currentStateDefinition->type !== StateDefinitionType::FINAL) {
        return null;
    }

    $id = $currentStateDefinition->id;
    if (!isset($behaviorDefinition[$id])) {
        return null;
    }

    $resultBehavior = $behaviorDefinition[$id];
    $arguments      = null;

    if (!is_callable($resultBehavior)) {
        if (str_contains((string) $resultBehavior, ':')) {
            [$resultBehavior, $arguments] = explode(':', (string) $resultBehavior);
            $arguments = explode(',', $arguments);
        }

        $resultBehavior = resolve($resultBehavior);
    }

    $params = InvokableBehavior::injectInvokableBehaviorParameters(
        actionBehavior: $resultBehavior,
        state: $this->state,
        eventBehavior: $this->state->currentEventBehavior,
        actionArguments: $arguments,
    );

    return $resultBehavior(...$params);
}
```

Key changes:
1. `new $resultBehavior()` → `resolve($resultBehavior)` — enables constructor DI
2. Positional call → `injectInvokableBehaviorParameters()` — type-hint based injection
3. `$arguments` parsing preserved for colon-separated args

## Breaking Change

**Technically BC** for anyone relying on positional order where it differs from injection. In practice, the common `(Context, Event)` order works identically under both approaches. The fix makes `(Event, Context)` and `(Context, State)` signatures work correctly — they were silently broken before.

**This is a bugfix, not a feature.** But since we're already doing v8 for @always and actor changes, bundle it.

## Test Scenarios

1. **Basic result with injection**: ResultBehavior with `(ContextManager $context)` — works
2. **Result with event**: `(ContextManager $context, EventBehavior $event)` — event injected correctly
3. **Reversed parameter order**: `(EventBehavior $event, ContextManager $context)` — both injected correctly (was broken before)
4. **Result with State**: `(ContextManager $context, State $state)` — State injected (was impossible before)
5. **Result with arguments**: `ResultClass:arg1,arg2` — arguments passed via injection
6. **Result with constructor DI**: ResultBehavior with `__construct(SomeService $service)` — resolved via container
7. **Callable result (closure)**: `'results' => ['done' => fn(ContextManager $c) => $c->get('total')]` — still works, injection applied to closures too
8. **No parameters**: `__invoke()` — works, no injection needed
9. **Existing tests still pass**: GreenResult (class, no params) and closure result (no params) from StateDefinitionTest

Note: `ForwardContext` is endpoint-specific and not available in `Machine::result()` context. `MachineController::resolveAndRunResult()` passes it; `Machine::result()` passes null. This is correct — ForwardContext only exists in forwarded endpoint flows.
