## Motivation

The existing `machine-decomposition.md` covers _when_ to split a machine into parent/child. But once you have multiple machines, new questions arise:

- How should machines communicate?
- What structure should the system have?
- How do you prevent a tangle of cross-machine messaging?
- How should child machines expose their status?
- Where do timers belong?

These are system-level design questions. The current docs don't address them. Users building real systems (e-commerce with payment + shipping + verification, lending with credit check + document review + approval) need this guidance.

## API Reference (Verified)

These are the exact EventMachine APIs used in the examples, verified against source code:

| Feature | Syntax | Notes |
|---------|--------|-------|
| Delegation | `'machine' => ChildMachine::class` | State config key |
| Done routing (catch-all) | `'@done' => 'target_state'` | Any child final state |
| Done routing (specific) | `'@done.approved' => 'target_state'` | Only when child reaches `approved` final state |
| Fail routing | `'@fail' => 'target_state'` | Only on exception/job failure, NOT business logic |
| Timeout | `'timeout' => 300` + `'@timeout' => 'target'` | Integer seconds, async-only (`queue` required) |
| Child output | `'output' => ['key1', 'key2']` | On child final state, filters context for parent |
| Entry action | `'entry' => 'actionName'` | State config key |
| After timer | `'EVENT' => ['target' => 'state', 'after' => Timer::minutes(2)]` | `after` key on transition config, `Timer` value object |
| sendTo | `$this->sendTo(string $machineClass, string $rootEventId, EventBehavior\|array $event)` | Sync, in behaviors via InvokableBehavior |
| dispatchTo | `$this->dispatchTo(...)` | Async variant, same params |
| sendToParent | `$this->sendToParent(ContextManager $context, EventBehavior\|array $event)` | Sync, throws if no parent |
| Context access | Parameter injection: `function __invoke(ContextManager $context)` | NOT `$this->context` |
| Forward | `'forward' => ['EVENT_TYPE']` | Accepts class FQCNs or strings |

**Critical `@done`/`@fail` distinction:**
- `@done` = child reached a final state (business success OR business failure — any final state)
- `@done.{state}` = child reached a SPECIFIC final state (resolution: specific first → catch-all)
- `@fail` = exception thrown / job crashed (infrastructure failure ONLY)

## Section Plan

### 1. Introduction (Core Principle)

> A multi-machine system is not a collection of independent machines — it is an organized structure where machines communicate through well-defined interfaces. The parent orchestrates, children report status through their states, and the hierarchy keeps the system comprehensible.

Brief context: machine-decomposition told you _when_ to split. This page tells you how to _organize_ the resulting system.

Mention that EventMachine's delegation pattern implements the **handshaking rule** automatically: the parent's delegation state (e.g., `processing_payment`) acts as the "busy" state, and the `@done`/`@fail` transition acts as the "done" acknowledgment. No manual handshaking needed.

### 2. The Command–State Interface

**Core idea:** Parent sends commands (events) to child. Parent reads child's state to make decisions. Child does NOT read parent's state.

**Example domain:** Order workflow with PaymentMachine child.

- Parent delegates to PaymentMachine via `machine` key (command down)
- Parent reads `@done`/`@done.{state}` (state up) to decide next step
- Child's final states are its "report" to the parent

**Anti-pattern: Parent leaks its state to child via `with`**

```php ignore
// Anti-pattern: parent passes its own state to child
'awaiting_payment' => [
    'machine' => PaymentMachine::class,
    'with'    => ['order_id', 'order_total', 'order_status'],  // ← parent state leaked to child
    '@done'   => 'shipping',
    '@fail'   => 'payment_error',
],
```

```php no_run
// Anti-pattern: child reads parent state to decide behavior
use Tarfinlabs\EventMachine\Behavior\ActionBehavior;
use Tarfinlabs\EventMachine\ContextManager;

class CapturePaymentAction extends ActionBehavior
{
    public function __invoke(ContextManager $context): void
    {
        // Child uses parent's order_status — tight coupling
        if ($context->get('order_status') === 'cancelled') {
            return;  // child shouldn't care about parent's state
        }

        // ... capture logic ...
    }
}
```

**Fix:** Parent handles cancellation itself, child only knows about payment:

```php ignore
// Parent handles cancellation — child doesn't need to know
'awaiting_payment' => [
    'machine' => PaymentMachine::class,
    'with'    => ['order_id', 'order_total'],  // only payment-relevant data
    'on'      => ['ORDER_CANCELLED' => 'cancelled'],
    '@done'   => 'shipping',
    '@fail'   => 'payment_error',
],
```

**Takeaway:** Information flows one way: commands down (`machine` key, `forward`, `sendTo`), states up (`@done`, `@done.{state}`, `sendToParent`). Pass only IDs and values the child needs via `with` — never parent state or status flags.

### 3. Design Your Child States for the Parent

**Core idea:** A child machine's final states should tell the parent everything it needs to know. Use `@done.{finalState}` to route the parent based on which final state the child reached. Use `output` to control which context keys the parent sees.

**Example domain:** Credit check in a loan application.

**Anti-pattern: Single final state hides important information**

```php ignore
// Child: CreditCheckMachine — parent can't distinguish outcomes
'states' => [
    'checking'  => [...],
    'completed' => ['type' => 'final'],  // approved? rejected? bureau down?
],
```

```php ignore
// Parent: no way to route differently
'processing_credit' => [
    'machine' => CreditCheckMachine::class,
    'with'    => ['applicant_id'],
    '@done'   => 'credit_decision',  // always goes to same state, must read context to decide
],
```

**Fix: Explicit final states + `@done.{state}` routing + `output`**

```php ignore
// Child: CreditCheckMachine — each outcome is a distinct final state
'states' => [
    'querying_bureau' => [
        'entry' => 'queryCreditBureauAction',
        'on'    => [
            'SCORE_RECEIVED'      => 'evaluating',
            'BUREAU_QUERY_FAILED' => 'bureau_unavailable',
        ],
        'after' => [
            Timer::seconds(30) => 'bureau_unavailable',
        ],
    ],
    'evaluating' => [
        'on' => [
            '@always' => [
                ['target' => 'approved', 'guards' => 'isCreditScoreSufficientGuard'],
                ['target' => 'rejected'],
            ],
        ],
    ],
    'approved' => [
        'type'   => 'final',
        'output' => ['credit_score', 'credit_limit'],  // only expose what parent needs
    ],
    'rejected' => [
        'type'   => 'final',
        'output' => ['credit_score', 'rejection_reason'],
    ],
    'bureau_unavailable' => [
        'type'   => 'final',
        'output' => ['error_code'],
    ],
],
```

```php ignore
// Parent: routes differently based on child's final state
'processing_credit' => [
    'machine'                   => CreditCheckMachine::class,
    'with'                      => ['applicant_id'],
    '@done.approved'            => 'approved',
    '@done.rejected'            => 'rejected',
    '@done.bureau_unavailable'  => 'retrying_credit_check',
    '@fail'                     => 'failed',  // exception/crash only
],
```

**Resolution order:** `@done.{state}` (specific) → `@done` (catch-all) → no transition.

**Takeaway:** Design child final states as a "report card" for the parent. Use `@done.{finalState}` for routing and `output` on final states to control what context the parent receives. Reserve `@fail` for infrastructure failures (exceptions, job crashes) — not business logic outcomes.

### 4. Timer Placement

**Core idea:** Business timeout timers belong on the machines that talk to the outside world (leaf machines). If a parent machine needs `@timeout` to guard against a child hanging, the child's design is incomplete.

**Example domain:** Document verification with external API.

**Anti-pattern: Timer on parent to guard against child**

```php ignore
// Anti-pattern: parent uses @timeout because child might hang
'verifying_documents' => [
    'machine' => DocumentVerificationMachine::class,
    'queue'   => 'default',
    'timeout' => 300,
    '@done'   => 'documents_verified',
    '@fail'   => 'verification_failed',
    '@timeout' => 'verification_timed_out',
],
```

Why would the child hang? If it calls an external API, the _child_ should own that timeout. The parent shouldn't need to know about the child's integration details.

**Fix: Timer on child (the machine that talks to the external service)**

```php ignore
// Child: DocumentVerificationMachine — owns its business timeout
'states' => [
    'calling_api' => [
        'entry' => 'callVerificationApiAction',
        'on'    => [
            'VERIFICATION_COMPLETED' => 'verified',
            'VERIFICATION_REJECTED'  => 'rejected',
        ],
        'after' => [
            Timer::minutes(2) => 'api_timed_out',
        ],
    ],
    'api_timed_out' => [
        'entry' => 'handleApiTimeoutAction',
        'on'    => [
            '@always' => [
                ['target' => 'calling_api', 'guards' => 'canRetryGuard'],
                ['target' => 'failed'],
            ],
        ],
    ],
    'verified' => ['type' => 'final', 'output' => ['verification_id']],
    'rejected' => ['type' => 'final', 'output' => ['rejection_reason']],
    'failed'   => ['type' => 'final', 'output' => ['failure_reason']],
],
```

```php ignore
// Parent: no timer needed — child guarantees completion
'verifying_documents' => [
    'machine'            => DocumentVerificationMachine::class,
    'queue'              => 'default',
    'with'               => ['document_ids'],
    '@done.verified'     => 'documents_verified',
    '@done.rejected'     => 'documents_rejected',
    '@done.failed'       => 'verification_failed',
    '@fail'              => 'failed',
],
```

**When parent `@timeout` IS appropriate:** Infrastructure protection — when the queue infrastructure itself might fail (worker crash, Redis down, job lost). This guards against the child _never completing at all_, not against slow business logic.

**Takeaway:** Business timeouts go on leaf machines (the ones calling external APIs, waiting for webhooks). Parent `@timeout` is for infrastructure protection only. If you need a parent timer for business logic, the child's design is incomplete.

### 5. Keep the Hierarchy Clean

**Core idea:** Each machine should communicate only with its direct parent and children. "Wild links" — messages that skip hierarchy levels — make the system unpredictable.

**Example domain:** E-commerce with order → payment → fraud check chain.

**Anti-pattern: Grandparent bypasses parent**

```php no_run
use Tarfinlabs\EventMachine\Behavior\ActionBehavior;
use Tarfinlabs\EventMachine\ContextManager;

// Anti-pattern: OrderWorkflowMachine action sends directly to FraudCheckMachine
// (should go through PaymentMachine)
class HandlePaymentReceivedAction extends ActionBehavior
{
    public function __invoke(ContextManager $context): void
    {
        // Order machine reaching past PaymentMachine to FraudCheckMachine
        $this->sendTo(
            machineClass: 'App\\Machines\\FraudCheckMachine',
            rootEventId: $context->get('fraud_check_id'),
            event: ['type' => 'FRAUD_DATA_SUBMITTED', 'payload' => ['amount' => $context->get('amount')]],
        );
    }
}
```

PaymentMachine doesn't know FraudCheckMachine received a command. It can't account for this in its transitions. System behavior becomes unpredictable.

**Fix: Route through the hierarchy**

```php ignore
// PaymentMachine owns the fraud check as its child
'processing' => [
    'machine'        => FraudCheckMachine::class,
    'with'           => ['transaction_id', 'amount'],
    '@done.clean'    => 'capturing',
    '@done.flagged'  => 'awaiting_manual_review',
    '@fail'          => 'failed',
],
```

The parent (OrderWorkflow) delegates to PaymentMachine. PaymentMachine delegates to FraudCheckMachine. Each machine talks only to its direct neighbors.

**When `sendTo()` / `dispatchTo()` IS appropriate:**
- **Sibling coordination:** Two machines at the same level need to synchronize (e.g., InventoryMachine notifies ShippingMachine that stock is reserved)
- **Notifications:** Fire-and-forget messages to logging, analytics, or notification machines
- **Cross-domain events:** Explicitly modeled communication between separate domain boundaries (e.g., order domain → accounting domain)

**Takeaway:** Prefer hierarchical delegation (`machine` key) over direct messaging (`sendTo`). Use `sendTo` for sibling coordination and fire-and-forget notifications, not for bypassing the hierarchy.

### 6. Composition Patterns

Brief overview of three patterns, with when to use each:

**Pattern A: Hierarchical (Parent → Children)**

The most common pattern. Parent orchestrates sequential/branching workflow, children specialize.

```
OrderWorkflow
  ├── ValidationMachine     (@done → awaiting_payment)
  ├── PaymentMachine        (@done.captured → shipping, @done.declined → payment_failed)
  │     └── FraudCheckMachine  (@done.clean → capturing)
  ├── ShippingMachine       (@done → completed)
  └── RefundMachine         (@done → refunded)
```

**When:** Sequential or branching workflows where the parent decides what happens next based on child outcomes. Most EventMachine systems use this pattern.

**Pattern B: Parallel Regions**

Independent sub-flows running simultaneously within one machine.

```
OrderFulfillment (parallel state)
  ├── region: payment   (processing → captured)
  └── region: inventory (reserving → reserved)
  @done → shipping (when both regions complete)
```

**When:** Independent sub-flows that must all complete before proceeding. Regions cannot communicate with each other — they share no context keys. See [Parallel Patterns](./parallel-patterns).

**Pattern C: Coordinator + Peer Machines**

Independent machines notify a coordinator via `sendTo`/`dispatchTo`. No parent-child relationship.

```
OrderMachine ──sendTo──→ AccountingMachine
ShippingMachine ──sendTo──→ AccountingMachine
RefundMachine ──sendTo──→ AccountingMachine
```

**When:** Cross-domain communication where machines belong to different bounded contexts. The coordinator reacts to events from multiple independent sources. Use `dispatchTo` (async) to avoid coupling the sender to the coordinator's response time.

### 7. Guidelines

1. **Commands down, states up.** Parent delegates via `machine` key, reads `@done.{state}`. Child never reads parent state — only values passed via `with`.

2. **Design child final states for the parent.** Use `@done.{finalState}` for routing and `output` on final states to filter context. Each distinct business outcome the parent cares about should be a separate final state.

3. **`@fail` is for crashes, not business logic.** `@fail` fires on exceptions and job failures. Business failures (rejected, declined, unavailable) are final states routed via `@done.{state}`.

4. **Timers at the edges.** Business timeouts on leaf machines (the ones calling external APIs, waiting for webhooks). Parent `@timeout` only for infrastructure protection.

5. **One hop only.** Each machine talks to its direct parent and children. No skipping levels. `sendTo` for siblings and notifications, not hierarchy bypass.

6. **Start bottom-up.** Design leaf machines first (they're closest to the domain reality), then compose parent machines around them.

7. **Aim for 3-level max.** Most systems need at most: orchestrator → domain machines → integration machines. Deeper hierarchies add complexity without value.

## Key Differences from Source Material (Wagner et al.)

| Book Concept | EventMachine Adaptation | Notes |
|---|---|---|
| Command–State interface | `machine` key (command) + `@done.{state}` (state) | Book uses raw state watching; EM uses lifecycle hooks |
| Handshaking (busy/done) | Automatic — delegation state = busy, @done = done | No manual handshaking needed |
| "States = full control info" | `@done.{finalState}` + `output` key | First-class support for rich child status reporting |
| Timers at edges only | `after` on leaf states, `@timeout` for infra only | Book's rule maps directly |
| No wild links | Prefer `machine` key over `sendTo` | `sendTo` exists for legitimate cross-domain use |
| Hierarchical structure | Parent → child delegation chain | Non-hierarchical via `sendTo`/`dispatchTo` for peers |
| Bottom-up design | Design leaf machines first | Same principle, software domains instead of hardware |
