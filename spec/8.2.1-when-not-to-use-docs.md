# Documentation: "When Not to Use a State Machine"

## Status
**Phase:** Plan reviewed and revised (v2)
**Version:** v7 documentation addition
**Date:** 2026-03-18
**File:** `docs/getting-started/when-not-to-use.md`

## Problem Statement

EventMachine documentation comprehensively covers "how to build" and "how to design well" but never addresses "should I even use this?" A developer seeing a `status` column on an Eloquent model naturally asks: "Should this be a state machine?" — and the docs give no structured answer.

The existing coverage is minimal:
- `what-is-event-machine.md:185-188` — three vague bullets ("Simple CRUD", "Stateless request/response", "One-time operations")
- `machine-decomposition.md` — "Too-Tiny Machines" anti-pattern, but from the perspective of someone already using machines
- `comparison.md` — compares EventMachine to other libraries, not to "no library at all"

## Research Summary

34 sources surveyed (blog posts, official docs, HN/Reddit/Lobsters discussions, academic references). Key community perspectives:

### Arguments FOR liberal state machine adoption
- **Kevin Burke:** "Any model with a status column should use a state machine" — the strongest pro-SM voice
- **Scott Logic / Richard Clayton:** "Your code already HAS a state machine — it's just implicit and bug-prone. Make it explicit."
- **Jannik Wempe:** "If you already use state machines elsewhere in the project, the marginal cost of using one for a simple case is near zero" (consistency argument)

### Arguments FOR restraint
- **Edward Radau:** "Sledgehammer to crack a nut" — for simple scenarios with few states and straightforward transitions, if/else is fine
- **David Khourshid (XState creator):** "You don't need a library for state machines. A switch statement suffices for simple cases."
- **Martin Fowler (YAGNI):** Don't add infrastructure until you feel the pain
- **Skorks:** Developers build systems incrementally — a state machine feels premature before the complexity exists

### Unique insights worth incorporating
- **Boolean explosion metric** (statecharts.dev): n booleans = 2^n implicit states. When implicit > explicit by 2x, time to formalize.
- **Implicit state machine** (Scott Logic): Every status column IS a state machine — the question is whether it's well-defined or scattered across if/else.
- **Symfony taxonomy:** Three tiers — enum (no enforcement), state machine (single state, enforced transitions), workflow (concurrent states).
- **GoCardless (Lawrence Jones):** If you persist state, do it properly with a transition table — not just a status column.

## Content Structure

### 1. Opening: The Status Column Question

**Goal:** Frame the page's purpose. Establish credibility by being honest about when EventMachine isn't needed.

**Content:**
- "Not every status column needs a state machine."
- Opening example: A blog post with `draft → published → archived`. Three states, one direction, no branching. You don't need a state machine for this — an enum and a `publish()` method on the model are perfectly fine.
- Quote from David Khourshid (XState creator): "You don't need a library for state machines."
- Quote from Fowler (YAGNI): Start simple, graduate when you feel the pain.

**Tone:** Honest, not defensive. A library confident enough to tell you when NOT to use it earns more trust than one that insists you always need it.

### 2. Three Tiers of State Management

**Goal:** Give a clear mental model. Not everything is binary (enum vs EventMachine) — there's a spectrum.

| Tier | Tool | Enforcement | History | Use When |
|------|------|-------------|---------|----------|
| 1 | Status enum + Eloquent | None | None | Linear progression, ≤4 states, informational status |
| 2 | Spatie Model States | Transition rules | Optional | 4-10 states, need transition validation, state-specific behavior |
| 3 | EventMachine | Full statechart | Built-in event sourcing | Branching, guards, timers, delegation, audit trail |

**For each tier, show the same example** — an invoice lifecycle (`draft → sent → paid`):
- Tier 1: Enum cast + `$invoice->markAsSent()` method with simple `if` check
- Tier 2: Spatie state classes with `transitionTo()` and allowed transitions
- Tier 3: EventMachine definition — only shown to illustrate what it looks like at this scale (and why it's overkill for this case)

**Key message:** Each tier adds power AND complexity. Use the lowest tier that meets your needs. The Tier 3 example for a simple invoice is intentionally verbose — that's the point.

### 3. The Decision Flowchart

**Goal:** A visual, step-by-step decision path that produces a recommendation.

> **v1 plan had a points-based scorecard. Removed because:**
> - Negative scores made the math fragile (a linear system scored −6, needing +9 of positives just to reach Tier 2 — nearly impossible)
> - Weighted points felt arbitrary and required "correct" calibration
> - A decision tree is faster to use and harder to misapply

**Format:** Mermaid flowchart (consistent with the rest of the docs).

```
Start: "Do you have a status column?"
  → No → "You don't need a state machine."
  → Yes → "Is the flow purely linear (A→B→C, no branching)?"
    → Yes → "Are there guard conditions or side effects on transitions?"
      → No → TIER 1: Enum + Eloquent
      → Yes → "Are guards simple (token check, null check) or complex (business rules, external calls)?"
        → Simple → TIER 1: Enum + Eloquent (with validation in model)
        → Complex → TIER 2: Spatie or consider TIER 3
    → No (has branching) → "Do you need ANY of: timers, parallel states, child delegation, event sourcing, cross-machine messaging?"
      → No → TIER 2: Spatie Model States
      → Yes → TIER 3: EventMachine
```

**Below the flowchart, a quick-reference complexity signals table** (no scoring — just presence/absence):

| If you need this... | You need at least... |
|---------------------|---------------------|
| Just track current status | Tier 1 — Enum |
| Enforce allowed transitions | Tier 2 — Spatie |
| State-specific behavior (polymorphic methods) | Tier 2 — Spatie |
| Complete transition history / audit trail | Tier 3 — EventMachine |
| Time-based automation (expiry, reminders) | Tier 3 — EventMachine |
| Parallel regions (concurrent flows) | Tier 3 — EventMachine |
| Child machine delegation / orchestration | Tier 3 — EventMachine |
| Cross-machine communication | Tier 3 — EventMachine |
| Rebuild state from any point in time | Tier 3 — EventMachine |

### 4. The Implicit State Machine

**Goal:** Address the strongest argument for state machines — the one Kevin Burke and Scott Logic make.

**Core argument:**
> "Every system with a status column already HAS a state machine. The question is whether it's explicit (well-defined, enforced) or implicit (scattered if/else, bug-prone)."
> — Scott Logic

**Show the evolution using a blog post example:**
1. Start: `$post->status = 'published'` — no validation, anyone can set anything
2. Bug: Someone publishes a post that's already archived. Users see a "new" post that's actually old.
3. Fix: Add `if ($this->status !== 'draft') throw ...` in the `publish()` method
4. Another bug: A scheduled job also publishes posts but doesn't check status
5. More fixes: Add the same if/else in the controller, the job, the observer, the API...
6. Breaking point: Five files now contain status validation logic. A new developer asks: "Can an archived post go back to draft?"

**Key message:** The implicit state machine isn't wrong when it's small. It becomes a liability when the same validation logic is scattered across 3+ files. That's when you formalize it — with Spatie (Tier 2) or EventMachine (Tier 3) depending on what else you need.

### 5. Overkill Signals

**Goal:** Concrete anti-patterns that indicate over-engineering.

#### Signal 1: The Linear Pipeline
```
draft → published → archived
```
Three states, one direction, no branching. An enum and a method per transition is clearer than a full machine definition.

#### Signal 2: The CRUD Wrapper
The "machine" just wraps `$model->update(['status' => 'x'])` with no guards, no actions, no side effects. All you've added is indirection.

#### Signal 3: The External-Only Machine
Every transition is triggered by an HTTP request or a simple method call. The machine never self-progresses, never fires timers, never spawns children. A controller with a `match` statement does the same job with less ceremony.

#### Signal 4: The Two-State Toggle
`active ↔ inactive`, `enabled ↔ disabled`, `published ↔ unpublished`. This is a boolean in disguise.

#### Signal 5: The Copy-Paste Machine
A developer sees an EventMachine in the project and creates one for every new status column "for consistency." Consistency has merit (Jannik Wempe's argument), but only when the new flow genuinely benefits from machine features. Consistency of tooling doesn't justify complexity where none is needed.

### 6. Graduation Signals — When to Upgrade

**Goal:** Tell users when they SHOULD switch from enum to EventMachine.

**The Graduation Checklist — if you observe 3+ of these, it's time to formalize:**

1. You have `if ($status === 'X')` checks in 3+ different files
2. You've had a production bug caused by an invalid state transition
3. You have 2+ boolean flags that interact with the status field (boolean explosion)
4. You need to answer "who changed the status, when, and why?" (audit trail)
5. Different API responses or UI elements depend on the current status
6. You've written `// only valid when status is X` comments in multiple places
7. A new team member asked "wait, can it go from X to Y directly?"
8. Side effects (email, webhook, notification) must fire exactly once on specific transitions
9. Multiple processes (queue workers, cron jobs, API calls) can update status concurrently
10. Your status field has grown from 3 to 7+ values over the project's lifetime
11. You need a timer: "expire after 30 days", "send reminder after 24 hours"
12. The same flow pattern appears in multiple models — reuse opportunity

**The Boolean Explosion Formula:**
> If you have `n` boolean flags alongside your status column, you have up to `2^n × status_count` implicit states. When this number exceeds your explicit state count by 2x or more, it's time to formalize.

Example: A support ticket with 4 statuses (`open`, `in_progress`, `resolved`, `closed`) and 3 boolean flags (`is_urgent`, `has_attachment`, `is_escalated`). That's 2^3 × 4 = 32 implicit states. A state machine with hierarchical states or parallel regions might model this as 6-8 explicit states.

### 7. Side-by-Side Examples

**Goal:** 4 universally recognizable scenarios. Each with the verdict, reasoning, and a brief code sketch.

#### Example A: Blog Post Lifecycle — Verdict: Enum

| Aspect | Detail |
|--------|--------|
| States | `draft → published → archived` |
| Triggers | Author clicks "Publish" or "Archive" |
| Guards | None (any draft can be published, any published post can be archived) |
| Timers | None |
| Side effects | None beyond the status change itself |

**Why enum:** Linear, 3 states, no branching, no guards, no side effects. A `StatusEnum` and two model methods (`publish()`, `archive()`) is the simplest correct solution.

```php
// This is enough.
enum PostStatus: string {
    case Draft     = 'draft';
    case Published = 'published';
    case Archived  = 'archived';
}
```

#### Example B: User Invitation — Verdict: Enum (possibly Tier 2)

| Aspect | Detail |
|--------|--------|
| States | `pending → accepted` or `pending → expired` |
| Triggers | User clicks invite link (accept) or cron job (expire) |
| Guards | Token must match, invitation must not be expired |
| Timers | Expiry after 7 days — but handled by a simple cron job, not a state timer |

**Why enum:** Only 3 states. The branching (`accepted` vs `expired`) is minimal. The "guard" is just a token + timestamp check, not a complex business rule. A cron job for expiry is simpler than a state machine timer for this case. If you later add "resend invitation" or "revoke invitation", re-evaluate.

#### Example C: E-Commerce Order — Verdict: EventMachine

| Aspect | Detail |
|--------|--------|
| States | `pending`, `paid`, `shipped`, `delivered`, `cancelled`, `refunded` |
| Triggers | User actions, payment webhooks, shipping API callbacks, timers |
| Guards | "Can only refund within 30 days", "Can only ship if stock > 0" |
| Timers | "Auto-cancel if unpaid after 24 hours" |
| Audit | "Show me the complete order history" — compliance, customer support |
| Side effects | Send confirmation email on payment, notify warehouse on ship, update inventory |

**Why EventMachine:** Non-linear (branching to `cancelled` from multiple states), real business-rule guards, timer-driven auto-cancellation, mandatory audit trail, side effects that must fire exactly once. This is what state machines are designed for.

#### Example D: Support Ticket — Verdict: EventMachine

| Aspect | Detail |
|--------|--------|
| States | `open`, `assigned`, `in_progress`, `waiting_on_customer`, `escalated`, `resolved`, `closed` |
| Triggers | Agent actions, customer replies, SLA timers, manager escalation |
| Guards | "Can only escalate if open > 4 hours", "Can only close if customer confirmed" |
| Timers | "Auto-escalate if no response in 4 hours", "Auto-close 7 days after resolution" |
| Parallel | — |
| Backward | `waiting_on_customer → in_progress` (customer replies), `resolved → open` (customer reopens) |

**Why EventMachine:** 7 states with non-trivial branching, backward transitions (reopen, reassign), SLA-driven timers, guards based on time elapsed and customer interaction. The timer and guard logic alone justify EventMachine.

### 8. Closing: Start Simple, Graduate When It Hurts

**Goal:** End with the practical takeaway.

Content:
- Start with Tier 1 (enum). It's the simplest correct solution.
- When the implicit state machine starts causing pain (see Graduation Checklist), evaluate Tier 2 or Tier 3.
- Tier 2 (Spatie) if you need transition enforcement and state-specific behavior but nothing more.
- Tier 3 (EventMachine) if you need timers, event sourcing, delegation, parallel states, or cross-machine communication.
- The migration path from enum to EventMachine is natural: status values become states, allowed transitions become `on` keys, if/else validation becomes guards, side effects become actions.
- For more detail on what EventMachine provides vs alternatives, see [Comparison](/getting-started/comparison).

**Final line:** "The best state machine is the one you don't build until you need it — and the one you do build when the alternative is a bug factory."
