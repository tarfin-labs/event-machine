## Problem

`simulateChildDone()`, `simulateChildFail()`, and `simulateChildTimeout()` only work with `machine` delegation. Job actors (`'job' => SomeJob::class`) throw because class validation compares against `$invokeDefinition->machineClass`, which is empty for jobs:

```php
// TestMachine.php line 1576
if ($invokeDefinition->machineClass !== $childClass) {
    // machineClass = '' for job actors â†’ always mismatches
}
```

Additionally, `simulateChildFail` and `simulateChildTimeout` have **no class validation at all** â€” a pre-existing gap.

---

## Decision: Extend, Don't Duplicate

Separate `simulateJobDone/Fail/Timeout` methods would be wrong because:

1. **Unified architecture** â€” `MachineInvokeDefinition`, `hasMachineInvoke()`, `routeChildDoneEvent()`, `CHILD_MACHINE_DONE` are all shared. No separate infrastructure exists.
2. **API already uses "child"** â€” `simulateChildDone`, `assertChildInvoked`, `fakingChild`. "Child" = "anything a state delegates to".
3. **User burden** â€” separate methods force checking whether config uses `machine:` or `job:` to pick the right method.
4. **DRY** â€” identical method bodies except class check.

---

## Part 1: Code Fix

### Private Helper: `resolveDelegateClass`

Extract shared logic to avoid 3x duplication:

```php
private function resolveDelegateClass(StateDefinition $stateDefinition, string $action): array
{
    if (!$stateDefinition->hasMachineInvoke()) {
        throw new AssertionFailedError(
            "Cannot simulate child {$action}: current state [{$stateDefinition->id}] does not have a child delegation."
        );
    }

    $invokeDefinition = $stateDefinition->getMachineInvokeDefinition();
    $delegateClass = $invokeDefinition->isJob()
        ? $invokeDefinition->jobClass
        : $invokeDefinition->machineClass;

    return [$invokeDefinition, $delegateClass];
}
```

Each `simulateChild*` method calls this, then validates `$delegateClass !== $childClass`.

### Changes Per Method

| Method | Before | After |
|--------|--------|-------|
| `simulateChildDone` | `machineClass` check | `resolveDelegateClass` + validate |
| `simulateChildFail` | No class check | `resolveDelegateClass` + validate |
| `simulateChildTimeout` | No class check | `resolveDelegateClass` + validate |
| Error messages | "machine invoke definition" | "child delegation" |

### Usage

```php
// Machine delegation â€” unchanged
->simulateChildDone(PaymentMachine::class, result: ['id' => 'pay_1'])

// Job actors â€” now works with the same method
Queue::fake();
->simulateChildDone(QueryPhonesJob::class, result: ['phones' => [...]])
```

`Queue::fake()` is needed because `handleJobInvoke()` calls `dispatch(ChildJobJob)` when the machine enters the job state. `simulateChildDone()` then simulates the completion routing.

### Not Included

| Proposed | Decision | Reason |
|----------|----------|--------|
| `simulateJobDone/Fail/Timeout` | Skip | DRY violation, architecture unified |
| `assertJobDispatched` | Skip | `Queue::assertPushed(ChildJobJob::class)` is standard Laravel |
| `fakingJob()` | Skip | Jobs don't extend Machine. `Queue::fake()` + `simulateChildDone` sufficient |
